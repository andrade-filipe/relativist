## Context

The REVIEW of bundle 2.26-B/C/D identified SF-002: SPEC-19 R43 mandates that `coordinator_free_rounds` defaults to `true` when `delta_mode` is `true`. The current `impl Default for GridConfig` (`merge/types.rs:342-354`) sets BOTH fields to `false` unconditionally. There is no builder or post-construction hook that flips `coordinator_free_rounds` when a caller enables `delta_mode`. A CLI user passing `--delta-mode` today runs the delta loop WITHOUT the coordinator-free-round optimization ‚Äî contrary to R43.

R43 is a MUST. The 2.26-D task breakdown (`SPEC-19-section-3.5-3.6-config-amendments-tasks.md` L25-32) listed R41/R42 as in-scope and omitted R43 ‚Äî a scope-definition miss. This task closes the gap.

            );
            self.coordinator_free_rounds = true;
        }
        self
    }
}

## Notes

- **Why this matters in practice today:** the coordinator-free-round optimization (skip merge when `has_border_activity = false` on all workers) is a performance win ‚Äî it shortens rounds when workers reduce locally without cross-partition interaction. Without R43 enforcement, `--delta-mode` users pay for the merge round unnecessarily on every quiet round. Spec R43's motivation is performance alignment of the delta protocol's design.
- **Why option (c) by default ‚Äî unconditional override:** R43 reads "MUST default to `true`". A strict reading makes the default override-able; a strictest reading makes it mandatory under `delta_mode`. Option (c) picks the strictest (simpler invariant, no user-intent tracking infrastructure). If real users later ask to force `coordinator_free_rounds=false` under delta_mode for debugging, DC-0397-A is revised and option (b) introduces the tracking bit. The cost of later flipping is low (~10 LoC + 1 new field).
- **Why not change `coordinator_free_rounds` to `Option<bool>`:** option (a) would require touching every read site of the field (potentially many). Spec reads as if the field is a simple `bool`; preserve that API shape.
- **Interaction with TASK-0394:** independent. Worker-side minting does not depend on `coordinator_free_rounds`.
- **Interaction with TASK-0395 (G1 parity):** G1 parity tests in TEST-SPEC-0385 UT-0385-06..08 run with `coordinator_free_rounds` value left at the normalized default (because `GridConfig::default()` + explicit `delta_mode=true` flows through `normalize` as soon as TASK-0397 ships). Expectation: parity still holds ‚Äî coordinator-free-rounds is a performance optimization, not a correctness switch.
- **Interaction with TASK-0396 (R20 dispatcher):** independent. Router observes `cfg.delta_mode`; normalization is orthogonal.
- **Documentation impact:** update `coordinator_free_rounds` field doc-comment to describe R43 normalization behavior. Existing doc-comment at `types.rs:320-339` may already mention R44 (`coordinator_free_rounds=true + delta_mode=false` MAY work) ‚Äî extend with R43 direction.

## Design Choices Flagged for Spec-Critic (Stage 1.5)

- **DC-0397-A ‚Äî unconditional override vs. user-choice preservation.** Options:
  - (a) Two-phase with `Option<bool>` sentinel; effective value via accessor.
  - (b) `coordinator_free_rounds_user_set: bool` tracking bit; normalize conditionally.
  - (c) Unconditional: `if delta_mode then coordinator_free_rounds = true`, silently override user's explicit `false`.
  - **Default plan:** (c) with a tracing warning. Rationale: simplest, spec-strict. **Spec-critic: confirm or flip to (b).**
- **DC-0397-B ‚Äî where `normalize` is called.** Options:
  - (a) Only in `build_grid_config*` (CLI construction path).
  - (b) In `build_grid_config*` AND as a public builder method callers can invoke programmatically.
  - (c) Implicitly in `Default::default()` (controversial ‚Äî changes semantics of `Default`).
  - **Default plan:** (a)+(b) ‚Äî CLI normalizes automatically, programmatic callers opt in. Do NOT change `Default`'s semantics. **Spec-critic: confirm.**
- **DC-0397-C ‚Äî naming.** `normalize` vs. `enforce_invariants` vs. `with_r43_defaults`. Default: `normalize`. Flagged to avoid bike-shedding.
