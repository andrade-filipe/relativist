## Context

Each worker must connect to the coordinator with retry logic using exponential backoff. The backoff starts at 1 second, doubles at each attempt up to a maximum of 16 seconds, with at most 10 attempts. This replicates the behavior of `connectWithRetry` in the Haskell prototype (AC-003, lines 331-352). If all 10 attempts fail, the function returns a `ProtocolError::Io` with `ConnectionRefused`.

## Acceptance Criteria

- [ ] `connect_with_retry` is implemented in `src/protocol/worker.rs`
- [ ] Function is async, takes `&NodeConfig` as parameter
- [ ] Attempts up to 10 TCP connections to `(config.host, config.port)`
- [ ] Backoff sequence: 1s, 2s, 4s, 8s, 16s, 16s, 16s, 16s, 16s, 16s
- [ ] Returns `Ok(TcpStream)` on successful connection
- [ ] Returns `Err(ProtocolError::Io(_))` after 10 failed attempts
- [ ] Logs each failed attempt with `tracing::warn!` including attempt number and delay

## Notes

- The maximum of 10 attempts and 16s cap are hardcoded constants, not configurable. This matches the Haskell prototype exactly.
- Consider extracting the retry constants (max_attempts, initial_delay, max_delay) as associated constants for testability.
- The `tokio::time::sleep(delay)` call between retries is essential; do NOT use `std::thread::sleep` as it blocks the async runtime.
- The tracing log should include enough context for debugging: attempt number, total attempts, error message, next delay.
