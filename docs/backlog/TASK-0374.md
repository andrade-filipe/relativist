## Context

Completes the SPEC-03 6-rule dispatch table by adding the three
asymmetric rules that the coordinator must resolve when they occur at a
border:

| Rule | Agents consumed | New agents | New cross-partition borders |
|------|-----------------|------------|-----------------------------|
| CON-DUP | 2 (one CON + one DUP) | 4 (two CON + two DUP) | up to 4 auxiliary wires; some local to one worker, some crossing to the other |
| CON-ERA | 2 (CON + ERA) | 2 (two ERA) | 2 new ERA → existing connections (may inherit border status) |
| DUP-ERA | 2 (DUP + ERA) | 2 (two ERA) | 2 new ERA — mirror of CON-ERA |

CON-DUP is the hard case: the commutation rule "crosses" the wires of a
CON and a DUP, producing 4 new agents with auxiliary ports. Four new
wires are created. Any subset of these wires may cross the border
between the two partitions — so the resolver must:

1. Compute each new agent's 2 auxiliary-port targets (via standard
   commutation rule topology).
2. For each target, determine whether it is LOCAL to one worker, LOCAL
   to the other, or crosses the border.
3. Generate a fresh `border_id` for each new crossing wire.
4. Emit `BorderDelta`s to each worker instructing them to create the
   new agents and link their ports.
5. Call `graph.add_border_states(new_entries)` (R15 part 3 — the
   primitive is already shipped).
6. Call `graph.remove_border(bid)` for the consumed original border.

CON-ERA and DUP-ERA are simpler: the erasure rule replaces the non-ERA
agent with 2 new ERA agents (one per auxiliary port), each connected to
the former auxiliary-port target. No new borders because ERA agents
have no auxiliary ports — they just inherit the connections.

## Notes

- **Why CON-DUP is the hard case:** it creates 4 new agents and 4 new
  wires. The other commutation / erasure rules create ≤ 2 new agents.
  CON-DUP is also the ONE rule that can create new cross-partition
  borders from a previously-local wire — making
  `graph.add_border_states` load-bearing.

- **New agent ID allocation:** the coordinator doesn't own the
  workers' `id_range`s. The two workers each have their own
  `IdRange` (from `Partition.id_range`). The new agents created by
  commutation must be assigned to the workers in pairs (2 new agents
  per worker — mirror of the original). The coordinator tells each
  worker "create 2 new agents of these symbols" in the delta; the
  worker allocates IDs from its `id_range` locally and reports them
  back. **But this requires a 2-phase protocol (coordinator proposes,
  worker acks with IDs, coordinator finalizes deltas).** Flagged as
  DC-B5 — may need a sync round.

## Design Choices Flagged for Spec-Critic (Stage 1.5)

- **DC-B5 — new agent ID allocation.** Who picks the `AgentId`s for
  the 4 new commutation agents? Options:
  (a) coordinator picks from a reserved `IdRange` block carved out of
  each worker's range at setup (requires splitting each worker's
  `id_range` in half — one half for local reductions, one for
  coordinator-allocated);
  (b) workers allocate locally after receiving "create 2 Con + 2 Dup"
  deltas, then report back the IDs in round N+1 (2-phase);
  (c) coordinator uses a shared `IdRange` at a "coordinator-reserved"
  range outside any worker's range (requires each worker to also
  accept "create agent X in arena slot Y not in my range" — breaks
  the `id_range` invariant).
  **spec-critic please rule.** This is a SPEC-19 design gap (OQ-2 in
  the spec flags the CON-DUP expansion delta format but not the ID
  allocation question specifically).

- **DC-B6 — does CON-ERA / DUP-ERA require new border IDs?** The 2
  new ERA agents inherit the auxiliary-port connections of the
  consumed non-ERA agent. If one of those connections was itself a
  border (the consumed CON had an auxiliary port connected to a
  FreePort), the border is preserved — just with the other endpoint
  re-pointed from the old CON's auxiliary port to the new ERA's
  principal port. No new border ID needed — this is a `apply_deltas`
  update to the existing border, not an `add_border_states` call.
  **Flag for spec-critic confirmation.**
