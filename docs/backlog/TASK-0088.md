## Context

Before the grid loop begins, the coordinator must open a TCP listener socket on a configurable port and accept connections from exactly `num_workers` workers. There is a configurable timeout (default 120 seconds) after which the coordinator aborts if not all workers have connected. This is the `WaitingWorkers` state of the coordinator FSM (R26). The function returns a vector of `TcpStream` handles, one per worker, ready for the grid loop.

## Acceptance Criteria

- [ ] `accept_workers` is implemented in `src/protocol/coordinator.rs`
- [ ] Function is async, takes `&NodeConfig` as parameter
- [ ] Binds a `TcpListener` to `(config.host, config.port)` (R17)
- [ ] Accepts exactly `config.num_workers` connections
- [ ] Times out after `config.worker_connect_timeout` (R24), returning `ProtocolError::Timeout`
- [ ] Returns `ProtocolError::WorkerCountMismatch` if timeout expires with fewer workers
- [ ] Logs each worker connection with `tracing::info!` (worker index, remote address)
- [ ] Returns `Ok((listener, Vec<TcpStream>))` on success

## Notes

- The `TcpListener` is returned alongside the streams because the coordinator may need to keep it alive for the duration of the grid loop (depending on OS behavior with accepted sockets).
- The function uses `tokio::time::timeout` to wrap the accept loop, not a manual timer.
- Workers connect in arbitrary order. The index in the returned `Vec<TcpStream>` determines the worker's assignment (stream[i] receives partition[i]).
- Connection loss during execution is handled separately (R25) -- this task only covers the initial accept phase.
