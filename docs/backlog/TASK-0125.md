## Context

`SecurityConfig` is the central configuration struct for the security module. It aggregates the detected security tier, the optional authentication token, optional TLS configurations (feature-gated), and operational limits (max message size, max connections, idle timeout). It is assembled from CLI flags during coordinator/worker startup and passed to the protocol layer. This task defines the struct and a `default()` implementation for Tier 1 (zero-config development).

## Acceptance Criteria

- [ ] `SecurityConfig` struct is defined in `src/security/mod.rs`
- [ ] `SecurityConfig` has fields: `tier`, `token`, `tls_server` (feature-gated), `tls_client` (feature-gated), `max_message_size`, `max_connections`, `idle_timeout`
- [ ] `SecurityConfig` derives `Debug`, `Clone`
- [ ] `SecurityConfig::development()` returns a Tier 1 config with defaults: no token, no TLS, 256 MiB max message, 1024 max connections, 30s idle timeout
- [ ] Default `max_message_size` is `268_435_456` (256 MiB, R30)
- [ ] Default `max_connections` is `1024` (R32)
- [ ] Default `idle_timeout` is `Duration::from_secs(30)` (R33)

## Notes

- The `SecurityConfig` struct uses `#[cfg(feature = "tls")]` on individual fields, not on the entire struct. This way the same struct exists with or without TLS, but the TLS fields are compiled out when not needed.
- The `Debug` derive is safe here because `AuthToken`'s custom `Debug` impl (TASK-0123) already redacts the token value.
- The `development()` constructor is a convenience for Tier 1; Tier 2 and Tier 3 configs are built by the CLI/config layer (TASK-0138).
