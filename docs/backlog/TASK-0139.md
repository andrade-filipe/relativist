## Context

SPEC-10 Section 6 defines 10 test requirements (T1-T10) that validate the security module end-to-end. While individual tasks include unit tests for their components, this task creates integration tests that exercise the full security flow: token lifecycle (generate -> encode -> transmit -> validate), TLS handshake (feature-gated), tier detection with real CLI args, connection acceptance with authentication, and the complete connection flow from TCP connect through authentication to grid loop readiness.

## Acceptance Criteria

- [ ] Integration test file: `tests/security_integration.rs` (or `src/security/tests.rs` module)
- [ ] T1: Two `AuthToken::generate()` calls produce different tokens
- [ ] T2: `AuthToken::from_base64(token.to_base64()).unwrap()` followed by `verify()` returns `true` against original
- [ ] T3: Correct token -> RegisterAck; wrong token -> RegisterNack + close; missing token when required -> RegisterNack + close; token provided in Tier 1 -> accepted (ignored)
- [ ] T4 (feature-gated `#[cfg(feature = "tls")]`): Valid cert + valid CA -> handshake success; valid cert + wrong CA -> failure; (expired cert test if feasible)
- [ ] T5: Oversized message rejected without reading payload (uses `max_payload_size` from SPEC-06 R9, not `max_message_size`)
- [ ] T6: Idle connection timeout closes the connection
- [ ] T7: Rejected worker does not prevent other workers from connecting
- [ ] T8: Coordinator without `--bind` binds to localhost only
- [ ] T9: Correct SecurityTier detected for all flag combinations
- [ ] T10: `format!("{:?}", token)` MUST contain `"[REDACTED]"` and MUST NOT contain the raw token bytes or base64 encoding

#[cfg(test)]
mod tests {
    use relativist::security::*;
    use relativist::security::token::AuthToken;

    /// T1: Token generation produces unique tokens.
    #[test]
    fn test_token_generation_uniqueness() {
        let t1 = AuthToken::generate();
        let t2 = AuthToken::generate();
        // Cannot use assert_ne! because AuthToken does not implement PartialEq.
        // Instead verify that verify() returns false.
        assert!(!t1.verify(&t2));
    }

    /// T2: Token base64 roundtrip (uses verify(), not ==).
    #[test]
    fn test_token_base64_roundtrip() {
        let token = AuthToken::generate();
        let encoded = token.to_base64();
        let decoded = AuthToken::from_base64(&encoded).unwrap();
        assert!(token.verify(&decoded));

## Test Expectations

- All 10 SPEC-10 test requirements (T1-T10) are covered
- TLS tests (T4) only run with `--features tls`
- Tests use in-memory transports or localhost TCP for isolation
- No external dependencies (no real certificates from a CA, no network access)
- Self-signed test certificates are generated as part of test setup (or pre-generated in `tests/certs/`)

## Notes

- This is marked as L complexity because it covers many test scenarios, but each individual test is small. The total may exceed 200 LoC, which is acceptable for a test-only task.
- TLS integration tests require self-signed certificates. These can be generated with `rcgen` (a Rust crate for certificate generation) in test setup, or pre-generated as PEM files in `tests/certs/`. Pre-generated files are simpler and avoid an extra dev-dependency.
- T8 (localhost binding test) is difficult to test in CI without network namespace isolation. Consider marking it as `#[ignore]` with a comment explaining it requires manual verification.
- T6 (idle timeout test) requires a controlled delay. Use `tokio::time::pause()` to control time in tests without real delays.
