## Context

SPEC-13 R52 (added during adversarial review, SC-009) requires a `SerializingChannelTransport` that wraps `ChannelTransport` and performs a bincode serialize/deserialize round-trip on every `send`/`recv`. The plain `ChannelTransport` bypasses serialization entirely (in-memory message passing), which means it cannot catch serde `Serialize`/`Deserialize` derive bugs or bincode compatibility issues. The serializing variant exercises the full serialization path while remaining in-process (no TCP).

At minimum, one integration test per benchmark (SPEC-08) SHOULD use the serializing variant to verify that `Message` serialization is correct (SPEC-06 R14: `deserialize(serialize(msg)) == msg`).

## Acceptance Criteria

- [ ] `SerializingChannelTransport` struct is defined, wrapping a `ChannelTransport`
- [ ] `SerializingChannelTransport` implements the `Transport` trait
- [ ] `send()` serializes the message to bytes via bincode, then deserializes back, then passes the deserialized message to the inner `ChannelTransport::send()`
- [ ] `recv()` delegates to the inner `ChannelTransport::recv()`, then performs a serialize/deserialize round-trip on the received message before returning
- [ ] `close()` delegates to the inner `ChannelTransport::close()`
- [ ] A `SerializingChannelTransport::pair()` constructor is provided, returning two connected transports (mirroring `ChannelTransport::pair()`)
- [ ] A round-trip test verifies that all `Message` variants survive the serialization cycle
- [ ] At least one integration test exercises the full grid cycle using `SerializingChannelTransport` instead of `ChannelTransport`

## Key Types / Signatures

```rust
/// A wrapper around `ChannelTransport` that performs a bincode
/// serialize/deserialize round-trip on every send/recv.
///
/// This catches serde derive bugs that `ChannelTransport` would miss
/// because it passes messages in-memory without serialization.
/// See SPEC-13 R52, SPEC-06 R14.
pub struct SerializingChannelTransport {
    inner: ChannelTransport,
}

impl SerializingChannelTransport {
    /// Create a connected pair of serializing channel transports.
    pub fn pair() -> (Self, Self) {
        let (a, b) = ChannelTransport::pair();
        (Self { inner: a }, Self { inner: b })
    }

    /// Round-trip a message through bincode serialization.
    fn round_trip(msg: &Message) -> Result<Message, ProtocolError> {
        let bytes = bincode::serialize(msg)
            .map_err(|e| ProtocolError::InvalidMessage(format!("serialize: {}", e)))?;
        let deserialized: Message = bincode::deserialize(&bytes)
            .map_err(|e| ProtocolError::InvalidMessage(format!("deserialize: {}", e)))?;
        Ok(deserialized)
    }
}

#[async_trait::async_trait]
impl Transport for SerializingChannelTransport {
    async fn send(&mut self, msg: &Message) -> Result<(), ProtocolError> {
        let verified = Self::round_trip(msg)?;
        self.inner.send(&verified).await
    }

    async fn recv(&mut self) -> Result<Message, ProtocolError> {
        let msg = self.inner.recv().await?;
        Self::round_trip(&msg)
    }

    async fn close(&mut self) -> Result<(), ProtocolError> {
        self.inner.close().await
    }
}
```

## Notes

- This is a SHOULD requirement (R52), not a MUST. It is important for test fidelity but does not block core functionality.
- The serialization correctness property (SPEC-06 R14: `deserialize(serialize(msg)) == msg`) is independently tested by property tests in Phase 5. This transport provides an additional layer of confidence by exercising serialization in integration tests.
- The performance overhead of the round-trip is negligible for integration tests but should NOT be used in benchmarks (SPEC-09) unless explicitly testing serialization overhead.
