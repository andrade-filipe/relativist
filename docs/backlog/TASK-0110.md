## Context

SPEC-13 R25 defines the worker's transition table. This task implements the pure transition function for the worker FSM, following the same stimulus-response pattern as the coordinator (but simpler, since the worker has fewer states and no context to track beyond the current state).

The transition table from SPEC-13 R25 (Revised v2):

| From | Event | To | Actions |
|------|-------|----|---------|
| Init | Connected | Idle | LogTransition |
| Idle | ReceivePartition(P) | Reducing | LogTransition |
| Reducing | ReductionComplete(P') | Returning | LogTransition |
| Returning | SendComplete | Idle | LogTransition |
| Idle | Shutdown | Done | CloseConnection, LogTransition |
| Reducing | ReductionError(e) | Error | SendMessage(Error(e)), LogTransition |
| Any | ConnectionLost | Error | LogTransition, ShutdownSelf |

> **Note on ConnectionLost:** The worker does NOT attempt to reconnect. Per SPEC-06 R25, the coordinator aborts the grid loop upon connection loss. Worker reconnection is a fault tolerance concern (Z5, out of scope for v1).
> **Note on worker registration (updated for SPEC-06 v3):** Registration behavior is tier-dependent (SPEC-06 R2a). In Tier 1 (no auth), registration MAY be implicit. In Tier 2/3 (auth enabled), the worker MUST send a Register message per SPEC-10 R14-R17.

## Acceptance Criteria

- [ ] `worker_transition(state, event) -> (WorkerState, Vec<WorkerAction>)` is implemented
- [ ] All 7 transitions from the table above are implemented
- [ ] Every state change produces a `LogTransition` action
- [ ] `ConnectionLost` from any state transitions to `Error` with `ShutdownSelf` (no reconnection)
- [ ] `Init + Connected -> Idle` produces only `LogTransition` in Tier 1 (in Tier 2/3, worker MUST send Register message per R2a -- handled at the runtime layer, not the FSM transition)
- [ ] Unhandled event+state combinations transition to `Error`
- [ ] The function is pure (no I/O, no async)

## Notes

- The worker FSM is simpler than the coordinator FSM: no context struct is needed because the worker does not track accumulated state (no counting of peers, no collection of partitions).
- The actual `reduce_all` call is NOT inside the transition function. The async runtime (TASK-0113) calls `reduce_all` and then feeds the `ReductionComplete` event back into the FSM.
