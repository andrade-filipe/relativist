## Context

When the `otel` Cargo feature is enabled, Relativist adds a `tracing_opentelemetry::OpenTelemetryLayer` to the subscriber registry (R26). This layer bridges `tracing` spans to OpenTelemetry spans, which are exported via OTLP/HTTP to a collector (Jaeger, Tempo, etc.). The OTel exporter endpoint is configured via the `OTEL_EXPORTER_OTLP_ENDPOINT` environment variable (R27), defaulting to `http://localhost:4318`.

This task modifies the `init_tracing()` function (from TASK-0144) to conditionally add the OTel layer when the `otel` feature is enabled. It also configures OTel resource attributes (R28): `service.name`, `service.version`, and `host.name`.

```rust
// src/observability/otel.rs

#[cfg(feature = "otel")]
use opentelemetry::trace::TracerProvider;
#[cfg(feature = "otel")]
use opentelemetry_sdk::trace as sdktrace;
#[cfg(feature = "otel")]
use tracing_opentelemetry::OpenTelemetryLayer;
#[cfg(feature = "otel")]
use tracing_subscriber::Registry;

use super::config::ProcessRole;

/// Build an OpenTelemetry tracing layer for the given process role (R26, R27, R28).
///
/// Returns `None` if OTel initialization fails (logged as WARNING).
#[cfg(feature = "otel")]
pub fn build_otel_layer(
    role: ProcessRole,
) -> Option<OpenTelemetryLayer<Registry, sdktrace::Tracer>> {
    let service_name = match role {
        ProcessRole::Coordinator => "relativist-coordinator",
        ProcessRole::Worker => "relativist-worker",
    };

    // Configure OTLP/HTTP exporter (R27)
    // Endpoint from OTEL_EXPORTER_OTLP_ENDPOINT or default http://localhost:4318
    let exporter = match opentelemetry_otlp::SpanExporter::builder()
        .with_http()
        .build()
    {
        Ok(e) => e,
        Err(err) => {
            tracing::warn!(error = %err, "failed to build OTel exporter, disabling OTel");
            return None;
        }
    };

    // Configure resource attributes (R28)
    let resource = opentelemetry_sdk::Resource::builder()
        .with_service_name(service_name)
        .with_attribute("service.version", env!("CARGO_PKG_VERSION"))
        .with_attribute("host.name", hostname::get()
            .map(|h| h.to_string_lossy().into_owned())
            .unwrap_or_else(|_| "unknown".to_string()))
        .build();

    let provider = sdktrace::SdkTracerProvider::builder()
        .with_batch_exporter(exporter)
        .with_resource(resource)
        .build();

    let tracer = provider.tracer(service_name);

    Some(OpenTelemetryLayer::new(tracer))
}
```

## Notes

- The OTel layer is added alongside (not replacing) the `fmt::Layer`. Both layers receive the same spans and events from the `tracing` registry.
- OTel initialization can fail for various reasons (network issues, invalid endpoint, missing collector). The function returns `None` gracefully rather than panicking, because observability should not prevent the main application from running.
- The `OTEL_EXPORTER_OTLP_ENDPOINT` environment variable is an OTel standard; no custom env var parsing is needed.
- The exact `opentelemetry-otlp` and `tracing-opentelemetry` APIs may differ by version. Check crate documentation at implementation time. The `opentelemetry` ecosystem evolves rapidly.
- A `hostname` crate dependency may need to be added for R28's `host.name` attribute. Alternatively, use `std::env::var("HOSTNAME")` as a simpler approach.
