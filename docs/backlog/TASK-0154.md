## Context

The actual handler implementations are in TASK-0155 (health/ready) and TASK-0156 (metrics). This task provides the router skeleton with the state types and route declarations.

## Key Types / Signatures

```rust
#[cfg(feature = "metrics")]
use axum::{routing::get, Router};
#[cfg(feature = "metrics")]
use std::sync::{atomic::AtomicU8, Arc};

/// Shared state for the metrics HTTP server.
#[cfg(feature = "metrics")]
#[derive(Clone)]
pub struct AppState {
    /// Prometheus metrics registry.
    pub registry: Arc<prometheus_client::registry::Registry>,
    /// Coordinator FSM state (encoded as u8: 0=Init, 1=WaitingForWorkers, ...).
    pub coordinator_state: Arc<AtomicU8>,
}

/// Build the axum Router for observability HTTP endpoints (R19, R20, R21).
///
/// Routes:
/// - GET /metrics  — Prometheus text exposition (TASK-0156)
/// - GET /health   — Liveness check (TASK-0155)
/// - GET /ready    — Readiness check (TASK-0155)
#[cfg(feature = "metrics")]
pub fn metrics_router(
    registry: Arc<prometheus_client::registry::Registry>,
    coordinator_state: Arc<AtomicU8>,
) -> Router {
    let state = AppState {
        registry,
        coordinator_state,
    };

    Router::new()
        .route("/metrics", get(metrics_handler))
        .route("/health", get(health_handler))
        .route("/ready", get(ready_handler))
        .with_state(state)
}
```

## Notes

- The `coordinator_state` uses `AtomicU8` with a simple numeric encoding of FSM states. This avoids pulling in the full FSM type from the coordinator module, keeping the observability module decoupled.
- The encoding is: `0=Init`, `1=WaitingForWorkers`, `2=Partitioning`, `3=Dispatching`, `4=WaitingForResults`, `5=Merging`, `6=CheckTermination`, `7=Done`. This mapping must be documented and consistent with the coordinator FSM.
- `axum` 0.8 uses `Router::new().route(path, method_router)` syntax. Check exact API compatibility at implementation time.
