## Context

TASK-0389 lands the `delta_mode` field with an adequate docstring. This task
polishes it to match the documentation standard set by `coordinator_free_rounds`
(which has a full multi-paragraph doc comment including an "IC concept"
explanatory note, per the codebase convention
`feedback_ic_code_documentation.md` — "Relativist code must have clear
comments explaining IC concepts").

We also add a **doctest** on `GridConfig` that demonstrates the intended usage
pattern:

```rust
/// # Examples
///
/// Opt into the delta protocol from a builder pattern:
///
/// ```
/// use relativist_core::merge::GridConfig;
///
/// let cfg = GridConfig {
///     num_workers: 4,
///     delta_mode: true,
///     ..GridConfig::default()
/// };
/// assert!(cfg.delta_mode);
/// assert_eq!(cfg.num_workers, 4);
/// // All other fields retain defaults:
/// assert!(!cfg.strict_bsp);
/// assert!(!cfg.coordinator_free_rounds);
/// ```
///
/// Default is v1 (backwards-compatible, R42):
///
/// ```
/// use relativist_core::merge::GridConfig;
///
/// let cfg = GridConfig::default();
/// assert!(!cfg.delta_mode);
/// ```
```

The doctest is compile-and-run tested by `cargo test --doc`, so it doubles as
(a) living documentation, (b) a compile-time assertion that `GridConfig` is
`pub`-constructible with the new field, and (c) a small R42 sanity check at
the doctest level.

## Notes

- **Why a separate task?** Splitting the doctest off keeps TASK-0389 focused
  on the minimal type change. A developer working TASK-0389 can ship it
  quickly; TASK-0393 is a polish pass that does not block 2.26-D's critical
  path.
- **Doctest discipline.** The doctest MUST avoid asserting anything about
  `delta_mode = true` runtime behaviour — because there is none until
  2.26-C's `run_grid_delta` lands. The doctest asserts only field
  settability and default invariants (R42).
- **Public API surface.** Adding a doctest to `GridConfig` makes it visible
  on `docs.rs` / `cargo doc`. No API change.
