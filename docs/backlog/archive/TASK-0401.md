# TASK-0401: Resolver-to-wire transport — populate `PendingCommutation.target_symbols` + `local_wiring` from `CommutationBatch`

**Spec:** SPEC-19 §3.3 R23 (wire payload content), R23a clauses 1-4 (slot namespace, mint-then-wire ordering, slot-marker decoding, concrete-id pass-through — this task emits the wire hints; the worker decodes them in TASK-0402), §3.4 R33 (`LocalWiringHint` shape), §3.4 R34 (serialization gate — transport preserves bytes), §3.6 R48b (empty `local_wiring` is legal).
**Requirements:** R23 wire content on the C→W direction; R23a determinism guarantee (resolver emits in deterministic CommutationBatch order; this task preserves that order verbatim on the wire).
**Priority:** P0 (without this transport, the worker receives empty `local_wiring` and every minted sibling stays DISCONNECTED — the exact bug D-005 exists to fix).
**Status:** TODO
**Depends on:** TASK-0400 (wire struct rewrite must ship first so `PendingCommutation` exposes `target_symbols` + `local_wiring` fields to populate).
**Blocked by:** TASK-0400.
**Estimated complexity:** S-M (~60-100 LoC production + ~100-160 LoC tests)
**Bundle:** D-005 Option A.

## Context

`CommutationBatch` already carries both `target_symbols: Vec<Symbol>` and `local_wiring: Vec<(u8, u8, PortRef)>` inside the resolver (`relativist-core/src/merge/border_resolver.rs:358`). What is missing is the transport step from `CommutationBatch → PendingCommutation` at the point where the resolver converts internal batches into wire-layer `PendingCommutation` values. Today that conversion loses `target_symbols` (falls back to the legacy `(symbol_type, arity)` pair — removed in TASK-0400) and drops `local_wiring` entirely. TASK-0401 restores fidelity: a 1-to-1 copy of both fields, preserving emission order so R23a's determinism guarantee holds on the wire.

The conversion entry point is `package_resolutions_with_pending` in `border_resolver.rs` (shipped by TASK-0398 as the D-004 plumbing-aware variant of `package_resolutions`). It already walks every `BorderResolution` and packages outgoing dispatches; it already has access to the owning `CommutationBatch` for each resolution. The task is additive: extend the copy to carry the two new fields into the `PendingCommutation` it constructs.

This task does NOT introduce new resolver logic. The `CommutationBatch.local_wiring` vector — populated at `border_resolver.rs:820-866` (CON-DUP) and `:1023-1080` (CON-ERA / DUP-ERA) — is consumed as-is. No additional slot-marker emissions. No re-ordering. No filtering.

### 3. Transport `local_wiring`

- [ ] Replace the `local_wiring: Vec::new()` placeholder with the conversion from `CommutationBatch.local_wiring: Vec<(u8, u8, PortRef)>` to `Vec<LocalWiringHint>`:
  ```rust

      .local_wiring
      .iter()
      .map(|&(src_slot, src_port, target)| LocalWiringHint {
          src_slot,
          src_port,
          target,
      })
      .collect(),
  ```
  (or use `into_iter()` + consuming move if the resolver can afford to drop the source `Vec`. Per-case — developer discretion. The Clone cost of `PortRef` is trivial — it's `Copy`.)
- [ ] Preserve emission order (R23a determinism guarantee). The iterator above walks `local_wiring` in index order; no sorting, no deduplication.
- [ ] Do NOT add a duplicate-key check at the resolver side. Duplicate `(src_slot, src_port)` detection is pinned to the worker's pre-pass via R23a clause 6 / R33c case 5 HashSet (TASK-0402 territory). The resolver is allowed to emit duplicates in principle; TASK-0402 rejects them.

```rust
// border_resolver.rs (private, optional helper)
fn commutation_batch_to_pending(
    batch: &CommutationBatch,
    request_id: u32,
) -> PendingCommutation {
    debug_assert!(
        !batch.target_symbols.is_empty(),
        "NF-004: resolver must never emit empty target_symbols"
    );
    PendingCommutation {
        request_id,
        target_symbols: batch.target_symbols.clone(),
        local_wiring: batch
            .local_wiring
            .iter()
            .map(|&(src_slot, src_port, target)| LocalWiringHint {
                src_slot,
                src_port,
                target,
            })
            .collect(),
    }
}
```

## Notes

- **R23a determinism guarantee on the wire.** `bincode`'s default `Vec<T>` encoding is length-prefixed + element-in-order, so sequential wire emission preserves the resolver's internal order without extra care. The explicit `.iter().map().collect()` pattern cannot reorder — documented here for future audit. Reviewer should confirm during Stage 4.
- **Hint count per rule (fixture-dependent):** CON-DUP typically emits 4 hints (2 output agents × 2 aux ports each, some routed as sibling-slot placeholders, others to concrete border FreePorts which become `PendingNewBorder` NOT `LocalWiringHint`). Exact count depends on whether the CON-DUP redex is at the partition border or interior. The per-rule unit tests should use small synthetic fixtures from `merge::test_fixtures` (if exists) or construct minimal Partitions inline.
- **Pre-NF-001 fan-out collapse.** If `package_resolutions_with_pending` pre-TASK-0401 emitted one `PendingCommutation` per agent slot inside a batch (pre-NF-001 was `(symbol_type, arity)` scalar — so one PC per slot was the only way to encode heterogeneous siblings), the collapse to one-PC-per-batch is the primary LoC saving of this task. Grep for the loop: if `for (slot_idx, symbol) in batch.target_symbols.iter().enumerate()` inside the construction, that is the fan-out. Replace with a single construction per batch. Net effect: wire-size drops by `(batch.target_symbols.len() - 1) * sizeof(request_id + discriminants)` per batch; request_id space pressure also drops.
- **No coordinator-side behavior change.** The coordinator's BorderGraph state (D-004 plumbing, TASK-0398) consumes `request_id` as-is — TASK-0401 does not alter `request_id` encoding. `register_minted_agents` (D-004) is unaffected.
