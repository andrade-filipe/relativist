## Context

The REVIEW of bundle 2.26-B/C/D (2026-04-23) identified MF-001: the worker's `handle_round_start` signature consumes only 5 of the 7 fields that `Message::RoundStart` carries on the wire, and the emission of `Message::RoundResult` hard-codes `minted_agents: Vec::new()`. The wire protocol (shipped in 2.26-A) and the resolver (shipped in 2.26-B) produce `local_reconnections` and `pending_commutations` correctly; the worker side simply does not read or act on them. The cross-partition reductions that depend on this wiring (CON-DUP commutation, CON-ERA / DUP-ERA erasure) cannot complete — the coordinator emits the intent, the worker silently drops it, and the coordinator's round-N+2 finalizer receives an empty echo.

TEST-SPEC-0381 (which defined the original 5-argument shape) was scoped per DC-C4 / DC-C6 and did not cover `local_reconnections` or `pending_commutations` — they were meant to come in the lifecycle completion slice but fell through scope between 2.26-C (coordinator dispatch) and 2.26-D (worker lifecycle narrative only, not worker-code). This task fills that gap.

This task does NOT change the wire format (`Message::RoundStart` and `Message::RoundResult` structure on the wire is frozen by 2.26-A and remains untouched). It only changes the worker-side handler that deserializes those two fields and the handler that serializes `minted_agents` in the outgoing result.

## Notes

- **Why the signature change is backward-compatible at test-call level:** all existing UT-0381-01..09 tests pass `vec![], vec![], vec![]` for the three prior vectors and would need to add two more `vec![]` at the end. That's a mechanical touch per test; no semantic change.
- **Why `local_reconnections` is step 1.5 of the R24 pipeline:** the resolver emits `local_reconnections` alongside `border_deltas` to describe reconnections inside the post-resolution net that are NOT cross-worker border deltas but still must be applied before the worker runs `reduce_all`. Applying them after `border_deltas` (which handle cross-worker edges) and before `reduce_all` (which consumes any redexes the reconnections expose) matches the resolver's post-condition assumption.
- **Why `pending_commutations` processing happens BEFORE `reduce_all`:** the minted agents are fresh participants that the local reduction may or may not touch (depending on what `resolve_con_dup` emitted as `local_wiring`). They must exist in the partition before `reduce_all` runs. Round N+2 wire-through completes the picture via the next `RoundStart`'s `border_deltas`.
- **Why `id_range` exhaustion is an error action, not a panic:** worker minting failure is a recoverable protocol error (the coordinator can choose to repartition or abort cleanly). Panicking would kill the worker process and leave the coordinator waiting for a result forever.
- **Relation to MF-002 (TASK-0395):** TASK-0395's integration tests will exercise the code in this task end-to-end by driving `run_grid_delta` on a CON-DUP workload. If TASK-0394 ships broken, TASK-0395's G1 parity test will fail — this is the intended correctness gate.

## Design Choices Flagged for Spec-Critic (Stage 1.5)

- **DC-0394-A — allocation strategy on `slot_hint` ignore.** The `PendingCommutation` struct carries a `slot_hint: u32` field that was originally intended to let the resolver suggest which slot the minted agent should occupy within the worker's allocation space. The current resolver emits `slot_hint = SLOT_MARKER_BASE + offset` as a placeholder. This task **ignores `slot_hint` entirely** and allocates from `partition.id_range` in the usual way. Rationale: `slot_hint` was scoped as advisory only (DC-B5); treating it as binding would couple the resolver to the worker's internal ID allocation scheme. If a future optimization wants the resolver to dictate specific IDs (e.g., for cache locality), that's a spec amendment, not this task's scope. **Spec-critic: please confirm this is consistent with DC-B5's "advisory" wording, or flag for a separate ruling.**
- **DC-0394-B — order preservation of `minted_agents`.** Spec R26 says `minted_agents` carries a response per `PendingCommutation`, but does NOT explicitly mandate input-order preservation. This task preserves input order (iterate `pending_commutations` in vector order, push `MintedAgent` in the same order) because the coordinator's round-N+2 finalizer will look up `MintedAgent.request_id` anyway, but SOME implementations could legitimately reorder for allocator efficiency. **Spec-critic: rule whether order preservation is a MUST or may be a SHOULD.** Default plan: MUST (simpler contract, zero cost).
