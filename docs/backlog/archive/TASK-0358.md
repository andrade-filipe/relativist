# TASK-0358: `TransportConfig.use_zero_copy` field + `--use-zero-copy` CLI flag

**Spec:** SPEC-18 §3.9
**Requirements:** R36, R37
**Priority:** P0 (critical path — wires the bundle into the transport layer)
**Status:** TODO
**Depends on:** TASK-0356, TASK-0357
**Blocked by:** TASK-0356, TASK-0357
**Estimated complexity:** S (~50 LoC)
**Bundle:** SPEC-18 §3.5 (item 2.24) — Zero-Copy Archive

## Context

SPEC-18 §3.9 R36 mandates the `use_zero_copy` field on `TransportConfig`:
false by default (bincode v2 everywhere), opt-in per node via configuration.
R37 mandates the matching CLI flag on both `CoordinatorArgs` and `WorkerArgs`
so operators can enable the archive path on the command line without code
changes. This task adds the field, the two CLI flags, the CLI-to-config
threading (`build_transport_config` or equivalent in
`relativist-core/src/config.rs`), and the coordinator/worker call-site
switch from `send_frame_with_threshold(..)` to `send_frame_v2(.., cfg.use_zero_copy)`.

When the `zero-copy` cargo feature is disabled, the field still exists
(unconditional) but setting it to `true` produces an
`ArchiveValidationFailed("zero-copy feature disabled".into())` on the
receive side (behavior from TASK-0357 — tuple variant per spec-critic
DC-2, 2026-04-16). The CLI flag is also unconditional —
it parses in any build, but logs a warning when `cfg(not(feature =
"zero-copy"))` and the user passed `--use-zero-copy`. Keeping the
field/flag unconditional avoids `#[cfg]` fan-out into every call site and
matches how existing non-feature-gated options (e.g., `compression_threshold`)
are handled.

- `send_frame_v2` is introduced by TASK-0356 and accepts

- `recv_frame` (TASK-0357) enforces R22 and rejects mis-flagged frames with

  builds is a "you've been warned; we will fail loudly at recv time"
  safety net, not the actual enforcement.

## Notes

- **Why unconditional (not `#[cfg(feature = "zero-copy")]`):** Same reasoning
  as TASK-0354's `ArchiveValidationFailed` variant. Making the field /
  flag conditional would force every caller to gate its field access
  with `#[cfg]`, and would mean the operator's `--use-zero-copy` invocation
  fails to parse on a feature-less build (surprising). Better to parse,
  warn at startup, and let recv-side validation catch any peer that
  actually sends archives.
- **Why warn, not error, on the `!feature && flag=true` path:** the flag
  being set is not itself a bug — the operator might have intended to
  compile with the feature but built the wrong binary. A warning gives
  them the observability to notice before runtime. The recv-side rejection
  (TASK-0357) is the actual enforcement.
- **Migration of control-message send sites:** not strictly required by
  this task's acceptance criteria, but recommended for consistency.
  Developer should do the trivial migration during REFACTOR stage if not
  in the initial commit. Every `send_frame_with_threshold(w, m, t)` call
  becomes `send_frame_v2(w, m, t, cfg.transport.use_zero_copy)` with no
  behavioral change for control variants (R22 falls through to bincode).
- **Backward compat:** `send_frame_with_threshold` and the 2-arg `send_frame`
  remain in the module (TASK-0356 keeps them). Any non-migrated call site
  continues to use bincode v2 exclusively, same as today.
