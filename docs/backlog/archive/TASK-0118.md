# TASK-0118: Feature-gated module stubs for tls, metrics, otel

**Spec:** SPEC-13
**Requirements:** R9, R37, R38, R39 (SPEC-13)
**Priority:** P2 (nice to have)
**Status:** TODO
**Depends on:** TASK-0115
**Blocked by:** none
**Estimated complexity:** S (< 50 LoC)

## Context

SPEC-13 R39 says feature-gated modules should provide no-op stubs when the feature is disabled, so calling code does not need extensive `#[cfg]` blocks. This task adds the skeleton for feature-gated functionality in `observability` and `security` modules: when `metrics` is disabled, a no-op metrics init; when `tls` is disabled, a no-op TLS config; when `otel` is disabled, a no-op OTel init. The actual implementations are in Phase 7 (Security) and Phase 8 (Observability).

## Acceptance Criteria

- [ ] `observability` module has `#[cfg(feature = "metrics")]` block with a `init_metrics()` stub
- [ ] `observability` module has `#[cfg(not(feature = "metrics"))]` block with a no-op `init_metrics()`
- [ ] `observability` module has `#[cfg(feature = "otel")]` / `#[cfg(not(feature = "otel"))]` pair for `init_otel()`
- [ ] `security` module has `#[cfg(feature = "tls")]` / `#[cfg(not(feature = "tls"))]` pair for `tls_config()` returning `Option<TlsConfig>` (Some when enabled, None when disabled)
- [ ] Calling code can call these functions unconditionally without `#[cfg]` attributes
- [ ] `cargo check` passes with default features (all stubs are no-ops)
- [ ] `cargo check --features full` passes (stub signatures match real signatures)

## Notes

- These are intentionally minimal stubs. The real implementations come in Phase 7 (SPEC-10, Security) and Phase 8 (SPEC-11, Observability).
- The no-op pattern ensures that `main.rs` and other modules can call `init_metrics()` and `init_otel()` unconditionally, simplifying the startup sequence.
- SPEC-13 R38: default features are empty, so a plain `cargo build` produces a binary with only TCP, logging, and token auth.
