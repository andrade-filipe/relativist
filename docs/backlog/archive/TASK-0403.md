## Context

`grid_delta_integration_tests.rs::LocalDeltaDispatch` is the in-crate test-only stand-in for a real TCP-connected worker. It implements the `WorkerDispatch` trait by (a) taking the coordinator's `RoundStartDispatch` payload, (b) synthesizing an equivalent `Message::RoundStart`, (c) running the worker's `handle_round_start` path inline, and (d) producing a `RoundResultPayload` back to the coordinator. It is the mechanism by which UT-0385-06..08 test G1 parity of the delta protocol against v1 `run_grid` — without actually opening a TCP socket.

Today, `LocalDeltaDispatch` DOES NOT forward `CommutationBatch.target_symbols` or `CommutationBatch.local_wiring` when building the synthesized `PendingCommutation`. It uses the pre-NF-001 `(symbol_type, arity)` fallback and omits `local_wiring`. This is the test-path symmetric of the wire-path gap that TASK-0401 fixes in `border_resolver.rs`. TASK-0403 closes the loop: the test path must be byte-equivalent to the wire path, OR (better) share a single resolver helper so both paths populate the PC identically.

Once that is done, flipping `const SKIP_ASYMMETRIC: bool = true;` → `false` at the top of `grid_delta_integration_tests.rs` exposes UT-0385-08 to the 3 asymmetric IC rules (CON-DUP, CON-ERA, DUP-ERA). The three tasks preceding TASK-0403 combine to make those rules pass: TASK-0400 defines the wire shape, TASK-0401 populates it from the resolver, TASK-0402 applies it at the worker. With all three shipped, UT-0385-08 MUST pass on all 6 fixtures × 2 strict modes, with:

- `canonicalize(out_delta) == canonicalize(out_v1)` on every case.
- `metrics.total_interactions == metrics_v1.total_interactions` on every case.

This is the D-005 acceptance gate, equivalent to D-003's and D-004's.

## Notes

- **Why no new unit tests?** UT-0385-08's parameterized matrix IS the acceptance contract. Adding more tests in TASK-0403 would dilute that gate. If TASK-0400/0401/0402 had gaps, they surfaced in their own TEST-SPECs.
- **Why the 34-line comment removal?** TASK-0399 authored that block as a diagnostic when the wire-structural gap was discovered mid-DEV. Leaving it behind would be misleading — SKIP_ASYMMETRIC is now false. Replace with a short citation to D-005 close (3 lines: one-liner "D-005 closed 2026-04-23", one-liner R23/R33/R48 spec ref, one-liner UT-0385-08 gate ref).
- **If UT-0385-08 fails on ANY case:** this is NOT a TASK-0403 bug. It means a gap exists in TASK-0400/0401/0402 that slipped past their own Stage 2/3/4/5 gates. Escalation path:
  1. Capture the failing canonicalize/metrics diff.
  2. Isolate which PC-application step broke (slot-marker decode? in-round reducibility? echo slot-0 echo?).
  3. File a BLOCK back to TASK-0402 (most likely culprit — the mint-then-wire protocol is the hardest of the three).
  4. DO NOT mask the failure by reverting `SKIP_ASYMMETRIC` — the canary flip is the ship signal, and masking it would mean shipping broken code.
- **LAN-mode correctness.** Because LocalDeltaDispatch now uses the SAME transport path as the wire, a production LAN worker running a real `handle_round_start` over TCP will behave identically to the test harness on the same inputs. This is the core TCC claim that Option A preserves: G1 parity is rooted in the same codepath whether the worker is local-test or LAN-production. Option B would have forked the paths — which TASK-0403 explicitly rejects via Option A selection.
- **Interaction with `cleanup_t1_violations` helper** (TASK-0399 shipped): that helper stays. Its role is orthogonal (T1 invariant cleanup between rounds). If UT-0385-08 fails, don't suspect `cleanup_t1_violations` first — suspect the R24.1.6b wire step.
- **Metrics parity subtlety.** `metrics.total_interactions` parity means the delta worker must execute the SAME number of interaction-rule firings as the v1 sequential path on each fixture. TASK-0402's R24 ordering invariant is what preserves this: minted-to-minted principal redexes enqueued during R24.1.6b are consumed during R24.2 `reduce_all`, contributing to the count in the same round — consistent with v1 which would have executed those same redexes sequentially after the CON-DUP / CON-ERA step.
