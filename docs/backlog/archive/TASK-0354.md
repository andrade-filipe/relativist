## Context

Adds the `ArchiveValidationFailed(String)` tuple variant to
`ProtocolError` (R26, R35 — verbatim per SPEC-18 §3.8). The variant is used by:
1. TASK-0357: `recv_frame` rejects malformed rkyv archives.
2. TASK-0357: `recv_frame` rejects non-hot-path messages with `FLAG_ARCHIVED`
   set (R22 enforcement).
3. TASK-0357: `recv_frame` rejects `FLAG_ARCHIVED` frames when
   `cfg(not(feature = "zero-copy"))` (Option B in bundle index — flagged for
   spec-critic).

The variant is **unconditional** in the enum (no `#[cfg(...)]` gate) so that
exhaustive `match` patterns in callers compile in both feature configurations.
In the `cfg(not(feature = "zero-copy"))` build, the variant is constructed
exactly once in TASK-0357 (the "feature disabled" rejection branch). In the
`cfg(feature = "zero-copy")` build, it is also constructed by the rkyv
validation failure branch.

## Notes

- **Variant shape — tuple `ArchiveValidationFailed(String)` per SPEC-18 R35
  verbatim.** Spec-critic ruled (DC-2, 2026-04-16) that R35 mandates the
  tuple form; the consistency-with-`UnknownFlags`/`ChecksumMismatch`
  argument is overruled by spec compliance. Precedent in this codebase:
  `DecompressionFailed(String)` (TASK-0346) was implemented as the tuple
  form for the same reason. See
  `docs/spec-reviews/SPEC-18-section-3.5-design-choices-2026-04-16.md`
  DC-2 for the full rationale.
- The variant is unconditional (no `cfg`) because:
  - Adding `#[cfg(feature = "zero-copy")]` would make exhaustive `match` on
    `ProtocolError` differ between feature configurations — a footgun for
    every caller.
  - The cost of carrying one extra unused variant in the no-feature build
    is one enum tag (1 byte at runtime, zero compile-time complexity).
- No clippy `dead_code` warning because enum variants on `pub enum` are
  always considered used (they're public API).
