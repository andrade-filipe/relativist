## Context

This task introduces the new `relativist-core/src/merge/border_resolver.rs`
file and wires it into `merge/mod.rs`. Ships the `BorderResolver` struct
(or the module-level namespace for free-function dispatch ÔÇö see design
note) plus the `materialize_agent` helper that, given a `BorderState` side
and the worker's `Partition`, returns the `(AgentId, Symbol)` of the agent
whose principal port anchors that side of the border.

**No resolution logic in this task.** This task lands the compile-green
skeleton: types, helper, module-doc, unit test for `materialize_agent`.
Downstream tasks (TASK-0373, TASK-0374) add the rule-dispatch bodies.

**Pure-core invariant (R19 inherited):** `border_resolver.rs` lives in
`merge/` and MUST NOT import `tokio`, `async_trait`, or `crate::protocol::*`.
The 2.26-A wire variants do NOT appear in this file. Coordinator glue
(the async wire-layer caller) lives in `coordinator.rs` under 2.26-C; this
file supplies the pure logic that coordinator calls into.

**DC-B1 (resolved, Option A):** The resolver consumes a read-only slice of
the coordinator's cached partitions (`&[Partition]`, indexed by
`WorkerId as usize`). Cache maintenance ÔÇö applying `BorderDelta`s, minted-
agent registration from `RoundResult.minted_agents` (SPEC-19 R26/R33 per
DC-B5), and applying `local_reconnections` (SPEC-19 R23/R33 per DC-B3) to
each worker's cached partition ÔÇö is 2.26-C territory; this bundle's
resolver is a pure function of the cache state at round boundaries.

**DC-B4 (resolved, Option B):** Per spec-critic DC-B4 + SPEC-19 ┬º3.5 R40c
(amendment owned by 2.26-D), the resolver assumes border-adjacent agents
are pinned in the worker's local `reduce_all` for the round in which
they are border-principal. No race window exists; `materialize_agent`
finding `None` is a cache-maintenance bug (DC-B1), not a race artefact.

## Key Types / Signatures

```rust
//! Coordinator-side border-redex resolution (SPEC-19 ┬º3.2 R13-R15 parts 1-2;
//! SPEC-19 ┬º3.3, item 2.26).
//!
//! When `BorderGraph::detect_border_redexes` yields an active redex, the
//! coordinator materializes the two agents from the involved workers'
//! partitions, runs the appropriate `interact_*` rule (SPEC-03) locally,
//! and packages the resulting port reconnections as `BorderDelta`s.
//!
//! **Pure-core invariant (R19 inherited from `merge/`):** no `tokio`, no
//! `async_trait`, no `crate::protocol::*` imports. The async wire-layer
//! caller lives under `coordinator.rs` (item 2.26-C) and calls INTO this
//! module ÔÇö not the other way around.

use crate::net::{AgentId, PortRef, Symbol};
use crate::partition::types::Partition;

/// Given a principal-port endpoint (`AgentPort(id, 0)`) on a worker's
/// partition, return the live agent's `(AgentId, Symbol)` pair. Returns
/// `None` if the port is not an `AgentPort`, if the referenced agent
/// slot is vacant, or if the port slot is not principal.
///
/// Callers MUST only invoke this for `is_redex == true` border sides
/// (both endpoints principal). The `None` return path is defensive so
/// the resolver can surface a typed error instead of panicking.
pub(crate) fn materialize_agent(
    partition: &Partition,
    port: PortRef,
) -> Option<(AgentId, Symbol)> {
    match port {
        PortRef::AgentPort(id, 0) => partition
            .subnet
            .agents
            .get(id as usize)?
            .as_ref()
            .map(|agent| (id, agent.symbol)),
        _ => None,
    }
}
```

## Notes

- **Why a module, not an immediate `BorderResolver` struct:** item 2.26-B
  needs multiple coordination points (agent materialization, rule
  dispatch, delta packaging). A struct adds no value yet; module-level
  pub(crate) functions are sufficient. If TASK-0375 decides a stateful
  `BorderResolver` is cleaner (holds config or scratch buffers), this
  task's skeleton is trivially upgradeable ÔÇö the struct just gains
  methods that internally call the free function.
- **Why `pub(crate)` on `materialize_agent`:** the function is an
  implementation detail of the resolver. External callers should go
  through the higher-level `resolve_border_redex` entry point added in
  TASK-0373.
- **Why this task is NOT a no-op:** it pins the module invariants in
  `//!` doc BEFORE any resolution code lands, preventing a future
  reviewer from accidentally importing `tokio` or `protocol` into the
  file when writing the dispatch body.

## Design Choices Flagged for Spec-Critic (Stage 1.5)

- **DC-B1 ÔÇö where to source the agent data.** Two options:
  (a) coordinator caches a copy of each worker's `Partition` (the
  `InitialPartition` payload from 2.26-A R31); (b) coordinator requests
  the two agents over the wire on demand (a new `AgentRequest` message).
  Option (a) is assumed by this bundle (the `materialize_agent` helper
  takes `&Partition`). Option (b) would require a new CÔåÆW request round
  trip per border redex ÔÇö doubling the latency. **spec-critic please rule.**

- **DC-B2 ÔÇö what happens if `materialize_agent` returns `None`.** Two
  options: (i) panic (programmer error ÔÇö invariant violation); (ii)
  return a typed `ResolverError::AgentMissing { bid, side }`. This task
  ships `Option` (soft-fail) so downstream tasks can pick (i) or (ii).
  **spec-critic please rule on the error semantics.**
