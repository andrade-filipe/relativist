## Context

After distributing partitions, the coordinator awaits `PartitionResult` from all workers before proceeding to merge (R22). Each received message is validated: it must be a `PartitionResult` with the correct round number, or an `Error` (which causes immediate abort). The function tracks `bytes_received` and `network_recv_time` for metrics (R33, R34). A configurable timeout (default 600 seconds) aborts if collection takes too long (R30). If a connection is lost, the coordinator aborts (R25).

## Notes

- Collection MAY be concurrent or sequential (R22). Sequential is simpler and acceptable because workers finish at different times. The spec does not require concurrent collection.
- The round number in `PartitionResult` must be validated to match the current round (`assert_eq` in pseudocode). A mismatch indicates a protocol desynchronization, which should be treated as an error.
- The function does NOT perform merge -- that is a separate concern. It only collects and validates the results.
