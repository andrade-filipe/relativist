## Context

R13 (coordinator requests agents from workers and runs `interact_*`
locally) and R14 (coordinator-side `interact_*` invocation does not
violate the layer boundary because SPEC-13 R6-R8 permits coordinator
access to core modules) describe the *coordinator-side integration* with
`BorderGraph`. R15 parts 1 and 2 (dispatch deltas back to workers; update
the graph from the resolution) similarly describe coordinator behaviour.

Per the bundle scope-decision, **NONE of these coordinator behaviours
ship in this bundle** — they ship under item 2.26 in a later bundle. But
the data-structure primitives (R12 detect, R16 remove, R15 part 3 add
via TASK-0364) MUST be designed in a way that the future coordinator can
call them cleanly.

This task ships **documentation only** to record:
1. The expected coordinator-side call sequence (R13-R15 lifecycle).
2. The exact primitives from this bundle that the coordinator will call.
3. The R19 pure-core invariant guard (so a future patch landing
   `async fn` in this file is rejected by review automatically).
4. A pointer to the future item 2.26 bundle for the actual call code.

No production code changes. No new tests. The deliverable is a single
augmented `//!` module doc block at the top of `border_graph.rs`.

//! Coordinator-side data structure tracking border-wire connectivity.
//!
//! SPEC-19 §3.2 (R8-R19, item 2.35) — BorderGraph replaces the full merged
//! net for delta-mode rounds. The coordinator updates this as workers
//! report border deltas; principal-vs-principal pairs become detectable
//! border redexes that the coordinator resolves locally before
//! redistributing the resulting deltas.
//!
//! # Coordinator-side lifecycle (R13-R15, ships under item 2.26)
//!
//! Per round, the coordinator:
//! 1. Calls [`BorderGraph::detect_border_redexes`] to enumerate the
//!    `(border_id, &BorderState)` pairs that need resolution.
//! 2. For each redex, requests the two agents from their owning workers
//!    via the wire protocol (item 2.26 territory — not shipped here).
//! 3. Runs the matching `interact_*` rule from
//!    [`crate::reduction::rules`] locally (R13/R14). SPEC-13 R6-R8
//!    permits this coordinator-side core access; it is the documented
//!    "coordinator boundary exception".
//! 4. For each affected worker, calls [`BorderGraph::apply_deltas`] with
//!    the resulting per-side updates.
//! 5. For each consumed border (CON-CON / DUP-DUP / ERA-ERA annihilation),
//!    calls [`BorderGraph::remove_border`] (R16).
//! 6. For CON-DUP commutations that produce new border wires, calls
//!    [`BorderGraph::add_border_states`] with `Vec<AddBorderEntry>`
//!    inputs; the graph derives `is_redex` from side endpoints (R15
//!    part 3 primitive; R9 invariant).
//! 7. Dispatches `Message::RoundStart { border_deltas, ... }` to each
//!    worker (item 2.26).
//!
//! # Pure-core invariant (R19)
//!
//! This module MUST NOT import `tokio`, `async_trait`, or anything under
//! [`crate::protocol`]. The bundle (SPEC-19 §3.2) ships a grep-based CI
//! guard:
//!
//! ```text
//! grep -E '^use\s+(tokio|crate::protocol|async_trait)' \
//!   relativist-core/src/merge/border_graph.rs
//! # MUST return zero matches.
//! ```
//!
//! Item 2.26's coordinator integration calls INTO this module from the
//! protocol layer; the dependency edge stays one-directional
//! (`protocol/` → `merge/`, never the reverse).
//!
//! # Out of scope (item 2.26 territory)
//!
//! - Coordinator-side dispatch loop (R13).
//! - Coordinator-side `interact_*` call site (R14).
//! - `Message::RoundStart` / `Message::RoundResult` wire-format
//!   extensions (R20-R36).
//! - `GridConfig.delta_mode` flag (R20).
//! - Worker-side delta emission and stateful-worker lifecycle
//!   (R20-R30).
//! - The `run_grid_delta` BSP loop (SPEC-19 §3.3, §4.3).
```

## Notes

- **Why ship documentation as its own task instead of folding it into
  TASK-0360:** the doc references method names from TASK-0361 (`from_partition_plan`),
  TASK-0362 (`apply_deltas`), TASK-0363 (`detect_border_redexes`), and
  TASK-0364 (`remove_border`, `add_border_states`). Writing it after
  those signatures stabilise prevents stale doc links. Splitting also
  isolates doc-only changes from code-changing tasks for cleaner review.
- **Why the grep guard lives in CI rather than as a `#[deny]` lint:**
  Rust has no built-in lint for "this file may not import tokio". A
  custom proc-macro lint would be heavy. A grep in CI is dirt-cheap and
  matches the precedent set by SPEC-18 §3.5's design-choice discrimination
  comment grep (DC-3). Item 2.26's bundle SHOULD add the grep to the
  `.github/workflows/` config (out of scope here; this task documents
  the invariant for that future task to enforce).
- **Why no test:** documentation alone has no behavioural contract to
  test. The `cargo doc` zero-warnings gate is the only mechanical check.
- **Per bundle DC-1..DC-4:** if spec-critic flips any of the design
  choices, this task's doc block must be re-aligned. The amendment is
  straightforward (text edits in the lifecycle section); LoC unchanged.
