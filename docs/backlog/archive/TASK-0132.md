## Context

When TLS is enabled, the coordinator must perform a TLS handshake on each accepted TCP connection before any wire protocol messages are exchanged (R28). The TLS handshake wraps the raw `TcpStream` into a `tokio_rustls::server::TlsStream<TcpStream>`, which implements `AsyncRead + AsyncWrite` and can be used transparently by the existing `send_frame`/`recv_frame` functions (R24). This task implements the server-side TLS accept function that sits between TCP accept and the authentication handshake.

## Key Types / Signatures

```rust
use tokio::net::TcpStream;
use tokio_rustls::TlsAcceptor;
use tokio_rustls::server::TlsStream;

/// Performs a TLS handshake on an accepted TCP connection (R28).
///
/// Wraps the raw TcpStream into a TlsStream that encrypts all
/// subsequent communication. The returned stream implements
/// AsyncRead + AsyncWrite, so the framing layer (send_frame/recv_frame)
/// works transparently on top of it (R24).
///
/// Called after TCP accept but before the Register handshake.
pub async fn tls_accept(
    stream: TcpStream,
    config: &TlsServerConfig,
) -> Result<TlsStream<TcpStream>, SecurityError> {
    let acceptor = TlsAcceptor::from(config.config.clone());
    acceptor.accept(stream).await.map_err(|e| {
        SecurityError::TlsConfig(format!("TLS handshake failed: {}", e))
    })
}
```

## Notes

- The `TlsAcceptor::from(Arc<ServerConfig>)` clones the `Arc`, so the config can be reused for multiple connections.
- The TLS handshake is a separate async operation that may fail independently of the TCP connection. A failed handshake should log at WARN with the remote address but not terminate the accept loop.
- The coordinator's accept flow with TLS becomes: TCP accept -> TLS handshake -> Register handshake -> grid loop. Without TLS: TCP accept -> Register handshake -> grid loop.
- `TlsStream<TcpStream>` erases the TLS layer at the type level for downstream consumers. The `send_frame`/`recv_frame` functions accept any `AsyncRead + AsyncWrite`, so they work with both `TcpStream` and `TlsStream<TcpStream>`.
- R28a: TLS applies only to `TcpTransport`. `ChannelTransport` (SPEC-13, R31) does not support TLS and MUST NOT attempt TLS handshakes. Integration tests using `ChannelTransport` MAY run at Tier 1 or Tier 2 regardless of the `tls` feature flag. Tier 3 integration tests MUST use `TcpTransport` (localhost).
