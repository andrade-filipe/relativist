**Blocked by:** none

## Context

SPEC-03 Section 4.5 (Revised v3) introduces a `link` helper function that wraps `Net::connect` with a guard for removed agents. This is a module-private function in `src/reduction/rules.rs` that all interaction functions use for external wire reconnection.

The `link` helper serves two purposes:

1. **R25 (self-referencing auxiliary ports):** When annihilation rules (CON-CON or DUP-DUP) encounter auxiliary ports that point back to the pair being consumed, the saved `PortRef` values reference removed agents after `remove_agent`. The `link` guard detects this (checks `agents[id].is_none()`) and makes the link a no-op, preventing ghost entries in the port array that would violate I2.

2. **R26 (FreePort boundary sentinels):** During local reduction in partitioned sub-nets, auxiliary ports may be connected to `FreePort(bid)` boundary sentinels. `link` delegates to `Net::connect`, which writes `FreePort(bid)` to the `AgentPort` side's port array but is a no-op for the `FreePort` side (no port array slot). This one-sided write is acceptable because the `free_port_index` is reconstructed post-reduction (SPEC-05).

The approach of wrapping `Net::connect` (SPEC-02) rather than modifying it was chosen because: (a) it keeps `connect` simple and in SPEC-02's territory, (b) the removed-agent guard is only relevant for annihilation rules, and (c) `link` provides a uniform wrapper for documenting FreePort behavior.

**Visibility:** `link` is `fn` (module-private), NOT `pub`. Only interaction functions in `rules.rs` call it. Internal wires in `interact_comm` use `net.connect()` directly because both endpoints are freshly created agents (never removed, never FreePort).

## Key Types / Signatures

```rust
/// Safe link: wraps Net::connect with a guard for removed agents (R25).
///
/// If either endpoint is an AgentPort whose agent has been removed
/// (agents[id] is None), the link is a no-op. This handles the
/// self-referencing auxiliary port edge case in annihilation rules.
///
/// For non-annihilation rules (commutation, erasure), the guard is
/// never triggered because auxiliary ports of the active pair always
/// point to agents outside the pair (or to FreePort sentinels).
///
/// FreePort behavior (R26): FreePort is not considered "removed".
/// When one endpoint is FreePort(bid), connect writes FreePort to
/// the AgentPort side's port array. The FreePort side is a no-op
/// in set_port (no port array slot). This one-sided write is
/// acceptable: free_port_index is reconstructed post-reduction
/// (SPEC-05, Section 4.3).
fn link(net: &mut Net, a: PortRef, b: PortRef) {
    let is_removed = |net: &Net, p: &PortRef| -> bool {
        if let PortRef::AgentPort(id, _) = p {
            net.agents[*id as usize].is_none()
        } else {
            false // FreePort is not "removed"; connect handles it
        }
    };
    if is_removed(net, &a) || is_removed(net, &b) {
        return;
    }
    net.connect(a, b);
}
```

## Notes

- This function is a key addition in SPEC-03 v3 (SC-001, SC-002). In v2, interaction functions called `net.connect()` directly. In v3, external wires go through `link` for safety.
- The guard is strictly a defense against the "read neighbors, remove, reconnect" pattern: after `remove_agent`, saved PortRef values may point to gone agents. The guard catches this.
- For internal wires in `interact_comm` (where both endpoints are freshly created agents), `net.connect()` is called directly. The `link` guard would never trigger there, so using `connect` directly is cleaner and documents the invariant that internal wires are always safe.
- The `is_removed` closure uses a non-capturing reference to `net` for clarity. The implementer MAY inline the check or use a standalone helper function if preferred.
