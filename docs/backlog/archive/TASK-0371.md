## Context

SPEC-06 R5 (reinforced by SPEC-18 R3 and R4, and SPEC-19 R37) mandates
that `Message` enum variants be append-only, with stable discriminants
across protocol versions. Any refactor that reorders, removes, or
re-numbers variants is a breaking wire-protocol change.

This task ships a single byte-level regression test that encodes each
of the 12 current variants (0..=11) under `bincode_v2` and asserts the
first byte of the encoded output matches the expected discriminant
value. Under bincode v2 varint encoding (SPEC-18 R3), discriminants
0..=250 encode as exactly one byte equal to the discriminant value.

If any future task accidentally reorders variants, this test fails
loudly at the expected position.

## Notes

- **Why a byte-level test rather than a `std::mem::discriminant`
  comparison:** `std::mem::discriminant` returns an opaque token that
  doesn't expose the wire encoding. The wire encoding is what we
  actually care about for protocol stability. The byte-level assertion
  makes the contract explicit.
- **Varint assumption (SPEC-18 R3):** valid for discriminants 0..=250.
  Once the enum exceeds 251 variants, the encoding switches to 3 bytes
  and this test needs updating (comment in the test body flags this:
  "assertion holds while variant count ≤ 251; extend encoding check
  if more variants are appended").
- **Why not a compile-time `#[repr(u8)]`:** `Message` is not
  `#[repr(u8)]` (it carries non-`Copy` payloads with different sizes),
  so the Rust discriminant isn't directly exposed. The bincode
  encoding IS the wire discriminant.
- **Why this test is in `types.rs` not `delta_wire_tests.rs`:** the
  discriminant check is a pure-bincode property, not a wire-frame
  property. Colocating with the other serde round-trip tests keeps
  the test concerns cleanly stratified.
