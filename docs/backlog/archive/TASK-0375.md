# TASK-0375: Delta packaging — group `BorderResolution` outputs by worker for wire dispatch

**Spec:** SPEC-19 §3.2 R15 part 1 (send port-update deltas to affected workers), §3.3 R23 (item 2.26)
**Requirements:** R15 part 1 (delta dispatch to workers)
**Priority:** P1 (consumed by TASK-0376 integration test; consumed by 2.26-C when the wire-layer caller materializes)
**Status:** TODO
**Depends on:** TASK-0373, TASK-0374
**Blocked by:** TASK-0374
**Estimated complexity:** S (~80 LoC)
**Bundle:** SPEC-19 §3.3 (item 2.26-B) — Coordinator-side border-redex resolution

## Context

Ships the final step of the resolver pipeline: given a batch of
`BorderResolution` structs (one per border redex detected this round),
group the outputs by worker into a single `Vec<(WorkerId,
RoundStartDispatch)>` where `RoundStartDispatch` is a dispatcher-side
struct containing the three payload fields that 2.26-A's `RoundStart`
wire message carries:

- `border_deltas: Vec<BorderDelta>`
- `resolved_borders: Vec<u32>`
- `new_borders: Vec<(u32, PortRef)>`

This task is the seam between the pure-core resolver (TASK-0372..0374)
and the async wire-layer caller that ships under 2.26-C (worker
lifecycle) or whichever 2.26 sub-bundle finally wires up
`run_grid_delta`. The output of this task is a pure-core data
structure; its consumer (the coordinator BSP loop) lives outside
2.26-B's scope.

**Hard scope line:** this task MUST NOT touch
`relativist-core/src/protocol/messages.rs` or `coordinator.rs` — those
are 2.26-A and 2.26-C territory. This task's output struct is a new
pure-core type. The wire-side mapping (`RoundStartDispatch → RoundStart`)
ships under 2.26-C.

## Key Types / Signatures

```rust
/// Per-worker payload for the `RoundStart` wire message (2.26-A).
/// Pure-core mirror of the wire variant — the coordinator converts
/// this into `Message::RoundStart` at dispatch time under 2.26-C.
#[derive(Debug, Clone, Default)]
pub(crate) struct RoundStartDispatch {
    pub(crate) border_deltas: Vec<BorderDelta>,
    pub(crate) resolved_borders: Vec<u32>,
    pub(crate) new_borders: Vec<(u32, PortRef)>,
}

pub(crate) fn package_resolutions(
    resolutions: Vec<BorderResolution>,
    num_workers: usize,
) -> Vec<(WorkerId, RoundStartDispatch)> {
    let mut per_worker: Vec<RoundStartDispatch> =
        (0..num_workers).map(|_| RoundStartDispatch::default()).collect();

    for resolution in resolutions {
        // Fan out worker_deltas → per_worker[w].border_deltas.
        // Fan out resolved_borders → per_worker[both_sides].resolved_borders.
        // Fan out new_borders → per_worker[both_sides].new_borders.
    }

    per_worker
        .into_iter()
        .enumerate()
        .map(|(w, dispatch)| (w as WorkerId, dispatch))
        .collect()
}
```

## Notes

- **Why `RoundStartDispatch` instead of directly constructing
  `Message::RoundStart`:** `Message::RoundStart` lives in
  `protocol/messages.rs` (added by 2.26-A TASK-0368). Constructing it
  here would cross the pure-core boundary — `merge/` imports
  `crate::protocol::*` would break R19. The 2.26-C worker-lifecycle
  bundle gets the `RoundStartDispatch → Message::RoundStart` mapping
  for free because it already depends on both.

- **Why worker 2 (in the 3-worker test above) still gets an empty
  dispatch:** SPEC-19 R23 says the coordinator sends `RoundStart` to
  EVERY worker every round, even if the deltas are empty. This is
  how workers advance their round counter and continue the BSP loop.
  The caller (2.26-C) handles the dispatch; this task just packages
  the payload.

## Dependencies on Sibling Bundles

- **Blocks on 2.26-A TASK-0366 (partial):** `RoundStartDispatch` is
  NOT a wire type and does NOT depend on `Message::RoundStart` being
  defined. So this task can actually ship BEFORE TASK-0366 if
  necessary. But logically, `RoundStartDispatch` is designed to
  mirror the wire variant's fields 1:1, so TASK-0366's field names
  are referenced.

- **Consumed by 2.26-C:** the worker-lifecycle bundle will add a
  `RoundStartDispatch → Message::RoundStart` converter in the
  coordinator. That converter does NOT ship in 2.26-B.

## Design Choices Flagged for Spec-Critic (Stage 1.5)

- **DC-B7 — Who tracks which worker owns which side of a resolved
  border?** `BorderResolution` from TASK-0373/0374 contains
  `resolved_borders: Vec<u32>` (the IDs, not the worker assignment).
  To fold into `per_worker[w].resolved_borders`, this task needs the
  `BorderState.worker_a / worker_b` info. Options:
  (a) embed `(bid, worker_a, worker_b)` triples in
  `BorderResolution.resolved_borders`;
  (b) pass the `BorderGraph` (post-mutation state) to
  `package_resolutions` so it can look up owners;
  (c) both TASK-0373/0374 already embed the worker assignments in
  `BorderResolution.worker_deltas` (the `(WorkerId, Vec<BorderDelta>)`
  tuples), so option (a) is redundant if the two workers' entries
  always appear in `worker_deltas`.
  Current plan: option (a) — embed the worker IDs in
  `resolved_borders`. **Spec-critic please rule.** This affects the
  `BorderResolution` struct shape flagged in TASK-0373, so this DC
  may cascade to TASK-0373 amendment.
