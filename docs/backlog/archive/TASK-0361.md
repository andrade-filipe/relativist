## Context

R10 mandates that `BorderGraph` be initialised from a `PartitionPlan`'s
`borders` map and the per-partition `free_port_index` data, with all
`BorderState` entries populated to reflect the post-`split()` topology.

The spec §4.2 pseudocode references a `find_border_owner(plan, border_id,
side)` helper that returns `(WorkerId, PortRef)` — the worker that owns a
given side of a given border. The helper walks `plan.partitions` looking
for the partition whose `free_port_index` contains `border_id`; the
matching `PortRef` value in that map is the current endpoint, and the
partition's `worker_id` is the owner.

A subtlety: a single partition's `free_port_index` only has ONE entry per
border (the local side it owns). The TWO sides of a border live in TWO
DIFFERENT partitions. So `find_border_owner` must visit each partition
and collect the side(s) where each border appears; the spec sketch's
`find_border_owner(plan, border_id, side_index)` API obscures this. The
implementation MUST iterate `plan.partitions` once and build all
`BorderState` entries in a single pass.

## Acceptance Criteria

- [ ] `BorderGraph` gets a new associated function:
      `pub fn from_partition_plan(plan: &crate::partition::types::PartitionPlan)
      -> Self`. (Public API — coordinators call this.)
- [ ] The constructor:
  1. Initialises `borders` (empty `HashMap`), `worker_borders` (a
     `vec![Vec::new(); plan.partitions.len()]`), and `active_redexes`
     (empty `HashSet`).
  2. Iterates `plan.partitions` once. For each `partition`, iterates
     `partition.free_port_index` (a `HashMap<u32, PortRef>`). For each
     `(border_id, port_ref)` entry, accumulates a side observation in a
     temporary `HashMap<u32, Vec<(WorkerId, PortRef)>>` (border_id → list
     of (owner, endpoint) sightings; the list MUST end up with exactly 2
     entries per `border_id` per the partitioning C3 invariant).
  3. After the iteration, materialises `BorderState` entries from the
     temporary map. For each `border_id` with sightings `[(wa, sa), (wb,
     sb)]`, constructs `BorderState { border_id, side_a: sa, side_b: sb,
     worker_a: wa, worker_b: wb, is_redex: is_principal_pair(sa, sb) }`,
     inserts into `borders`, pushes `border_id` into both
     `worker_borders[wa as usize]` and `worker_borders[wb as usize]`, and
     if `is_redex` inserts `border_id` into `active_redexes`.
  4. If any `border_id` has fewer than 2 OR more than 2 sightings, the
     constructor MUST panic with a clear message (this is a partitioning
     C3 violation, an invariant bug — fail fast). The panic message MUST
     include the offending `border_id` and the actual sighting count.
- [ ] The constructor MUST visit `plan.borders` keys to validate that every
      border declared by the plan ends up with exactly 2 sightings (no
      orphan borders that have a `borders[id]` entry but no
      `free_port_index` entry on either side). If a declared border has 0
      sightings, panic with a message naming `border_id` (this catches a
      `split()` bug that silently drops a border).
- [ ] The constructor MUST NOT visit `partition.net.agents` or any port
      array — it is a pure metadata pass over `free_port_index`. Time
      complexity is O(B) per R18 (B = total border count summed across
      partitions = 2× distinct border count).
- [ ] `find_border_owner` is **NOT** a public function. The spec §4.2
      pseudocode uses it as an explanatory device; the actual
      implementation folds the lookup into the single-pass iteration above.
      A `pub(crate)` helper MAY exist for testability if the developer
      prefers, but the constructor MUST NOT do two passes.
- [ ] Inline `#[cfg(test)] mod tests` extends the existing module with:
      - `from_partition_plan_empty`: a `PartitionPlan` with one partition
        and zero borders → `BorderGraph` has empty `borders`,
        `worker_borders[0].is_empty()`, empty `active_redexes`.

      - `from_partition_plan_principal_aux`: 2-partition plan with 1

      - `from_partition_plan_two_borders`: 2-partition plan with 2 borders

## Key Types / Signatures

```rust
impl BorderGraph {
    /// Initialise from a `PartitionPlan` after `split()` (SPEC-19 R10).
    ///
    /// Single O(B) pass over each partition's `free_port_index` collects
    /// the two sides of every border, then materialises `BorderState`
    /// entries with `is_redex` precomputed and the `active_redexes` set
    /// seeded.
    ///
    /// # Panics
    ///
    /// Panics if any border in `plan.borders` does not have exactly 2
    /// sightings across the partitions' `free_port_index` maps. This is a
    /// partitioning C3 (FreePort bijectivity) invariant violation —
    /// `split()` is buggy if it ever happens.
    pub fn from_partition_plan(plan: &crate::partition::types::PartitionPlan) -> Self {
        use crate::partition::types::WorkerId;

        let mut borders: std::collections::HashMap<u32, BorderState> =
            std::collections::HashMap::new();
        let mut worker_borders: Vec<Vec<u32>> =
            vec![Vec::new(); plan.partitions.len()];
        let mut active_redexes: std::collections::HashSet<u32> =
            std::collections::HashSet::new();

        // Pass 1: collect sightings.
        let mut sightings: std::collections::HashMap<u32, Vec<(WorkerId, PortRef)>> =
            std::collections::HashMap::new();
        for partition in &plan.partitions {
            for (&border_id, &port_ref) in &partition.free_port_index {
                sightings
                    .entry(border_id)
                    .or_default()
                    .push((partition.worker_id, port_ref));
            }
        }

        // Pass 2: validate + materialise.
        for &border_id in plan.borders.keys() {
            let observed = sightings.remove(&border_id).unwrap_or_default();
            assert_eq!(
                observed.len(),
                2,
                "partitioning C3 invariant violated: border_id {} has {} sightings (expected 2)",
                border_id,
                observed.len()
            );
            let (wa, sa) = observed[0];
            let (wb, sb) = observed[1];
            let is_redex = is_principal_pair(sa, sb);
            if is_redex {
                active_redexes.insert(border_id);
            }
            borders.insert(
                border_id,
                BorderState { border_id, side_a: sa, side_b: sb, worker_a: wa, worker_b: wb, is_redex },
            );
            worker_borders[wa as usize].push(border_id);
            worker_borders[wb as usize].push(border_id);
        }

        // Pass 3: catch any sighting whose border_id is NOT in plan.borders
        // (another C3 inversion: orphan free_port_index entry).
        if let Some((&orphan, _)) = sightings.iter().next() {
            panic!(
                "partitioning C3 invariant violated: free_port_index entry for border_id {} has no matching plan.borders entry",
                orphan
            );
        }

        BorderGraph { borders, worker_borders, active_redexes }
    }
}
```

## Notes

- **Why one collection pass instead of the spec's `find_border_owner`
  pseudocode:** the pseudocode is O(B²) (for each border, scan all
  partitions for both sides) but the spec doesn't actually mandate the
  pseudocode shape — only the resulting state. R18's "O(B) for
  initialization" complexity bound makes the single-pass approach
  obligatory.
- **Why panic on C3 violation instead of returning `Result`:** the spec
  C3 invariant (SPEC-04 R8 — FreePort bijectivity) is a structural
  guarantee from `split()`. A C3 violation is a programming bug in
  partitioning, not a runtime condition that the coordinator can recover
  from. Panicking with a precise message is the right response per CLAUDE.md
  ("no unwrap in production" exception: panics that document an invariant
  bug are explicit assertions of a contract).
- **`BorderState` field-order ambiguity:** when a border has 2 sightings
  collected by hash-map iteration, the order of `(side_a, side_b)` is
  effectively arbitrary (HashMap iteration order). This is FINE because
  R11's `apply_deltas` looks up by `worker_id` (matching `worker_a` vs
  `worker_b`), not by side index, and `is_principal_pair` is symmetric.
  TEST-SPEC-0361 should use `BTreeMap`-based fixtures or assert
  set-equality (`{worker_a, worker_b} == {0, 1}`) instead of positional
  equality, to avoid flakiness.
