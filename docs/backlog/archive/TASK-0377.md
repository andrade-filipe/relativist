## Context

The §3.2 bundle (TASK-0360..0365) shipped `border_graph.rs` with a
grep guard verifying no `tokio` / `protocol` / `async_trait` imports
— that's R19, the pure-core invariant. This bundle adds a second
pure-core file (`border_resolver.rs`). The guard MUST extend to it.

The guard is partially embedded in each earlier task's acceptance
criteria ("grep guard passes"), but that's a manual check executed by
the `developer` agent during Stage 3. This task makes the guard
**programmatic and automatic** via one of:

(a) a `#[test]` that runs
`std::process::Command::new("grep")` against the file (fragile — grep
might not be on every dev machine; fails on Windows);

(b) a compile-time guard: a `#[cfg(...)]` trick that fails to compile
if certain identifiers are reachable in the file;

(c) a `build.rs` check that reads the file and panics on forbidden
imports;

(d) a module-level doc-comment that's enforced by reviewer discipline
+ an assertion in the module's `#[cfg(test)]` block that imports a
non-pure identifier and checks the resolver doesn't re-export it.

**Default plan:** option (a) — programmatic grep — with a Windows-
compatible fallback (pure-Rust parser: read the file, filter lines
starting with `use`, assert none match the forbidden prefixes). This
mirrors the pattern the §3.2 bundle used in its "acceptance gate"
section of the bundle-index file.

## Acceptance Criteria

- [ ] New test in `relativist-core/src/merge/border_resolver.rs`:
      ```rust
      #[test]
      fn pure_core_no_forbidden_imports() {
          let src = include_str!("border_resolver.rs");
          for line in src.lines() {
              let trimmed = line.trim_start();
              if !trimmed.starts_with("use ") {
                  continue;
              }
              assert!(
                  !trimmed.starts_with("use tokio"),
                  "R19 violation: border_resolver.rs imports tokio"
              );
              assert!(
                  !trimmed.starts_with("use async_trait"),
                  "R19 violation: border_resolver.rs imports async_trait"
              );
              assert!(
                  !trimmed.starts_with("use crate::protocol"),
                  "R19 violation: border_resolver.rs imports crate::protocol"
              );
          }
      }
      ```
      **Note:** `include_str!` on the current file is a common pattern
      for in-source pure-Rust pre-commit checks. Test runs in any
      environment that builds the crate (no `grep` dependency).

## Notes

- **Why this is a separate task, not part of TASK-0372:** TASK-0372
  ships the skeleton; its compile-time acceptance criterion covers
  "grep guard passes" as a manual check. This task upgrades the manual
  check to a programmatic one that survives beyond Stage 3. A future
  2.26-C developer who accidentally writes `use tokio::time::sleep;`
  at the top of `border_resolver.rs` (e.g., while adding a helper that
  turned out to need async) will see a red test immediately — the
  pipeline itself blocks the regression.

- **Why not a `build.rs` check:** `build.rs` runs before the crate
  compiles. Changes to the file would not retrigger the check
  automatically unless `cargo:rerun-if-changed` is set. An in-source
  test is simpler and runs on every `cargo test` invocation.

- **Why not move this to a crate-wide lint:** clippy/rustc don't
  natively support "this file may not import this crate". A workspace-
  level enforcement would require a custom lint tool (e.g.,
  `cargo-deny` module rules) — not worth the infra overhead for one
  file. The in-source test is lightweight and CI-transparent.

## Design Choices Flagged for Spec-Critic (Stage 1.5)

- **DC-B8 — scope of the import guard.** Strict reading of R19 says
  the guard should cover ALL pure-core files in `merge/`, not just
  `border_resolver.rs`. Options:
  (a) ship the guard only for the new file (minimal 2.26-B scope);
  (b) extend the guard to every `merge/*.rs` file (broader safety
  net);
  (c) turn the guard into a shared helper in a `dev-only` module
  (`merge/internal/pure_core_guard.rs`) that takes a file path and
  runs the check — then each pure-core file opts in with a one-liner
  test.

  Current plan: (a). **Spec-critic please rule whether (b) or (c)
  is in scope for this bundle or should be deferred to a separate
  hardening task.**

- **DC-B9 — guard granularity.** Should the test also forbid
  `crate::coordinator::*` imports (since `coordinator.rs` imports
  `protocol`, so transitively bringing it in via `coordinator`
  re-exports would re-introduce the async dependency)? Currently the
  test only forbids direct `tokio` / `protocol` / `async_trait`. Any
  future file that re-exports protocol types under `merge/` would
  pass this guard but violate the intent. Proposal: extend the
  forbidden list with `use crate::coordinator` and `use crate::worker`
  to cover the transitive-leak case. **Flag for spec-critic.**
