## Context

When a worker connects to a TLS-enabled coordinator, it must initiate a TLS handshake before sending any wire protocol messages (R28). The worker uses `tokio_rustls::TlsConnector` with its `TlsClientConfig` to wrap the `TcpStream` into a `TlsStream<TcpStream>`. The server name for SNI (Server Name Indication) is derived from the coordinator's hostname. This task implements the client-side TLS connect function.

## Key Types / Signatures

```rust
use tokio_rustls::TlsConnector;
use tokio_rustls::client::TlsStream;
use rustls::pki_types::ServerName;

/// Initiates a TLS handshake on an established TCP connection (R28).
///
/// Wraps the TcpStream into a client-side TlsStream that encrypts
/// all subsequent communication. Uses the provided TlsClientConfig
/// to verify the coordinator's certificate against the trusted CA (R26).
///
/// Called after TCP connect but before the Register message.
pub async fn tls_connect(
    stream: TcpStream,
    config: &TlsClientConfig,
    server_name: &str,
) -> Result<TlsStream<TcpStream>, SecurityError> {
    let server_name = ServerName::try_from(server_name.to_owned())
        .map_err(|e| SecurityError::TlsConfig(format!("invalid server name: {}", e)))?;

    let connector = TlsConnector::from(config.config.clone());
    connector.connect(server_name, stream).await.map_err(|e| {
        SecurityError::TlsConfig(format!("TLS handshake failed: {}", e))
    })
}
```

## Notes

- The server name for SNI must match the Common Name (CN) or Subject Alternative Name (SAN) in the coordinator's certificate. For self-signed certificates in a TCC context, this is typically `"localhost"` or the coordinator's hostname.
- If the coordinator's certificate uses an IP address (common in grid computing), the `ServerName::IpAddress` variant handles this. `ServerName::try_from` supports both DNS names and IP addresses.
- The TLS connect is called after `TcpStream::connect()` succeeds but before sending the `Register` message.
- Error messages from `tls_connect` may contain details about certificate validation failures, which is acceptable for the worker-side log (the worker is a trusted operator entity). The coordinator-side does NOT log such details to clients.
