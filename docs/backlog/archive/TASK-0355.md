# TASK-0355: 16-byte aligned receive buffer (`rkyv::util::AlignedVec`)

**Spec:** SPEC-18 §3.5
**Requirements:** R25
**Priority:** P0 (critical path — needed by TASK-0357)
**Status:** TODO
**Depends on:** TASK-0352, TASK-0353
**Blocked by:** TASK-0352, TASK-0353
**Estimated complexity:** S (~50 LoC)
**Bundle:** SPEC-18 §3.5 (item 2.24) — Zero-Copy Archive

## Context

Rkyv archives MUST be deserialized from a 16-byte-aligned buffer (R25).
`vec![0u8; len]` makes no alignment promise; on most platforms it's 8-byte
aligned at best, sometimes only 1-byte aligned for small vectors. Calling
`rkyv::access::<ArchivedPartition>(&unaligned_buf)` on an unaligned buffer
returns an alignment error, which would surface as
`ProtocolError::ArchiveValidationFailed("alignment violation ...")` from
TASK-0354.

This task introduces the helper used by TASK-0357: when `FLAG_ARCHIVED` is
set in the frame header, the payload read buffer is allocated via
`rkyv::util::AlignedVec` (16-byte alignment) instead of `Vec<u8>`. The bare
`Vec<u8>` path remains in use for the bincode v2 path (no alignment
requirement) and for the `cfg(not(feature = "zero-copy"))` build.

## Key Types / Signatures

```rust
#[cfg(feature = "zero-copy")]
use rkyv::util::AlignedVec;

#[cfg(feature = "zero-copy")]
/// Reads exactly `len` bytes from `reader` into a 16-byte-aligned buffer
/// (SPEC-18 R25). Used by `recv_frame` when `FLAG_ARCHIVED` is set so that
/// `rkyv::access` can safely produce an `&Archived*` ref without an
/// alignment fault.
///
/// Why `AlignedVec` and not `vec![0u8; len]`: `Vec<u8>` makes no alignment
/// promise. On most 64-bit platforms it is 8-byte aligned; rkyv requires
/// 16. An unaligned buffer surfaces as
/// `ProtocolError::ArchiveValidationFailed("alignment ...")`.
pub(crate) async fn read_aligned_payload<R: AsyncReadExt + Unpin>(
    reader: &mut R,
    len: usize,
) -> Result<AlignedVec, ProtocolError> {
    let mut buf = AlignedVec::with_capacity(len);
    buf.resize(len, 0); // zero-init; safer than set_len + read_exact
    reader
        .read_exact(buf.as_mut_slice())
        .await
        .map_err(ProtocolError::ConnectionLost)?;
    debug_assert_eq!(buf.as_ptr() as usize % 16, 0,
        "AlignedVec must yield 16-byte-aligned start address");
    Ok(buf)
}
```

## Test Expectations

Under `#[cfg(all(test, feature = "zero-copy"))]`:

- `aligned_payload_alignment_for_len_1` — read 1 byte from a `tokio_test::io`
  reader, assert `(ptr as usize) % 16 == 0`.
- `aligned_payload_alignment_for_len_16` — read 16 bytes, assert alignment.
- `aligned_payload_alignment_for_len_256` — read 256 bytes, assert alignment.
- `aligned_payload_alignment_for_len_4096` — read 4096 bytes, assert
  alignment.

Each test constructs a deterministic payload (e.g., `(0..len).map(|i| i as
u8).collect::<Vec<_>>()`), feeds it through an in-memory channel, calls
`read_aligned_payload`, asserts (1) the bytes match the input and (2) the
start address satisfies the 16-byte alignment.

The default-features build runs zero of these tests (the entire `mod`
or each `#[test]` is gated under `cfg(feature = "zero-copy")`).

## Dependencies Context

- `rkyv::util::AlignedVec` lives in `rkyv` 0.8 with the `alloc` feature
  (enabled in TASK-0352).
- `AlignedVec` API:
  - `with_capacity(usize) -> Self` — allocates aligned storage.
  - `resize(new_len: usize, value: u8)` — grows/shrinks, zero-fills new bytes.
  - `as_mut_slice(&mut self) -> &mut [u8]` — for `read_exact` to fill.
  - `as_ptr(&self) -> *const u8` — for the alignment assertion.
  - `len(&self)`, `is_empty(&self)`, `Deref<Target = [u8]>` — standard.
- `tokio::io::AsyncReadExt::read_exact(&mut self, buf: &mut [u8])` is the
  same API used by the existing `recv_frame`.

## Notes

- **Why `resize(len, 0)` instead of `set_len(len)`:** `set_len` is `unsafe`
  and requires the bytes to be initialized before use. `resize(len, 0)`
  zero-initializes and avoids the `unsafe` block. The performance cost is
  one extra pass over the buffer (~10 GB/s on modern memory) — negligible
  compared to the LZ4 decompression and rkyv access that follow. CLAUDE.md
  explicitly forbids `unsafe` without a `// SAFETY:` comment, and we'd
  rather not litigate the safety argument for a non-bottleneck path.
- **`debug_assert_eq!` not `assert!`:** the alignment is an `AlignedVec`
  invariant; if it ever fails, that's a rkyv bug, not a runtime condition
  to handle. Debug-only assertion is appropriate.
- **Where this is called from:** `recv_frame` (TASK-0357) inserts a branch
  after the unknown-flags + size checks: when `header.is_archived()` AND
  `cfg(feature = "zero-copy")`, call `read_aligned_payload` instead of
  `vec![0u8; header.length as usize]` + `read_exact`. The alignment branch
  IS the only behavior change for the rkyv path's read step.
