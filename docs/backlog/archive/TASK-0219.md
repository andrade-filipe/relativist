## Context

SPEC-04 v3 added a precondition to the split algorithm (Section 4.5, SC-003 resolution): the input net MUST NOT contain any stale FreePort (Boundary) sentinels from a previous round. All boundary FreePorts MUST have been resolved by the preceding merge (SPEC-05). In debug mode, the split function SHOULD scan for FreePort values above `max_existing_lafont_freeport_id` and assert that none exist. This precondition is automatically satisfied when merge correctly resolves all borders (SPEC-05, R3-R6) and `reduce_all` does not reintroduce boundary FreePorts (the reduction engine does not generate FreePort values).

This assertion catches a subtle bug class: if merge fails to resolve some borders, the next split would compute `max_freeport_id` incorrectly (it would include stale boundary IDs), potentially causing border ID collisions.

## Notes

- This assertion is a safety net for multi-round grid cycles. In the first round (split on a fresh net), there are no stale boundaries by definition.
- The assertion is best-effort because without metadata about the previous round's border ID range, it cannot definitively distinguish Lafont vs. stale-boundary FreePorts. However, in practice the split function computes `max_freeport_id` which would include stale boundaries, causing the new border IDs to start above them. The real danger is if stale boundaries cause the `max_freeport_id` computation to produce an unexpectedly high starting point.
- For round N > 0, the coordinator could pass the expected Lafont FreePort ceiling as a parameter. This is left as a future enhancement.
- This function is O(P) where P is the port array length, same as `max_freeport_id`.
