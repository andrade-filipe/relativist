## Context

SPEC-11 R6 mandates that key functions are annotated with `#[tracing::instrument]` to create spans with structured fields. The `split()` function in the partition module is one of the five required instrumented functions. Its span must include `input_agents`, `k` (number of partitions), and `strategy` as structured fields.

This task adds the `#[instrument]` attribute to the existing `split()` function. It does NOT change the function's logic -- only adds observability annotations.

## Acceptance Criteria

- [ ] `split()` function in `src/partition/` has `#[tracing::instrument]` attribute
- [ ] Span includes field `input_agents` (number of agents in the input net)
- [ ] Span includes field `k` (number of partitions requested)
- [ ] Span includes field `strategy` (name of the partitioning strategy)
- [ ] The `skip` parameter is used to avoid serializing large arguments (e.g., the entire `Net`)
- [ ] Function compiles and existing behavior is unchanged

## Notes

- The `skip(net, strategy)` is critical: without it, `#[instrument]` would try to `Debug`-format the entire `Net` and strategy object into the span, which is expensive and noisy.
- The `strategy_name` parameter or a `Display` impl on the strategy type provides the human-readable strategy name. The exact mechanism depends on Phase 3's API design. Adjust the `fields()` expression to match.
- If `split()` is an `async fn`, use `#[instrument]` directly (it works with async). If it's synchronous, `#[instrument]` still works.
