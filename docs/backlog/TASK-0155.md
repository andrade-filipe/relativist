## Context

The `/health` (liveness) and `/ready` (readiness) endpoints are standard Kubernetes-style probes that enable orchestration systems to monitor the coordinator's state. The `/health` endpoint always returns 200 OK with body `"ok"` (if the process is alive and responding, it is healthy). The `/ready` endpoint checks the coordinator FSM state: it returns 200 OK when the coordinator has passed initialization (state >= `WaitingForWorkers`), and 503 Service Unavailable when still in `Init`.

## Key Types / Signatures

```rust
use axum::{extract::State, http::StatusCode};

/// Liveness probe: always returns 200 OK (R21).
///
/// If the process can respond to HTTP, it is alive.
#[cfg(feature = "metrics")]
pub async fn health_handler() -> &'static str {
    "ok"
}

/// Readiness probe: returns 200 if coordinator is past Init, 503 otherwise (R22).
///
/// State encoding: 0=Init (not ready), >=1 = ready.
/// States >=1: WaitingForWorkers, Partitioning, Dispatching,
/// WaitingForResults, Merging, CheckTermination, Done.
#[cfg(feature = "metrics")]
pub async fn ready_handler(
    State(state): State<AppState>,
) -> StatusCode {
    let fsm_state = state
        .coordinator_state
        .load(std::sync::atomic::Ordering::Relaxed);
    if fsm_state >= 1 {
        StatusCode::OK
    } else {
        StatusCode::SERVICE_UNAVAILABLE
    }
}
```

## Notes

- The `health_handler` is deliberately trivial. In v1, there is no internal health check beyond "can respond to HTTP". Future versions could check event loop responsiveness.
- The `ready_handler` uses `Ordering::Relaxed` because exact consistency is not required for a readiness probe. A brief delay in reflecting the latest state is acceptable.
- The state encoding (0=Init, 1=WaitingForWorkers, etc.) must be consistent with the coordinator FSM. Consider defining named constants for the state values in a shared location.
