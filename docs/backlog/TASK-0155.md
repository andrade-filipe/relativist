## Context

The `/health` (liveness) and `/ready` (readiness) endpoints are standard Kubernetes-style probes that enable orchestration systems to monitor the coordinator's state. The `/health` endpoint always returns 200 OK with body `"ok"` (if the process is alive and responding, it is healthy). The `/ready` endpoint checks the coordinator's `AtomicBool` readiness flag (R22a): it returns 200 OK when `is_ready == true` (coordinator has passed initialization), and 503 Service Unavailable when `is_ready == false` (`Init` or `Error` state).

Per R22, the ready states are: `WaitingForWorkers`, `Partitioning`, `Dispatching`, `WaitingForResults`, `Merging`, `CheckTermination`, `Done`. The not-ready states are: `Init` and `Error`.

## Key Types / Signatures

```rust
use axum::{extract::State, http::StatusCode};

/// Liveness probe: always returns 200 OK (R21).
///
/// If the process can respond to HTTP, it is alive.
#[cfg(feature = "metrics")]
pub async fn health_handler() -> &'static str {
    "ok"
}

/// Readiness probe: returns 200 if coordinator is ready, 503 otherwise (R22, R22a).
///
/// Uses AtomicBool flag set by the coordinator FSM:
/// - true when in WaitingForWorkers..Done
/// - false when in Init or Error
#[cfg(feature = "metrics")]
pub async fn ready_handler(
    State(state): State<AppState>,
) -> StatusCode {
    if state.is_ready.load(std::sync::atomic::Ordering::Relaxed) {
        StatusCode::OK
    } else {
        StatusCode::SERVICE_UNAVAILABLE
    }
}
```

## Notes

- The `health_handler` is deliberately trivial. In v1, there is no internal health check beyond "can respond to HTTP". Future versions could check event loop responsiveness.
- The `ready_handler` uses `Ordering::Relaxed` because exact consistency is not required for a readiness probe. A brief delay in reflecting the latest state is acceptable.
- The `AtomicBool` approach is robust against enum reordering (unlike the previous `AtomicU8` ordinal comparison) and correctly handles the `Error` state returning 503.
- **Change history:** Originally used `AtomicU8` ordinal comparison (`>= 1` = ready). SPEC-11 Revised v2 (SC-003, SC-012, R22a) replaced with `AtomicBool` and added explicit `Error` state handling.
