## Context

Lands the sole **worker → coordinator per-round** delta-protocol variant:

- `RoundResult { round: u32, border_deltas: Vec<BorderDelta>, stats:
  WorkerRoundStats, has_border_activity: bool,
  minted_agents: Vec<MintedAgent> }` at discriminant **9**.

  The trailing `minted_agents: Vec<MintedAgent>` was added by the
  2026-04-17 SPEC-19 amendment (DC-B5 + R33 + R48): it carries the
  worker's response to the coordinator's `PendingCommutation` requests
  from the previous `RoundStart`. Each entry echoes back the
  `request_id` and the freshly allocated `AgentId`. Typical payload
  size: 0-2 entries per round; empty on rounds with no CON-DUP border
  resolutions.

Carries the per-round reply from each worker after it has run
`reduce_all` on its local partition and diffed its `free_port_index`
against `previous_border_state` (SPEC-19 R25). The `stats` field is the
existing `WorkerRoundStats` struct (already `Serialize + Deserialize`
per v1; SPEC-19 §3.1 bundle added `has_border_activity` to this struct).

**The `has_border_activity: bool` field on the variant is redundant
with `stats.has_border_activity`** — but SPEC-19 R26 names it as a
top-level field of the message. Per DC-A2 of
`docs/spec-reviews/SPEC-19-section-3.4-design-choices-2026-04-17.md`:
R26's top-level `has_border_activity` field is duplicated on the wire
per spec literal, BUT the duplication is graph-enforced at construction
— the worker-side `RoundResult` builder MUST satisfy
`top_level == stats.has_border_activity` via debug_assert, and a
regression test verifies the invariant. The canonical source of truth
is `stats.has_border_activity` (per §3.1 amendment,
`merge/types.rs` L149, derivative of `compute_border_activity`); the
top-level field exists for coordinator pattern-match ergonomics only
(it is a wire copy of the canonical value).

## Notes

- **Duplication `has_border_activity` vs `stats.has_border_activity`
  (ambiguity DC-A1):** the spec names both. The bundle index flags this
  for spec-critic. Decision here: follow R26 verbatim. Rationale: the
  cost is one extra byte per RoundResult (varint-encoded bool); the
  benefit is a top-level field the coordinator can inspect without
  destructuring the stats struct. If spec-critic rules this out, this
  task gets a trivial amendment (drop the field + update the test to
  read `stats.has_border_activity` instead). LoC impact: -5.
- **Why a separate variant from v1's `PartitionResult`:** v1's variant
  carries a full `Partition`. The delta protocol's `RoundResult`
  carries only `Vec<BorderDelta>` — an order-of-magnitude smaller
  payload in typical rounds. The two variants coexist because they
  serve different protocols (full-partition v1 vs delta v2); the
  `delta_mode` config flag (TASK in sub-bundle 2.26-D) selects which
  flow the worker uses.
- **R36 verification:** frame-layer responsibility, lands in TASK-0370.
  This task ships only the bincode-layer correctness.
- **Why four tests for one variant:** the field count is 4 (tied for
  the largest variant in the enum), `border_deltas` carries a
  non-trivial type, `stats` is a nested struct with its own fixed-size
  array, and `has_border_activity` is a duplicated field whose
  independence from `stats.has_border_activity` must be guarded by a
  regression test.

**Details:**
- **Canonical vs cache:** `stats.has_border_activity` is the CANONICAL
  source of truth — it is the derivative computed by
  `compute_border_activity` (SPEC-19 §3.1 amendment, `merge/types.rs`
  L149, grid.rs L32 / L127-136 / L1599 convergence-read-path). The
  top-level `msg.has_border_activity` is a **wire copy** for
  coordinator pattern-match ergonomics. R26 mandates both fields
  appear on the wire.
- **Graph-enforced equality:** the worker-side `RoundResult`
  construction site MUST carry
  `debug_assert!(msg.has_border_activity == msg.stats.has_border_activity,
  "RoundResult invariant: top-level has_border_activity MUST equal
  stats.has_border_activity (SPEC-19 R26, §3.1 amendment)")` — this
  fires ONLY in debug builds, vanishes in release. The worker
  builder ships in 2.26-C; this task lands the invariant-enforcing
  TEST STUB as `#[ignore]` so the contract is discoverable now
  without blocking the wire-layer ship.
- **Drift vulnerability closed:** without this amendment, a maintainer
  could compute `has_border_activity` once and write it to
  `stats.has_border_activity` but forget the top-level copy (or vice
  versa) — causing silent convergence-detector false positives /
  negatives (the exact symmetry-breaking bug BSP cannot tolerate per
  SPEC-05 R5 + T4 strong confluence). This is structurally identical
  to §3.2 DC-4's `BorderState.is_redex` ruling (graph recomputes vs.
  caller-trusted): make the primitive coupling-safe, not trust-based.
- **Concrete code changes expected:**
  - Variant inline doc comment updated to state canonical-vs-cache
    status (already applied above).
  - New test `test_round_result_activity_matches_stats_activity`
    (serde-layer equality check for both true/true and false/false

## DC-B5 Amendment (2026-04-17)

**Verdict source:** `docs/spec-reviews/SPEC-19-section-3.3-2.26B-design-choices-2026-04-17.md` (DC-B5), ratified by ESPECIALISTA-EM-SPECS in SPEC-19 §3.3 R26 + §3.4 R32 + R33 + R48 (2026-04-17 amendment, see SPEC-19 §9 Changelog).

**What changed:** `RoundResult` grew from 4 fields to 5. The new
`minted_agents: Vec<MintedAgent>` field carries the worker's response
to the coordinator's `PendingCommutation` requests from the previous
`RoundStart` — 2-phase AgentId allocation per Option (B) of the DC-B5
verdict.

**Why this matters:** the `MintedAgent` echo closes the cross-round
correlation that lets the coordinator finalize CON-DUP border-redex
resolutions at round N+2 (detection at N, dispatch at N+1, echo at
N+2). The `request_id` field is the correlation key (R48).

**What this task ships vs what 2.26-C ships:**
- TASK-0369 (this task): wire-layer serde/bincode for `minted_agents`,
  order-preservation guarantees.
- 2.26-C: worker-side `MintedAgent` allocation (from the worker's
  reserved AgentId range, NOT overlapping `u32::MAX - 10_000 .. u32::MAX`
  per R48), debug_assert that `request_id` matches an incoming
  `PendingCommutation`, and the `RoundResult` builder site.
- 2.26-B: coordinator-side `PendingCommutation` dispatch at round N+1
  (after detecting a CON-DUP border redex at round N), nack path for
  unknown-`request_id` responses (R48 protocol-violation).

**Test count delta:** +1 test vs pre-amendment plan (`test_round_result_minted_agents_multi_order_preserved`).

**LoC estimate:** M → M (+20 LoC: 1 new field + 1 new test + updated
existing tests for the new field + doc comment). Still within 200-LoC
atomic budget.

**Spec backing:** SPEC-19 §3.3 R26, §3.4 R32/R33/R48 (2026-04-17 amendment). See SPEC-19 §9 Changelog for provenance.
