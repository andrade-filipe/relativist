## Context

During local reduction by a worker, the reduction rules may alter what is connected to a boundary FreePort. An agent connected to `FreePort(bid)` may be consumed, reconnected, or replaced. The `free_port_index` built at split time (TASK-0052) can become stale. Before merge (SPEC-05), the index must reflect the actual port array state. SPEC-04, Section 4.6 recommends lazy reconstruction (approach 2) as the baseline: scan the port array to rebuild the index. This avoids coupling the reduction engine to the partition index, at the cost of O(A_i * PORTS_PER_SLOT) per partition.

## Acceptance Criteria

- [ ] Function `rebuild_free_port_index` scans a partition's subnet port array
- [ ] Finds all ports connected to `FreePort(bid)` where bid != u32::MAX (DISCONNECTED)
- [ ] Returns a fresh `HashMap<u32, PortRef>` mapping `bid -> AgentPort(a, p)`
- [ ] Correctly handles all three scenarios from Section 4.6: reconnection, erasure, propagation via CON-DUP
- [ ] The function replaces (not merges with) the existing `free_port_index`

## Notes

- This function is called at a well-defined point: after local reduction completes and before merge begins. It is NOT called during reduction.
- The spec explicitly recommends this approach as baseline for simplicity (Section 4.6). Migration to active notification (approach 1) is a future optimization guided by benchmarks.
- DISCONNECTED entries (FreePort(u32::MAX)) must be excluded from the index. They are transient states, not real boundary markers.
