## Context

Security operations (token parsing, TLS configuration loading, authentication validation) need a dedicated error type. `SecurityError` centralizes all security-related errors in the `security` module, following the same pattern as `ProtocolError` (TASK-0081) in the protocol module. Variants cover token decoding failures, TLS configuration errors (feature-gated), authentication failures, and invalid security configuration combinations (e.g., TLS without token).

## Acceptance Criteria

- [ ] `SecurityError` enum is defined in `src/security/mod.rs`
- [ ] Enum derives `Debug` and implements `std::fmt::Display` via `thiserror::Error`
- [ ] Variants include: `InvalidBase64`, `InvalidTokenLength`, `AuthenticationFailed`, `ConfigurationError`, and `TlsError` (feature-gated)
- [ ] `TlsError` variant is behind `#[cfg(feature = "tls")]`
- [ ] Error messages in `Display` do NOT reveal internal state (R37)
- [ ] `SecurityError` implements `std::error::Error`

## Key Types / Signatures

```rust
use thiserror::Error;

/// Errors from security operations (SPEC-10).
#[derive(Debug, Error)]
pub enum SecurityError {
    /// Token string is not valid base64.
    #[error("invalid token encoding")]
    InvalidBase64,

    /// Decoded token is not exactly 32 bytes.
    #[error("invalid token length (expected 32 bytes)")]
    InvalidTokenLength,

    /// Worker presented an invalid or missing token.
    #[error("authentication failed")]
    AuthenticationFailed,

    /// Invalid combination of security flags (e.g., TLS without token).
    #[error("security configuration error: {0}")]
    ConfigurationError(String),

    /// TLS-related error (loading certs, handshake failure, etc.).
    #[cfg(feature = "tls")]
    #[error("TLS error: {0}")]
    TlsError(String),

    /// I/O error (e.g., writing token file).
    #[error("I/O error: {0}")]
    Io(#[from] std::io::Error),
}
```

## Notes

- The `AuthenticationFailed` message is intentionally generic per R37: it must NOT indicate whether the token was wrong, missing, or malformed. The coordinator logs the specific reason internally at WARN level, but the error sent to the client is always the same string.
- The `TlsError` variant wraps a `String` rather than the underlying `rustls::Error` to avoid leaking the `rustls` type into the public API when the feature is disabled.
