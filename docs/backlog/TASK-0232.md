# TASK-0232: Fix self-loop annihilation in interact_anni

**Spec:** SPEC-03
**Requirements:** R25 (extended scope — intra-agent self-loops)
**Priority:** P0 (blocker for Phase 11)
**Status:** TODO
**Depends on:** TASK-0024 (interact_anni)
**Blocked by:** none
**Estimated complexity:** S (< 50 LoC)

## Context

SPEC-03 R25 defines handling for self-referencing auxiliary ports during annihilation: when aux ports of agents `a` and `b` point back to the pair being consumed. The current implementation uses a `link` helper guard that makes links to removed agents no-ops.

However, R25's guard only covers **inter-agent** cross-references (e.g., `a.p1 <-> b.p2`). It does NOT handle **intra-agent** self-loops (e.g., `a.p1 <-> a.p2`), which are a valid IC construct. Specifically, Church(0)'s inner lambda (`lambda x. x`) has `lam_x.p1 <-> lam_x.p2` (SPEC-14 R5).

When an agent with a self-loop is annihilated with another agent:
1. The self-loop agent's aux ports both resolve to itself (already being removed)
2. The `link` guard makes BOTH cross-pattern links no-ops
3. The partner agent's external ports are left DISCONNECTED
4. This violates invariant T1 (port linearity)

**Root cause:** The `link` guard treats self-loop ports as "dead agent references" and silently drops the links, but the correct behavior is to connect the partner's external ports to each other (since the self-loop agent acts as identity).

## Fix

Detect self-loops **before** removing agents in `interact_anni`. When one agent has a self-loop (`p1 <-> p2`), connect the OTHER agent's external auxiliary ports directly to each other. When both have self-loops, return immediately (nothing external to connect).
