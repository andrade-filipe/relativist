# TASK-0128: Implement token validation in coordinator accept flow

**Spec:** SPEC-10
**Requirements:** R13, R14, R15, R16, R17
**Priority:** P0 (critical path)
**Status:** TODO
**Depends on:** TASK-0124, TASK-0125, TASK-0127, TASK-0088
**Blocked by:** none
**Estimated complexity:** M (50-200 LoC)

## Context

When a worker connects to the coordinator, the first message must be `Register`. If the coordinator has token authentication enabled (Tier 2 or 3), it must validate the token in the `Register` message using constant-time comparison. On success, it sends `RegisterAck` and adds the worker to the pool. On failure (wrong token, missing token, or non-Register message), it sends `RegisterNack`, closes the connection, and logs the rejection at WARN level with the remote address but NOT the rejected token value.

This function sits between `accept_workers` (TASK-0088) and the grid loop, handling the authentication handshake for each connection.

## Acceptance Criteria

- [ ] `authenticate_worker` function is implemented in `src/security/mod.rs` (or a dedicated `src/security/auth.rs`)
- [ ] Function is async, takes a stream reference, the expected token (`Option<&AuthToken>`), and returns `Result<(), SecurityError>`
- [ ] Reads the first message from the stream, expects `Message::Register`
- [ ] If expected token is `None` (Tier 1): accepts unconditionally, ignores any token in the Register message
- [ ] If expected token is `Some`: validates `Register.auth_token` using `AuthToken::verify` (R15)
- [ ] On missing token (Register.auth_token is None when token is required): sends `RegisterNack`, returns error (R16)
- [ ] On wrong token: sends `RegisterNack { reason: "authentication failed" }`, returns error (R16)
- [ ] On success: sends `RegisterAck { worker_id }` (R17)
- [ ] Logs rejection at WARN with remote address but NOT the token value (R16)
- [ ] Logs acceptance at INFO with worker_id and remote address
- [ ] Non-Register first message: sends `RegisterNack { reason: "protocol error" }`, returns error (R37)

## Notes

- This function uses `recv_frame` and `send_frame` from the protocol module to read/write messages. It does NOT directly read/write bytes.
- The `remote_addr` parameter is a string for logging purposes. The actual `SocketAddr` is available from `TcpStream::peer_addr()` at the call site.
- In Tier 1, the function still expects a `Register` message (to verify the connecting client speaks the Relativist protocol), but it does not check the auth_token field.
- The function must be called per-worker in the accept loop, after TCP connection (and TLS handshake if applicable) but before adding to the worker pool.
