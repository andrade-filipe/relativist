# TASK-0127: Extend Message enum with Register, RegisterAck, RegisterNack

**Spec:** SPEC-10
**Requirements:** R14, R16, R17, R19
**Priority:** P0 (critical path)
**Status:** TODO

**Blocked by:** none
**Estimated complexity:** S (< 50 LoC)

## Context

The wire protocol `Message` enum (TASK-0082, SPEC-06) currently has 4 variants for the grid loop. SPEC-10 Section 4.3 requires 3 additional variants for the registration handshake, defined as separate payload structs: `Register(RegisterPayload)` (worker -> coordinator, carrying protocol version and optional auth token), `RegisterAck(RegisterAckPayload)` (coordinator -> worker, confirming registration with assigned WorkerId), and `RegisterNack(RegisterNackPayload)` (coordinator -> worker, rejecting registration with a reason). These are the first messages exchanged on each connection, before the grid loop begins. The complete `Message` enum has seven variants in total.

## Acceptance Criteria

- [ ] `RegisterPayload` struct is defined with fields: `protocol_version: u8` (current version: 1) and `auth_token: Option<[u8; 32]>` (R19, R36)
- [ ] `RegisterAckPayload` struct is defined with field: `worker_id: WorkerId`
- [ ] `RegisterNackPayload` struct is defined with field: `reason: String`
- [ ] All three payload structs derive `Debug`, `Clone`, `Serialize`, `Deserialize`
- [ ] `Message::Register(RegisterPayload)` variant is added to the Message enum
- [ ] `Message::RegisterAck(RegisterAckPayload)` variant is added to the Message enum
- [ ] `Message::RegisterNack(RegisterNackPayload)` variant is added to the Message enum
- [ ] `auth_token` is `Option<[u8; 32]>`: `None` in Tier 1, `Some(bytes)` in Tier 2/3 (R14, R19)
- [ ] `protocol_version` enables fast rejection of non-Relativist connections (R36)
- [ ] `RegisterNackPayload::reason` is always `"authentication failed"` for auth failures (R16, R35)
- [ ] Doc comments specify direction (Worker->Coordinator or Coordinator->Worker)

## Key Types / Signatures

```rust
/// Registration messages extending SPEC-06's Message enum.
/// These variants MUST be added to the Message enum defined in SPEC-06 Section 4.1.
///
/// Direction: Worker -> Coordinator
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RegisterPayload {
    /// Protocol version for fast rejection of incompatible clients.
    /// Current version: 1.
    pub protocol_version: u8,
    /// Authentication token. None when running in Tier 1 (no auth).
    /// Some(raw_bytes) when running in Tier 2 or 3.
    pub auth_token: Option<[u8; 32]>,
}

/// Direction: Coordinator -> Worker (success)
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RegisterAckPayload {
    /// The WorkerId assigned to this worker by the coordinator.
    pub worker_id: WorkerId,
}

/// Direction: Coordinator -> Worker (failure)
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RegisterNackPayload {
    /// Human-readable reason for rejection.
    /// MUST be generic (e.g., "authentication failed") and MUST NOT
    /// reveal internal state (R35).
    pub reason: String,
}

/// Messages in the communication protocol (SPEC-06 + SPEC-10).
#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum Message {
    // === Registration handshake (SPEC-10, Section 4.3) ===

    /// Worker registration request. First message on every new connection.
    /// Worker -> Coordinator.
    Register(RegisterPayload),

    /// Registration accepted. Coordinator -> Worker.
    RegisterAck(RegisterAckPayload),

    /// Registration rejected. Coordinator -> Worker.
    /// Connection MUST be closed after sending this.
    RegisterNack(RegisterNackPayload),

    // === Grid loop (SPEC-06) ===
    // ... existing 4 variants (AssignPartition, Shutdown, PartitionResult, Error) ...
}
```

## Notes

- The `auth_token` field uses `Option<[u8; 32]>` instead of `Option<AuthToken>` because the wire protocol transmits raw bytes (R14). The coordinator converts between `AuthToken` and `[u8; 32]` at the validation boundary.
- The `RegisterNackPayload::reason` field is a `String` for extensibility, but R16/R35 require that authentication errors always use the generic `"authentication failed"` message. Protocol errors may use `"protocol error"`.
- The `protocol_version: u8` field in `RegisterPayload` enables fast rejection of non-Relativist connections (R36). Current version is 1. When the protocol version is unrecognized, the coordinator MUST close the connection immediately.
- The three payload structs are separate from the Message enum for clarity. They extend (not replace) the four SPEC-06 variants, making a total of seven `Message` variants.
- This task modifies the `Message` enum defined in Phase 5 (TASK-0082). It is a non-breaking addition (new variants, no changes to existing ones).
