## Context

`send_frame` is the core function for writing a message to the TCP socket. It serializes the `Message` with bincode, computes a CRC32C checksum of the payload, writes the 8-byte header (length + checksum, little-endian), writes the payload, and flushes. It returns the total bytes written (header + payload), which is needed for network metrics (R34). The function is generic over `AsyncWriteExt` to support both real TCP streams and in-memory buffers for testing.

## Acceptance Criteria

- [ ] `send_frame` is implemented in `src/protocol/frame.rs`
- [ ] Function is async and generic over `W: AsyncWriteExt + Unpin`
- [ ] Message is serialized with bincode using explicit configuration: `bincode::config::standard().with_little_endian().with_fixed_int_encoding()` (R11, SPEC-06 v3)
- [ ] CRC32C checksum is computed using `crc32fast::hash(&payload)`
- [ ] Header is written as 8 bytes: length (LE u32) + checksum (LE u32)
- [ ] Payload bytes are written after the header
- [ ] Writer is flushed after writing
- [ ] Returns `Ok(total_bytes)` where `total_bytes = FRAME_HEADER_SIZE + payload.len()`
- [ ] Returns `Err(ProtocolError::Serialize(_))` on bincode failure
- [ ] Returns `Err(ProtocolError::ConnectionLost(_))` on write/flush failure (renamed from `Io` in SPEC-06 v3)

/// Serializes a message and sends it as a frame over the TCP socket.

/// Returns the total number of bytes written (header + payload).
///
/// Steps:
/// 1. Serialize Message with bincode -> payload: Vec<u8>.

/// 3. Write header (length + checksum) as 8 bytes little-endian.
/// 4. Write payload.
/// 5. Flush the buffer.

    writer: &mut W,

## Notes

- The function uses `crc32fast::hash(&payload)` which computes CRC32C (Castagnoli) with hardware acceleration on x86 (SSE4.2). No need to manage a `Hasher` instance manually for single-shot computation.
- SPEC-06 v3 R11 now mandates explicit bincode config: `bincode::config::standard().with_little_endian().with_fixed_int_encoding()`. Bincode v2.x defaults differ from v1 defaults; explicit config is required.
- Flushing is important: without flush, the data may remain in the write buffer and not be sent over TCP, causing the receiver to hang.
- The framing cost is O(n) where n is payload size (R39): one serialization pass + one CRC32 pass + one write.
