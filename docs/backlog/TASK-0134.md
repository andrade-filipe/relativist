# TASK-0134: Implement connection limits

**Spec:** SPEC-10

## Context

The coordinator should enforce a maximum number of concurrent TCP connections (default: 1024) as a defensive measure against resource exhaustion (R32). Connections beyond this limit are rejected immediately. This is a lightweight guard that prevents the coordinator from being overwhelmed by excessive connections, whether from misconfigured workers, port scanners, or accidental connection storms.

## Acceptance Criteria

- [ ] `ConnectionLimiter` struct is defined in `src/security/mod.rs`
- [ ] `ConnectionLimiter::new(max: usize)` creates a limiter with the given maximum
- [ ] `ConnectionLimiter::try_acquire(&self) -> bool` returns `true` if under limit, `false` otherwise
- [ ] `ConnectionLimiter::release(&self)` decrements the active count
- [ ] Uses `AtomicUsize` for thread-safe counting (no mutex needed)
- [ ] When `try_acquire` returns `false`, the caller rejects the connection with TCP reset
- [ ] Rejections are logged at WARN level with the current count and max

```rust
use std::sync::atomic::{AtomicUsize, Ordering};

/// Enforces a maximum number of concurrent connections (R32).
///
/// Uses atomic operations for lock-free counting.
/// Default maximum: 1024 (DEFAULT_MAX_CONNECTIONS).
#[derive(Debug)]
pub struct ConnectionLimiter {
    current: AtomicUsize,
    max: usize,
}

impl ConnectionLimiter {
    /// Create a new limiter with the given maximum.
    pub fn new(max: usize) -> Self {
        Self {
            current: AtomicUsize::new(0),
            max,
        }
    }

    /// Attempt to acquire a connection slot.
    ///
    /// Returns `true` if the connection is allowed (count < max),
    /// `false` if the limit has been reached.
    pub fn try_acquire(&self) -> bool {
        /* atomic compare-and-swap loop */
    }

    /// Release a connection slot when a connection closes.
    pub fn release(&self) {
        self.current.fetch_sub(1, Ordering::Relaxed);
    }

    /// Current number of active connections.
    pub fn active(&self) -> usize {
        self.current.load(Ordering::Relaxed)
    }
}
```

## Notes

- The `try_acquire` method must use a compare-and-swap (CAS) loop, not a simple load-then-increment, to avoid race conditions where two threads both read the count as below-max and both increment.
- `Ordering::Relaxed` is sufficient here because we only need atomicity of the counter, not happens-before guarantees with other memory operations.
- The connection limiter protects the accept loop. It does NOT limit the number of registered workers (that is controlled by `num_workers` in `NodeConfig`).
- This is a SHOULD requirement (R32), so it is P2 priority. The system works without it, but it provides an important safety net.
