## Context

Token comparison must use constant-time equality to prevent timing side-channel attacks (R15). An attacker who can measure response times for registration attempts could deduce the token byte-by-byte if standard comparison short-circuits on the first differing byte. The `subtle` crate provides `ConstantTimeEq` for this purpose. This task adds the `verify` method to `AuthToken` using constant-time comparison.

## Acceptance Criteria

- [ ] `AuthToken::verify(&self, other: &AuthToken) -> bool` is implemented
- [ ] Comparison uses `subtle::ConstantTimeEq` (not standard `==` operator)
- [ ] The `PartialEq` derive on `AuthToken` (from TASK-0123) is NOT used for security-sensitive comparison; `verify` is the only method used in the authentication path
- [ ] Method returns `true` if tokens are identical, `false` otherwise
- [ ] A doc comment explicitly states this is for authentication and must not be replaced with `==`

## Key Types / Signatures

```rust
use subtle::ConstantTimeEq;

impl AuthToken {
    /// Constant-time comparison to prevent timing side-channel attacks (R15).
    ///
    /// SECURITY: This method MUST be used for all authentication decisions.
    /// Do NOT use the `PartialEq` (`==`) operator for token validation,
    /// as it may short-circuit and leak timing information.
    pub fn verify(&self, other: &AuthToken) -> bool {
        self.0.ct_eq(&other.0).into()
    }
}
```

## Test Expectations

- Unit test: `token.verify(&token.clone())` returns `true`
- Unit test: `token.verify(&AuthToken::generate())` returns `false` (with overwhelming probability)
- Unit test: `token.verify` with a token differing in only the last byte returns `false`
- Unit test: `token.verify` with a token differing in only the first byte returns `false`

## Notes

- The `subtle` crate is the standard Rust crate for constant-time operations, used by `rustls`, `ring`, and other security-critical libraries. It provides `ConstantTimeEq` which guarantees the comparison does not short-circuit.
- We keep `PartialEq` derived on `AuthToken` for non-security contexts (test assertions, debug comparisons), but the `verify` method is the one used in the actual authentication code path. The doc comment makes this distinction explicit.
- The `.into()` call converts `subtle::Choice` (a constant-time boolean) into a standard `bool`.
