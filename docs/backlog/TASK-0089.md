## Context

In each round of the grid loop, the coordinator sends partitions to all workers concurrently. R21 requires that all `AssignPartition` writes are initiated before awaiting any `PartitionResult`. This is implemented using `futures::future::join_all` or `tokio::join!` to send all partitions in parallel. The function also tracks `bytes_sent` (including header bytes, R34) and `network_send_time` for metrics (R33). A configurable timeout (default 60 seconds) aborts if distribution takes too long (R30).

## Notes

- The concurrent send is a deliberate improvement over the Haskell prototype, which sends sequentially (AC-003, lines 388-391). This reduces the latency gap between the first and last worker receiving their partition.
- `join_all` requires collecting futures into a `Vec`. Each future borrows a different element of `worker_streams` (disjoint mutable borrows). This may require splitting the slice manually or using indexed access with unsafe (or better, using `iter_mut().zip()`).
- If any send fails, the entire distribution is considered failed. There is no partial retry (fault tolerance is out of scope, R25).
