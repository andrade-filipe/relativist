# TASK-0136: Implement message size pre-validation in recv_frame

**Spec:** SPEC-10
**Requirements:** R29, R30, R31
**Priority:** P0 (critical path)
**Status:** TODO
**Depends on:** TASK-0086, TASK-0125
**Blocked by:** none
**Estimated complexity:** S (< 50 LoC)

## Context

The `recv_frame` function (TASK-0086) already checks payload size against `max_payload_size`. This task ensures the check is integrated with the security module's `SecurityConfig.max_message_size` and that the `--max-message-size` CLI flag flows through correctly. It also adds the `ProtocolError::MessageTooLarge` error variant (if not already present as `PayloadTooLarge`) and ensures the rejection happens BEFORE any memory allocation for the payload (R29). This is primarily an integration/wiring task connecting TASK-0086's implementation with SPEC-10's requirements.

## Acceptance Criteria

- [ ] `recv_frame` uses the `max_message_size` from `SecurityConfig` (or the value passed through from it) for its size check (R29)
- [ ] The length field in the frame header is checked against `max_message_size` BEFORE `vec![0u8; length]` allocation (R29)
- [ ] Default `max_message_size` is `268_435_456` bytes (256 MiB, R30)
- [ ] When a message exceeds the limit, `ProtocolError::MessageTooLarge` (or `PayloadTooLarge`) is returned (R31)
- [ ] The oversized payload is NOT read from the socket (R31)
- [ ] The connection is closed after rejection (R31)
- [ ] `--max-message-size` CLI flag is documented as accepting bytes or human-readable format (R30)

## Files to Create/Modify

- `src/protocol/frame.rs` -- **modify**: verify/enhance size check in recv_frame
- `src/protocol/error.rs` -- **modify**: ensure `MessageTooLarge` or `PayloadTooLarge` variant exists
