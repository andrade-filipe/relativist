# TASK-0159: Optional trace context in wire protocol messages

**Spec:** SPEC-11
**Requirements:** R29, R30
**Priority:** P2 (nice to have)
**Status:** TODO
**Depends on:** TASK-0158, Phase 5
**Blocked by:** none
**Estimated complexity:** M (50-200 LoC)

## Context

When the `otel` feature is enabled, the wire protocol (SPEC-06) may include an optional trace context header (trace_id + span_id, 32 bytes total) in `AssignPartition` and `PartitionResult` messages (R30). This enables distributed tracing across the coordinator-worker boundary: the coordinator creates a root span for each grid round, and workers create child spans under the coordinator's trace context (R29).

When `otel` is NOT enabled, the trace context header is completely omitted (zero overhead, R30). This is achieved via conditional compilation: the wire protocol message format includes the trace context field only when the `otel` feature is active.

## Key Types / Signatures

```rust
/// Trace context for distributed tracing across coordinator-worker boundary (R30).
///
/// Contains the W3C Trace Context identifiers needed to link worker spans
/// to the coordinator's grid round span.
///
/// Only present when the `otel` feature is enabled.
#[cfg(feature = "otel")]
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct TraceContext {
    /// 128-bit trace identifier (16 bytes).
    pub trace_id: [u8; 16],
    /// 64-bit span identifier (8 bytes).
    pub span_id: [u8; 8],
}

/// Extract the current span's trace context for inclusion in a wire protocol message.
///
/// Returns `None` if there is no active span or if OTel is not properly initialized.
#[cfg(feature = "otel")]
pub fn extract_trace_context() -> Option<TraceContext> {
    use opentelemetry::trace::TraceContextExt;
    use tracing_opentelemetry::OpenTelemetrySpanExt;

    let span = tracing::Span::current();
    let context = span.context();
    let span_ref = context.span();
    let sc = span_ref.span_context();

    if sc.is_valid() {
        Some(TraceContext {
            trace_id: sc.trace_id().to_bytes(),
            span_id: sc.span_id().to_bytes(),
        })
    } else {
        None
    }
}

/// Create a remote parent context from a received TraceContext.
///
/// Used by workers to link their reduction span to the coordinator's grid round span.
#[cfg(feature = "otel")]
pub fn inject_trace_context(ctx: &TraceContext) -> opentelemetry::Context {
    use opentelemetry::trace::{SpanContext, SpanId, TraceId, TraceFlags};

    let span_context = SpanContext::new(
        TraceId::from_bytes(ctx.trace_id),
        SpanId::from_bytes(ctx.span_id),
        TraceFlags::SAMPLED,
        true, // remote
        Default::default(),
    );

    opentelemetry::Context::new().with_remote_span_context(span_context)
}
```

## Notes

- The trace context is 24 bytes total (16 + 8). This is negligible overhead compared to partition data sizes.
- The `#[cfg(feature = "otel")]` on the message struct fields means the wire protocol format is different depending on compile-time features. This is acceptable because coordinator and workers are always built from the same source with the same features enabled.
- The `inject_trace_context` approach creates a remote parent context. Workers should create their spans as children of this context using `tracing_opentelemetry::OpenTelemetrySpanExt::set_parent()`.
- This is P2 because distributed tracing is a "nice to have" for v1. The core observability (structured logging + metrics) is sufficient for the benchmark suite (SPEC-09). OTel tracing adds visibility for debugging but is not required for the TCC evaluation.
