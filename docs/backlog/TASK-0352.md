## Context

Adds `rkyv` as an **optional** dependency on `relativist-core` and creates the
`zero-copy` cargo feature that gates the entire rkyv code path (R20). All
subsequent tasks (TASK-0353..0359) gate their rkyv-specific code on
`#[cfg(feature = "zero-copy")]` (or `#[cfg_attr(feature = "zero-copy", ...)]`
for derives). When the feature is OFF, the codebase compiles, tests pass, and
no rkyv code is reached. The feature MUST NOT be in `default`.

## Notes

- Rkyv 0.8 features:
  - `validation` — required for `rkyv::access` (the safe API mandated by R24).
  - `alloc` — enables `AlignedVec` (R25) and the `rancor::Error` type.
  - **Do not** enable `bytecheck` directly; `validation` pulls it in.
- Rkyv 0.8.x changed APIs from 0.7 (e.g., `Archive` trait, `to_bytes` returns
  `AlignedVec` not `Vec<u8>`). All subsequent tasks (TASK-0353..0359) assume
  the 0.8.x API surface. If the developer chooses 0.7.x (not recommended),
  the API signatures in TASK-0353/0356/0357 must be updated.
- Why not in `full`: the `full` feature is the "everything I might need in
  production" bundle. `zero-copy` is an experimental optimization gated
  behind a separate flag because it adds non-trivial dependency surface
  (~10 transitive crates including `bytecheck`, `ptr_meta`, `rend`).
