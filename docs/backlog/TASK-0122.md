## Context

Relativist uses a three-tier security model (Development, PrivateNetwork, Production) selected exclusively by the combination of CLI flags provided at startup. Tier detection MUST be purely flag-based with no inference from bind address or other context (R1). The coordinator must detect the active tier at startup and log it at INFO level. Tier detection is a pure function of two boolean inputs: whether a token is configured and whether TLS is configured. This task defines the `SecurityTier` enum and the detection function, which is used by `SecurityConfig` construction and by the coordinator startup sequence.

## Notes

- The tier is detected, not directly selected by the user. There is no `--tier` CLI flag. The coordinator infers the tier purely from which flags are present. No other context (bind address, environment, etc.) MUST influence tier detection (R1, R3).
- When the `tls` feature is not compiled in, `has_tls` will always be `false` at the call site, so the `Production` variant is unreachable but still defined (for completeness and forward compatibility).
- R4 says TLS without token SHOULD be rejected. We implement it as an error because there is no valid use case and it protects against misconfiguration.
