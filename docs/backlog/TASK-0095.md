## Context

The `send_frame` and `recv_frame` functions are generic over `AsyncWriteExt` and `AsyncReadExt` respectively. For unit testing the protocol without starting real TCP connections, we need an in-memory transport: a pair of connected async byte streams where bytes written to one end can be read from the other. Tokio provides `tokio::io::duplex()` which creates a bidirectional in-memory channel. This task provides helper functions that create in-memory transports and demonstrates their use with `send_frame`/`recv_frame`.

## Notes

- `tokio::io::duplex(max_buf_size)` creates a pair of `DuplexStream`s that implement both `AsyncRead` and `AsyncWrite`. The `max_buf_size` controls the internal buffer; if the writer fills the buffer, it blocks until the reader drains it.
- This approach is standard in the Tokio ecosystem for testing protocols without network overhead.
- The corruption test requires writing raw bytes to the stream (bypassing `send_frame`) to inject a corrupted frame. Use `AsyncWriteExt::write_all` to write a manually-constructed frame with a wrong checksum or flipped payload byte.
- Consider also testing edge cases: zero-length payload (Shutdown message), very large payload, multiple messages in sequence.
