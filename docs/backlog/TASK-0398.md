## Context

The REVIEW of 2026-04-23 closed MF-001 (worker-side R23/R26) and MF-002 (G1 parity for symmetric IC rules), but the TASK-0395 sub-agent discovered during DEV that `RoundResultPayload` — the pure-core struct returned by `WorkerDispatch::dispatch_round_start` — does NOT carry the `minted_agents: Vec<MintedAgent>` field that `Message::RoundResult` (wire) carries. The coordinator's BSP loop therefore silently drops the worker-side DC-B5 echo. The `PendingPortRef::Pending { commutation_id, agent_slot, port_slot }` tokens emitted by the resolver's CON-DUP / CON-ERA / DUP-ERA branches are never resolved to concrete `PortRef::AgentPort(minted_id, port)`, and `BorderResolution.pending_new_borders` is also dropped by `package_resolutions` (file `border_resolver.rs:189-190` explicit comment: "`pending_new_borders` is INTENTIONALLY dropped — those entries stay at coordinator state until round N+2 finalization"). The "coordinator state" slot was never implemented.

This task ships the **pure-core plumbing** (types + BorderGraph methods) without yet touching `run_grid_delta_inner`. TASK-0399 then wires the loop and flips `SKIP_ASYMMETRIC = false` for full D-003 closure.

### 3. Add `BorderGraph::encode_request_id` / `decode_request_id` helpers

- [ ] Add two free functions in `border_resolver.rs` (or a new sibling module `merge/commutation_id_codec.rs` — prefer inline in border_resolver.rs next to `CommutationId` type alias):
  ```rust
  /// Encode a (commutation_id: u64, agent_slot: u8) pair into a u32
  /// wire-layer request_id per DC-B5 §3.4 TASK-0370 convention.
  ///
  /// Format: `request_id = ((commutation_id & 0x0FFF_FFFF) << 4) | (agent_slot & 0xF)`.
  /// This reserves the upper 4 bits of commutation_id (losing the
  /// top 36 bits of a u64) and the low 4 bits for agent_slot
  /// (max 16 agents per CommutationBatch — DC-B5 invariant).
  ///
  /// debug_assert!s that `commutation_id < 2^28` and `agent_slot < 16`.
  /// A single BSP run creating >= 2^28 commutations is out of scope
  /// (same assumption as `BorderIdAllocator`'s u32 capacity).
  pub(crate) fn encode_request_id(commutation_id: CommutationId, agent_slot: u8) -> u32;

  /// Inverse of `encode_request_id`. Returns `(commutation_id_low28, agent_slot)`.
  /// The returned `commutation_id_low28` is the low-28-bit projection
  /// of the original u64; matching against `CommutationBatch.commutation_id`
  /// thus compares the low 28 bits only (sound under the <2^28 assumption).
  pub(crate) fn decode_request_id(request_id: u32) -> (u32, u8);
  ```
- [ ] These are the SHARED encode/decode contract between (a) any caller that fans `CommutationBatch → wire-layer PendingCommutation` (e.g., `LocalDeltaDispatch` in `grid_delta_integration_tests.rs`, currently uses ad-hoc `(id & 0x0FFFFFFF) * 16 + slot`; TASK-0399 migrates it to these shared helpers) and (b) `BorderGraph::register_minted_agents` decoding the echo.

### 5. Add `BorderGraph::register_minted_agents`

- [ ] New method on `BorderGraph`:
  ```rust
  /// DC-B5 round-N+2 finalizer (SPEC-19 §3.3 R26 / R48). Consumes the
  /// `minted_agents` echo from a `RoundResultPayload`, resolves all
  /// matching `PendingPortRef::Pending` tokens in
  /// `self.pending_new_borders` to concrete `PortRef::AgentPort(minted_id, port_slot)`,
  /// and promotes every fully-resolved `PendingNewBorder` to an
  /// `AddBorderEntry` via `self.add_border_states`.
  ///
  /// Partial resolution is allowed: a PendingNewBorder with one side
  /// still Pending stays in self.pending_new_borders until a later
  /// register_minted_agents call resolves the remaining side.
  ///
  /// R48 invariant: the method REJECTS with `GridError::ProtocolViolation`
  /// if any MintedAgent.request_id decodes to a (commutation_id, agent_slot)
  /// pair that matches NO outstanding PendingPortRef::Pending in
  /// self.pending_new_borders (stray request_id). Duplicate request_ids
  /// within the same mints slice are lenient (QA-0394-A): both get
  /// registered; the second overwrites self.resolved_mints entry (OK
  /// under the DC-B5 "workers may mint duplicate agents for duplicate
  /// requests; coordinator treats the first as canonical" weakening).
  ///
  /// Order of resolved_entries passed to add_border_states preserves
  /// self.pending_new_borders input order (stability for deterministic
  /// BSP replay).
  pub(crate) fn register_minted_agents(
      &mut self,
      worker_id: WorkerId,
      mints: &[MintedAgent],
  ) -> Result<(), GridError>;
  ```
- [ ] Body outline:
  1. For each MintedAgent in mints: decode request_id, store in self.resolved_mints[(commutation_id_low28, agent_slot)] = minted_agent_id. Track which request_ids matched at least one Pending token (for R48).
  2. Walk self.pending_new_borders, partition into "fully resolved" and "still pending" based on whether BOTH side_a and side_b can be promoted to Concrete via self.resolved_mints.
  3. For each fully-resolved entry: construct AddBorderEntry { border_id, side_a: concrete, side_b: concrete, worker_a, worker_b }; accumulate into a Vec<AddBorderEntry>.
  4. Check if add_border_states should REPLACE an existing border_id (DC-B6 CON-ERA / DUP-ERA preserving case): if self.borders.contains_key(border_id), call self.remove_border(border_id) first, then self.add_border_states. Otherwise just add_border_states.
  5. If any MintedAgent.request_id didn't match any Pending token in this round, return `GridError::ProtocolViolation(format!("R48: stray MintedAgent.request_id {request_id} from worker {worker_id}"))`.
  6. Replace self.pending_new_borders with the "still pending" partition.
- [ ] `WorkerId` parameter is used for the R48 error message (easier to diagnose the guilty worker); no logic branches on it.
- [ ] New `GridError` variant if not already present: `ProtocolViolation(String)`. Check `src/error.rs`; add if missing.

## Notes

- **Encoding assumption:** `commutation_id < 2^28` — documented in docstrings with `debug_assert`. Single BSP run approaching this limit is out of scope (matches `BorderIdAllocator` u32 ceiling assumption).
- **Why helpers are `pub(crate)` and not `pub`:** `CommutationId` type alias is `pub(crate)`; `PendingNewBorder` is `pub(crate)`. A `pub` helper taking `CommutationId` compiles but fires `private_interfaces` lint (same class of issue as TASK-0396 `run_grid_entry`). Keep everything `pub(crate)` — callers are in-crate.
- **Why not extend `package_resolutions` signature in place:** breaks TASK-0395's `LocalDeltaDispatch` at `grid_delta_integration_tests.rs`. Companion function `package_resolutions_with_pending` is additive; TASK-0399 migrates `LocalDeltaDispatch` to the new function; `package_resolutions` then has a dead-code `#[allow]` note (or gets deleted as part of TASK-0399 cleanup — flag for that task).
- **R48 duplicate-request_id policy:** the LENIENT behavior pinned by QA-0394-A (worker mints both agents despite duplicate request_id, echoes both in minted_agents) means `register_minted_agents` receives a mint slice with duplicates possible. DC-B5 weakening: the coordinator uses the FIRST mint as canonical for each (commutation_id, agent_slot); subsequent duplicates overwrite `self.resolved_mints` entry silently (last-write-wins for the HashMap), but both minted AgentIds exist in the worker's partition arena (the coordinator cache lazy-syncs on FinalStateResult). This is SUBOPTIMAL but ACCEPTABLE under R48's current lenient phrasing. UT-0398-06 pins the behavior.
- **Interaction with `QA-0394-A`:** that probe asserts the worker's LENIENT behavior (mints both). This task's behavior is analogous at the coordinator side — ACCEPT both, use the first. Flag for spec-critic review whether strictening is wanted; default: LENIENT.
- **Interaction with `BorderGraph::add_border_states`:** returns `()` today (no Result). Handles panics on duplicate border_id. DC-B6 case (CON-ERA / DUP-ERA preserving an EXISTING border_id): `register_minted_agents` must call `remove_border` first before `add_border_states`. Confirm this is the correct DC-B6 semantics by re-reading `resolve_non_era_era` at border_resolver.rs — flagged for Stage 4 REVIEW.

## Design Choices Flagged for Spec-Critic (Stage 1.5)

- **DC-0398-A — CommutationId low-28-bit truncation.** Options:
  - (a) Accept the truncation; debug_assert commutation_id < 2^28; document in docstring. (Default plan.)
  - (b) Extend the wire format to carry full u64 commutation_id alongside u32 request_id (breaks 2.26-A wire format — rejected).
  - (c) Coordinator keeps a HashMap<u32, (CommutationId, u8)> reverse index; skip the encode/decode functions. (More state; more error-prone on LocalDeltaDispatch outbound encoding.)
  - **Default:** (a). **Spec-critic: confirm or flip to (c) if the HashMap is considered architecturally cleaner.**
- **DC-0398-B — R48 duplicate-request_id handling.** Options:
  - (a) Lenient on both sides (worker + coordinator accept duplicates; second mint overwrites self.resolved_mints entry). Consistent with QA-0394-A.
  - (b) Strict at coordinator (coordinator panics or returns ProtocolViolation on duplicate request_id in mints slice).
  - **Default:** (a). **Spec-critic: confirm.** If (b) is chosen, QA-0394-A needs an updating companion test for coordinator-side rejection.
- **DC-0398-C — enqueue_pending_borders visibility.** Options:
  - (a) pub(crate) — matches the lint constraint from TASK-0396.
  - (b) pub — requires also promoting PendingNewBorder to pub (current is pub(crate)).
  - **Default:** (a). **Spec-critic: confirm.**
