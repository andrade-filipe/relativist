## Context

The `SecurityConfig` struct (TASK-0125) must be constructed from CLI flags and environment variables at coordinator/worker startup. This task implements the builder logic that parses `--token`, `--token auto`, `--token-file`, `RELATIVIST_TOKEN`, `--tls-cert`, `--tls-key`, `--tls-ca`, `--max-message-size`, and `--insecure` flags into a fully resolved `SecurityConfig`. The builder detects the security tier (R3), generates or decodes the token, loads TLS configs, and performs all validation (R4, R8).

## Acceptance Criteria

- [ ] `SecurityConfig::from_cli(args)` or `SecurityConfigBuilder` is implemented in `src/security/mod.rs`
- [ ] No flags -> `SecurityConfig::development()` (Tier 1, R2)
- [ ] `--token auto` -> generates new token via `AuthToken::generate()`, Tier 2 (R9)
- [ ] `--token <base64>` -> decodes token via `AuthToken::from_base64()`, Tier 2
- [ ] Worker: `--token <value>` or `RELATIVIST_TOKEN` env var -> token loaded (R13), CLI takes precedence
- [ ] `--token-file <path>` -> token written to path (R12), defaults to `DEFAULT_TOKEN_FILE`
- [ ] `--tls-cert` + `--tls-key` + `--token` -> Tier 3
- [ ] `--tls-cert` without `--tls-key` (or vice versa) -> `ConfigurationError` (R25)
- [ ] TLS flags without `--token` -> `ConfigurationError` (R4)
- [ ] TLS flags without `tls` feature compiled -> `ConfigurationError` with helpful message
- [ ] Tier is detected via `detect_tier()` (R3)
- [ ] Tier is logged at INFO level at startup (R3)
- [ ] Token is displayed and written to file if generated (R11, R12)

## Notes

- The `SecurityArgs` struct is a bridge between the CLI parser (Phase 6) and the security module. The CLI parser populates it from clap, then passes it to `SecurityConfig::for_coordinator()` or `SecurityConfig::for_worker()`.
- `RELATIVIST_TOKEN` env var is read with `std::env::var("RELATIVIST_TOKEN")` inside `for_worker()`.
- When TLS feature is not compiled in but TLS flags are provided, the error message should say something like `"TLS flags provided but the 'tls' feature is not enabled. Rebuild with: cargo build --features tls"`.
- The `for_coordinator` and `for_worker` distinction exists because: coordinators generate tokens and load server TLS; workers consume tokens and load client TLS.
