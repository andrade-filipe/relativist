## Context

This task wires Phase 3 into the `run_grid` function: after local reduction of all partitions, the coordinator calls `merge` (TASK-0065/0066) to recombine the net, then calls `reduce_all` (SPEC-03) to resolve border redexes and any residual redexes (R15-R18). This is a deliberate improvement over the Haskell prototype, which only resolved border redexes selectively. Strong confluence guarantees that resolving everything at once is correct. After resolution, metrics are recorded and the round counter is incremented.

```rust
// Inside run_grid loop body, after Phase 2:

// === PHASE 3: MERGE + RESOLVE BORDERS ===
let t_merge = Instant::now();

let (mut merged_net, border_redex_count) = merge(
    plan.partitions,
    &plan.borders,
);

metrics.border_redexes_per_round.push(border_redex_count);

// Resolve border redexes AND residual redexes (R15-R18)
let border_interactions = reduce_all(&mut merged_net);

metrics.merge_time_per_round.push(t_merge.elapsed());
metrics.border_interactions_per_round.push(border_interactions as u64);
metrics.total_interactions += local_interactions + border_interactions as u64;

current_net = merged_net;
metrics.rounds += 1;
```

## Notes

- SPEC-05 R15 explicitly states that `reduce_all` is called on the ENTIRE merged net, not just on border redexes. This automatically handles: (1) border redexes, (2) CON-DUP cascades from border resolution, (3) any residual redexes from partition queues. Strong confluence (T4) guarantees correctness regardless of reduction order.
- The `border_interactions` count (R19) includes ALL interactions during the post-merge `reduce_all`, not just border redexes. This is a simplification: distinguishing border-originated interactions from residual-originated interactions would require tracking within the reduction engine, which adds complexity for minimal analytical value.
- After this task, the full `run_grid` loop should be functional (no more `todo!()`).
