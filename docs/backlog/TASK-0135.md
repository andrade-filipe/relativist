# TASK-0135: Implement idle connection timeout

**Spec:** SPEC-10
**Requirements:** R33
**Priority:** P2 (defensive measure)
**Status:** TODO

## Context

The coordinator should close connections that have been idle (no messages sent or received) for longer than a configurable timeout (default: 30 seconds) as a defensive measure against stale connections and resource leaks (R33). This is implemented using `tokio::time::timeout` wrapping the recv_frame calls in the connection handler. If the timeout fires, the connection is closed and logged.

## Acceptance Criteria

- [ ] `idle_timeout_wrapper` function (or equivalent pattern) is documented/implemented in `src/security/mod.rs`
- [ ] Wraps an async read operation with `tokio::time::timeout(idle_timeout, ...)`
- [ ] On timeout: returns `SecurityError::ConnectionTimeout` (or equivalent)
- [ ] Default idle timeout: 30 seconds (from `SecurityConfig.idle_timeout`)
- [ ] Timeout resets on each successful message exchange
- [ ] Connection closure on timeout is logged at WARN level with remote address

## Key Types / Signatures

```rust
use std::time::Duration;
use tokio::time::timeout;

/// Wraps an async recv operation with an idle timeout (R33).
///
/// If no message is received within `idle_duration`, returns a timeout error.
/// The caller is responsible for closing the connection.
///
/// Usage in the coordinator's per-connection handler:
/// ```rust,ignore
/// let msg = with_idle_timeout(
///     config.idle_timeout,
///     recv_frame(&mut stream, config.max_message_size),
/// ).await?;
/// ```
pub async fn with_idle_timeout<F, T, E>(
    idle_duration: Duration,
    future: F,
) -> Result<T, SecurityError>
where
    F: std::future::Future<Output = Result<T, E>>,
    E: Into<SecurityError>,
{
    match timeout(idle_duration, future).await {
        Ok(Ok(result)) => Ok(result),
        Ok(Err(e)) => Err(e.into()),
        Err(_elapsed) => Err(SecurityError::ConnectionTimeout),
    }
}
```

## Notes

- This is a SHOULD requirement (R33), hence P2 priority.
- The idle timeout applies per-message-wait, not to the entire connection lifetime. Each successful recv resets the "clock" because `with_idle_timeout` is called per-recv.
- The `ConnectionTimeout` variant needs to be added to `SecurityError` (in TASK-0121 or as part of this task). It is a simple addition: `#[error("connection idle timeout")] ConnectionTimeout`.
- For the grid loop, where the coordinator waits potentially long for workers to reduce, the timeout is not used (the distribute/collect timeouts from `NodeConfig` apply instead). The idle timeout is primarily for the registration phase and any unexpected silence during operation.
