# TASK-0126: Implement token file write

**Spec:** SPEC-10
**Requirements:** R11, R12
**Priority:** P1 (important but not blocking)
**Status:** TODO
**Depends on:** TASK-0123
**Blocked by:** none
**Estimated complexity:** S (< 50 LoC)

## Context

When the coordinator generates or receives an authentication token, it must (a) log the token at INFO level exactly once (R11), and (b) write it to a file with restricted permissions for workers to read (R12). The default file path is `./relativist-token`, configurable via `--token-file`. The file must have mode `0600` (owner read/write only) on Unix systems. This task implements the token display and file write functions.

## Acceptance Criteria

- [ ] `display_token(token: &AuthToken)` logs the base64-encoded token at INFO level exactly once (R11)
- [ ] `write_token_file(token: &AuthToken, path: &Path) -> Result<(), SecurityError>` writes the base64 token to a file (R12)
- [ ] File is created with mode `0600` on Unix (owner read/write only)
- [ ] On non-Unix platforms, file is created normally (best-effort, no permission setting)
- [ ] `DEFAULT_TOKEN_FILE` constant is `"./relativist-token"` (R12)
- [ ] The function overwrites any existing file at the path
- [ ] Errors (I/O failures) are propagated as `SecurityError::Io`

## Notes

- The `display_token` function is intentionally separate from `write_token_file` because R11 requires the token to be logged exactly once. The caller (coordinator startup) calls `display_token` first, then `write_token_file`, and after that point the token value must never appear in logs again.
- On Windows, Unix file permissions (`0600`) are not applicable. The function should set permissions to read/write for the owner only using the best available mechanism, or skip permission setting with a logged warning.
- The `#[cfg(unix)]` / `#[cfg(not(unix))]` conditional compilation handles the platform difference.
