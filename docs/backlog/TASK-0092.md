## Context

`run_coordinator` is the top-level function that orchestrates the distributed grid loop. It composes the sub-functions from TASK-0088 through TASK-0091 into the full coordinator FSM: accept workers -> (partition -> distribute -> collect -> merge) loop -> shutdown. This is the distributed equivalent of the local `grid_loop` from SPEC-05, where Phase 2 (local reduction) is replaced by network communication with remote workers. The function populates `GridMetrics` with network metrics per round.

## Acceptance Criteria

- [ ] `run_coordinator` is implemented in `src/protocol/coordinator.rs`
- [ ] Function is async, accepts `Net`, `&NodeConfig`, `&GridConfig`, `&dyn PartitionStrategy`
- [ ] Phase 0: calls `accept_workers` to wait for all workers (TASK-0088)
- [ ] Loop: checks normal form (empty redex queue after draining stale redexes) and round limit

- [ ] After loop: calls `shutdown_workers` (TASK-0091)
- [ ] Populates `GridMetrics` with all per-round metrics: agents, partition_time, network_send_time, network_recv_time, bytes_sent, bytes_received, worker_stats, border_redexes, local/border interactions
- [ ] Returns `Ok((Net, GridMetrics))` on success

## Notes

- This function is essentially the pseudocode from SPEC-06 Section 4.6 translated to Rust.
- The function uses `Instant::now()` to time each phase (partitioning, sending, receiving, merging) for metrics.
- All network I/O is async via tokio (R20).
- Connection loss aborts the entire grid loop (R25). No retry logic.
- The function MUST collect metrics in the exact same structure as the local grid_loop so that benchmarks (SPEC-09) can compare local vs distributed modes.
- This is the most integration-heavy task in Phase 5. It should be implemented last, after all sub-functions are tested individually.
