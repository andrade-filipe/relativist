## Context

SPEC-13 R21 defines the coordinator's transition table: a pure function that takes the current state and an event and returns the new state plus a list of actions. This is the core FSM logic, deliberately separated from async I/O. The function is synchronous and testable without tokio. Every transition must be logged at INFO level (R23).

The transition table from SPEC-13 R21 (Revised v2):

| From | Event | To | Actions |
|------|-------|----|---------|
| Init | ConfigLoaded | WaitingForWorkers | BindListener, LogTransition |
| WaitingForWorkers | WorkerConnected(id) [count < min] | WaitingForWorkers | LogTransition |
| WaitingForWorkers | WorkerConnected(id) [count >= min] | Partitioning | InvokeSplit, LogTransition |
| Partitioning | SplitComplete(parts) | Dispatching | LogTransition |
| Dispatching | AllDispatched | WaitingForResults | StartTimer(collect_timer), LogTransition |
| WaitingForResults | PartitionReturned(id, P) [not all] | WaitingForResults | --- |
| WaitingForResults | PartitionReturned(id, P) [all received] | Merging | CancelTimer, InvokeMergeAndReduce, LogTransition |
| WaitingForResults | PhaseTimeout(id) | Error | LogTransition, ShutdownAll |
| Merging | MergeComplete(net, _) | CheckTermination | LogTransition |
| CheckTermination | [is_normal_form == true] | Done | WriteOutput, ShutdownAll, LogTransition |
| CheckTermination | [is_normal_form == false] | Partitioning | InvokeSplit, LogTransition |
| Any | FatalError(e) | Error | LogTransition, ShutdownAll |

> **Note on worker registration (updated for SPEC-06 v3):** Registration behavior is tier-dependent (SPEC-06 R2a). In Tier 1 (no auth), registration MAY be implicit upon TCP connection acceptance. In Tier 2/3 (auth enabled), the Register/RegisterAck/RegisterNack handshake is mandatory per SPEC-10 R14-R17.
> **Note on Merging state:** Encompasses both merge (SPEC-05 R1-R11) and post-merge reduce_all (SPEC-05 R15-R18). InvokeMergeAndReduce performs both steps as a single unit.

## Acceptance Criteria

- [ ] `transition(state, event, ctx) -> (CoordinatorState, Vec<CoordinatorAction>)` is implemented
- [ ] `CoordinatorContext` holds mutable state needed for transition decisions: `connected_workers`, `expected_workers`, `received_partitions`, `round`, `current_net`
- [ ] All 12 transitions from the table above are implemented (including CheckTermination branching)
- [ ] Every state change produces a `LogTransition` action
- [ ] `WaitingForWorkers -> Partitioning` produces `InvokeSplit` action (not called inline)
- [ ] `WaitingForResults [all received] -> Merging` produces `InvokeMergeAndReduce` action
- [ ] `Merging -> CheckTermination` is an intermediate step; termination decision happens in CheckTermination
- [ ] `CheckTermination` checks `is_normal_form` from the MergeComplete event stored in context
- [ ] `CheckTermination [not normal form] -> Partitioning` produces `InvokeSplit` action
- [ ] `FatalError` from any state transitions to `Error`
- [ ] The function is pure (no I/O, no async, no side effects beyond the returned actions)
- [ ] Unhandled event+state combinations produce a `FatalError` transition

## Notes

- The `CoordinatorContext` is a pragmatic addition not shown in the spec's pseudocode. The spec's `transition(state, event)` signature is simplified; in practice, the function needs access to accumulated state (how many workers have registered, how many partitions have been received). The context struct keeps this state separate from the FSM state enum.
- The context is passed as `&mut` rather than returned, which is a minor deviation from pure-functional style but avoids unnecessary cloning.
- The actual async event loop that calls this function is in TASK-0112 (coordinator startup wiring).
