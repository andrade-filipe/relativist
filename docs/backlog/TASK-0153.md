# TASK-0153: Implement coordinator metric aggregation from worker reports

**Spec:** SPEC-11
**Requirements:** R13
**Priority:** P1 (important, not blocking)
**Status:** TODO

**Estimated complexity:** M (50-200 LoC)

## Context

When the coordinator receives `ReturnPartition` messages containing `WorkerMetricsReport` data, it must aggregate the worker-reported metrics into its own Prometheus registry (R13). This is the mechanism by which per-worker reduction statistics become visible through the single `/metrics` scrape endpoint.

Aggregation includes: observing worker reduction durations in a histogram, incrementing total redex counters, and (optionally) tracking per-rule interaction counts. This task implements the aggregation function that bridges `WorkerMetricsReport` to `CoordinatorMetrics`.

- [ ] Can aggregate reports from multiple workers in the same round

## Key Types / Signatures

```rust
/// Aggregate a worker's metrics report into the coordinator's Prometheus registry (R13).
///
/// Called by the coordinator after receiving each `ReturnPartition` message.
/// Multiple calls per round (one per worker) are expected and safe.
#[cfg(feature = "metrics")]
pub fn aggregate_worker_report(
    metrics: &CoordinatorMetrics,
    report: &WorkerMetricsReport,
    payload_bytes: u64,
) {
    // Observe worker reduction duration into round_duration histogram
    // (or a dedicated worker_reduce_duration histogram if added).
    // This provides distribution of per-worker reduction times.

    // Increment return_bytes_total by the payload size.
    metrics.return_bytes_total.inc_by(payload_bytes);

    // Optionally: log per-rule interaction counts at DEBUG level for
    // correlation with reduction efficiency analysis (DISC-006 v2).
    tracing::debug!(
        redexes_reduced = report.redexes_reduced,
        reduce_duration_secs = report.reduce_duration_secs,
        "aggregated worker metrics report"
    );
}
```

## Notes

- The aggregation is intentionally simple in v1: worker metrics are summed/observed into the coordinator's global counters. Per-worker breakdowns (e.g., a `worker_id` label on reduction duration) could be added later but are not required by SPEC-11.
- The `payload_bytes` parameter is the serialized size of the `ReturnPartition` message, not computed from the `WorkerMetricsReport`. It is provided by the collect phase which knows the frame size.
- This function is called on the coordinator's main event loop. It must be fast (no allocations, no I/O). All Prometheus counter/histogram operations are lock-free atomic operations.
