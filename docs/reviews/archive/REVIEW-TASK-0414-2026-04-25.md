## 1. Summary

The implementation lands all 2 new `CoordinatorState` variants, all 9 new `CoordinatorEvent` variants, all 9 new `CoordinatorAction` variants, plus four helper enums (`LeaveKind`, `RetainedSlot`, `DepartureKind`, `TimerKind`) as required by SPEC-20 §3.8 A2. Enum ordering is correct (new variants appended at end for discriminant stability). `serde::Serialize` is intentionally one-way (log/observability only), correctly documented, and not present on wire-protocol paths. The `#[repr(u32)]` with explicit discriminants on `TimerKind` (NF-008 closure) is correctly implemented. The 11 inline tests cover construction, `PartialEq` reflexivity, cross-inequality, Debug format, and discriminant-stability — appropriate depth for a type-surface-only task.

Four issues prevent a clean PASS. The most consequential is that the pre-existing `transition()` function continues to pass raw `TimerId` literals `0` at two call sites (`StartTimer(0, ...)` and `CancelTimer(0)`) after `TimerKind` is now available in the same file — this directly violates NF-008 intent and will silently diverge from SPEC-20 §4.1.4 as soon as TASK-0436 adds the new transitions. The second significant issue is `LeaveKind`'s placement: defined as a local stub in `coordinator.rs` but required by both the FSM layer (coordinator) and the wire-protocol layer (TASK-0418 `Message::LeaveRequest`); its current location is architecturally inverted and sets up a cross-task coupling that will require a breaking rename/move at TASK-0418. Two further LOW issues complete the picture.

**Cleared for Stage 5 QA after the four MEDIUM fixes land.** The wildcard `transition()` arm is confirmed present, correctly logs, and is the established absorber for unhandled events — it is sound for the current type-surface-only task scope; the concern is about what TASK-0436 must do (see MF-002).

## 2. Findings

### Must-Fix (MEDIUM)

#### MF-001 — `StartTimer` / `CancelTimer` use raw `u32` literals `0` after `TimerKind` is available

**Category:** Spec Violation / Code Quality (magic numbers)
**Principle/Spec:** SPEC-20 §4.1.3 NF-008; Clean Code (no magic numbers)
**File:** `relativist-core/src/coordinator.rs:412, 432`

**Problem:** `TimerKind` and the `TimerId = kind as u32` contract are defined in this same file at lines 78–89. Yet the two pre-existing `transition()` call sites that fire the collect timer still use the raw literal `0`:

```rust
// line 412
actions.push(CoordinatorAction::StartTimer(0, Duration::from_secs(600)));
// line 432
actions.push(CoordinatorAction::CancelTimer(0));
```

`TimerKind::Collect = 3` (not `0`). `0` is `TimerKind::InitialWait`. This is both wrong-by-spec and will actively mislead TASK-0436 authors who add the SPEC-20 §4.1.4 `InitialWait` / `JoinWindowMin` / `JoinWindowMax` transitions to the same `match` block while the collect timer already occupies `TimerId = 0`.

**Before:**
```rust
actions.push(CoordinatorAction::StartTimer(0, Duration::from_secs(600)));
// ...
actions.push(CoordinatorAction::CancelTimer(0));
```

**After:**
```rust
actions.push(CoordinatorAction::StartTimer(
    TimerKind::Collect as TimerId,
    Duration::from_secs(600),
));
// ...
actions.push(CoordinatorAction::CancelTimer(TimerKind::Collect as TimerId));
```

**Why:** `TimerKind::Collect = 3`. Using `0` here means the collect timer is registered under `InitialWait`'s id — a semantic collision that will silently cause TASK-0436's `InitialWait` timer management to cancel or re-arm the collect timer. Fixing this now, while the new enum is in the same commit, costs two lines and prevents a latent cross-task bug.

#### MF-002 — `LeaveKind` is defined in `coordinator.rs` but will need to live in a shared types module by TASK-0418

**Category:** Architecture (dependency direction / future merge conflict risk)
**Principle/Spec:** SPEC-13 R28 (dependency direction: core pure, protocol depends on core); SPEC-20 §4.1.2 R21 (wire schema carries `LeaveKind`)
**File:** `relativist-core/src/coordinator.rs:32–44`

**Problem:** `LeaveKind` is annotated "local stub pending TASK-0418." TASK-0418 adds `Message::LeaveRequest { kind: LeaveKind }` to the wire-protocol `Message` enum in `protocol/`. When that lands the developer will face a binary choice:

1. `protocol/` imports `coordinator::LeaveKind` — but `protocol/` is above `coordinator` in the module dependency chain (protocol depends on coordinator's types, not the reverse), so this is architecturally unsound (inverted dependency).
2. `LeaveKind` is redefined or re-exported in `protocol/` — creating a duplicate definition that diverges silently until someone notices the two `LeaveKind` types are different.

Neither option is clean. The correct location is a shared types module (`net/`, `partition/types.rs`, or a new `types.rs` in the crate root) that both `coordinator.rs` and `protocol/` import.

**Before (current):**
```rust
// coordinator.rs — sole definition of LeaveKind
pub enum LeaveKind {
    AfterResult,
    Urgent,
}
```

**After (recommended):**
Move `LeaveKind` to `relativist-core/src/partition/types.rs` alongside `WorkerId`, since it is a departure classification tied to worker identity. Then:
```rust
// coordinator.rs
use crate::partition::LeaveKind; // re-export from shared location
```
```rust
// protocol/ (TASK-0418)
use crate::partition::LeaveKind; // same import, no duplication
```

If moving to `partition/types.rs` is premature for the current bundle, the minimum acceptable fix for TASK-0414 is to add a `// TODO(TASK-0418): move this to partition/types.rs before implementing Message::LeaveRequest` comment directly on the enum definition and flag this in the TASK-0418 task file. This does not resolve the architectural issue, but it makes the coupling explicit and prevents the silent-diverge scenario.

**Why:** SPEC-13 R28 and the crate's inviolable dependency direction (`net <- reduction <- partition <- merge <- protocol <- coordinator/worker`) require shared wire-relevant types to live in a layer that both protocol and coordinator can import without circularity. Leaving `LeaveKind` in `coordinator.rs` inverts this at TASK-0418 time.

#### MF-003 — `CoordinatorAction::InvokeSplitAndDispatch` uses a bare `u32` payload instead of a named wrapper

**Category:** Code Quality (primitive obsession / naming)
**Principle/Spec:** Rust idioms (newtype pattern); SPEC-20 §4.1.2 (`InvokeSplitAndDispatch(K_eff)`)
**File:** `relativist-core/src/coordinator.rs:283`

**Problem:** The variant is:
```rust
InvokeSplitAndDispatch(u32 /* K_eff */),
```
The inline comment carries semantic intent that the type system does not. At every call site in the future (TASK-0430, TASK-0431, TASK-0436), the author must remember that this `u32` means "effective worker count" — not a `WorkerId`, not a round number, not an arbitrary count. SPEC-20 §4.1.2 names this value `K_eff` explicitly as a distinct concept.

A lightweight newtype prevents confusion at call sites where multiple `u32` values are in scope:

**Before:**
```rust
InvokeSplitAndDispatch(u32 /* K_eff */),
```

**After (minimal — newtype alias):**
```rust
/// Effective worker count for a round (remote workers + 1 if hybrid). SPEC-20 R2.
pub type EffectiveWorkerCount = u32;

InvokeSplitAndDispatch(EffectiveWorkerCount),
```

A full newtype struct (`pub struct EffectiveWorkerCount(u32)`) is the most correct form per the crate's newtype-IDs convention but is heavier; a type alias is acceptable here because `K_eff` is a count, not an identifier, and arithmetic on it is legitimate. The comment alone is insufficient — it disappears at call sites.

**Why:** Every other domain-specific `u32` in this file (`WorkerId`, `TimerId`, `round`) is either a named type alias or carries an in-context name. Bare `u32` in a public action enum creates "what does this number mean" confusion for TASK-0436 authors and violates the crate's primitive-obsession standard.

#### MF-004 — `TimerKind` is not exported from `lib.rs` / the crate's public API surface

**Category:** Architecture (visibility / API completeness)
**Principle/Spec:** SPEC-20 §4.1.3 (NF-008 — log tooling must be able to decode `TimerId -> TimerKind`); `pub(crate)` unless truly public API (project standard)
**File:** `relativist-core/src/lib.rs:26` (imports `pub mod coordinator`)

**Problem:** `TimerKind` is defined `pub` inside `pub mod coordinator`, which means it is technically reachable as `relativist_core::coordinator::TimerKind`. However, SPEC-20 §4.1.3 states that the purpose of `#[repr(u32)]` is specifically to allow log tooling to decode `TimerId -> TimerKind` without per-build metadata. If `TimerKind` is buried inside `coordinator`, observability crates and the future `observability/` module (SPEC-11) must reach through `coordinator` to access it — violating the "observability has no dependency on coordinator business logic" expectation.

This is a MEDIUM rather than HIGH because the enum IS accessible via `coordinator::TimerKind`; the issue is discoverability and architectural correctness of the access path, not a compile failure.

**Suggested fix:** Add a re-export at the crate root or from the relevant shared types module so that `TimerKind` is accessible as `relativist_core::TimerKind` (or `relativist_core::coordinator::TimerKind` at minimum, which already works). More importantly, document in the `TimerKind` Rustdoc or in `lib.rs` that observability consumers should import it from `coordinator::TimerKind`. If `TimerKind` is later moved to a shared types module (a natural consequence of MF-002), this is resolved automatically.

**Why:** The spec's NF-008 rationale is portable, build-independent decoding. Burying `TimerKind` in `coordinator` couples any decoder to the coordinator module's full import chain. For now this is note-worthy but not blocking — the encoding is correct and complete per §4.1.3.

#### SF-001 — `transition()` wildcard arm does not distinguish "unhandled new SPEC-20 event" from "genuine unexpected event"

**Category:** Code Quality (diagnostic clarity)
**File:** `relativist-core/src/coordinator.rs:500–507`

The wildcard `(state, event)` arm logs a `WARN` and returns an empty actions list. This is sound as a catch-all for TASK-0414 (which adds no transition logic). However, once TASK-0436 wires the new transitions, any new SPEC-20 event that is accidentally omitted from the match arms will silently fall through to this same `WARN` — indistinguishable from a genuinely unexpected event. A developer debugging TASK-0436 who fires `WorkerJoined` in state `WaitingForResults` should get a `WARN` that is identifiable as "SPEC-20 R10b event handled by catch-all — check TASK-0436 arms" rather than a generic "unexpected event."

No code change needed before TASK-0436. Add a `// NOTE(TASK-0436): Before landing transition wiring, replace this wildcard with exhaustive arms; any SPEC-20 event falling through here is a missing transition row.` comment above the wildcard arm now so TASK-0436 authors see it.

#### SF-002 — `CoordinatorContext` is missing `hybrid_coordinator` and `elastic_*` config fields needed by all TASK-0436 guard conditions

**Category:** Architecture (forward-readiness / spec alignment)
**Spec:** SPEC-20 §4.1.4 transition table — almost every new row has a guard condition (`K=0 && hybrid_coordinator`, `elastic_departure`, etc.)
**File:** `relativist-core/src/coordinator.rs:303–343`

`CoordinatorContext` holds `min_workers`, `connected_workers`, `total_workers`, `round`, and `net`. None of the SPEC-20 guard-condition fields (`hybrid_coordinator: bool`, `elastic_join: bool`, `elastic_departure: bool`, `solo_budget: u32`) are present. TASK-0430 (hybrid FSM) will add these, but they are directly consumed by transitions that TASK-0436 owns. Without them, TASK-0436 cannot implement a single guarded transition from §4.1.4 without first amending the context struct — creating a dependency that neither task currently declares.

This is a LOW for TASK-0414 because the task contract explicitly defers transition logic to TASK-0436. However, the reviewer recommends adding an explicit `// NOTE: TASK-0430 or TASK-0436 must extend CoordinatorContext with hybrid_coordinator, elastic_join, elastic_departure, solo_budget before any §4.1.4 guarded transition can be implemented.` comment in `CoordinatorContext`'s doc block or struct body so the interdependency is visible to the next task author.

#### SF-003 — `UT-0414-09` (`existing_variants_unchanged`) tests serde output but does not cover the two new variants' serde output regression-proof

**Category:** Test Quality (coverage gap)
**File:** `relativist-core/src/coordinator.rs:1001–1029`

```rust

("\"SoloReducing\"", CoordinatorState::SoloReducing),
```

Replace the standalone spot-checks at lines 1025–1028 with entries in the `expected` vec.

#### SF-004 — Missing `#[non_exhaustive]` or explicit compile-time guard against silent TASK-0436 catch-all reliance

**Category:** Code Quality / Architecture (future-proofing)
**Spec:** TASK-0414 acceptance criteria note: "existing FSM transition function is untouched (kept exhaustive via `#[non_exhaustive]` or placeholder)"
**File:** `relativist-core/src/coordinator.rs:155–218` (CoordinatorEvent), `229–297` (CoordinatorAction)

The task description listed `#[non_exhaustive]` as one option for keeping the function sound after new variants land. Neither `CoordinatorEvent` nor `CoordinatorAction` carry `#[non_exhaustive]`, and none of the new TASK-0414 variants are listed in the `transition()` match arms. The wildcard arm absorbs them silently. This is SOUND for the current state because TASK-0436 owns the transition logic — but it means that any consumer crate that pattern-matches on `CoordinatorEvent` directly will need no changes when TASK-0436 lands new match arms (there is no compiler forcing of exhaustion). For an internal-crate enum this is LOW severity; for a `pub` enum in a `pub mod`, callers outside the crate would silently absorb unhandled variants.

Given that `CoordinatorEvent` is pub and the crate has a public API surface, consider whether `#[non_exhaustive]` is appropriate here. If the enum is intended as a closed, implementation-internal type, change `pub enum CoordinatorEvent` to `pub(crate) enum CoordinatorEvent`. If it must remain `pub`, `#[non_exhaustive]` prevents external callers from writing exhaustive matches that silently miss new variants.

## 5. Transition Exhaustiveness Assessment

The `transition()` function at line 350 matches `(&ctx.state, event)`. The final arm `(state, event) => { tracing::warn!(...) }` is a wildcard that absorbs all unhandled `(state, event)` pairs, including every new SPEC-20 pair, and returns an empty `Vec<CoordinatorAction>`. This is sound for the current task (type-surface only) because:

1. No production code drives `WorkerJoined`, `WorkerLeft`, `MembershipWindowClosed`, etc. into `transition()` yet.
2. The wildcard logs a `WARN`, making accidental fires visible.
3. TASK-0436 is the designated owner of these arms — the wildcard is a placeholder, not a permanent omission.

The risk is that the wildcard will silently absorb any accidentally-missing arm in TASK-0436. See SF-001 for the recommended mitigation (add a comment reminding TASK-0436 to replace the wildcard with exhaustive arms before shipping).
