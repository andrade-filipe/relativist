## 1. Summary

TASK-0413 implements the surgical branching described in SPEC-20 §3.8 A1: two pure helpers (`handle_connection_loss`, `handle_phase_timeout`) encode the conditional gate, and `ConnectionLossOutcome` is the typed return that lets the future call sites dispatch without re-examining the flag. The implementation is intentionally a Phase-A placeholder — no production call site yet, no FSM wiring — consistent with the task's stated scope (gate only; TASK-0438 / TASK-0415 follow).

The default path (`elastic_departure = false`) is provably byte-identical to the pre-SPEC-20 R25 behavior: it returns `ConnectionLossOutcome::Abort(error_description.to_owned())` with no side effects whatsoever. UT-0413-09 explicitly asserts both helpers return `Abort` on the default path, and UT-0413-03 verifies the description is propagated verbatim. These are the two most load-bearing tests in the suite and they are correct.

The two `#[ignore]`-gated integration tests (UT-0413-11, UT-0413-12) are well-documented with the TASK-0415 blocking reason and are safely inert until the `GridConfig.elastic_departure` field lands.

The findings below are all MEDIUM-or-lower. Three are about precision issues that matter for future maintainers; none is a live correctness defect.

## 2. Findings

### MEDIUM

#### RV-001 — `ConnectionLossOutcome` is shared between two semantically distinct events but carries no event-kind discriminant

- **Category:** Code Quality (SRP / expressiveness)
- **Location:** `relativist-core/src/protocol/coordinator.rs:43-50` (enum definition); `handle_connection_loss` and `handle_phase_timeout` both return it
- **Evidence:** `ConnectionLossOutcome` is returned by both `handle_connection_loss` (R19: TCP connection drop) and `handle_phase_timeout` (R18: timeout during `WaitingForResults`). SPEC-20 §3.3 R18 and R19 are distinct events that land in different FSM rows in §4.1.3 (`PhaseTimeout(id)` row vs `WorkerConnectionLost(id)` row). The enum's `RecoveryTriggered { worker_index }` variant carries no discriminant to tell the call site which event fired. When TASK-0438 wires the FSM, the call site must already know whether it came from `handle_connection_loss` or `handle_phase_timeout` — so it will correctly dispatch `CoordinatorEvent::WorkerConnectionLost` vs `CoordinatorEvent::PhaseTimeout`. But the shared return type makes that responsibility implicit and untypeable: a caller who calls `handle_phase_timeout` and then reads `RecoveryTriggered` cannot ask "was this a timeout or a connection drop?" from the value alone.
- **Why it matters:** TASK-0438 will wire these outcomes into the FSM. If both helpers return the same `RecoveryTriggered { worker_index }`, TASK-0438's call site is forced to encode the dispatch kind in surrounding context rather than in the outcome value. That is a temporal coupling: knowing the FSM event to emit requires remembering which helper was called. This is exactly the scenario ISP and SRP warn against — the type is doing two jobs.
- **Suggested fix (two options; Option A preferred):**

  **Option A (preferred) — add an `event_kind` field to `RecoveryTriggered`:**
  ```rust
  /// Distinguishes the FSM event that triggered the recovery path,
  /// so TASK-0438's call site can emit the correct `CoordinatorEvent`
  /// variant without re-examining external context.
  #[derive(Debug, PartialEq, Eq)]
  pub(crate) enum DepartureEventKind {
      /// TCP connection closed unexpectedly (SPEC-20 §3.3 R19).
      ConnectionLost,
      /// `collect_timeout` elapsed for the worker (SPEC-20 §3.3 R18).
      PhaseTimeout,
  }

  pub(crate) enum ConnectionLossOutcome {
      Abort(String),
      RecoveryTriggered {
          worker_index: usize,
          kind: DepartureEventKind,
      },
  }
  ```
  Update `handle_connection_loss` to set `kind: DepartureEventKind::ConnectionLost` and `handle_phase_timeout` to set `kind: DepartureEventKind::PhaseTimeout`. Update UT-0413-02, UT-0413-05, UT-0413-07, UT-0413-10 to assert the `kind` field. The existing `assert_eq!` checks on `worker_index` need a `kind` arm added — no test is deleted.

  **Option B (minimal) — keep the shared type but document the disambiguation contract explicitly in the Rustdoc:**
  Add to the `RecoveryTriggered` variant's doc comment: "Call sites MUST already know which helper produced this outcome and emit the corresponding `CoordinatorEvent` (ConnectionLost → `WorkerConnectionLost(worker_index)`, PhaseTimeout → `PhaseTimeout(worker_index)`) to the FSM (TASK-0438). The enum itself does not carry an event discriminant; see TASK-0438 for the wiring."
  This is cheaper but defers the disambiguation burden to TASK-0438.

- **Severity rationale:** MEDIUM because Option B defers a typing debt to TASK-0438 and a future author may miss the comment; Option A catches a wrong dispatch at compile time. The fix is small (~10 lines + test updates) and has no risk of breaking the test floor.

#### RV-002 — `handle_phase_timeout` doc comment misstates which SPEC-20 ref governs the elastic path

- **Category:** Code Quality (Rustdoc precision / spec citation)
- **Location:** `relativist-core/src/protocol/coordinator.rs:98-101` (function-level doc), `:121-123` (inline comment)
- **Evidence:** The function Rustdoc reads: "`elastic_departure = true`: emits `tracing::warn!`, records the `WorkerConnectionLost` / `PhaseTimeout` event placeholder". The `/` notation implies `WorkerConnectionLost` is relevant here, but `handle_phase_timeout` maps exclusively to R18 (PhaseTimeout detection), not R19 (connection-loss detection). The correct FSM event for a phase timeout in the elastic path is `CoordinatorEvent::PhaseTimeout(worker_index)`, not `CoordinatorEvent::WorkerConnectionLost(worker_index)` (see SPEC-20 §4.1.3 transition table row: `WaitingForResults | PhaseTimeout(id) [elastic_departure] | WaitingForResults | ReclaimPartition(id, ...), RemoveWorker(id), LogDeparture(Timeout)`). Citing `WorkerConnectionLost` in the PhaseTimeout helper is a cross-contamination of the two R18/R19 channels.

  The inline TODO comment at line 124-125 reinforces this:
  ```rust
  // TODO(TASK-0438): emit CoordinatorEvent::WorkerConnectionLost(worker_index)
  ```
  The TODO incorrectly names `WorkerConnectionLost`; it should name `CoordinatorEvent::PhaseTimeout(worker_index)`.

- **Suggested fix:**
  In the `handle_phase_timeout` function doc, replace:
  ```
  records the `WorkerConnectionLost` / `PhaseTimeout` event placeholder
  ```
  with:
  ```
  records the `PhaseTimeout(worker_index)` event placeholder (SPEC-20 §3.3 R18)
  ```
  In the inline TODO comment (line 124-125), replace:
  ```rust
  // TODO(TASK-0438): emit CoordinatorEvent::WorkerConnectionLost(worker_index)
  ```
  with:
  ```rust
  // TODO(TASK-0438): emit CoordinatorEvent::PhaseTimeout(worker_index)
  // into the FSM transition table here. For now this is a warn-only placeholder.
  ```
  (The `handle_connection_loss` TODO at line 85-86 is already correct — it names `WorkerConnectionLost`. No change there.)

- **Severity rationale:** MEDIUM because TASK-0438's author will read these TODOs as the primary dispatch guide, and an incorrect event name in the TODO directly causes a wrong FSM transition. The fix is a one-line doc edit and a one-line comment edit.

#### RV-003 — UT-0413-12 ignore comment contains an inaccurate precondition claim

- **Category:** Test Quality (documentation precision)
- **Location:** `relativist-core/src/protocol/coordinator.rs:1576-1600` (UT-0413-12)
- **Evidence:** The ignore message and body comment for UT-0413-12 read:
  ```
  #[ignore = "depends on TASK-0415 GridConfig.elastic_departure field for run_coordinator wiring"]
  ```
  and the body comment says:
  ```
  // This test does NOT require GridConfig.elastic_departure because
  // the default GridConfig already has elastic_departure = false.
  // Gated #[ignore] until TASK-0415 stabilizes the GridConfig shape
  ```
  This is internally contradictory. The `#[ignore]` message says it depends on `elastic_departure`, while the body says it does NOT require `elastic_departure` because the default is `false`. The real reason this test is gated is that `run_coordinator` does not yet read `grid_config.elastic_departure` (that wiring belongs to a downstream task, presumably TASK-0436 or wherever `run_coordinator` is modified to call `handle_connection_loss`). The test is blocked by the `run_coordinator` wiring, not by the `GridConfig` field itself.

- **Suggested fix:** Replace the ignore message and the leading comment block with a single coherent explanation:
  ```rust
  // UT-0413-12: verifies that the R25 DEFAULT PATH (elastic_departure = false)
  // is unchanged by TASK-0413. Gated #[ignore] because run_coordinator does not
  // yet call handle_connection_loss / handle_phase_timeout — that wiring is
  // pending the downstream dispatcher task (see TASK-0436 / TASK-0438).
  // GridConfig.elastic_departure field (TASK-0415) is NOT the blocker here;
  // the default GridConfig already has elastic_departure = false once that
  // field lands. Remove the #[ignore] when run_coordinator's collect_results
  // path branches on elastic_departure.
  #[ignore = "blocked by run_coordinator wiring, not GridConfig shape (TASK-0436/TASK-0438)"]
  ```

- **Severity rationale:** MEDIUM because the wrong blocking reason will cause the test to remain gated longer than necessary — a developer who lands TASK-0415 (GridConfig field) and searches for unignore candidates will read the ignore message, conclude the dependency is satisfied, and be confused when the test still fails or does not exercise the right path.

#### RV-004 — `handle_connection_loss` and `handle_phase_timeout` symmetry is a strength; a shared private helper could make the symmetry explicit and enforce it

- **Category:** Code Quality (DRY / single level of abstraction)
- **Location:** `relativist-core/src/protocol/coordinator.rs:70-92` and `:109-131`
- **Evidence:** Both helpers share an identical structure:
  ```rust
  if elastic_departure {
      tracing::warn!( ... );
      // TODO(TASK-0438): emit CoordinatorEvent::...
      ConnectionLossOutcome::RecoveryTriggered { worker_index }
  } else {
      ConnectionLossOutcome::Abort( <format-string> )
  }
  ```
  The only differences are the `tracing::warn!` message and the `Abort` format string. If a future edit adds a metric emission or a structured field to one branch, there is a risk that the other branch drifts without a reviewer noticing. The symmetry is documented in the module-level comment and is an intentional design decision (the task spec explicitly says "same branching for PhaseTimeout(id)"), but the code does not make this structural identity explicit.
- **Suggested fix (optional — no code change required):** Add a brief comment above `handle_phase_timeout` noting the structural symmetry with `handle_connection_loss`:
  ```rust
  // Structurally symmetric with handle_connection_loss above (TASK-0413 design note):
  // the only differences are the tracing message content and the Abort format string.
  // When TASK-0438 wires the FSM, both branches must emit their respective
  // CoordinatorEvent variants; the symmetry should be preserved.
  ```
- **Severity rationale:** LOW — the symmetry is already evident to a careful reader; this is a comment suggestion only.

#### RV-005 — `#[allow(dead_code)]` placement is correct but the explanation could be more precise

- **Category:** Code Quality (documentation)
- **Location:** `relativist-core/src/protocol/coordinator.rs:41` (on the enum) and `:69` and `:108` (on the two helpers)
- **Evidence:** The three `#[allow(dead_code)]` annotations carry `// wired by TASK-0415 (GridConfig.elastic_departure) + downstream Phase B/C/D.` The enum allow is sound — the enum variants are referenced only from the test helpers inside `mod tests`, which Rust's dead-code analysis may or may not count depending on how the analysis walks the crate. The function allows are sound for the same reason: no production call site exists. However, the comment says "wired by TASK-0415", which implies TASK-0415 alone will eliminate the dead-code warning. In fact, TASK-0415 adds `GridConfig.elastic_departure` but does NOT wire the call sites to `handle_connection_loss` / `handle_phase_timeout` — that wiring is a downstream task (TASK-0436 or similar). The attribute is honest but the comment overpromises.
- **Suggested fix:** Amend to:
  ```rust
  #[allow(dead_code)] // production call site lands when run_coordinator is wired (TASK-0436+); GridConfig field lands separately (TASK-0415).
  ```
- **Severity rationale:** LOW — the attribute is correct regardless of the comment; this is a precision fix only.

## 3. Spec compliance analysis

**SPEC-06 R25 (verbatim):** "If a connection with a worker is lost during execution, the coordinator MUST abort the grid loop and return an error."

**A1 conditional clause (SPEC-20 §3.8):** "... unless `GridConfig.elastic_departure = true`, in which case the `WaitingForResults × PhaseTimeout(id)` and the `WaitingForResults × ConnectionLost(id)` transitions are handled per SPEC-20 §3.3 R18-R19."

**Compliance assessment:**
- `elastic_departure = false` path: `handle_connection_loss` returns `Abort(description)` with zero side effects. `handle_phase_timeout` returns `Abort(format!("phase timeout after {:.2}s", ...))`. Both are byte-identical to the v1 behavior. UT-0413-09 asserts this for both helpers jointly. COMPLIANT.
- `elastic_departure = true` path: both helpers emit `tracing::warn!` and return `RecoveryTriggered { worker_index }` without aborting. The actual reclaim (SPEC-20 §3.3 R18-R19 "coordinator reclaims state and removes the worker from `W_active`") is deferred to TASK-0438 via the TODO comments. COMPLIANT FOR THIS TASK'S SCOPE (gate only).
- `PhaseTimeout` coverage: `handle_phase_timeout` covers the `WaitingForResults × PhaseTimeout(id)` row per A1 and per SPEC-13 R21. COMPLIANT.
- `ConnectionLost` coverage: `handle_connection_loss` covers the `WaitingForResults × ConnectionLost(id)` row per A1. COMPLIANT.

The implementation faithfully models Phase A of the A1 amendment. The open question for downstream tasks is whether the shared `ConnectionLossOutcome` return type (RV-001) will cause TASK-0438's FSM wiring to emit the wrong event variant — hence RV-001 is MEDIUM rather than HIGH.

## 4. Architecture compliance analysis

- **Core layer purity:** Both helpers live in `src/protocol/coordinator.rs`, which is the async/tokio layer. Neither helper has any async boundary, no I/O, and no tokio dependency. The only side effect is `tracing::warn!`, which is acceptable in the protocol layer. The helpers are trivially pure in the sense relevant to testability — they take primitives and return an enum. ALIGNED.
- **No FSM wiring in this task:** The helpers are explicit placeholder stubs with `TODO(TASK-0438)` comments. This is the correct Phase A separation: TASK-0413 adds the decision type and gate; TASK-0436 / TASK-0438 wire the FSM transition table. The `#[allow(dead_code)]` annotations are an honest acknowledgment that no production call site exists. ALIGNED.
- **`GridConfig.elastic_departure` not yet on `GridConfig`:** The helpers take `elastic_departure: bool` as a plain parameter rather than `&GridConfig`. This is the right design for Phase A: it avoids a direct dependency on `GridConfig`'s not-yet-complete shape (TASK-0415 adds the field) while keeping the helpers testable today. ALIGNED.
- **Dependency direction:** `protocol/coordinator.rs` depends on `merge::GridConfig` (indirectly via the `run_coordinator` signature) but the new helpers in this task take only primitives. No new dependency was introduced. ALIGNED.
- **Backward compat with 1181 / 1224 floor:** The new code is behind a `#[cfg(test)]`-inaccessible `#[allow(dead_code)]` guard in production; all existing tests exercise the unchanged `run_coordinator`, `accept_workers`, `distribute_partitions`, `collect_results`, `shutdown_workers` paths. ALIGNED.

## 5. Strengths

**Surgical scope discipline.** The task does exactly what the backlog says: one conditional gate, two pure helpers, no FSM wiring. The module-level doc comment explains the phasing (Phase B+ owns the reclaim logic) clearly and cites the downstream task IDs. Future maintainers reading the file understand immediately what is placeholder and what is complete.

**Backward-compat test is the right shape.** UT-0413-09 asserting BOTH helpers together under `elastic_departure = false` is more robust than two separate tests, because it prevents the scenario where one helper is accidentally modified to return `RecoveryTriggered` on the default path. The joint assertion is a good invariant formulation.

**Verbatim error propagation is explicit and tested.** UT-0413-03 verifies that the `Abort(String)` carries the original error description character-for-character. This is the right level of specificity for a backward-compat sentinel — it would catch a future refactoring that accidentally decorates the message.

**Worker index zero handled as a valid index.** UT-0413-07 is a meaningful regression guard given that `0` is often used as a "null" sentinel in index-based code. Its presence shows spec-awareness.
