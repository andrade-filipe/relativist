## 1. Summary

The implementation correctly realises SPEC-20 §3.8 A7 as a pure structural concatenation. Field handling for every live member of the current `Net` struct (`agents`, `ports`, `redex_queue`, `next_id`, `root`, `freeport_redirects`) is enumerated, defended by an explicit precondition contract (`debug_assert!`), and matched 1:1 by the test suite. Borrow / ownership use destructuring to consume both inputs by value — idiomatic and matches the spec's `union(self, other) -> Net` signature exactly. The `union_root_follows_self` rule is implemented and tested. The bridge to ARG-006 P12 is correctly cited in the Rustdoc (verified against `docs/theory-bridge.md` lines 59-62).

Every Stage-3 gate the orchestrator confirmed is preserved: 1181 default / 1224 zero-copy library tests, no clippy warnings, formatted. The 10 `net_union.rs` integration tests are deterministic, citation-aligned, and exercise every claimed semantic of `union` (empty-left, empty-right, both-empty, disjoint-ids, border preservation, internal-wire preservation, root-follows-self, debug-precondition panic, wireless side, duplicate border ids).

The findings below are MEDIUM-or-lower: minor Rustdoc precision improvements, a fixture-leakage concern on the test side, two cross-spec follow-up reminders, and one observability hook to consider out-of-scope but worth recording.

This is a strong opening DEV task and sets a good bar for the ~50 remaining Phase-A tasks in the SPEC-20 amendment cluster. **Cleared for Stage 5 QA after the four MEDIUM fixes land.**

## 2. Findings

### MEDIUM

#### RV-001 — `Net::empty()` semantic redundancy with `Net::new()` is documented but the public-API surface should pick a winner

- **Category:** Code Quality (DRY / API design)
- **Location:** `relativist-core/src/net/core.rs:83-90`
- **Evidence:** `Net::empty()` is a one-line `#[inline]` alias for `Net::new()`. The Rustdoc concedes the byte-equality. The motivation given (spec parity — `a.union(Net::empty())`) is real but creates a permanent two-named entrypoint to the same value. SPEC-02 already established `Net::new()` (the existing constructor) and `Net::default()`. Adding a third spelling means future callers can pick any of three for the same operation, and the type's "constructor surface" now reads as inconsistent.
- **Why it matters:** This is the first DEV task of a 50-task bundle. Whatever pattern lands here propagates. We should not normalise "spec-parity aliases" as an idiom — they explode into an N+1 maintenance problem (every spec citation might claim its own preferred constructor name).
- **Suggested fix (pick ONE):**
  - **(A) Preferred:** drop `Net::empty()`; in the Rustdoc for `Net::new()` add a `# Naming` paragraph noting that SPEC-20 §3.8 A7 spells the empty Net `Net::empty()` in prose and that the implementation chose `Net::new()` for parity with `Default::default()`. Update `tests/net_union.rs` lines 163, 188, 337, 342 and the test-spec doc to use `Net::new()`. Leave the integration-test comment that explains the rename so a reader chasing the spec sees the alias decision.
  - **(B) Alternative:** keep `Net::empty()` but mark `Net::new()` as the canonical constructor and route `Net::empty` through a `#[doc(alias = "new")]` attribute and an explicit "Prefer `Net::new` for new code; this exists for spec-text parity only" warning in the Rustdoc. Less clean than (A).
- **Severity rationale:** MEDIUM (not HIGH) because the duplication is harmless at runtime; the cost is API hygiene over a long bundle.

#### RV-002 — Test fixture `force_agent_id_range` reaches into `Net`'s public field surface; document the test-only invariant

- **Category:** Test Quality (encapsulation)
- **Location:** `relativist-core/tests/net_union.rs:79-99`
- **Evidence:** The helper writes directly to `net.agents`, `net.ports`, and `net.next_id`. These are `pub` on `Net` (see `core.rs:27, 32, 41`), so this compiles without unsafe — but it is precisely the "callers MUST NOT index into `agents` directly" pattern that `Net::get_agent`'s Rustdoc warns against (`core.rs:255-257`). This is a defensible test-only fixture *because* TASK-0411 (`remap_partition_ids`) will land the production-side equivalent and the fixture mimics its postcondition; however, the file does not say that.
- **Why it matters:** Without an explicit comment, the next task author who reads `net_union.rs` may copy this pattern into other integration tests, which would normalise direct `Net` field manipulation across the bundle. We want the opposite: every other test should drive `Net` through `create_agent` / `connect` / `remove_agent`.
- **Suggested fix:** Add a 3-line module-level comment at the top of `net_union.rs` (above the `use` block) stating: "The `force_agent_id_range` fixture below intentionally reaches into `Net`'s public fields to simulate the disjoint-`AgentId` postcondition that `remap_partition_ids` (TASK-0411, SPEC-04 amendment A4) will establish in production. Do NOT replicate this pattern in other integration tests — drive `Net` through its method API." Also consider gating the helper's body behind a `// TEST FIXTURE — see TASK-0411` banner comment so a future grep finds it.
- **Severity rationale:** MEDIUM because the fixture is correct and the tests pass; the risk is precedent leakage to future tasks.

#### RV-003 — `union`'s `freeport_redirects` collision policy is documented as "other wins" but the Rustdoc claim that "targets cannot reference colliding agents" is too strong

- **Category:** Code Quality (Rustdoc precision)
- **Location:** `relativist-core/src/net/core.rs:354-356, 439`
- **Evidence:** The Rustdoc says: "`freeport_redirects`: union of the two maps; if a key collides, `other` wins. Under the disjoint-`AgentId` precondition the targets cannot reference colliding agents." The first half is precise (`HashMap::extend` overwrites on key collision). The second half is misleading: the `freeport_redirects` map is keyed by **border `u32` ids** (not `AgentId`s), and the disjoint-`AgentId` precondition says nothing about disjoint border-id ranges (EC-3 explicitly tests duplicate border id 10 surviving). So when both sides happen to redirect the same border `u32`, `other`'s redirect target wins — and that target may be a `FreePort(other_border)` or an `AgentPort(other_owned_id, p)`. Both are legal under the precondition; the prose's "cannot reference colliding agents" reasoning is unrelated to why the collision policy is sound.
- **Why it matters:** SPEC-20 §4.2.2 expects `union` callers to round-trip the result through `split()`, which rebuilds border-id allocations from scratch (`PartitionPlan::allocate_border_ids`, A3). So `freeport_redirects` collisions during union are observationally invisible in production. But the Rustdoc currently sells a soundness argument that is wrong-by-construction; a future maintainer who relies on the stated invariant for a different caller will be misled.
- **Suggested fix:** Replace the second sentence with: "Border `u32` keys are NOT guaranteed disjoint (the disjointness precondition is on `AgentId`s, not on border ids — see EC-3). When the same border-id appears in both redirect maps, `other`'s entry overwrites `self`'s. This is observationally invisible because SPEC-20 §4.2.2 round-trips the union result through `split()`, which re-allocates all border ids via `PartitionPlan::allocate_border_ids` (SPEC-04 A3) and discards the inherited redirect map." Also consider adding a `tests/net_union.rs` UT that asserts a same-border collision is resolved as "other wins" so the policy is regression-proofed (call it `ec_4_union_freeport_redirect_collision_other_wins` and gate on the assumption that splits always follow).
- **Severity rationale:** MEDIUM because it is documentation-only, but documentation that misstates a soundness argument is exactly the kind of thing that bites at the next caller.

#### RV-004 — Port-array transfer relies on a precondition (slots owned by `other` are DISCONNECTED in `ports_a`) that holds by construction but is not asserted

- **Category:** Code Quality / Architecture (defensive contract)
- **Location:** `relativist-core/src/net/core.rs:432-436`
- **Evidence:** The "copy `other`'s port slots" loop writes any `target != DISCONNECTED` from `ports_b` into `ports[idx]`. The implementation's correctness assumes that every port-array index `idx` corresponding to an `other`-owned `AgentId` is `DISCONNECTED` in `ports_a` (otherwise `self`'s old wire would be silently overwritten by `other`'s wire, possibly orphaning a `self` port target). Under the disjoint-`AgentId` precondition, every index `idx ∈ [other_owned_id * 3 .. (other_owned_id + 1) * 3)` is unreachable from any `self`-owned agent's wires, so `ports_a[idx] == DISCONNECTED` by induction over `connect`. This is correct, but neither the code nor the Rustdoc states the invariant.
- **Why it matters:** TASK-0411 will land `remap_partition_ids`, and TASK-0412 will land `reconstruct(_, _, reclaimed)`. Both will compose with `union`. If a future caller violates the disjointness precondition in a way that the per-agent `debug_assert!` (line 415) does not catch — for example, by handing in two nets where `agents_a[5] == None` but `ports_a[15..18]` contain stale targets from a removed agent (which `remove_agent` zeroes — but a `set_port` followed by `remove_agent` cleanup edge case could leave a stale slot if the contract on `set_port` is bent) — the silent overwrite would corrupt the result without firing the agent-overlap assert.
- **Suggested fix:** Add a `debug_assert!` inside the port-copy loop that fires when `target != DISCONNECTED`, the slot is `< ports_a.len()`, and the original `ports_a[idx] != DISCONNECTED`. Also extend the `# Field handling` Rustdoc paragraph for `ports` to read: "for every index `idx` belonging to an `other`-owned `AgentId`, `ports_a[idx]` is `DISCONNECTED` by the disjoint-`AgentId` precondition; this is asserted under `debug_assertions`." If the assertion is judged too costly even in debug, document the invariant in the Rustdoc and add a single-shot `debug_assert!` outside the loop (e.g., on the first `other`-owned slot encountered).
- **Severity rationale:** MEDIUM because the invariant holds by construction in every current and planned caller (TASK-0411 / TASK-0412 / TASK-0440 / TASK-0443), so this is defence-in-depth, not a live bug.

### LOW

#### RV-005 — `Net::union`'s decomposition pattern (destructure both inputs) is idiomatic but worth pinning as the bundle norm

- **Category:** Code Quality (Rust idioms)
- **Location:** `relativist-core/src/net/core.rs:373-390`
- **Evidence:** Two `let Net { … } = …;` destructures consume both inputs. This is clean, avoids borrow-checker contention, and is the most natural way to express `(self, other) -> Net`. It also lets the compiler refuse to compile if a future Net field is added without explicit handling — which is the property SPEC-22 D-009 (`free_list: VecDeque<AgentId>`) will lean on.
- **Suggested fix:** No code change. Add a one-line comment above the first destructure: `// Destructure both inputs by value: forces every Net field to be accounted for, so a future SPEC-22 D-009 field-add fails to compile until handled.` This documents the *load-bearing* intent of the destructure (vs. a one-off micro-optimisation) and makes the SPEC-22 transition explicit.
- **Severity rationale:** LOW — pure documentation hint; the code is already correct.

#### RV-006 — `redex_queue` concatenation order (`self`'s first) is intentional but undocumented

- **Category:** Code Quality (Rustdoc precision)
- **Location:** `relativist-core/src/net/core.rs:438`
- **Evidence:** The Rustdoc says "`redex_queue`: the two queues are concatenated (`self`'s first)." This is correct (`redex_a.extend(redex_b)` puts `self`'s entries first), but the **observational consequence** is that during the next `reduce_n(budget)` call after a union, redexes from the `self` partition are dequeued before redexes from `other`. Because all entries are stale-checked at dequeue time (SPEC-02 R17), correctness is unaffected; ordering only changes which redex would have fired first under a `reduce_n` budget cap. SPEC-20 §4.2.2 immediately re-`split()`s the union result, so this never matters in production. But for `Net::union`'s callers outside SPEC-20 (none planned, but the API is `pub`), the ordering is observable.
- **Suggested fix:** Add to the `# Field handling` `redex_queue` bullet: "The interleaving (self's queue first) is observationally invisible under SPEC-20 §4.2.2's call pattern — the union result is immediately re-`split()`ed, and `split()` discards `redex_queue` entries — but is part of the public contract and asserted by the existing test suite via the multiset-equality checks (no UT pins the ordering directly because no caller depends on it)."
- **Severity rationale:** LOW — completeness of the contract.

#### RV-007 — Observability instrumentation on `union` is appropriate to defer, but record the deferral

- **Category:** Observability
- **Location:** N/A — `Net::union` has zero `tracing` / metrics emission
- **Evidence:** SPEC-20 §3.6 specifies metrics for the departure-recovery cycle (e.g., `partitions_reclaimed_total`, `union_calls_total`); the call site in §4.2.2 invokes `Net::union` on the recovery hot path. The current implementation emits no `tracing::trace!`. This is the correct decision for a pure-layer primitive in `src/net/` (no async, no I/O, no observability dependencies — see CLAUDE.md "Core layer is pure"), and the metric should be emitted at the §4.2.2 call site (TASK-0440 / TASK-0443), not inside `union`. But nothing in the present PR records that decision, so a future spec-critic round may flag it as missing.
- **Suggested fix:** Add a one-line `# Observability` paragraph to the Rustdoc: "This is a pure-layer primitive (SPEC-13 module boundaries: `net/` is sync, no `tracing`). Departure-recovery metrics (SPEC-20 §3.6 `union_calls_total`, etc.) MUST be emitted by the §4.2.2 call sites (TASK-0440 / TASK-0443), NOT inside this function."
- **Severity rationale:** LOW — preserves the architecture decision in code, not just in conversation.

The `# Precondition` block explicitly cites both SPEC-20 §4.2.2 and the upstream A4 + R13 mechanics that establish the precondition. ARG-006 P12 (mixed-trace recoverability) is named and verified against `docs/theory-bridge.md` lines 59-62. The Rustdoc reads like an executable summary of A7. Future tasks should match this density.

**Field enumeration is complete and explicit.** The implementation explicitly handles every one of the six current `Net` fields (agents, ports, redex_queue, next_id, root, freeport_redirects), and the Rustdoc lists each one with its merge policy. The destructure-both-inputs pattern (RV-005) makes this a compiler-enforced invariant — if SPEC-22 lands `free_list: VecDeque<AgentId>` later, the code stops compiling until a maintainer chooses a merge policy. This is the right shape for a primitive that will outlive its current callers.

**Test-spec mapping is 1:1.** Every UT-0410-NN / EC-N from TEST-SPEC-0410 has a function with a matching name; deviations (e.g., UT-0410-02's `root` semantics being `None` rather than `a.root` because of root-follows-self) are flagged in inline comments. Multiset / set-based equality functions (`live_agent_ids`, `live_symbols_multiset`, `collect_border_ids`, `collect_internal_wires`) keep tests robust to internal-ordering changes — important because future arena compaction (SPEC-22) may renumber.

**Borrow / ownership is idiomatic.** Consuming `self` and `other` by value is correct for an aggregating primitive; the destructure pattern avoids `clone`s. The implementation is allocation-light: a single `agents.resize` and `ports.resize`, two `extend`s, one move into the result struct.

Acknowledge in the SPEC-22 backlog.

This is **out of scope** for TASK-0410 (Tier 2 DEV) but should be tracked as an ESPECIALISTA-SPECS task before SPEC-02 ships its next revision.
