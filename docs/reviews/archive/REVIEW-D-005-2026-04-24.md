## Root cause

The minted CON-DUP agents land in the workers' partitions with their **principal ports at
`DISCONNECTED`**, not wired to `FreePort(100/101/110/111)`. Every downstream consumer treats
those agents as dead and drops them:

1. `cleanup_t1_violations` (grid_delta_integration_tests.rs:252–271) removes every agent
   whose principal is `DISCONNECTED` — all 4 minted CON-DUP agents qualify.
2. Even without cleanup, `merge::core::merge` (merge/core.rs:137–164) would fail step 3's
   border-restoration pass because the workers' `free_port_index` never learned the newly-
   promoted border_ids.

### Evidence trail (code citations)

**(a) `resolve_con_dup` routes all four external principal wires to `pending_new_borders`, not
to `local_wiring`** — border_resolver.rs:823–862 calls `emit_external_principal` four times;
for our fixture `t_c1..t_d2` are all `FreePort(10x)`, so `emit_external_principal` falls to
the `_ =>` arm (border_resolver.rs:955–974) and pushes a `PendingNewBorder`. The
`CommutationBatch.local_wiring` for each worker contains ONLY the intra-worker aux-aux
slot-marker wire (border_resolver.rs:870–871). **No principal-port entry exists in
`local_wiring`.**

**(b) `apply_pending_commutation` only wires what `local_wiring` provides** —
worker.rs:337–491. After `net.create_agent` (worker.rs:364) every port slot is
`DISCONNECTED` (net/core.rs:121 initialises new ports to `DISCONNECTED`). The wire loop
(worker.rs:386–457) processes only `local_wiring` entries. Principal slot 0 of each minted
agent is therefore never touched — it stays `DISCONNECTED`.

**(c) `register_minted_agents` updates `border_graph` but never the workers** —
border_graph.rs:686–763. Promotion to `add_border_states` (border_graph.rs:559–599) touches
`self.borders`, `self.worker_borders`, and `self.active_redexes`, but emits NO `BorderDelta`
/ `new_borders` back to the workers. The workers' `Partition.free_port_index` never learns
the new border_ids, and their `subnet` never gets the principal-to-`FreePort(bid)` wire.

**(d) Convergence breaks the loop BEFORE the next-round dispatch materialises the borders**
— grid.rs:597 (strict) and grid.rs:639 (lenient) both call `check_delta_convergence` after
Round 2. At that point `border_graph.active_redexes` is empty (the 6 newly-added borders are
`FreePort`-vs-`AgentPort`, not principal-vs-principal — see `is_principal_pair` at
border_graph.rs:583), `local_redexes == 0` (no firings — all principals are DISCONNECTED),
and `has_border_activity == false` (the minted principals are on non-border `FreePort`s
outside `border_id_start..border_id_end`, so they don't hit `free_port_index` —
helpers.rs:92–97). Predicate returns `true`, loop breaks. **Round 3, which would have
carried the promoted borders via `new_borders`, never fires.**

**(e) Final merge can't rescue** — `run_grid_delta_final_collect` (grid.rs:806–850)
reconstructs the plan from the collected (worker-side) partitions. Merge step 3
(core.rs:137–164) reads each partition's `free_port_index` for the border_id; since neither
worker knows the new border_ids, both `current_a` and `current_b` are `None` and the
`(None, None)` branch silently drops the wire. Even if `cleanup_t1_violations` were removed,
the minted agents would remain with `DISCONNECTED` principals in the merged net.

### Refuted hypotheses

- **H1 (cleanup false-positive on live principals)** — PARTIALLY TRUE but misidentified
  root. The cleanup IS dropping the agents, but the premise "minted principals land with
  FreePort(BORDER_ID)" is wrong: they land DISCONNECTED because (a)/(b)/(c) above never wire
  them. Fixing cleanup alone is insufficient (merge still can't restore the wire).
- **H2 (strict vs lenient asymmetry in `package_resolutions_with_pending`)** — REFUTED. The
  iteration-order swap confirmed both modes fail identically at the same assertion with the
  same diff. `commutation_batch_to_pending` has no strict/lenient branch
  (border_resolver.rs:387–409). The bug is orthogonal to BSP mode.
- **H3 (merge drops cross-partition minted edges via stale `BorderGraph`)** — REFUTED. The
  coordinator's `border_graph.borders` DOES contain the 6 promoted borders (phase 4 of
  `register_minted_agents`, border_graph.rs:758–760 calls `add_border_states`). The bug is
  on the WORKER side — their `free_port_index` is never told.
- **H4 (slot-marker residue leaks into `Partition.subnet`)** — REFUTED.
  `apply_pending_commutation` decodes clause-3 slot markers before calling `Net::connect`
  (worker.rs:415–430). The marker never reaches the subnet. Only the clause-6 rejection
  path (`FreePort` in local_wiring) would propagate a `MalformedLocalWiring` — that path
  isn't hit because the resolver correctly avoids emitting FreePort in local_wiring for
  CON-DUP.

## Fix surface

**Primary fix (recommended)** — materialise the promoted borders into the workers' subnets
so their principals are wired at the moment the coordinator promotes a pending border.

- **File:** `relativist-core/src/merge/grid.rs`, function `run_grid_delta_inner`, lines
  **~550–562** (immediately after `register_minted_agents` and before the existing
  `apply_deltas` loop).
- **Change sketch:** after `register_minted_agents` resolves `PendingNewBorder`s into
  `border_graph.borders`, synthesise for each newly-added border_id a `new_borders` entry
  `(border_id, side_a_port)` for `worker_a` and `(border_id, side_b_port)` for `worker_b`
  (where `side_x_port` is the concrete `PortRef` from `BorderState.side_a/b`). Apply each
  via `apply_border_deltas_to_partition(partitions_vec[w], &[], &[], &[(bid, side)])`. Also
  apply the mirror operation to the worker-side partition that `handle_final_state_request`
  will eventually return — the simplest way is to plumb the same per-worker `new_borders`
  list into a new `WorkerDispatch::dispatch_apply_promoted_borders` entry point, OR fold it
  into the existing `dispatch_round_start` as a final "reconciliation round" before
  `FinalStateRequest`. Developer picks the cleaner option at Stage 6.
- **Alternative (smaller blast radius, test-only):** in
  `LocalDeltaDispatch::dispatch_final_state_request`
  (grid_delta_integration_tests.rs:201–238), before invoking `handle_final_state_request`,
  patch each worker's `ctx.delta_state.partition` with the
  coordinator-view new_borders for that worker. Requires plumbing `&BorderGraph` into the
  dispatch. This fixes the test but leaves the future TCP path broken — **insufficient for
  close**; flagged as NIT.

**Secondary diagnostic surface** — the test harness's `cleanup_t1_violations`
(grid_delta_integration_tests.rs:252–271). Once the primary fix lands, minted principals
will be wired to `FreePort(border_id)` (not `DISCONNECTED`), so cleanup becomes a no-op for
these fixtures. No functional change needed, but its existence is a SMELL — it masks the
protocol gap diagnosed here. After the fix, consider whether it's still needed (NIT).

**Out of scope for this bundle**: redesigning the Round-N+2 / N+3 protocol ordering. The
spec (SPEC-19 §3.3 R23/R23a, §3.6 R48/R48a/R48b) is silent on how the coordinator notifies
workers of borders promoted by `register_minted_agents`. This is a latent spec ambiguity;
flag in NITs for spec-critic.

## Blast radius

The fix touches one loop body in `run_grid_delta_inner` after `register_minted_agents`. Tests
that could regress:

| Test area | Fixtures | Risk |
|-----------|----------|------|
| CON-CON / DUP-DUP / ERA-ERA (symmetric) | no mints → no promoted borders → new loop body is no-op | **none** |
| `run_grid_delta_*` strict/lenient convergence tests (grid.rs:1490 onwards) | no CON-DUP redexes in these fixtures | **none** |
| `BorderGraph::register_minted_agents` unit tests (TASK-0398) | operate on synthetic graphs without workers | **none** |
| `apply_pending_commutation` unit tests (worker.rs:2102–2616) | operate on raw `Net`, not partitions | **none** |
| UT-0385-06, UT-0385-07 (existing CON-CON G1 parity) | symmetric → no mints | **none** |
| UT-0385-08 (this file) — 6 fixtures × 2 modes | CON-CON/DUP-DUP/ERA-ERA unaffected; CON-DUP/CON-ERA/DUP-ERA all share this bug and will flip to green | **positive** |

Guard against regression: ensure the new synthesis only fires when
`register_minted_agents` actually promoted a border (i.e. `resolved_entries` was non-empty
in that call). Stage 5 adds explicit probes (see below).

## NITs

- **NIT-1:** The orchestration plan's claim "CON-DUP strict=true passes" is wrong. Both
  modes fail. Plan prose should be corrected when Stage 6 closes the bundle.
- **NIT-2:** `cleanup_t1_violations` (grid_delta_integration_tests.rs:252) is a
  test-harness workaround for a protocol gap, not a proper fix. After the primary fix
  lands, revisit whether it can be deleted or at least downgraded to a `debug_assert!(
  dead_ids.is_empty(), ...)` — T1 violations surfacing in the FINAL partition indicate
  protocol drift.
- **NIT-3:** SPEC-19 §3.3 is silent on how the coordinator propagates to workers the
  borders that `register_minted_agents` promoted. The fix is forced to invent the protocol.
  Flag for a spec-critic round: SPEC-19 §3.3 needs an R23b (or R24.1.6d) stating that
  promoted borders MUST be issued via `new_borders` in the same round as the mint echo is
  consumed, OR a dedicated `Message::BordersPromoted` handshake before FinalStateRequest.
- **NIT-4:** `register_minted_agents` (border_graph.rs:686) returns `Result<(), GridError>`
  but the caller at grid.rs:552 `?`s straight through — losing the per-worker context for
  diagnosis. Consider wrapping the error with the worker id and round number. Not blocking.
- **NIT-5:** `commutation_batch_to_pending` (border_resolver.rs:387) is marked
  `#[allow(dead_code)]` with a stale comment "Consumed by TASK-0403". It now has a live
  consumer (`LocalDeltaDispatch`); drop the allow and the comment when touching this area.
