The brief (D-011 Phase F-1) defines three explicit Round-1 questions:

## 2. Findings

### F1 (HIGH) — R18a measurement-point ambiguity for the streaming path

**Section:** R18a, second sentence.
**Issue:** R18a says the value is captured "AFTER the streaming pipeline has finished producing all `AgentBatch`es and the worker arenas are populated". This phrase has two possible interpretations:

- (a) After the LAST `AgentBatch` has been generated by the generator iterator, regardless of whether the partitioner has ingested it.
- (b) After the partitioner's `generate_and_partition_chunked_with_chunk_size_and_lifetime` returns and all worker partitions are finalized.

The two points are NOT equivalent in the streaming path. Between (a) and (b), the partitioner accumulates per-partition `Vec<Option<Agent>>` (or `SparseNet` per SPEC-21 §4.9 SC-006 closure) which becomes a non-trivial memory allocation — for `dual_tree(50000)` at `chunk_size=10000`, the partitioner-side accumulators dominate over the per-batch transient cost.

If the harness samples at point (a), R18a underestimates construction memory; if at point (b), it overestimates relative to a hypothetical "pure streaming dispatch" architecture but it correctly reflects the actual implementation path (which is the path the M5 gate validates).

**Severity:** HIGH. The §4.9.2 STREAMING acceptance gate ("R18a bounded by O(chunk_size + |retained_borders|)") is unfalsifiable without disambiguation.

**Resolution required:** R18a must explicitly fix point (b) — AFTER `generate_and_partition_chunked_*` returns and BEFORE the first `AssignPartition` is dispatched.

### F2 (HIGH) — R37c reference-net allocation poisons R18a

**Section:** R37c second paragraph (the "the reference eager net MUST be discarded" clause).
**Issue:** R37c requires construction-phase isomorphism to a reference eager net. If the reference net is allocated BEFORE R18a's sampling point on the streaming path, then `peak_memory_during_construction` reflects the SUM of (streaming arena + reference eager net) — exactly the memory profile R18a is supposed to refute. The current spec text acknowledges this and offers two escape hatches: "perform the construction-isomorphism check in a separate measurement run, or sample VmHWM BEFORE the reference net is allocated and assert the watermark is monotone-increasing in subsequent samples."

The first escape hatch (separate run) is operationally heavy: it doubles the rodada wall-clock and requires the harness to thread two distinct execution paths through one `Benchmark::execute` call. The second escape hatch (monotone watermark assertion) is mathematically correct but the spec does not say which sample populates `BenchmarkResult.peak_memory_during_construction` — the pre-reference-allocation value? The post-reference-but-pre-reduction value?

**Severity:** HIGH. Implementer ambiguity will produce Tier 3 numbers that do not match SPEC-21 R10's contract.

**Resolution required:** R37c must explicitly state that R18a is sampled BEFORE the reference net is allocated. The reference net's allocation MUST happen AFTER R18a is captured.

### F3 (MEDIUM) — `agent_count_at_construction_complete` ill-defined for SparseNet under streaming

**Section:** R18c, third bullet.
**Issue:** R18c says: "Sparse path (`representation == Sparse`): `SparseNet::agents.len() as u64`". But the spec also permits `chunk_size.is_some()` AND `representation == Sparse` simultaneously (no exclusion is stated), in which case the construction is a streaming pipeline that produces a `SparseNet` accumulator. R18c does not say whether to sum across `AgentBatch.agents.len()` (streaming bullet) or use `SparseNet::agents.len()` (sparse bullet) when both apply. These values WILL differ if the streaming pipeline emits forward references that are later resolved by overwriting; the streaming-sum overcounts.

**Severity:** MEDIUM. Phase D (SparseNet micro-bench) is dual_tree-only and does not exercise streaming, so the conflict does not bite in the production rodada — but the spec is internally underspecified.

**Resolution required:** Either (a) explicitly forbid `chunk_size.is_some() && representation == Sparse` in v2 (deferred to a future spec), OR (b) define R18c as `SparseNet::agents.len()` when `representation == Sparse`, regardless of `chunk_size`. Option (b) is preferred because it preserves orthogonality of the two axes.

### F4 (MEDIUM) — CSV column-order spec is duplicated against R39a; risk of drift

**Section:** R18 backward-compat block (the "joinability gate" subsection).
**Issue:** The amendment block enumerates the v1 column order verbatim — but R39a (line 595) ALSO enumerates the same column order. Two copies of the same list will drift. If a future amendment changes R39a's column list, the joinability gate's frozen list will silently no longer match.

**Severity:** MEDIUM. Maintenance hazard.

**Resolution required:** The joinability gate SHOULD reference R39a as the source of truth for the v1 column order, not duplicate it. The new column order (R18a–R18g additions) MUST be added explicitly to R39a's schema.

### F5 (LOW) — `RecyclePolicy` ownership cycle vs. SPEC-22 §3.1

**Section:** §4.1 (the new enum) and R18g.
**Issue:** R18g says `RecyclePolicy` is "owned by SPEC-22 §3.1 (R10a) and re-exported into the `bench` module". But SPEC-22 §3.1 R10a does not define a Rust enum named `RecyclePolicy` — it specifies the BEHAVIOR of two recycling postures. The Rust enum is introduced for the FIRST time by this amendment to SPEC-09 §4.1. SPEC-22 owns the SEMANTICS but not the TYPE.

**Severity:** LOW. Ownership ambiguity. Resolvable by either moving the enum to SPEC-22 or rephrasing SPEC-09 to claim co-ownership.

**Resolution required:** Add a sentence clarifying that the `RecyclePolicy` enum is FIRST DEFINED HERE (SPEC-09 §4.1) for cross-module use, with semantics governed by SPEC-22 R10b/R10c. Wire-format / persistence concerns (if any) defer to SPEC-22.

### F6 (LOW) — `max_pending_lifetime` field defined but never crosswalked to SPEC-21

**Section:** §4.1 `BenchmarkSuiteConfig::max_pending_lifetime`.
**Issue:** The field's docstring says "Honored only when `chunk_size.is_some()`" but §4.9.1 only mentions that the bench harness MUST propagate the value into `merge::generate_and_partition_chunked_with_chunk_size_and_lifetime`. It does not explain what happens if `chunk_size == None` — the harness silently ignores the field? The harness asserts the field equals its default? This is a minor implementer ambiguity.

**Severity:** LOW.

**Resolution required:** Add one sentence to §4.9.1 stating that `max_pending_lifetime` MUST be ignored (not validated, not propagated) when `chunk_size == None`.

### F7 (LOW) — Acceptance gate "no statistically significant regression" is procedurally underspecified

**Section:** §4.9.2 EAGER acceptance gate.
**Issue:** The gate says "no statistically significant regression" and defines it as "the Tier 3 median is within the v1 bootstrap 95% CI band". But `v1_local_baseline/phase2_summary.csv` does not currently include bootstrap CI columns (R32a is SHOULD, and the v1 campaign deferred CI computation to Python post-processing). If the v1 summary lacks `wall_clock_ci95_lo/hi` columns, the gate has no v1 reference band to test against.

**Severity:** LOW (operational, not spec-correctness). The gate is still meaningful if the post-processing step computes the v1 CI from `phase2_detail.csv` raw data.

**Resolution required:** Add a footnote stating that if `v1_local_baseline/phase2_summary.csv` lacks CI columns, the post-processing step MUST recompute them from `phase2_detail.csv` per R32a's offline-bootstrap escape hatch.

### F8 (INFO) — SPEC-22 §3.6 cross-reference target is mislabeled in the brief

**Section:** SPEC-22 cross-reference (insertion point).
**Issue:** The D-011 Phase F-1 brief says "append to SPEC-22 §3.6". But SPEC-22 has no §3.6 — its sections are §3.1 (free-list), §3.2 (SparseNet), §3.3 (invariants), §3.4 (config), §3.5 (cross-cutting MUSTs), §3.8 (amendments). The brief intended §3.1 (free-list lifecycle is governed there). The cross-reference was inserted at the end of §3.1 (after R12).

**Severity:** INFO. Brief-vs-spec mismatch, not a spec defect.

**Resolution:** Documented in the closure log; no spec change required.

## 3. Answers to the brief's three explicit questions

### Q1: Does R37c eliminate the streaming-bug-masking risk?

**Answer: PARTIALLY — pending F2 resolution.**

R37c correctly mandates a structural-equivalence gate at the construction boundary, which is the right place to catch bugs that silently produce a structurally-different but reducible net. The choice of `nets_isomorphic` (with R37a's lightweight fast-path on >5000 agents) is sound and consistent with SPEC-08.

However, F2 identifies a measurement-protocol contradiction: the reference eager net's allocation poisons R18a unless explicitly sequenced. Until F2 is resolved, the bench harness implementer faces a choice between (a) honoring R37c (correctness check) at the cost of inflating R18a, or (b) deferring R37c to a separate run at the cost of doubling rodada wall-clock. Neither choice is unambiguous from the current spec text.

### Q2: Are R18a–R18g sufficient to validate the M5 acceptance gate?

**Answer: YES — pending F1 resolution.**

R18a (peak construction memory), R18c (input-size invariant), and the §4.9.2 STREAMING gate's empirical scaling protocol (double `input_size` at fixed `chunk_size`, observe R18a within 2× envelope) are sufficient to validate SPEC-21 R10/R12's "memory scales with `chunk_size`" contract. R18d adds working-set watermark for the reduction phase, which validates SPEC-22 R32's free-list memory budget.

The only blocker is F1: until R18a's measurement point is disambiguated, the empirical envelope cannot be evaluated meaningfully.

### Q3: Are the new CSV columns backward-compatible with v1_local_baseline join keys?

**Answer: YES.**

The composite join key `(benchmark, input_size, workers, mode, repetition)` consists of the leftmost 5 columns of both schemas. The amendment preserves the v1 column order verbatim as the leftmost 22 columns; the 7 new columns are appended to the right. A pandas `pd.merge(on=['benchmark','input_size','workers','mode','repetition'])` will work without any column-mapping shim. F4 raises a maintenance concern (duplicated column list) but does not threaten correctness.
