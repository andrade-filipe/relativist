# TASK-0356: `send_frame` archive path (hot-path only, optional LZ4)

**Spec:** SPEC-18 §3.5
**Requirements:** R22, R23
**Priority:** P0 (critical path)
**Status:** TODO
**Depends on:** TASK-0353, TASK-0354
**Blocked by:** TASK-0353, TASK-0354
**Estimated complexity:** M (~100 LoC)
**Bundle:** SPEC-18 §3.5 (item 2.24) — Zero-Copy Archive

## Context

Adds the rkyv send path to `send_frame_with_threshold`. When the caller opts
in via a new parameter `use_zero_copy: bool` (sourced from
`TransportConfig.use_zero_copy` per TASK-0358) AND the message is one of
the two hot-path variants (`AssignPartition`, `PartitionResult`) per R22,
the payload is serialized via `rkyv::to_bytes::<rkyv::rancor::Error>(&value)`
and the `FLAG_ARCHIVED` bit is set in the frame header. Optional LZ4
compression (R23) is applied on top of the archived bytes when their size
is `>= compression_threshold` (combination of FLAG_ARCHIVED + FLAG_COMPRESSED
is explicitly permitted by R15).

For control messages (`Shutdown`, `Error`, `Register`, `RegisterAck`,
`RegisterNack`) per R22, the rkyv path MUST NOT be used even when
`use_zero_copy = true`; those messages always go through bincode v2.
Bincode is also the only path when `cfg(not(feature = "zero-copy"))` —
in that build the entire archive branch does not compile and `use_zero_copy`
is unused.

## Acceptance Criteria

- [ ] `relativist-core/src/protocol/frame.rs` introduces a new public
      function `send_frame_v2(writer, message, compression_threshold,
      use_zero_copy)` (or extends `send_frame_with_threshold` with a
      `use_zero_copy: bool` argument — task-splitter prefers a new function
      to keep the existing API stable; final shape per developer + reviewer).
- [ ] When `cfg(feature = "zero-copy")` AND `use_zero_copy == true` AND
      `message` is `Message::AssignPartition { partition, .. }` OR
      `Message::PartitionResult { partition, .. }`:
  - Call `rkyv::to_bytes::<_, rkyv::rancor::Error>(partition)` to get an
    `AlignedVec<u8>` of archived bytes. On failure, map the error via
    `ProtocolError::ArchiveValidationFailed(format!("serialize: {}", e))`
    (DC-4 mandated prefix — spec-critic 2026-04-16).
  - Compute CRC32C over those archived bytes (R12 — checksum over the
    uncompressed payload, where "uncompressed" here means
    "pre-LZ4-compression"; the rkyv archive itself IS the uncompressed
    payload from the wire's perspective).
  - If archived size `>= compression_threshold` AND
    `compression_threshold != usize::MAX`, apply
    `compress_payload(&archived)` and set `FLAG_COMPRESSED | FLAG_ARCHIVED`
    in the flags byte. Otherwise, set `FLAG_ARCHIVED` only.
  - **Note:** `AssignPartition` and `PartitionResult` carry MORE than just
    the `Partition` (round number, stats). The rkyv path MUST archive the
    enclosing struct, not just the inner field. Developer must define a
    small wire wrapper struct (e.g., `ArchiveAssignPayload { round: u32,
    partition: Partition }` and `ArchivePartitionResultPayload { round: u32,
    partition: Partition, stats: WorkerRoundStats }`) under
    `#[cfg(feature = "zero-copy")]` with the rkyv derives, so `to_bytes`
    serializes the full wire payload of the message. The bincode v2 path
    continues to use the `Message` enum directly.
- [ ] When `cfg(feature = "zero-copy")` AND `use_zero_copy == true` AND
      `message` is a control variant (Shutdown/Error/Register/RegisterAck/
      RegisterNack) per R22: the rkyv branch is NOT taken; falls through
      to bincode v2 (existing behaviour). No error — silent fallback per
      §3.5 first sentence ("MAY be used").
- [ ] When `cfg(not(feature = "zero-copy"))`: the function compiles without
      the rkyv branch; `use_zero_copy` argument may be dead-code (`#[allow(
      unused_variables)]`) or a `cfg`-gated parameter (developer choice;
      simpler to keep the parameter and `#[allow(dead_code)]` it).
- [ ] The bare `send_frame(writer, message)` function and
      `send_frame_with_threshold(writer, message, threshold)` keep their
      existing signatures and continue to call the bincode v2 path
      (use_zero_copy = false default). Backward compatibility for all
      existing call sites.
- [ ] `cargo build --workspace` succeeds (default features).
- [ ] `cargo build --workspace --features zero-copy` succeeds.
- [ ] `cargo test --workspace` count stays at 887+ (no regression in
      bincode path).
- [ ] `cargo test --workspace --features zero-copy` count grows by 4-6
      tests added in this task (covered by TEST-SPEC-0356).
- [ ] Clippy + fmt clean both ways.
- [ ] No `unwrap()`, no `unsafe`, no `println!`.
- [ ] Existing `send_frame` tests (T6-T8 round-trips, R5 CRC-on-uncompressed,
      etc.) all stay green.

## Key Types / Signatures

```rust
#[cfg(feature = "zero-copy")]
#[derive(rkyv::Archive, rkyv::Serialize, rkyv::Deserialize)]
struct ArchiveAssignPayload {
    round: u32,
    partition: Partition,
}

#[cfg(feature = "zero-copy")]
#[derive(rkyv::Archive, rkyv::Serialize, rkyv::Deserialize)]
struct ArchivePartitionResultPayload {
    round: u32,
    partition: Partition,
    stats: WorkerRoundStats,
}

pub async fn send_frame_v2<W: AsyncWriteExt + Unpin>(
    writer: &mut W,
    message: &Message,
    compression_threshold: usize,
    use_zero_copy: bool,
) -> Result<usize, ProtocolError> {
    #[cfg(feature = "zero-copy")]
    if use_zero_copy {
        match message {
            Message::AssignPartition { round, partition } => {
                let payload = ArchiveAssignPayload {
                    round: *round,
                    partition: partition.clone(),
                };
                let archived = rkyv::to_bytes::<rkyv::rancor::Error>(&payload)
                    .map_err(|e| ProtocolError::ArchiveValidationFailed(
                        format!("serialize: {}", e)
                    ))?;
                return send_archive_frame(writer, archived.as_slice(),
                    compression_threshold).await;
            }
            Message::PartitionResult { round, partition, stats } => {
                let payload = ArchivePartitionResultPayload {
                    round: *round,
                    partition: partition.clone(),
                    stats: stats.clone(),
                };
                let archived = rkyv::to_bytes::<rkyv::rancor::Error>(&payload)
                    .map_err(|e| ProtocolError::ArchiveValidationFailed(
                        format!("serialize: {}", e)
                    ))?;
                return send_archive_frame(writer, archived.as_slice(),
                    compression_threshold).await;
            }
            // R22: control messages always fall through to bincode.
            _ => {}
        }
    }
    // Default path: bincode v2 (existing send_frame_with_threshold body).
    send_frame_with_threshold(writer, message, compression_threshold).await
}

#[cfg(feature = "zero-copy")]
async fn send_archive_frame<W: AsyncWriteExt + Unpin>(
    writer: &mut W,
    archived: &[u8],
    compression_threshold: usize,
) -> Result<usize, ProtocolError> {
    let checksum = crc32fast::hash(archived);
    let (payload, flags): (Vec<u8>, u8) =
        if archived.len() >= compression_threshold && compression_threshold != usize::MAX {
            (compress_payload(archived), FLAG_COMPRESSED | FLAG_ARCHIVED)
        } else {
            (archived.to_vec(), FLAG_ARCHIVED)
        };
    // ... write 9-byte header + payload (mirror existing send_frame body) ...
}
```

  `send_frame_with_threshold` output (regression check).

## Notes

- **`rkyv::rancor::Error` to `ProtocolError` — APPROVED by spec-critic
  DC-4 (2026-04-16):** map rkyv send-side failures to
  `ProtocolError::ArchiveValidationFailed(format!("serialize: {}", e))`.
  The mandatory `"serialize: "` prefix recovers the diagnostic value of
  distinguishing send-side from recv-side failures through the reason
  string (log scrapers can split on the prefix). Rationale:
  - The conflation introduces no observable wire-format difference
    (send-side failures never produce a frame on the wire).
  - Rkyv send-side failures are pathological (OOM during alignment
    padding, unsupported field type that slipped past TASK-0353) — a
    dedicated variant would not pay for itself.
  - A separate `ArchiveSerializeFailed(String)` variant would require a
    spec amendment to R26/R35 to authorise the new variant; spec-critic
    explicitly rejected this option to avoid blocking the bundle.
  See `docs/spec-reviews/SPEC-18-section-3.5-design-choices-2026-04-16.md`
  DC-4 for the full rationale. The recv-side prefix vocabulary
  (`"access: "`, `"deserialize: "` in TASK-0357) is NOT mandated to
  unify with `"serialize: "` — different prefixes per direction are
  acceptable.
- **Why a separate `ArchiveAssignPayload` wrapper struct:** `Message` is
  an enum with serde derives. Deriving rkyv's `Archive` on `Message` would
  archive the discriminant, but the wire path uses the explicit
  `FLAG_ARCHIVED` bit + restricted variant set (R22) to communicate the
  message kind — there is no need to archive the enum. A small wrapper
  struct per hot-path variant is simpler than rkyv-deriving the entire
  `Message` enum, and matches SPEC-18 §4.6's intent ("rkyv serializes the
  Partition directly").
- **Why `partition.clone()` in the construction:** `to_bytes` takes
  `&value`, but the wrapper struct owns its fields. The clone copies the
  `Partition` once; for a multi-MB partition this is ~10ms — comparable to
  the LZ4 compression that follows. If profiling shows this is a hot
  bottleneck, the wrapper can be redesigned to borrow (`&'a Partition`)
  with rkyv's `with` adapters, deferred as a v2.1 optimization.
- **Backward compatibility:** the existing `send_frame(writer, msg)` and
  `send_frame_with_threshold(writer, msg, t)` are unchanged. New callers
  (e.g., `protocol/coordinator.rs`, `protocol/worker.rs`) opt in by
  switching to `send_frame_v2(_, _, t, transport_cfg.use_zero_copy)`. The
  switch in those modules is **out of scope** for this task and falls
  under TASK-0358 wiring.
