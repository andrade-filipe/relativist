# TASK-0103: Define RelativistError top-level error type

**Spec:** SPEC-13
**Requirements:** R15, R16, R17, R18 (SPEC-13)
**Priority:** P0 (critical path)
**Status:** TODO
**Depends on:** Phase 5, TASK-0081
**Blocked by:** none
**Estimated complexity:** M (50-200 LoC)

## Context

SPEC-13 R17 requires a top-level `RelativistError` that unifies all per-module error enums via `#[from]` conversions. The current `error.rs` has a minimal `RelError` with string-based variants. This task replaces it with the full typed error hierarchy: `NetError`, `ReductionError`, `PartitionError`, `MergeError`, `ProtocolError`, `CoordinatorError`, `WorkerError`, and the umbrella `RelativistError`. Per SPEC-13 R18, each error must be classifiable as transient or fatal.

This task defines the enums and the `exit_code()` method that maps error categories to exit codes (SPEC-07 R43): 0=success, 1=config, 2=communication, 3=internal.

## Acceptance Criteria

- [ ] `RelativistError` enum is defined with `#[from]` variants for all 7 module errors plus `Io` and `Config`
- [ ] `RelativistError` derives `Debug` and `thiserror::Error`
- [ ] `RelativistError::exit_code()` returns `1` for Config, `2` for Protocol/Coordinator(Protocol)/Worker(Protocol), `3` for all others
- [ ] Per-module error enums (`NetError`, `ReductionError`, `PartitionError`, `MergeError`) are defined as stubs (at minimum the variants from SPEC-13 R16)
- [ ] `CoordinatorError` and `WorkerError` enums are defined with `#[from]` for their sub-errors
- [ ] All error enums derive `Debug` and `thiserror::Error`
- [ ] Old `RelError` is replaced (or aliased) so existing code compiles

## Key Types / Signatures

```rust
use thiserror::Error;

/// Top-level error type for the Relativist binary.
#[derive(Debug, Error)]
pub enum RelativistError {
    #[error(transparent)]
    Net(#[from] NetError),
    #[error(transparent)]
    Reduction(#[from] ReductionError),
    #[error(transparent)]
    Partition(#[from] PartitionError),
    #[error(transparent)]
    Merge(#[from] MergeError),
    #[error(transparent)]
    Protocol(#[from] ProtocolError),
    #[error(transparent)]
    Coordinator(#[from] CoordinatorError),
    #[error(transparent)]
    Worker(#[from] WorkerError),
    #[error("I/O error: {0}")]
    Io(#[from] std::io::Error),
    #[error("configuration error: {0}")]
    Config(String),
}

impl RelativistError {
    /// Map error to process exit code (SPEC-07 R43).
    pub fn exit_code(&self) -> i32 {
        match self {
            Self::Config(_) => 1,
            Self::Protocol(_) | Self::Io(_) => 2,
            Self::Coordinator(CoordinatorError::Protocol(_)) => 2,
            Self::Worker(WorkerError::Protocol(_)) => 2,
            _ => 3,
        }
    }
}
```

## Notes

- SPEC-13 R18 classifies errors as Transient (retryable) or Fatal. The `exit_code()` method is a simplification for the CLI layer. The coordinator FSM (Phase 6 tasks below) will need richer classification; that logic lives in the coordinator module, not here.
- The old `RelError` name is replaced by `RelativistError`. If other modules import `RelError`, a `pub type RelError = RelativistError;` alias should be provided temporarily.
- Per-module errors may eventually move to their respective modules (e.g., `NetError` in `net/error.rs`). For now, centralizing them in `error.rs` is acceptable and simpler.
