## Context

Implements the resolution entry point
`resolve_border_redex(&mut BorderGraph, partitions: &[Partition], border_id)`
for the three SAME-SYMBOL rules:

| Rule | `SpecificRule` | Agents consumed | New agents | Border delta |
|------|----------------|-----------------|------------|--------------|
| CON-CON | Anni (cross) | 2 | 0 | `remove_border(bid)`; reconnect a.1↔b.2 and a.2↔b.1 through workers |
| DUP-DUP | Anni (parallel) | 2 | 0 | `remove_border(bid)`; reconnect a.1↔b.1 and a.2↔b.2 through workers |
| ERA-ERA | Void | 2 | 0 | `remove_border(bid)`; no new connections |

All three produce zero new cross-partition borders (no commutation
expansion). The emitted `BorderDelta`s describe the updates the two
workers must apply to their own partitions (the auxiliary ports of the
consumed agents are reconnected to whatever the OTHER agent's auxiliary
ports pointed to — which may be local (`AgentPort` in the same
partition) or may cross back to the other worker via an existing
border).

**Scope:** ship the three same-symbol rules + the top-level
`resolve_border_redex` entry point that dispatches on `Symbol` pairs.
CON-DUP / CON-ERA / DUP-ERA (commutation + asymmetric erasure) ship in
TASK-0374.

## Notes

- **Why the resolver does NOT directly call `interact_*`:** the
  `interact_*` functions mutate a `Net`. Here we don't have a
  unified net — the two agents live in two different `Partition`s
  that the coordinator doesn't "own" mutably. The resolver simulates
  the effect by reading auxiliary-port targets and emitting deltas
  that instruct each worker to apply the mutation on its side. This
  preserves R14's "same 6 interaction rules" guarantee by mirroring
  the pattern, without crossing the ownership boundary.

  **Alternative considered:** build a temporary `Net` with copies of
  the two agents, call `interact_*`, then diff back. Rejected —
  introduces agent-ID renumbering complexity. The simulate-and-diff
  path is simpler and has the same observable outcome.

- **Why deltas to BOTH workers on CON-CON:** the cross pattern
  swaps the auxiliary-port targets. Worker A's auxiliary ports
  (a.1, a.2) now point to whatever b.2 and b.1 used to point to —
  which are live targets inside Worker B's partition (or themselves
  cross borders). Worker A needs to know this; Worker B symmetrically.

- **ERA-ERA emits zero deltas** because ERA is arity 0 — no
  auxiliary ports to reconnect. The only effect is the border
  disappears.

## Design Choices Flagged for Spec-Critic (Stage 1.5)

- **DC-B3 — delta content when an auxiliary port of agent `a` points
  to a LOCAL port in Worker A's partition.** The delta for Worker A
  needs to describe the reconnection between a.1's former target and
  a.2's former target (after cross pattern). But these targets are
  both *local* to Worker A — there's no "border" to rewrite. Does
  Worker A need a `BorderDelta` for this local reconnection, or is a
  different delta type needed? Proposal: extend `BorderDelta` with an
  optional local-reconnection list, OR ship a separate
  `LocalReconnection { a: PortRef, b: PortRef }` delta type. **Flag
  for spec-critic.**

- **DC-B4 — race with local reduction.** If Worker A has already
  locally consumed agent `a` between emitting its `has_border_activity
  = true` and the coordinator detecting the redex, the resolver's
  `materialize_agent` will return `None` for that side. Current plan
  (this task): panic. Spec-critic should rule whether the delta
  protocol is supposed to prevent this race by strict BSP
  synchronization (answer likely yes), or whether the resolver needs
  to be race-tolerant (unexpected).
