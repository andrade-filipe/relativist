## Context

TASK-0367, 0368, 0369 shipped the bincode-layer correctness for the
five new `Message` variants. This task verifies the frame-layer
behaviour per SPEC-18 §3.5 integration:

- R34: CRC32 integrity — every encoded frame passes the existing
  `recv_frame` CRC check (which is already enforced by SPEC-18's
  frame impl; this task adds explicit tests hitting it for the 5
  new variants).
- R35: large-payload variants (`InitialPartition`, `FinalStateResult`)
  engage LZ4 compression when their encoded size ≥
  `compression_threshold` (default 1024 bytes; `DEFAULT_COMPRESSION_THRESHOLD`
  constant).
- R36: small-payload variants (`RoundStart`, `RoundResult`,
  `FinalStateRequest`) SHOULD skip compression when encoded size <
  threshold. Verified by asserting `FLAG_COMPRESSED` is UNSET in the
  frame header for small payloads and SET when we force a tiny
  threshold to exercise the compression path.

The frame-layer APIs (`send_frame`, `send_frame_with_threshold`,
`send_frame_v2`, `recv_frame`, `FLAG_COMPRESSED`, `FLAG_ARCHIVED`,
`DEFAULT_COMPRESSION_THRESHOLD`) are all in place as of SPEC-18's ship
(pipeline-state 2026-04-17). No new frame-layer code in this task —
only new tests and one shared fixture module.

**Scope boundary:** rkyv zero-copy archive path (SPEC-18 R22) is
explicitly out of scope — the spec restricts the rkyv fast path to
`AssignPartition` and `PartitionResult` only. The 5 new variants use
bincode-only, which `send_frame` / `send_frame_with_threshold` handle.

## Notes

- **Why `send_frame_with_threshold` and not `send_frame`:** the default
  threshold constant is fine for most tests, but the
  `test_round_start_forced_compression_when_threshold_is_one` test
  needs a non-default threshold. Using
  `send_frame_with_threshold(..., DEFAULT_COMPRESSION_THRESHOLD)` in
  the default-threshold tests keeps the API surface consistent.
- **Why not `send_frame_v2` directly:** `send_frame_v2` is the
  rkyv/archive v2 entry point (SPEC-18). The 5 new variants go through
  the plain bincode path (`send_frame` / `send_frame_with_threshold`),
  which handles the `FLAG_COMPRESSED` flag per the SPEC-18 §3.3
  threshold logic. R22 restricts `FLAG_ARCHIVED` to
  `AssignPartition` / `PartitionResult` only.
- **Why the CRC-tamper test only on `InitialPartition` and not all 5:**
  the CRC check is layer-agnostic — it operates on the encoded payload
  regardless of variant. One tamper test on the largest variant is
  sufficient to regression-guard the CRC path; testing it 5 times
  would be redundant. The existing SPEC-18 CRC-tamper tests already
  cover the primitive for other variants.
- **Threshold tuning for `RoundStart` small-payload test:** 3
  `BorderDelta`s × ~5 bytes each + 2 `u32`s + 2 `(u32, PortRef)`
  tuples + `round: u32` + enum discriminant ≈ 40-80 encoded bytes.
  Well under 1024. The `< 512` assertion in the test is a
  conservative sanity check.
- **R36 "SHOULD" wording:** the spec allows compression if the frame
  layer decides it's worthwhile. The default threshold of 1024 bytes
  means small deltas naturally skip compression — matching the spec
  SHOULD. The `forced_compression_when_threshold_is_one` test confirms
  the layer is capable of compressing if configured to.
- **Why 8 tests, not 5 (one per variant):** the 5 variants × 1 test
  each gives only bincode evidence. Adding the CRC-tamper test,
  forced-compression test, and small-size-sanity assertions makes the
  sub-bundle's R34/R35/R36 evidence self-contained. 8 tests is the
  minimum to cover the three requirements across the 5 variants.

## Spec-Critic Amendments (2026-04-17)

**Verdict source:** `docs/spec-reviews/SPEC-19-section-3.4-design-choices-2026-04-17.md` (Bonus-1 — R35 observability).

| Item | Verdict | Change |
|---|---|---|
| Bonus-1 | R35 "MUST benefit from compression" is non-observable as written — spec language "benefit" is informal; testable criterion must be frame_len < uncompressed | Add concrete wire-observability assertion to the large-payload compression tests |

**Details:**
- R35 states `InitialPartition` and `FinalStateResult` "MUST benefit
  from all wire optimizations". The three listed optimisations
  (`CompactSubnet`, bincode v2 varint, LZ4 compression) are all
  transparently applied by the frame layer — there is no opt-in
  switch. "MUST benefit" is thus a comment, not a testable MUST. A
  200-agent payload that LZ4 compresses to 98% of original still
  "had compression applied" but does NOT benefit.
- **Concrete code change expected:** extend
  `test_initial_partition_wire_roundtrip_compressed` AND
  `test_final_state_result_wire_roundtrip_compressed` with an
  additional assertion after `send_frame_with_threshold`:
  ```rust
  // R35 "benefit" criterion (spec-critic Bonus-1 2026-04-17):
  // compressed payload must be strictly smaller than the
  // uncompressed bincode encoding. If LZ4 hit e.g. 98% of original,
  // the wire "benefit" is illusory — this test should fail and the
  // real action is to revisit SPEC-18's threshold default (1024 B),
  // not SPEC-19.
  let uncompressed_len = bincode_v2::encode(&msg)
      .expect("encode must succeed")
      .len();
  let frame_len = /* bytes written to the wire, captured from the
                     send-side buffer or from the recv-side frame
                     header's payload_len */;
  assert!(
      (frame_len as usize) < uncompressed_len,
      "R35 benefit: compressed frame ({} bytes) must be < \
       uncompressed bincode ({} bytes)",
      frame_len, uncompressed_len
  );
  ```
- The assertion is ADDITIVE to the existing `FLAG_COMPRESSED` SET
  assertion — together they verify (a) compression was applied AND
  (b) compression was beneficial. Existing test count unchanged
  (the assertion augments existing tests rather than adding new
  `#[test]` fns); all other R34/R36 assertions proceed as originally
  specified.
