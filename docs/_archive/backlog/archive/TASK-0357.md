## Context

Wires the `FLAG_ARCHIVED` branch into `recv_frame`. The pipeline order is
load-bearing (R12, pinned by QA Probe 4 from the SPEC-18 Â§3.1-3.4 ship):

```
read 9-byte header
  â†’ reject unknown flags (existing)
  â†’ reject if length > max_payload_size (existing)
  â†’ if FLAG_ARCHIVED:
      â†’ if cfg(not(feature = "zero-copy")):
          return ArchiveValidationFailed("zero-copy feature disabled")
      â†’ else:
          read_aligned_payload (TASK-0355) â€” yields AlignedVec
          â†’ if FLAG_COMPRESSED: decompress into a new AlignedVec
          â†’ verify CRC32C against the (decompressed) payload bytes
          â†’ call rkyv::access::<ArchivedAssignPayload>(&payload)
            (or ArchivedPartitionResultPayload â€” discriminate by what?
             see Notes â€” the wire MUST distinguish the two)
          â†’ reject if access fails â†’ ArchiveValidationFailed
          â†’ rkyv::deserialize â†’ Message::AssignPartition { ... } /
                                  Message::PartitionResult { ... }
  â†’ else: existing bincode v2 path (TASK-0346/0347 logic)
```

Key invariants preserved:
- **R12 ordering:** decompress â†’ CRC verify â†’ rkyv access. CRC ALWAYS over the
  bytes the receiver hands to the deserializer (whether that's
  bincode-bytes or rkyv-archive-bytes).
- **R24 step 3:** use `rkyv::access` (validating). NEVER `rkyv::access_unchecked`
  in production code (the spec is explicit). Validation cost is acceptable
  vs full bincode deserialize.
- **R22:** non-hot-path messages with `FLAG_ARCHIVED` MUST be rejected with
  `ArchiveValidationFailed`. Since the wire format does NOT carry an
  explicit "which hot-path variant is this?" tag, see the Notes section
  for the discrimination strategy.

## Key Types / Signatures

```rust
pub async fn recv_frame<R: AsyncReadExt + Unpin>(
    reader: &mut R,
    max_payload_size: u32,
) -> Result<(Message, usize), ProtocolError> {
    // 1-3: existing header read + unknown-flag check + size check
    // ... (unchanged) ...

    // 4-7: split on FLAG_ARCHIVED
    if header.is_archived() {
        #[cfg(not(feature = "zero-copy"))]
        return Err(ProtocolError::ArchiveValidationFailed(
            "zero-copy feature disabled".into(),
        ));

        #[cfg(feature = "zero-copy")]
        {
            let wire = read_aligned_payload(reader, header.length as usize).await?;
            let payload: rkyv::util::AlignedVec = if header.is_compressed() {
                decompress_payload_aligned(&wire)
                    .map_err(ProtocolError::DecompressionFailed)?
            } else {
                wire
            };
            let computed_crc = crc32fast::hash(payload.as_slice());
            if computed_crc != header.checksum {
                return Err(ProtocolError::ChecksumMismatch {
                    expected: header.checksum,
                    computed: computed_crc,
                });
            }
            // SPEC-18 R22 discrimination: try AssignPartition first (hotter
            // coordinator->worker direction per DC-3), then PartitionResult;
            // reject otherwise. See docs/spec-reviews/
            // SPEC-18-section-3.5-design-choices-2026-04-16.md DC-3 for the
            // Assign-first ordering rationale.
            let message = if let Ok(archived) =
                rkyv::access::<ArchivedAssignPayload, rkyv::rancor::Error>(payload.as_slice())
            {
                let p: ArchiveAssignPayload =
                    rkyv::deserialize::<_, rkyv::rancor::Error>(archived)
                        .map_err(|e| ProtocolError::ArchiveValidationFailed(
                            format!("deserialize: {}", e),
                        ))?;
                Message::AssignPartition { round: p.round, partition: p.partition }
            } else if let Ok(archived) =
                rkyv::access::<ArchivedPartitionResultPayload, rkyv::rancor::Error>(payload.as_slice())
            {
                let p: ArchivePartitionResultPayload =
                    rkyv::deserialize::<_, rkyv::rancor::Error>(archived)
                        .map_err(|e| ProtocolError::ArchiveValidationFailed(
                            format!("deserialize: {}", e),
                        ))?;
                Message::PartitionResult { round: p.round, partition: p.partition, stats: p.stats }
            } else {
                return Err(ProtocolError::ArchiveValidationFailed(
                    "FLAG_ARCHIVED on non-hot-path message".into(),
                ));
            };
            return Ok((message, FRAME_HEADER_SIZE + header.length as usize));
        }
    }

    // Existing bincode v2 path (TASK-0346/0347) unchanged.
    // ...
}
```

## Notes

- **Discrimination strategy ("which hot-path variant is this?") â€”
  APPROVED by spec-critic DC-3 (2026-04-16):** Recommendation (a)
  try-then-try is approved with the **Assign-first ordering rule** and
  the **mandated source comment** captured in the Acceptance Criteria
  above. SPEC-18 R22-R24 specify only that hot-path messages MAY use
  the rkyv path; they don't define an in-band wire tag. The two wrapper
  structs (`ArchiveAssignPayload` vs `ArchivePartitionResultPayload`)
  have distinct rkyv archive layouts (different root struct sizes,
  different relative pointer offsets), so an `ArchivePartitionResultPayload`
  archive fed to `rkyv::access::<ArchivedAssignPayload>` fails validation
  deterministically (length + offset checks won't match) â€” try-then-try
  is therefore correct, not just convenient.

  Option (b) (1-byte sub-tag in the wire) was explicitly rejected by
  spec-critic because it would require either amending SPEC-18 Â§3.4 R14
  (frame header layout â€” adds a 10th byte) or burning one of the 6
  reserved flag bits, neither of which is justified for an optional
  optimisation feature on a hot path where the cost of a failed
  `rkyv::access` is ~0.1ms on typical partitions. See
  `docs/spec-reviews/SPEC-18-section-3.5-design-choices-2026-04-16.md`
  DC-3 for the full rationale.
- **`decompress_payload_aligned` helper:** if the developer chooses to
  preserve alignment through decompression, factor a tiny helper
  `pub(crate) fn decompress_payload_aligned(compressed: &[u8]) ->
  Result<rkyv::util::AlignedVec, String>` in `compression.rs` (or
  `frame.rs`). Wraps `lz4_flex::decompress_size_prepended` and copies into
  `AlignedVec`. Keep the existing `decompress_payload(...) -> Vec<u8>` for
  the bincode path. Cost: one extra memcpy on the rkyv-decompressed path.
- **`rkyv::access` cost vs full deserialize:** rkyv's validation walks the
  archive once to verify offsets/bounds; cost is roughly 10% of full
  bincode deserialize on typical partitions. Net win is still 5-10Ă— on the
  hot path because the deserialize itself is O(0) â€” `&ArchivedPartition`
  is a view into the buffer. The current implementation eagerly calls
  `rkyv::deserialize` to materialize a `Message`; a follow-up v2.1
  optimization can route the `&Archived*` ref directly into reduction
  code without materializing â€” out of scope for this task.
- **NO `access_unchecked` in production:** spec is explicit (R24 step 3).
  Tests MAY use it for benchmark-driven proofs, but `recv_frame` uses
  only `access`. Source-grep `rg "access_unchecked"` should return zero
  hits in `src/`.
- **Pre-existing tests that break:** QA Probe 3
  (`qa_probe_3_both_flags_set_archive_bit_currently_ignored`) explicitly
  documented its name as "currently_ignored" â€” anticipating this bundle.
  TEST-SPEC-0357 will rename + split it into the new
  `cfg(feature = "zero-copy")` and `cfg(not(feature = "zero-copy"))`
  variants.
