# TASK-0362: `BorderGraph::apply_deltas` (incremental redex set + DISCONNECTED handling)

**Spec:** SPEC-19 §3.2 (and §4.2 pseudocode)
**Requirements:** R11, R17, R18 (SHOULD: incremental redex set)
**Priority:** P0 (critical path — every round calls this)
**Status:** TODO
**Depends on:** TASK-0361
**Blocked by:** TASK-0361
**Estimated complexity:** M (~100 LoC)
**Bundle:** SPEC-19 §3.2 (item 2.35) — BorderGraph and Delta-Based Merge

## Context

R11 specifies the per-round update primitive: workers report a list of
`BorderDelta { border_id, new_target }`, the coordinator applies each delta
to the matching `BorderState` (replacing whichever side the reporting worker
owns), recomputes `is_redex`, and maintains the incremental
`active_redexes` set per R18 SHOULD so that TASK-0363's
`detect_border_redexes` is O(|redexes|) rather than O(B).

R17 covers the erasure case: when a worker erases an agent connected to a
`FreePort(border_id)`, it reports a delta whose `new_target` is the
`DISCONNECTED` sentinel (`PortRef::FreePort(u32::MAX)`, already exported
from `crate::net::types`). The graph MUST mark that side as disconnected;
when both sides become disconnected, the border ceases to exist and MUST
be removed from `borders`, `active_redexes`, and (per the spec note in §4.2
"worker_borders entries become stale but harmless") `worker_borders` MAY
remain stale.

`BorderDelta` does not yet exist as a Rust source-level type — it lives
only in the spec §3.3 pseudocode. This task ships a *minimal* in-tree
`BorderDelta` definition (just the two fields, no serde derives — wire
encoding is item 2.26's responsibility). The struct lives in
`border_graph.rs` for now; if item 2.26 needs it elsewhere it can be moved
later.

## Acceptance Criteria

- [ ] New `pub struct BorderDelta` defined in `border_graph.rs`:
      `pub border_id: u32`, `pub new_target: PortRef`, with derives
      `#[derive(Debug, Clone, Copy, PartialEq, Eq)]`. **No** serde / rkyv
      derives in this task (wire encoding is item 2.26's job; if needed
      item 2.26 can add the derives non-breakingly).
- [ ] `BorderGraph` gets new method:
      `pub fn apply_deltas(&mut self, worker_id: WorkerId, deltas: &[BorderDelta])`
      with the following behaviour per delta:
  1. Look up `state = self.borders.get_mut(&delta.border_id)`. If absent
     (border was already removed by a prior delta in the same call, or by
     an earlier call), **silently skip** the delta (do not panic — this is
     the "stale delta" tolerance specified by item 2.26's lifecycle: a
     worker can emit a delta for a border that the coordinator already
     resolved server-side in this same round).
  2. Determine which side belongs to the reporting worker:
     - If `state.worker_a == worker_id`: `state.side_a = delta.new_target`.
     - Else if `state.worker_b == worker_id`: `state.side_b = delta.new_target`.
     - Else: **silently skip** (delta from a worker that doesn't own
       either side — also stale).
  3. Snapshot `was_redex = state.is_redex`, recompute
     `state.is_redex = is_principal_pair(state.side_a, state.side_b)`.
  4. Update `active_redexes`:
     - If `state.is_redex && !was_redex` → `active_redexes.insert(delta.border_id)`.
     - If `!state.is_redex && was_redex` → `active_redexes.remove(&delta.border_id)`.
     - Else: no change.
  5. Handle R17 double-erasure: if BOTH `state.side_a == DISCONNECTED` AND
     `state.side_b == DISCONNECTED`, remove the entry — `self.borders.remove(&delta.border_id)`,
     `self.active_redexes.remove(&delta.border_id)`. (Per spec §4.2 note,
     `worker_borders` is left stale.)
- [ ] `apply_deltas` MUST be O(|deltas|) — no inner loop over `borders`,
      no iteration over `worker_borders`. Each delta is one `HashMap` lookup
      plus a constant number of field reads/writes.
- [ ] Inline `#[cfg(test)] mod tests` extends the existing module with
      (each test builds a fresh `BorderGraph` via TASK-0361's
      `from_partition_plan` over a small synthetic `PartitionPlan`):
      - `apply_delta_principal_to_principal_marks_redex`: border that
        starts with one principal + one auxiliary; delta from owner of
        the auxiliary side updates it to a principal port → `is_redex
        == true`, `active_redexes.contains(&bid)`.
      - `apply_delta_principal_to_aux_clears_redex`: border that starts
        as redex; delta from one side moves to auxiliary →
        `is_redex == false`, `!active_redexes.contains(&bid)`.
      - `apply_delta_wrong_worker_silent_skip`: delta with `worker_id`
        that owns neither side → `state` unchanged, `active_redexes`
        unchanged, no panic.
      - `apply_delta_unknown_border_silent_skip`: delta with `border_id`
        not in `borders` → no panic, `borders.len()` unchanged.
      - `apply_delta_disconnect_one_side`: border state intact, delta
        with `new_target = DISCONNECTED` from one owner → side updated,
        border still present, `is_redex == false`,
        `borders.contains_key(&bid) == true`.
      - `apply_delta_disconnect_both_sides_removes_border`: send two
        DISCONNECTED deltas (one per owner) → border absent from
        `borders`, absent from `active_redexes`. The `worker_borders`
        entries MAY remain (per spec §4.2 note); test MUST NOT assert
        their removal.
      - `apply_deltas_empty_slice_noop`: passing `&[]` → graph unchanged.
      - `apply_deltas_batch_atomicity`: pass 3 deltas in one call, only
        the relevant subset applied; assert each individual side state
        and `active_redexes` membership.
- [ ] No `unwrap()` in production. `if let Some(state) = ...` for the
      `borders.get_mut`.
- [ ] `apply_deltas` is fully synchronous; no `async`, no `await`. R19
      pure-core grep guard still passes.
- [ ] `cargo test --workspace` count grows by 8 — 918 → 926.
- [ ] Same +8 with `--features zero-copy` — 958 → 966.

## Key Types / Signatures

```rust
/// Per-round connectivity change reported by a worker (SPEC-19 §3.3).
///
/// Workers emit one of these for each border port whose connectivity
/// changed during local reduction. `new_target = DISCONNECTED` means the
/// agent that was connected to `FreePort(border_id)` was erased
/// (R17 case).
///
/// **Note:** wire encoding (`serde` / `rkyv` derives) is intentionally
/// omitted in this task. Item 2.26 (delta protocol) will add the derives
/// when it ships the wire-format extension for `Message::RoundResult`.
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct BorderDelta {
    pub border_id: u32,
    pub new_target: PortRef,
}

impl BorderGraph {
    /// Apply worker-reported deltas (SPEC-19 R11, R17, R18).
    ///
    /// O(|deltas|) — per-delta cost is one `HashMap` get_mut, one
    /// `PortRef` write, and a constant number of `HashSet` operations on
    /// `active_redexes`. Stale or unknown-worker deltas are silently
    /// skipped (item 2.26 lifecycle requirement).
    pub fn apply_deltas(&mut self, worker_id: WorkerId, deltas: &[BorderDelta]) {
        for delta in deltas {
            let Some(state) = self.borders.get_mut(&delta.border_id) else {
                continue; // stale: border already removed
            };

            if state.worker_a == worker_id {
                state.side_a = delta.new_target;
            } else if state.worker_b == worker_id {
                state.side_b = delta.new_target;
            } else {
                continue; // stale: this worker owns neither side
            }

            let was_redex = state.is_redex;
            state.is_redex = is_principal_pair(state.side_a, state.side_b);

            if state.is_redex && !was_redex {
                self.active_redexes.insert(delta.border_id);
            } else if !state.is_redex && was_redex {
                self.active_redexes.remove(&delta.border_id);
            }

            // R17: double-erasure removes the border.
            if state.side_a == DISCONNECTED && state.side_b == DISCONNECTED {
                self.borders.remove(&delta.border_id);
                self.active_redexes.remove(&delta.border_id);
                // worker_borders left stale per spec §4.2 note.
            }
        }
    }
}
```

## Notes

- **Silent-skip vs panic for unknown borders:** chosen per the bundle's
  scope-decision boundary with item 2.26. The delta BSP loop in item 2.26
  may emit overlapping deltas in the rare case where (a) a border is
  resolved coordinator-side mid-round and (b) the worker independently
  reports a delta for that same border before receiving the resolution
  message. Silent skip is the correct behaviour per "stale delta"
  tolerance. If spec-critic disagrees and prefers a `Result<(),
  ApplyDeltaError>` API, this task's acceptance criteria can flip — the
  test count delta stays the same.
- **Why `BorderDelta` lives here and not in a new `delta.rs` file:** keeps
  the bundle's blast radius to a single new file (`border_graph.rs`) plus
  the one-line `mod.rs` edit from TASK-0360. Item 2.26 will likely move
  it to `protocol/types.rs` when it adds serde derives — that move is
  trivial because no public API outside `merge/` references `BorderDelta`
  in this bundle.
- **`active_redexes` invariant (TEST-SPEC-0362 fuzz target):** for any
  reachable graph state, `active_redexes == {bid : borders[bid].is_redex
  == true}`. This invariant is preserved by `apply_deltas` (proof: the
  three branches insert/remove/no-op cover the three transition cases).
  TASK-0364 (`add_border_states` + `remove_border`) MUST also preserve it.
- **Why no `Result` return:** all "error" cases (stale border, wrong
  worker) are normal protocol behaviour, not errors. A `Result<(), ...>`
  would force callers to handle non-error cases explicitly, adding noise
  to item 2.26's coordinator loop. If a real error case ever emerges,
  switch to `Result` then.
