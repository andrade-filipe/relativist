## Context

Stitches TASK-0372..0375 together in a single integration-test file
(`relativist-core/tests/border_resolver_integration.rs`) that exercises
the full resolver pipeline against a controlled 2-partition fixture
with exactly 1 border redex:

```
┌── Partition 0 ──┐         ┌── Partition 1 ──┐
│                 │         │                 │
│   [agent_a]────────bid 0──────────[agent_b] │
│     (p0=principal)       (p0=principal)     │
│                 │         │                 │
└─────────────────┘         └─────────────────┘
```

The test:
1. Builds the fixture via `PartitionPlan { partitions: vec![p0, p1],
   borders: ... }`.
2. Constructs `BorderGraph::from_partition_plan(&plan)`.
3. Asserts `graph.detect_border_redexes() == [(0, ...)]`
   (precondition).
4. Calls `resolve_border_redex(&mut graph, &plan.partitions, 0)` →
   `BorderResolution`.
5. Wraps in `package_resolutions(vec![resolution], 2)` →
   `Vec<(WorkerId, RoundStartDispatch)>`.
6. Asserts the post-condition shape for each rule variant.
7. Asserts `graph.detect_border_redexes().is_empty()` and
   `graph.borders.contains_key(&0) == false` (border removed).

Integration test file (`tests/`) rather than inline `#[cfg(test)]`
because it exercises multiple public-ish functions across the resolver
module — this is the "ties it all together" test per bundle acceptance
gate.

## Notes

- **Why integration test (not just inline unit tests):** TASK-0373,
  TASK-0374, TASK-0375 each ship inline `#[cfg(test)]` for their
  PIECE. This task tests the COMPOSITION — precondition →
  `resolve_border_redex` → `graph` mutations → `package_resolutions`
  → per-worker dispatch. Inline tests don't capture the full
  pipeline; hence a separate integration test file.

- **Why 2 partitions, not more:** each border has exactly 2 sides
  (SPEC-19 C3 invariant). A 2-partition fixture is the minimum
  configuration that exercises a border redex. Larger fixtures (3+
  partitions, multiple simultaneous border redexes) are 2.26-C / 2.26-D
  scope — the end-to-end BSP-loop test lives there.

- **Why no `run_grid_delta` call in this test:** `run_grid_delta`
  requires the wire protocol + stateful worker lifecycle, both of
  which ship under 2.26-C. This task validates the pure-core
  resolver in isolation; the BSP-loop end-to-end test is 2.26-C's
  acceptance gate.

## Dependencies on Sibling Bundles

- **Depends on 2.26-A (wire protocol):** only LOGICALLY — the
  integration test does NOT construct `Message::RoundStart`. It
  validates `RoundStartDispatch` (the pure-core mirror). No compile-
  time dependency on 2.26-A's deliverables.

- **Unblocks 2.26-C:** once this bundle is GREEN, 2.26-C can
  compose `RoundStartDispatch → Message::RoundStart` with confidence
  that the resolver behaves correctly.
