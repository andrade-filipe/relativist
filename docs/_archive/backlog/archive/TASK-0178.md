## Context

The three I/O subcommands (`reduce`, `inspect`, `generate`) need their argument structs defined with clap derives so they can be integrated into the top-level CLI (SPEC-13). This task defines `ReduceArgs` (R24), `InspectArgs` (R28), and `GenerateArgs` (R33), plus the handler functions that wire the args to the I/O layer.

The handlers call `load_net` / `save_net` (TASK-0168), `net_summary` (TASK-0169), `format_reduction_summary` (TASK-0170), `reduce_all` (Phase 2), and `generate_example` (TASK-0171) to perform the actual work. They also handle the `--metrics` flag (R21), `--max-interactions` safety limit (R26), and `--output` writing (R18-R19).

This task defines the structs and handlers but does NOT integrate them into `main.rs` -- that integration belongs to the Phase 6 (CLI & Config) tasks that wire up the top-level clap command.

## Notes

- The handler functions are defined in the `io` module (not in `main.rs` or a CLI module) so they can be tested independently of the CLI framework. The Phase 6 CLI integration task will wire these handlers to the clap subcommand dispatch.
- `run_reduce` needs timing: use `std::time::Instant::now()` to measure reduction duration.
- The `--input-format` flag on `ReduceArgs` overrides extension detection for the input file only. The output format is always determined by the output file's extension (R18, R50 v2).
- `run_generate` confirmation message (R43) must include: example name, size, agent count, redex count, output path, and file size.
- The handlers use `todo!()` for the actual logic bodies. The implementation depends on Phase 2 (`reduce_all`, `reduce_n`) being available. If Phase 2 is not complete, the `run_reduce` handler can compile with `todo!()` and be filled in later.
