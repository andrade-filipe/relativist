# TASK-0396: R20 dispatcher fork — `run_grid` routes to `run_grid_delta` when `cfg.delta_mode`

**Spec:** SPEC-19 §3.3 R20 ("The delta protocol MUST be activated via a configuration flag `GridConfig.delta_mode: bool`, defaulting to `false`. When `delta_mode` is `false`, the v1 full-partition protocol ... is used unchanged. When `delta_mode` is `true`, the delta BSP loop is used.")
**Requirements:** R20 (delta_mode is an operational switch, not an inert field)
**Priority:** P1 (without this, SPEC-19 R20 is a dead letter — flag exists, does nothing observable; SF-001 closure from REVIEW-SPEC-19-section-3.3-3.5-3.6-item-2.26-BCD-2026-04-23)
**Status:** TODO
**Depends on:** **TASK-0394** (worker-side R23/R26 completion) so that `run_grid_delta` is correctness-complete before the dispatcher fork exposes it as a public path. (Technically the fork can be wired first and the bug shipped; but then SF-001 acceptance would fail QA.)
**Blocked by:** TASK-0394 (correctness); TASK-0395 (coordination) — the two changes share the test-support feature gate decision (DC-0395-A).
**Estimated complexity:** S (~40-60 LoC production + ~40 LoC tests)
**Bundle:** SPEC-19 §3.3 Refactor (post-REVIEW 2026-04-23) — SF-001 closure

## Context

The REVIEW of bundle 2.26-B/C/D identified SF-001: `GridConfig.delta_mode` is a field, threaded through the CLI, but **no call site reads it**. The public `run_grid` function in `merge/grid.rs:37-303` is oblivious. Callers must explicitly invoke `run_grid_delta`, which is `pub(crate)`. A user passing `--delta-mode` on the CLI will NOT route through the delta loop — the flag is inert.

The existing `run_grid_delta` entry (`grid.rs:339-385`) carries a `TODO(2.26-D)` at L346-349 ("once `GridConfig.delta_mode` lands, assert it here. Until then, this function is unconditionally delta-mode at entry."). 2.26-D landed the field but not the fork.

This task closes the gap by introducing a public dispatcher that reads `cfg.delta_mode` and routes to the appropriate entry point.

## Key Types / Signatures

```rust
// relativist-core/src/merge/grid.rs

/// SPEC-19 R20 dispatcher. Routes to `run_grid` (v1) when `config.delta_mode`
/// is false, and to `run_grid_delta` (v2) when true. Callers MUST provide a
/// WorkerDispatch when delta_mode is true; passing None panics.
pub fn run_grid_entry(
    net: Net,
    config: &GridConfig,
    strategy: &dyn PartitionStrategy,
    dispatch: Option<&mut dyn WorkerDispatch>,
) -> (Net, GridMetrics) {
    if config.delta_mode {
        let d = dispatch.expect(
            "SPEC-19 R20: delta_mode=true requires a WorkerDispatch; None was provided"
        );
        run_grid_delta(net, config, strategy, d)
    } else {
        // v1 path ignores `dispatch`
        run_grid(net, config, strategy)
    }
}
```

## Notes

- **Why this is P1 and not P0:** the 2-code-path problem is real but not a correctness gap per se — `run_grid_delta` works correctness-wise (modulo MF-001 which TASK-0394 fixes); users just can't reach it through the CLI. A CLI user running `relativist local --workers 2 compute add 3 5 --delta-mode` today would get the v1 baseline (because CLI calls `run_grid` directly), which is not wrong, just not what `--delta-mode=true` advertises.
- **Why `Option<&mut dyn WorkerDispatch>` and not two separate function signatures:** keeping a single entry point preserves the Option B ambition from the REVIEW ("public dispatcher is cleanest"). A split API (`run_grid_v1` + `run_grid_v2(... dispatch)`) is an alternative, but forces every caller to branch on `delta_mode` — the dispatcher fork centralizes the branch.
- **Why panic on `None` when `delta_mode=true` instead of returning `Result`:** panicking matches the "pre-condition assertion" convention elsewhere in `merge/grid.rs` (e.g., `run_grid_delta` already panics on `num_workers < 1` with `assert!(config.num_workers >= 1, ...)`). Converting to `Result` for one new variant would introduce API noise; the check is structural, not recoverable.
- **Relation to SF-002 (TASK-0397):** independent. R43 ("coordinator_free_rounds defaults to true when delta_mode=true") is a field-initialization issue, not a dispatcher issue.
- **Interaction with R42:** R42 ("default `delta_mode = false` preserves v1 behavior exactly") is already tested at `grid.rs:1968-2039`. Routing through `run_grid_entry` with `delta_mode=false` MUST preserve that guarantee byte-for-byte — this is a QA gate.
