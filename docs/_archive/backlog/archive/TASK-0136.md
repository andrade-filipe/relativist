# TASK-0136: Verify message size pre-validation in recv_frame (SPEC-06 integration)

**Spec:** SPEC-10
**Requirements:** R29, R30
**Priority:** P0 (critical path)
**Status:** TODO
**Depends on:** TASK-0086
**Blocked by:** none
**Estimated complexity:** S (< 50 LoC)

## Context

The `recv_frame` function (TASK-0086) checks payload size against `max_payload_size` (SPEC-06 R9, `NodeConfig.max_payload_size`). SPEC-10 R29 references this check as a security defense against memory exhaustion attacks but explicitly delegates ownership to SPEC-06 -- SPEC-10 does NOT define its own message size configuration (SC-004). This task verifies the integration is correct: the `recv_frame` size check uses the value from `NodeConfig.max_payload_size` (not any security config), the error variant uses `ProtocolError::PayloadTooLarge` (SPEC-06 Section 4.4 name), and the rejection happens BEFORE any memory allocation.

## Acceptance Criteria

- [ ] `recv_frame` uses `max_payload_size` from `NodeConfig` (SPEC-06 R9) for its size check (R29)
- [ ] The length field in the frame header is checked against `max_payload_size` BEFORE `vec![0u8; length]` allocation (R29)
- [ ] Default `max_payload_size` is `268_435_456` bytes (256 MiB, SPEC-06 R9)
- [ ] When a message exceeds the limit, `ProtocolError::PayloadTooLarge` is returned (R30, matching SPEC-06 Section 4.4 name)
- [ ] The oversized payload is NOT read from the socket (R30)
- [ ] The connection is closed after rejection (R30)
- [ ] There is NO `max_message_size` field in `SecurityConfig` (SC-004: message size is owned by SPEC-06)
- [ ] There is NO `--max-message-size` CLI flag (removed in Revised v2)

## Files to Create/Modify

## Notes

- Message size limits are owned by SPEC-06, not SPEC-10 (SC-004, SPEC-10 Section 5.6). SPEC-10 references the check as a security defense (R29) but does not duplicate the configuration.
- The error variant MUST be `PayloadTooLarge` (matching SPEC-06 Section 4.4), NOT `MessageTooLarge` (which was used in Draft v1 and removed in Revised v2).
- This task overlaps with TASK-0086 which implements the size check. The purpose of this task is to verify the security integration is correct and the error variant name matches the canonical spec.
- There is no `--max-message-size` CLI flag (removed in Revised v2). The payload size is configured via SPEC-06's `NodeConfig`.
