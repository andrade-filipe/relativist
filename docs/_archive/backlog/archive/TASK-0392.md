## Context

SPEC-19 §3.5 amends three SPEC-01 invariants to account for the delta protocol:

- **R38 (G1):** reformulates the Fundamental Property from
  `reduce_all(net) ~ extract_result(run_grid(net, n))` to
  `reduce_all(net) ~ extract_result(run_grid_delta(net, n))`, where
  `extract_result` = Final State Collection (R27-R29) then `merge()`. Full
  formal proof is pending (OQ-1, ARG-005).
- **R39 (D3):** amends Border Completeness sub-clauses D3a/b/c/d so that
  border-redex detection is incremental via `BorderGraph` rather than by
  exhaustive scan. Equivalence to the v1 exhaustive detection is the
  correctness claim (pending formal proof, Section 8).
- **R40 (D6):** reformulates Protocol Termination under the delta protocol,
  distinguishing **lenient delta mode** (R_delta_lenient = 1 in the common
  case) from **strict delta mode** (R_delta_strict ≤ N, same bound as v1
  strict), and anchoring termination on **Global Normal Form**: all workers
  report zero local redexes AND `BorderGraph` contains zero active pairs.

These amendments are *spec* text that already exists in SPEC-19 §3.5. The
task-splitter has no authority to edit specs. What we can (and should) do is
add **narrative notes in `docs/ROADMAP.md`** that cross-link from the
implementation roadmap (bundle 2.26) to the spec amendments, so future readers
of the ROADMAP (which is the canonical roadmap entry point, per user
preference `project_relativist_roadmap.md`) see the amendment context without
having to hunt through the spec.

The narrative notes do NOT restate the spec (that would be redundant). They
link to it and explain the practical implication for the implementation:

- G1 amendment means the correctness invariant for `run_grid_delta` is
  defined but the proof is a separate TCC deliverable (ARG-005). The
  engineering consequence: sub-bundle 2.26-C's convergence test is the
  operational guarantee; the formal proof is deferred.
- D3 amendment means `BorderGraph.detect_border_redexes()` (shipped in 2.35)
  is the *authoritative* border-redex oracle; exhaustive `findBorderRedexes`
  in v1 merge is retained for the `delta_mode = false` path only.
- D6 amendment means Global Normal Form is checked via
  (a) per-worker flag `has_border_activity` (shipped in TASK-0348) AND
  (b) `BorderGraph.is_empty()` (shipped in 2.35). The convergence check in
  2.26-C is the joint of the two.

## Proposed subsection skeleton (for the test-generator / developer to flesh out)

```markdown
#### 2.26 Invariant Amendments (SPEC-19 §3.5 — TASK-0392)

This is narrative documentation of the SPEC-19 amendments. The formal text
lives in SPEC-19 §3.5; the formal proofs are TCC work items (OQ-1, ARG-005,
DISC-011). No SPEC-01 edit is required — SPEC-19 §3.5 is the canonical
amendment location.

**G1 (Fundamental Property) — R38.**
The delta protocol keeps the merged net distributed across workers between
rounds. G1 is reformulated from
`reduce_all(net) ~ extract_result(run_grid(net, n))` to
`reduce_all(net) ~ extract_result(run_grid_delta(net, n))`, where
`extract_result` performs Final State Collection (R27-R29) followed by
`merge()`. The formal proof of recoverability of the sequential intermediate
state is deferred (Section 8 OQ-1 → DISC-011 → ARG-005). **Operational
guarantee in the implementation:** the convergence test in bundle 2.26-C
exercises a canonical workload end-to-end with `delta_mode = true` and
compares the decoded result to the `reduce_all` reference.

**D3 (Border Completeness) — R39.**
Border-redex detection is incremental via `BorderGraph.detect_border_redexes()`
(shipped in bundle 2.35, TASK-0374..0388). The v1 exhaustive scan in
`merge::find_border_redexes` is retained for the `delta_mode = false` path.
Equivalence between the two oracles is the correctness claim; formal proof
pending (Section 8).

**D6 (Protocol Termination) — R40.**
Termination is anchored on **Global Normal Form**: (a) every worker reports
`local_redexes == 0` AND `has_border_activity == false` (TASK-0348), AND
(b) `BorderGraph.is_empty()` (bundle 2.35). The joint check is performed by
the delta BSP loop in bundle 2.26-C. `R_delta_lenient = 1` in the absence of
cross-partition cascades; `R_delta_strict ≤ N` matches the v1 strict-mode
bound.

**Configuration mechanism.** The `GridConfig.delta_mode: bool` field
(TASK-0389) and the `--delta-mode` CLI flag (TASK-0390) toggle between the
v1 path and the amended path. Default is `false` (R42).
```

## Spec-Critic Amendments (2026-04-17)

**Details:**

- The original skeleton treated R38/R39/R40 uniformly under a single "proof
  deferred to Section 8" framing. Spec-critic AMB-D-3 reads the spec closely
  and rules that the deferral language is NOT uniform across the three
  amendments. The ROADMAP narrative MUST be split into two groups.
- **Group 1 — R38 (G1) and R39 (D3): PROOF PENDING.** Both carry explicit
  "pending formal proof" language in SPEC-19 §3.5 (R38 line 223-225: "The
  full formal proof of the recoverability property is pending (see Section
  8, Open Questions: DISC-011 + ARG-005). This spec defines the design; the
  theoretical proof is a separate deliverable." and line 225's parenthetical
  `(MUST for the reformulation; proof pending)`; R39 line 234: "…is the core
  correctness claim of the delta protocol and is pending formal proof
  (Section 8)."). The R38 and R39 paragraphs in the ROADMAP subsection MUST
  retain the phrase "formal proof deferred to Section 8 / ARG-005" and MUST
  be marked **PROOF PENDING** in the narrative. Operational guarantees:
  for G1, the 2.26-C convergence test vs `reduce_all` reference; for D3,
  the `BorderGraph.detect_border_redexes()` oracle shipped in 2.35.
- **Group 2 — R40 (D6): OPERATIONALLY COMPLETE AFTER 2.26-C.** R40 does
  NOT carry any "pending proof" clause. Its fourth bullet ("Progress
  guarantee") is a self-contained operational argument: each round
  consumes ≥ 1 interaction from the finite T7 budget ⇒ the protocol
  terminates in ≤ N rounds strict / 1 round lenient. The R40 paragraph
  in the ROADMAP subsection MUST NOT say "proof deferred". Instead it
  MUST read approximately: "Termination proof is operational: each round
  consumes ≥ 1 interaction from the finite budget T7, so the protocol
  terminates in ≤ N rounds strict / 1 round lenient. This is the full
  spec statement at §3.5 R40; no Section 8 deferral." The R40 paragraph
  MUST cross-reference sub-bundle 2.26-C's delta BSP loop as the
  operational discharge of the MUST.
- **Cross-reference (cross-bundle coupling):** R40 is **PARTIAL after
  2.26-D** (this sub-bundle ships only the config flag plumbing
  TASK-0389/0390 and the narrative documentation TASK-0392) and
  **COMPLETE after 2.26-C** (sub-bundle 2.26-C ships the delta BSP loop
  with the Global Normal Form termination check: joint per-worker
  `has_border_activity == false` from TASK-0348 + `BorderGraph.is_empty()`
  from 2.35 + zero local redexes). The subsection text MUST include a
  one-line status note to this effect inside the R40 paragraph, so
  future readers see the PARTIAL → COMPLETE transition without
  hunting through bundle indexes.
