## Context

R16 specifies the annihilation-cleanup primitive: when a border redex is
resolved by an annihilation rule (CON-CON, DUP-DUP) or void rule (ERA-ERA),
both agents are consumed and the border wire ceases to exist. The
coordinator (item 2.26) calls `BorderGraph::remove_border(border_id)` to
update the graph. This is distinct from R17's double-disconnect path
because R16 is triggered by a coordinator-side resolution (no DISCONNECTED
delta involved); R17 is triggered by a worker erasing an agent locally.

R15 part 3 specifies the CON-DUP expansion primitive: when CON-DUP
commutation produces 4 new agents whose auxiliary ports may inherit
FreePort connections, the coordinator MUST add new `BorderState` entries
for these new borders. Per the bundle scope-decision, only the *primitive*
ships in this bundle (the data-structure mutator); the coordinator-side
caller that constructs the `BorderState` values from the resolution result
is item 2.26's responsibility.

Per bundle DC-4, `add_border_states` takes a pre-built
`Vec<BorderState>` (the caller is responsible for setting the `is_redex`
bit; the implementation seeds `active_redexes` accordingly and updates
`worker_borders`). This matches `from_partition_plan`'s state-construction
shape.

## Acceptance Criteria

- [ ] `BorderGraph` gets a new method:
      `pub fn remove_border(&mut self, border_id: u32) -> Option<BorderState>`
      that:
  1. `Removes border_id` from `self.borders` (returning the removed
     state, or `None` if absent).
  2. Removes `border_id` from `self.active_redexes` (no-op if absent).
  3. **Does NOT** scan or rebuild `self.worker_borders` — entries become
     stale per the same justification as R17 in TASK-0362 (spec §4.2
     note). Stale `worker_borders` entries are filtered at lookup time
     by future readers.
  4. Returns the removed `BorderState` so the caller can audit the
     resolution (e.g., emit a `tracing::debug!` with the consumed
     endpoints).
- [ ] `BorderGraph` gets a new method:
      `pub fn add_border_states(&mut self, states: Vec<BorderState>)`
      that:
  1. For each `state` in `states`:
     - Inserts `state.clone()` (or moves the value out — developer
       choice) into `self.borders` keyed by `state.border_id`. If a
       border with this ID already exists, **panics** with a message
       naming the duplicate `border_id`. (The caller is responsible for
       generating fresh border IDs; a duplicate is a coordinator bug.)
     - If `state.is_redex == true`, inserts `state.border_id` into
       `self.active_redexes`.
     - Pushes `state.border_id` into `self.worker_borders[state.worker_a
       as usize]` and `self.worker_borders[state.worker_b as usize]`.
  2. If a `state` references a `worker_a` or `worker_b` that is
     out-of-bounds for `self.worker_borders` (i.e., `as usize >=
     self.worker_borders.len()`), **panics** with a message naming the
     border_id and the offending worker_id. (This catches a coordinator
     bug where a CON-DUP commutation generates new borders involving
     workers that don't exist.)
- [ ] Both methods MUST be O(|input|) — no scan of existing state. Time
      complexity unchanged from R18.
- [ ] Inline `#[cfg(test)] mod tests` extends the existing module:
      - `remove_border_present_returns_state`: graph with 1 border →
        `remove_border(bid)` returns `Some(state)`,
        `borders.is_empty() == true`.
      - `remove_border_absent_returns_none`: empty graph →
        `remove_border(999) == None`.
      - `remove_border_clears_active_redex`: redex border in graph →
        `remove_border` clears it from `active_redexes`.
      - `remove_border_leaves_worker_borders_stale`: graph with 1 border
        → `remove_border` → `worker_borders[0]` STILL contains the
        `border_id` (positive contract: stale entries are tolerated).
      - `add_border_states_inserts_with_redex_bit`: empty graph →
        `add_border_states([state{is_redex: true}])` → `borders.len() == 1`,
        `active_redexes` contains the new border_id.
      - `add_border_states_inserts_without_redex_bit`: empty graph →
        `add_border_states([state{is_redex: false}])` → `borders.len() == 1`,
        `active_redexes.is_empty()`.
      - `add_border_states_updates_worker_borders`: 2-worker graph →
        `add_border_states([state{worker_a: 0, worker_b: 1, ...}])` →

## Key Types / Signatures

```rust
impl BorderGraph {
    /// Remove a border, e.g. after CON-CON / DUP-DUP / ERA-ERA
    /// annihilation at the coordinator (SPEC-19 R16). Returns the removed
    /// state for caller-side audit (or `None` if the border was already
    /// absent).
    ///
    /// Stale `worker_borders` entries are tolerated (spec §4.2 note).
    pub fn remove_border(&mut self, border_id: u32) -> Option<BorderState> {
        self.active_redexes.remove(&border_id);
        self.borders.remove(&border_id)
    }

    /// Add freshly-created borders, e.g. from a CON-DUP commutation that
    /// produced 4 new agents whose auxiliary ports inherit FreePort
    /// connections (SPEC-19 R15 part 3, primitive only).
    ///
    /// Caller responsibility: pre-compute `state.is_redex` (use
    /// `is_principal_pair` if needed). Item 2.26's coordinator dispatch
    /// owns the construction logic.
    ///
    /// # Panics
    ///
    /// - If any `state.border_id` is already present in `self.borders`.
    /// - If any `state.worker_a` or `state.worker_b` is out-of-bounds for
    ///   `self.worker_borders.len()`.
    pub fn add_border_states(&mut self, states: Vec<BorderState>) {
        for state in states {
            assert!(
                !self.borders.contains_key(&state.border_id),
                "duplicate border_id {} in add_border_states",
                state.border_id
            );
            let max_worker = self.worker_borders.len();
            assert!(
                (state.worker_a as usize) < max_worker,
                "out-of-bounds worker {} for border_id {}",
                state.worker_a, state.border_id
            );
            assert!(
                (state.worker_b as usize) < max_worker,
                "out-of-bounds worker {} for border_id {}",
                state.worker_b, state.border_id
            );

            let bid = state.border_id;
            let wa = state.worker_a as usize;
            let wb = state.worker_b as usize;
            let is_redex = state.is_redex;

            self.borders.insert(bid, state);
            if is_redex {
                self.active_redexes.insert(bid);
            }
            self.worker_borders[wa].push(bid);
            self.worker_borders[wb].push(bid);
        }
    }
}
```

## Notes

- **Why `remove_border` returns `Option<BorderState>` not `()`:** the
  coordinator (item 2.26) wants the consumed endpoints for two reasons:
  (1) emit a structured trace event for observability ("border 42
  consumed: side_a was AgentPort(7, 0), side_b was AgentPort(8, 0)"),
  and (2) verify the resolution was consistent with the agents that the
  coordinator just reduced (defensive cross-check). Returning the state
  costs nothing (the value was about to be dropped).
- **Why `add_border_states` takes `Vec` not `&[BorderState]`:** moves the
  values directly into the `HashMap` without cloning. Item 2.26's caller
  builds the vec once per round and discards it; ownership transfer is
  the cheapest path. `&[BorderState]` would force a clone-per-state.
- **Why both methods panic instead of returning `Result`:** matches
  TASK-0361's panic-on-C3-violation precedent. A duplicate border_id or
  out-of-bounds worker is a coordinator bug, not a runtime condition. The
  panic message identifies the bad value precisely so the bug is fast to
  triage.
- **Why this task does not depend on TASK-0362/0363:** the two methods
  here only touch `self.borders`, `self.active_redexes`, and
  `self.worker_borders` — all three landed by TASK-0360+0361. The task
  CAN run in parallel with TASK-0362; the topological sort just happens
  to put it on a separate branch from TASK-0362→0363.
- **Per DC-4 of the spec-critic verdict, shipping Option B (graph recomputes `is_redex`) to enforce the R9 invariant at the primitive boundary.** See Spec-Critic Amendments below.

**New struct definition (mandated form):**

```rust
/// Input entry for [`BorderGraph::add_border_states`].
///
/// Five connectivity fields — **no `is_redex`**. The graph computes
/// `is_redex` via `is_principal_pair(side_a, side_b)` to enforce the
/// R9 invariant `state.is_redex == is_principal_pair(side_a, side_b)`
/// at the primitive boundary (spec-critic DC-4).
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct AddBorderEntry {
    pub border_id: u32,
    pub side_a: PortRef,
    pub side_b: PortRef,
    pub worker_a: WorkerId,
    pub worker_b: WorkerId,
}
```

**Updated `add_border_states` body (mandated form):**

```rust
pub fn add_border_states(&mut self, entries: Vec<AddBorderEntry>) {
    for entry in entries {
        assert!(
            !self.borders.contains_key(&entry.border_id),
            "duplicate border_id {} in add_border_states",
            entry.border_id
        );
        let max_worker = self.worker_borders.len();
        assert!(
            (entry.worker_a as usize) < max_worker,
            "out-of-bounds worker {} for border_id {}",
            entry.worker_a, entry.border_id
        );
        assert!(
            (entry.worker_b as usize) < max_worker,
            "out-of-bounds worker {} for border_id {}",
            entry.worker_b, entry.border_id
        );

        let is_redex = is_principal_pair(entry.side_a, entry.side_b);
        let bid = entry.border_id;
        let wa = entry.worker_a as usize;
        let wb = entry.worker_b as usize;

        let state = BorderState {
            border_id: entry.border_id,
            side_a: entry.side_a,
            side_b: entry.side_b,
            worker_a: entry.worker_a,
            worker_b: entry.worker_b,
            is_redex,
        };

        self.borders.insert(bid, state);
        if is_redex {
            self.active_redexes.insert(bid);
        }
        self.worker_borders[wa].push(bid);
        self.worker_borders[wb].push(bid);
    }
}
```

**New test (mandated — test #11):**

```rust
#[test]
fn add_border_states_enforces_is_redex_invariant() {
    // Part 1: principal-principal entry → stored is_redex == true
    let mut graph = /* 2-worker empty graph */;
    graph.add_border_states(vec![AddBorderEntry {
        border_id: 1,
        side_a: PortRef::AgentPort(7, 0),  // principal
        side_b: PortRef::AgentPort(8, 0),  // principal
        worker_a: 0,
        worker_b: 1,
    }]);
    assert!(graph.borders.get(&1).unwrap().is_redex);
    assert!(graph.active_redexes.contains(&1));

    // Part 2: principal-auxiliary entry → stored is_redex == false
    graph.add_border_states(vec![AddBorderEntry {
        border_id: 2,
        side_a: PortRef::AgentPort(9, 0),  // principal
        side_b: PortRef::AgentPort(10, 1), // auxiliary
        worker_a: 0,
        worker_b: 1,
    }]);
    assert!(!graph.borders.get(&2).unwrap().is_redex);
    assert!(!graph.active_redexes.contains(&2));
}
```

**Reason recorded:** spec-critic ruled Option B wins on invariant-preservation grounds — under Option A, a caller can pass `BorderState { side_a: principal, side_b: principal, is_redex: false }` and break the R9 invariant the moment the struct enters the graph. Option B makes the primitive **coupling-safe** — the caller cannot write the `is_redex` bit at all. Matches `from_partition_plan`'s pattern (which computes `is_redex` via `is_principal_pair` at TASK-0361 line 160, not caller-trusted).

**LoC estimate:** slightly up (~55 LoC; +5 LoC for the new struct + derives, +1 LoC for the `is_principal_pair` call). Within the S-complexity budget.

**Test count impact:** tests go from 10 → **11**. Bundle index Stage 2 test estimate (~25 lib tests) grows to ~26 — below the "hard floor" margin; no further bundle-index change required (already updated by spec-critic).

**R9 invariant enforcement:** graph-level. The caller of `add_border_states` now **cannot** produce a state with a broken `is_redex` bit — the field is not in the input type.

**No spec amendment required.** R15 part 3 is shape-agnostic; the primitive's contract is strengthened, not weakened.
