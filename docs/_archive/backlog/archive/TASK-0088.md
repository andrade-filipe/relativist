## Context

Before the grid loop begins, the coordinator must open a TCP listener socket on the configured address (`bind: SocketAddr`, default `127.0.0.1:9000`) and accept connections from exactly `num_workers` workers. The coordinator MUST also process the registration handshake per R2a: in Tier 1, registration is implicit (no Register message required); in Tier 2/3, the Register/RegisterAck/RegisterNack handshake is mandatory. There is a configurable timeout (default 120 seconds) after which the coordinator aborts if not all workers have registered. This corresponds to the `WaitingForWorkers` state (SPEC-13 R19; SPEC-06 R26 is historical). The function returns a vector of `TcpStream` handles, one per worker, ready for the grid loop.

> **SPEC-06 v3 changes:** R26 demoted to historical (superseded by SPEC-13 R19). `WorkerCountMismatch` moved to `CoordinatorError`. Phase 0 now includes registration handshake (R2a). Bind address uses `SocketAddr` (not separate host+port).

## Notes

- The `TcpListener` is returned alongside the streams because the coordinator may need to keep it alive for the duration of the grid loop (depending on OS behavior with accepted sockets).
- The function uses `tokio::time::timeout` to wrap the accept loop, not a manual timer.
- Workers connect in arbitrary order. The index in the returned `Vec<TcpStream>` determines the worker's assignment (stream[i] receives partition[i]).
- Connection loss during execution is handled separately (R25) -- this task only covers the initial accept phase.
