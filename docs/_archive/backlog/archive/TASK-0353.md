# TASK-0353: Derive rkyv `Archive`/`Serialize`/`Deserialize` on the 8 hot-path types

**Spec:** SPEC-18 §3.5, §4.6
**Requirements:** R21
**Priority:** P0 (critical path)
**Status:** TODO
**Depends on:** TASK-0352
**Blocked by:** TASK-0352
**Estimated complexity:** M (~150 LoC across multiple files)
**Bundle:** SPEC-18 §3.5 (item 2.24) — Zero-Copy Archive

## Context

Adds rkyv derive macros to the 8 types that cross the wire on the hot path
(R21). All derives are gated under `#[cfg_attr(feature = "zero-copy", derive(...))]`
so the default build is unaffected. The 8 types are:
`Net`, `Partition`, `CompactSubnet`, `Agent`, `Symbol`, `PortRef`, `IdRange`,
`WorkerRoundStats`.

The challenge: `PortRef` already has manual `Serialize`/`Deserialize` impls
(SPEC-18 §3.2 / TASK-0344) for the compact serde encoding. Rkyv's derive
operates on the type structure directly and does NOT compose with serde
attributes — it generates its own `ArchivedPortRef` type. The two encodings
coexist: serde for bincode v2 (control + non-zero-copy hot-path), rkyv for
zero-copy hot-path.

`Partition.subnet` has a `#[serde(serialize_with = "...", deserialize_with = "...")]`
adapter pointing at `CompactSubnet`. Per SPEC-18 §4.6, rkyv archives the
`Partition` directly (full `Net` including arena), so the serde adapter is
NOT consulted on the rkyv path. No additional rkyv attributes needed.

## Key Types / Signatures

Example pattern (apply to all 8 types):

```rust
#[derive(Debug, Clone, /* existing derives */)]
#[cfg_attr(feature = "zero-copy", derive(rkyv::Archive, rkyv::Serialize, rkyv::Deserialize))]
#[cfg_attr(feature = "zero-copy", rkyv(check_bytes))]
pub struct Partition {
    // ... existing fields, including the `#[serde(...)]` adapter on `subnet` ...
}
```

For `PortRef` (which has manual Serialize/Deserialize impls):

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
#[cfg_attr(feature = "zero-copy", derive(rkyv::Archive, rkyv::Serialize, rkyv::Deserialize))]
#[cfg_attr(feature = "zero-copy", rkyv(check_bytes))]
pub enum PortRef {
    AgentPort(AgentId, PortId),
    FreePort(u32),
}

// Existing manual `impl serde::Serialize for PortRef` / `impl serde::Deserialize`
// is preserved — rkyv operates on the type, serde uses the manual impls.
```

## Dependencies Context

- `rkyv = "0.8"` is now an optional dep (TASK-0352).
- `rkyv::Archive`, `rkyv::Serialize`, `rkyv::Deserialize` are the three derive
  macros (note the `rkyv::` prefix — there are no name clashes with the serde
  derives because rkyv exports its own `Serialize`/`Deserialize` traits in
  the `rkyv::` namespace).
- `Net` contains `Vec<Option<Agent>>` and `Vec<PortRef>` (the arena and port
  array per SPEC-02). Both compose with rkyv as long as the inner types
  derive `Archive`. This is why all 8 types must be done in a single task —
  partial derivation breaks compilation.

## Notes

- **`PortRef` manual serde impls are NOT replaced.** They are the SPEC-18
  R5-R8 compact encoding (1-byte DISCONNECTED, varint AgentId). Rkyv has its
  own archive layout for enums (discriminant + payload, fixed offsets for
  alignment) — that is the rkyv wire format on the FLAG_ARCHIVED path. The
  two encodings produce different bytes; that's expected and per spec.
- **Rkyv 0.8 syntax:** the `#[rkyv(...)]` attribute takes options like
  `check_bytes` (enables validation, required for `rkyv::access`) and
  `omit_bounds` (suppresses generic bound generation). The developer should
  consult `rkyv` docs for any field type that fails to derive.
- **`HashMap<u32, PortRef>` in `Partition.free_port_index`:** rkyv 0.8
  archives `HashMap` via the `alloc` feature (already enabled in TASK-0352).
  No additional configuration needed.
- **`[u64; 6]` in `WorkerRoundStats.interactions_by_rule`:** primitive arrays
  archive cleanly in rkyv. No attribute needed.
- **`f64` in `WorkerRoundStats.reduce_duration_secs`:** rkyv handles `f64`
  via `rend::f64_le` under the hood (cross-endian safe). No attribute needed.
- If derivation fails on `Net`'s arena (`Vec<Option<Agent>>`) due to bound
  inference, add `#[rkyv(bound(serialize = "...", deserialize = "..."))]` on
  the field. The exact bound expression depends on rkyv's macro output —
  the developer will iterate to GREEN.
