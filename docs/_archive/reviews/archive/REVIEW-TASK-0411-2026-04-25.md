## 1. Summary

Both primitives are correctly implemented. `allocate_border_ids` satisfies the SPEC-20 A3 contract (atomic allocation, cursor not advanced on error, overflow guard is sound). `remap_partition_ids` satisfies A4 (structure-preserving renumber, FreePort entries untouched, `free_port_index` remapped, `redex_queue` remapped, `root` remapped, `freeport_redirects` reset to empty). Error variants are well-named, spec-cited, and produce human-readable messages. The 18 tests (13 integration + 5 inline) cover all happy-path, error-path, and edge cases mandated by TEST-SPEC-0411; determinism is guaranteed by the pure-synchronous nature of both functions.

The blast radius of 9 modified files is appropriate and unavoidable: adding a required field to `PartitionPlan` forces every construction site to supply it. The developer correctly threaded the field through all sites. However, one non-trivial construction site (`reconstruct_partition_plan_from_collected` in `merge/grid.rs`) hard-codes `next_border_id: 0` in a production code path that participates in departure recovery — this is a latent semantic bug (RV-001, HIGH).

The `IdRange::len` implementation uses `saturating_sub` rather than a direct subtraction, which creates a subtle but undocumented silent behaviour on inverted ranges (RV-002, MEDIUM). One missing test case from the TEST-SPEC is not implemented (UT-0411-09 idempotence test is present, but the EC-1 "exact maximal then exhausted" follow-up allocation is present in `grid.rs` under a different name — no gap there). One architecture observation: the absence of a `Default` impl for `PartitionPlan` is the proximate cause of the blast radius, and a simple `impl Default` would have quarantined it (RV-003, MEDIUM). One documentation gap: the `agent_count()` method's cast from `usize` to `u32` has a silent truncation on nets with more than `u32::MAX` agents (RV-004, MEDIUM). Two LOW items round out the review.

This is a sound implementation of a foundational primitive that gates three successor tasks (TASK-0440, TASK-0442, TASK-0443). **Cleared for Stage 5 QA after the HIGH fix lands; the three MEDIUM fixes may be co-landed with QA running in parallel.**

## 2. Findings

### HIGH

#### RV-001 — `reconstruct_partition_plan_from_collected` hard-codes `next_border_id: 0` in a production departure-recovery path

- **Category:** Architecture (semantic correctness, Spec Compliance)
- **Principle/Spec:** SPEC-20 §3.8 A3 (R18a), SPEC-20 §4.2.2 departure recovery step 4
- **File:** `relativist-core/src/merge/grid.rs:1235-1239`
- **Evidence:**
  ```rust
  crate::partition::PartitionPlan {
      partitions,
      borders,
      next_border_id: 0,   // <-- always 0
  }
  ```
  `reconstruct_partition_plan_from_collected` is called from two production code paths: `run_grid_delta_final_collect` (delta-mode BSP final merge) and, after TASK-0440/0442/0443 land, the departure-recovery path. In the final-collect path the plan is passed directly to `merge()`, which destructures it with `..` and never reads `next_border_id` — so `0` is harmless today. However, the function's contract comment already acknowledges SPEC-20 A8 semantics and `reclaimed_partitions`: once TASK-0440/0443 calls this path in departure-recovery mode, the returned plan will be immediately used as the `PartitionPlan` that `allocate_border_ids` is called on (R24d). At that point a `next_border_id` of `0` would allow re-allocation of border IDs that were already assigned in the original split — breaking D3 (border completeness) and potentially D4 (ID uniqueness).
- **Problem:** The cursor `0` is correct only for the final-collect path (plan is passed to `merge()` which ignores `next_border_id`). It is wrong for any departure-recovery caller that subsequently calls `allocate_border_ids` on the reconstructed plan. The bug does not fire today because TASK-0440/0443 have not yet landed — but this function already signals A8 awareness, so the semantic hazard is present in this PR.
- **Fix:** Pass the active `next_border_id` from the `BorderGraph` (or an explicit parameter) to `reconstruct_partition_plan_from_collected`. The `BorderGraph` owns the highest-known border id at the point of reconstruction. Two options:

  Option A — add a parameter (minimal, explicit):
  ```rust
  fn reconstruct_partition_plan_from_collected(
      mut partitions: Vec<crate::partition::Partition>,
      border_graph: &crate::merge::border_graph::BorderGraph,
      reclaimed_partitions: Vec<crate::partition::Partition>,
      next_border_id: u32,  // caller supplies from the active PartitionPlan
  ) -> crate::partition::PartitionPlan {
      // ...
      crate::partition::PartitionPlan {
          partitions,
          borders,
          next_border_id,
      }
  }
  ```
  Callers that produce a plan only for `merge()` pass `0` or `border_graph.max_border_id() + 1` explicitly — making the intent visible at the call site.

  Option B — derive `next_border_id` from `border_graph.borders.keys().max().copied().map(|m| m + 1).unwrap_or(0)` inside the function itself. This is weaker because `BorderGraph` may not retain borders that were removed via `remove_border` — so the maximum surviving key underestimates the true cursor.

  **Recommended fix: Option A.** The two current callers are `run_grid_delta_final_collect` (pass `0` explicitly — no semantic change for today's merge path, documents the intent) and future TASK-0440/0443 callers (pass the actual cursor from the active plan).

- **Why:** This is the one construction site where `next_border_id: 0` is not a test-only harmless literal — it is in a production function that is explicitly designed to accept reclaimed partitions. Leaving it unaddressed here means TASK-0440 will silently inherit the bug without a failing test, because the cursor value is only observable when `allocate_border_ids` is called after reconstruction.

### MEDIUM

#### RV-002 — `IdRange::len` uses `saturating_sub` but the silent-zero-on-inversion behaviour is undocumented

- **Category:** Code Quality (Rustdoc precision, hidden behavior)
- **Principle:** Meaningful names / clear contracts
- **File:** `relativist-core/src/partition/types.rs:31-33`
- **Evidence:**
  ```rust
  pub fn len(self) -> u32 {
      self.end.saturating_sub(self.start)
  }
  ```
  `saturating_sub` returns `0` when `self.end < self.start`. The Rustdoc says "Returns 0 when `start >= end` (empty range)." This is technically correct, but `start > end` is an inverted (malformed) range, not merely an empty one. The two cases are semantically distinct: an empty range (`start == end`) is a well-formed zero-length allocation; an inverted range (`start > end`) is a construction error. Treating them identically with a saturating subtraction silently masks inverted ranges — a caller who constructs `IdRange { start: 200, end: 100 }` by mistake will receive `len() == 0` and `is_empty() == true`, with no indication that the range is malformed.
- **Fix (two-part):**

  1. Add a `debug_assert!(self.start <= self.end, "IdRange is inverted: start={} > end={}", self.start, self.end)` at the top of `len` and `is_empty` (mirrors the QA-001 boundary-vigilance pattern from TASK-0410).
  2. Expand the Rustdoc to distinguish the two zero cases:
  ```rust
  /// Returns the number of IDs in the range `[start, end)`.
  ///
  /// Returns `0` for both the empty-range case (`start == end`) and the
  /// inverted-range case (`start > end`). Inverted ranges are a construction
  /// error; a `debug_assert!` fires in debug builds.
  pub fn len(self) -> u32 {
      debug_assert!(self.start <= self.end, "IdRange inverted: start={} > end={}", self.start, self.end);
      self.end.saturating_sub(self.start)
  }
  ```
  No change to the logic — `saturating_sub` remains correct for the normal case; the assert catches misuse in development.

- **Why:** `remap_partition_ids` checks `agent_count > range_size` to detect an undersized range. If `new_range` is accidentally inverted, `len()` returns `0`, `agent_count > 0` triggers `NewRangeTooSmall` — correct behaviour by coincidence, but without a diagnostic that tells the caller their input was inverted. The assert converts a confusing error into an immediately actionable one.

#### RV-003 — Absence of `impl Default for PartitionPlan` is the proximate cause of the 9-file blast radius; a `Default` impl would quarantine future field additions

- **Category:** Architecture (blast-radius containment, Rust idioms)
- **Principle:** OCP (extensible without modifying existing code) / DIP (depend on abstraction surface)
- **File:** `relativist-core/src/partition/types.rs` (missing impl)
- **Evidence:** Every construction site that builds a `PartitionPlan` for test-only or bridge purposes (8 of the 9 modified files, excluding `partition/split.rs` which correctly sets `next_border_id: wire_class.border_id_end`) had to be updated to add `next_border_id: 0`. In the future, if SPEC-22 or a subsequent elastic-grid amendment adds another field to `PartitionPlan`, every one of these sites will require an update again. For construction sites that only care about `partitions` and `borders` (e.g., pure-merge paths, test fixtures, `reconstruct_partition_plan_from_collected`), a `Default` impl with `next_border_id: 0` would allow struct-update syntax `{ partitions, borders, ..Default::default() }`, confining future field-addition churn to a single location.
- **Fix:**
  ```rust
  impl Default for PartitionPlan {
      /// Default cursor is 0 (no border IDs allocated yet).
      ///
      /// Construction sites that build a `PartitionPlan` purely for
      /// `merge()` consumption (which ignores `next_border_id`) should use
      /// struct-update syntax: `PartitionPlan { partitions, borders, ..Default::default() }`.
      /// Sites that will call `allocate_border_ids` MUST supply an
      /// explicit `next_border_id` derived from the preceding `split()`'s
      /// `wire_class.border_id_end`.
      fn default() -> Self {
          Self {
              partitions: Vec::new(),
              borders: HashMap::new(),
              next_border_id: 0,
          }
      }
  }
  ```
  This does not require updating any existing construction site (it is additive). Future field additions only require updating `Default::default()`.
- **Why:** The 9-file blast radius is unavoidable for the initial `next_border_id` field introduction, but it should not repeat. A `Default` impl is the idiomatic Rust mechanism to express "most construction sites only need the obvious zero-value" without propagating struct-literal churn.

#### RV-004 — `Partition::agent_count` silently truncates on nets with more than `u32::MAX` live agents

- **Category:** Code Quality (arithmetic safety, undocumented assumption)
- **Principle:** No magic numbers / defensive contracts
- **File:** `relativist-core/src/partition/types.rs:93-95`
- **Evidence:**
  ```rust
  pub fn agent_count(&self) -> u32 {
      self.subnet.count_live_agents() as u32
  }
  ```
  `count_live_agents()` returns `usize`. The `as u32` cast truncates silently if the count exceeds `u32::MAX` (4 294 967 295). In practice, an IC net with more than ~4 billion live agents is not a realistic scenario with current hardware, but the truncation is invisible and produces a wrong value for `remap_partition_ids`'s precondition check: `agent_count > range_size` would compare a truncated count against a correct range size and might wrongly accept an undersized range.
- **Fix:** Replace the silent cast with an explicit guard:
  ```rust
  pub fn agent_count(&self) -> u32 {
      let n = self.subnet.count_live_agents();
      debug_assert!(
          n <= u32::MAX as usize,
          "Partition::agent_count: agent count {} exceeds u32::MAX; remap precondition check will be wrong",
          n
      );
      n as u32
  }
  ```
  In release builds the cast is still `as u32` (no runtime penalty); the assert fires in debug builds on any future over-sized net before the remap precondition is silently broken. This mirrors the QA-001 boundary-vigilance pattern from TASK-0410.
- **Why:** The `remap_partition_ids` precondition check (`agent_count > range_size`) is the primary safety gate before renumbering. A silently-truncated `agent_count` is the only plausible path by which a malformed input escapes the guard without an error.

### LOW

#### RV-005 — `remap_partition_ids` silently drops `free_port_index` entries where the AgentPort target maps to a non-existent old id

- **Category:** Code Quality (silent data loss, undocumented behaviour)
- **File:** `relativist-core/src/partition/remap.rs:164-177`
- **Evidence:**
  ```rust
  let new_free_port_index: HashMap<u32, PortRef> = partition
      .free_port_index
      .iter()
      .filter_map(|(&bid, &port_ref)| {
          let remapped = match port_ref {
              PortRef::AgentPort(old_id, port) => id_map
                  .get(&old_id)
                  .map(|&new_id| PortRef::AgentPort(new_id, port)),
              PortRef::FreePort(_) => Some(port_ref),
          };
          remapped.map(|r| (bid, r))
      })
      .collect();
  ```
  When `id_map.get(&old_id)` returns `None` (the AgentPort target in `free_port_index` does not correspond to a live agent), the entry is silently dropped via `filter_map`. A well-formed partition from `split()` guarantees that every `free_port_index` entry points to a live agent — so this cannot happen in production. However, the behaviour is undocumented. Similarly, lines 132-139 in the internal port rewrite path preserve `old_target` unchanged when `id_map.get(&ref_id)` returns `None`, with the comment "should not occur in a well-formed partition ... but we preserve it." The two silent-failure policies are inconsistent: the port array path preserves the stale reference; the `free_port_index` path drops it.
- **Suggested fix (documentation only, no logic change):** Add a `debug_assert!` in the `free_port_index` rebuild that fires when `id_map.get(&old_id)` returns `None` for an `AgentPort` target:
  ```rust
  PortRef::AgentPort(old_id, port) => {
      debug_assert!(
          id_map.contains_key(&old_id),
          "free_port_index entry bid={} points to dead agent {}; partition is malformed",
          bid, old_id
      );
      id_map.get(&old_id).map(|&new_id| PortRef::AgentPort(new_id, port))
  }
  ```
  Also add a sentence to the Rustdoc's `# Semantics` section: "For a well-formed partition (as produced by `split()`), every `free_port_index` entry's `AgentPort` target corresponds to a live agent; entries referencing dead agents are silently dropped. A `debug_assert!` fires in debug builds if this invariant is violated."
- **Why:** The inconsistent silent-failure policies between port array and `free_port_index` will confuse a future maintainer who reads both comment threads. A shared `debug_assert!` policy makes both paths consistent.

#### RV-006 — `with_next_border_id` is the only constructor but its use forces the field to appear as a separate "step" rather than as part of struct initialization; the Rustdoc conflates two use cases

- **Category:** Code Quality (Rustdoc precision, API ergonomics)
- **File:** `relativist-core/src/partition/types.rs:125-131`
- **Evidence:** The Rustdoc for `with_next_border_id` says: "Used in tests and by the coordinator's departure-recovery path (SPEC-20 §4.2.2) to create a plan whose border-id allocator begins at a known offset." But the departure-recovery path (TASK-0440/0443) has not yet landed, so this is a forward claim. More importantly, the function name `with_next_border_id` implies the builder-pattern convention ("consume self, set field, return self"), but it is actually a constructor that creates a new empty plan. This is mildly confusing — a reader who sees `PartitionPlan::with_next_border_id(100)` might expect it to take an existing plan and return a modified one.
- **Suggested fix (two-part):**
  1. Add a `# Naming note` to the Rustdoc: "`with_next_border_id` follows the constructor-with-named-argument pattern rather than the `Builder::with_field` mutating-builder pattern. It always creates a new empty plan."
  2. Separate the two use cases in the Rustdoc: first, "used in tests when a known cursor offset is required"; second, "reserved for the coordinator's departure-recovery path (TASK-0440/0443, SPEC-20 §4.2.2) — not yet called in production code".
- **Why:** Clear documentation here prevents a future developer from assuming the function modifies an existing plan when refactoring the coordinator departure path.

## 3. Strengths

**Arithmetic safety in `allocate_border_ids` is sound and well-commented.** The `available = u32::MAX - self.next_border_id` formulation avoids overflow by computing the gap first (the comment correctly notes "no overflow possible since `u32::MAX >= next_border_id`"). The `count > available` guard then establishes `count <= u32::MAX - start`, making `start + count <= u32::MAX` trivially safe. The accompanying comment `// Safe: count <= available = u32::MAX - start` makes the invariant explicit. The cursor is not advanced on error — the "atomic allocation" property is both correctly implemented and explicitly documented. This is exactly the QA-001 boundary-vigilance pattern applied correctly to a new context.

**`remap_partition_ids` is structurally complete.** Every component of a `Partition`/`Net` that can reference an `AgentId` is explicitly handled: `agents` array (new arena), `ports` array (full rewrite), `redex_queue` (filter-map with both-or-none semantics), `free_port_index` (remapped), `root` (remapped), `freeport_redirects` (reset to empty, correct because redirects are build-time artifacts). The `DISCONNECTED` sentinel and `FreePort` entries are correctly left untouched. This level of completeness mirrors `Net::union`'s field-enumeration discipline from TASK-0410 and sets the right standard for subsequent elastic-grid tasks.

**Test-spec mapping is 1:1 and the fixture is realistic.** Every UT and EC from TEST-SPEC-0411 is implemented. The `make_p0` fixture uses ids at offset 10 (not starting at 0), internal wires at non-principal ports, and a `FreePort(42)` border marker — matching the "agents at [10,14)" spec fixture exactly. The `make_two_agent_partition` inline fixture in `remap.rs` is independently constructed and thus cross-validates the integration fixture. The idempotence test (UT-0411-09) is present and meaningful.

**Cascading struct-literal updates are mechanically correct.** Every non-test `PartitionPlan` construction site correctly supplies `next_border_id`. In particular, `split.rs`'s `trivial_plan` (line 117) correctly uses `next_border_id: 0` (no borders in a trivial split), and the non-trivial path (line 84) correctly uses `next_border_id: wire_class.border_id_end` — these two are the semantically important sites, and both are right.

**`protocol/coordinator.rs` correctly forwards the cursor.** The coordinator's merge-plan reconstruction (line 484: `next_border_id: plan.next_border_id`) correctly propagates the cursor from the original split plan to the merge plan. This is the one coordination site where the value matters outside of `partition/` and it is handled correctly.

**Error variants satisfy NF-006.** `BorderIdSpaceExhausted { requested, available }` and `NewRangeTooSmall { partition_size, range_size }` are specific, carry full context, and produce readable messages ("border_id space exhausted: requested N ids but only M remain"). Both are cited in the Rustdoc of their respective callers and in `error.rs`. The pattern is consistent with existing `PartitionError` variants.

**Composition with TASK-0410 (`Net::union`) is clean.** The intended call sequence — `remap_partition_ids(reclaimed, fresh_range) → union(surviving_net, reclaimed.subnet) → split(union_result, K_eff_new)` — is fully supported. `remap_partition_ids` establishes the `AgentId` disjointness precondition that `Net::union` requires (SPEC-20 A7 / TASK-0410 Rustdoc). The TASK-0410 review (RV-002) flagged `force_agent_id_range` as a test fixture that "mimics the postcondition that `remap_partition_ids` (TASK-0411) will establish in production" — that prediction is now confirmed correct by this implementation.

**Serde/rkyv regression risk for new error variants is nil.** `PartitionError` is not serialized over the wire (it is a Rust error type, not a protocol message). The two new variants (`BorderIdSpaceExhausted`, `NewRangeTooSmall`) are local to the coordinator; they never cross a worker boundary. No wire-format change, no bincode/rkyv schema impact.

**`PartitionPlan` serde impact is contained.** `PartitionPlan` does derive `serde::Serialize/Deserialize` (types.rs:99). Adding `next_border_id: u32` is a backward-incompatible change to the bincode layout if any existing serialized `PartitionPlan` is read from disk or wire. However, `PartitionPlan` is not part of the wire protocol (it is a coordinator-local data structure, not a `Message` variant). The only persistence of `PartitionPlan` is in the `retain_partitions` / `checkpoint_partitions` path (SPEC-20 R23a-d, TASK-0440), which has not yet landed. There is no existing on-disk format to break.

**Successor task readiness:** TASK-0440 (v1 departure re-split) can proceed: it needs both `allocate_border_ids` (to rebase border IDs, R24d) and `remap_partition_ids` (to renumber reclaimed partitions, R30). Both are now available. TASK-0440 should incorporate the RV-001 fix as its first action (updating `reconstruct_partition_plan_from_collected`'s call site to pass the actual cursor).
