## 3. Should-Fix Issues

### SF-001: Backward-compat assertion in UT-0412-01 is structurally weak — it pins agent count, not structural identity

**Category:** Test Quality
**Principle:** Regression guarantee stated in TASK-0412 acceptance criterion: "When `reclaimed_partitions.is_empty()`, behavior is bit-identical to the existing 2-argument signature."
**File:** `relativist-core/src/merge/grid.rs`, UT-0412-01, all four fixtures (approximately lines 4501–4662)

**Problem:** Every fixture in `reconstruct_empty_reclaimed_matches_legacy` asserts `net_new.count_live_agents() == net_legacy.count_live_agents()`. Agent count is a coarse proxy for structural identity. Two nets can have the same agent count while differing in wire topology (e.g., different FreePort residuals after merge) — especially if a future change to `merge()` or `reconstruct_partition_plan_from_collected` introduces a bug only in the reclaimed-path branch. The legacy path and the new path are themselves both computed via `reconstruct_partition_plan_from_collected` (the new path is a strict wrapper), so the structural proof is trivially correct-by-construction in the current code — but the test does not capture this reasoning, and a refactor that separates the two code paths would break the identity property invisibly.

The stronger claim that the test should pin is: when `reclaimed_partitions.is_empty()`, the resulting `Net` must have the same live agents (by `AgentId` set or at least by multiset of symbols) and the same port topology. For non-border fixtures (A, B, C, fixtures used in UT-0412-02/03/EC-*) a multiset-of-symbols comparison is sufficient. For fixture D (border present), a wire-topology comparison would be the gold standard, but at minimum both legacy and new should produce a `Net` where agent `a` and agent `b` are directly wired at port 1 — which the current assertion does not verify.

**Before (current — all four fixtures):**
```rust
assert_eq!(
    net_new.count_live_agents(),
    net_legacy.count_live_agents(),
    "UT-0412-01 fixture D: two workers + border, empty reclaimed must match legacy"
);
```

**After (strengthen to symbol multiset + optional spot-wire check):**
```rust
// Agent count parity (necessary but not sufficient).
assert_eq!(
    net_new.count_live_agents(),
    net_legacy.count_live_agents(),
    "UT-0412-01 fixture D: agent count must match legacy"
);
// Symbol multiset parity (sufficient for non-border fixtures; also covers D).
let symbols_new: std::collections::BTreeMap<_, _> = net_new
    .live_agents()
    .map(|a| (a.id, a.symbol))
    .collect();
let symbols_legacy: std::collections::BTreeMap<_, _> = net_legacy
    .live_agents()
    .map(|a| (a.id, a.symbol))
    .collect();
assert_eq!(
    symbols_new,
    symbols_legacy,
    "UT-0412-01 fixture D: symbol map must match legacy (bit-exact regression check)"
);
```

Note: `live_agents()` is already used throughout the test file; this pattern adds zero new dependencies. If `AgentId` allocation is not deterministic across two separate calls to `merge()` (because both start from `next_id = 0`), substitute a `BTreeMap<Symbol, usize>` count-by-symbol instead of identity-keyed. The TASK-0410 review (REVIEW-TASK-0410 §3, "Test-spec mapping is 1:1") established `live_symbols_multiset` as a test utility in `tests/net_union.rs`; that helper or an equivalent should be added for TASK-0412's in-module tests.

**Why:** The acceptance criterion says "bit-identical". The current assertion does not verify identity — it verifies one scalar derived from the result. If the backward-compat path ever silently diverges (e.g., port order in `PartitionPlan.borders` map changes), the test would not catch it.

**Severity rationale:** MEDIUM because the correctness proof is clear-by-construction for the current implementation, but the test gives no regression protection against future refactors that separate the two code paths.

### SF-002: `reconstruct`'s `debug_assert!` checks `worker_id` overlap as proxy for `AgentId` overlap — the proxy's precondition is uncited

**Category:** Code Quality (Defensive contract)
**Principle:** DIP / documentation-as-contract
**File:** `relativist-core/src/merge/grid.rs`, lines 1277–1296

**Problem:** The `debug_assert!` comment correctly explains the proxy reasoning: "SPEC-20 R11 forbids WorkerId reuse, so a worker_id collision implies an AgentId-range collision (D4 breach)." However, SPEC-20 R11 is not cited in the assert message itself or the surrounding comment. A future reader — or a caller that constructs partitions outside the normal BSP pipeline — needs to be able to trace why `worker_id` disjointness is a sound proxy. If SPEC-20 R11's semantics are ever relaxed (e.g., in a compaction step), the proxy silently becomes incorrect while the debug_assert still fires.

Additionally, the proxy checks only `surviving_wids` vs. `reclaimed_partitions` — it does NOT check pairwise disjointness within `reclaimed_partitions` itself (i.e., two reclaimed partitions with the same `worker_id`). This is a gap in the precondition check: if `remap_partition_ids` misbehaves and produces two reclaimed partitions with identical `worker_id`, the merge will silently produce incorrect topology without any assertion firing.

**Before (lines 1277–1296):**
```rust
// SPEC-20 A8: debug-assert pairwise-disjoint worker_ids as a proxy for
// the AgentId-disjoint precondition. Each Partition carries exactly one
// worker_id, and SPEC-20 R11 forbids WorkerId reuse, so a worker_id
// collision implies an AgentId-range collision (D4 breach).
#[cfg(debug_assertions)]
{
    use std::collections::HashSet;
    let surviving_wids: HashSet<crate::partition::WorkerId> =
        surviving_partitions.iter().map(|p| p.worker_id).collect();
    for rp in &reclaimed_partitions {
        debug_assert!(
            !surviving_wids.contains(&rp.worker_id),
            ...
        );
    }
}
```

**After (extend to cover intra-reclaimed duplicates; add R11 citation):**
```rust
// SPEC-20 A8: debug-assert pairwise-disjoint worker_ids as a proxy for
// the AgentId-disjoint precondition (SPEC-20 R11: WorkerIds are never
// reused across active slots, so a worker_id collision implies an
// AgentId-range collision — D4 breach). The proxy is sound only while
// R11 holds; if R11 is relaxed, this assert must be upgraded to a
// direct AgentId-range disjointness scan.
#[cfg(debug_assertions)]
{
    use std::collections::HashSet;
    let surviving_wids: HashSet<crate::partition::WorkerId> =
        surviving_partitions.iter().map(|p| p.worker_id).collect();
    // Check reclaimed vs. surviving.
    for rp in &reclaimed_partitions {
        debug_assert!(
            !surviving_wids.contains(&rp.worker_id),
            "SPEC-20 A8 / R11: reclaimed worker_id {} overlaps a surviving \
             partition; caller MUST call remap_partition_ids (A4) first",
            rp.worker_id
        );
    }
    // Check within reclaimed set (two badly-remapped partitions would
    // also violate D4 while evading the surviving-only check above).
    let mut seen_reclaimed: HashSet<crate::partition::WorkerId> = HashSet::new();
    for rp in &reclaimed_partitions {
        debug_assert!(
            seen_reclaimed.insert(rp.worker_id),
            "SPEC-20 A8 / R11: two reclaimed partitions share worker_id {}; \
             remap_partition_ids must produce unique ids per partition",
            rp.worker_id
        );
    }
}
```

**Why:** The existing assert covers only the surviving-vs-reclaimed axis. The intra-reclaimed axis is entirely unguarded. Given that the correctness of the merge depends on total pairwise disjointness across all partitions, a half-coverage assert is architecturally weaker than the spec demands.

**Severity rationale:** MEDIUM because the missing check corresponds to a real, reachable failure mode (a buggy `remap_partition_ids` producing duplicate `worker_id`s among the reclaimed set), and the current test suite does not exercise intra-reclaimed collision at all.

### SF-003: `reconstruct_partition_plan_from_collected` is `fn` (private) but callers reference it both as the legacy baseline (UT-0412-01) and as a migration target — the naming contract is unclear

**Category:** Code Quality (Naming / SRP)
**Principle:** Meaningful names; OCP
**File:** `relativist-core/src/merge/grid.rs`, lines 1205–1240

**Problem:** `reconstruct_partition_plan_from_collected` is used in two semantically distinct roles:
1. As a **production helper** called by `run_grid_delta_final_collect` and `reconstruct` (the public entry point).
2. As a **test reference baseline** for UT-0412-01 ("what the 2-arg signature used to produce").

This dual use is defensible — the function IS the implementation that both paths share — but the Rustdoc does not explain it. The result is that UT-0412-01 constructs its "legacy" net by calling `reconstruct_partition_plan_from_collected` + `merge` directly, while the "new" net calls `reconstruct` (which internally calls the same helper). These two paths are currently *identical* by construction, making the assertion trivially true rather than testing a real behavioral invariant. This means UT-0412-01 does not actually validate the backward-compat claim against an *independent* baseline — it validates that `reconstruct` wraps its own helper correctly, which is a much weaker assertion.

The test-spec (TEST-SPEC-0412 UT-0412-01) explicitly called for the comparison to be against "a structural canonicalised expectation" or "a pre-amendment baseline captured as a structural expected value." The implementation chose to compare against the live helper call instead — which compiles and passes, but provides no regression safety if the helper itself regresses.

This is not a blocker, but the test comment on fixture D (lines 4558–4577) is the only place that explains the fixture shape for the border case. That explanation should be summarised in the helper's Rustdoc so future TASK-0443 authors understand what preconditions the helper expects.

**Suggested fix:** Add a `# Test-baseline contract` paragraph to `reconstruct_partition_plan_from_collected`'s Rustdoc: "UT-0412-01 uses this function directly as the 2-arg legacy baseline. The assertion in UT-0412-01 verifies that `reconstruct(bg, s, Vec::new())` is equivalent to calling this function with `Vec::new()` and then `merge()`. This is currently guaranteed by construction (both code paths ARE identical); the test anchors the intent so a future code split cannot silently diverge." Also note in the UT-0412-01 comments that the "legacy" reference is the same internal path, so the test's protection is against wrapper divergence, not independent implementation divergence.

**Severity rationale:** MEDIUM because it explains a subtle design choice that has already confused at least one reader (the orchestrator when diagnosing the Stage 3 failures), and the test provides weaker guarantees than the test-spec intended.

### NTH-001: `simple_era_partition` and `lone_con_partition` have inconsistent `AgentId` offset strategies — document the mixing rule

**Category:** Test Quality
**File:** `relativist-core/src/merge/grid.rs`, lines 3735–3754 and 4468–4494

`lone_con_partition(worker_id, base_free)` sets `subnet.next_id = worker_id` (offsets by 1-per-worker), while `simple_era_partition(worker_id)` sets `subnet.next_id = worker_id * 1000` (offsets by 1000-per-worker). When both fixtures are mixed in the same test — as they are in UT-0412-02 (`lone_con_partition(0,100)`, `lone_con_partition(1,200)`, `simple_era_partition(2)`) — the `AgentId` ranges are:

- `lone_con_partition(0)`: agent id = 0
- `lone_con_partition(1)`: agent id = 1
- `simple_era_partition(2)`: agent id = 2000

These are disjoint — correct — but the rule is implicit. If a future test adds `lone_con_partition(2, ...)`, it would produce agent id = 2, colliding with `simple_era_partition(2)`'s `next_id` offset of 2000 only if the ERA agent ends up at id 2000, which it does (not 2) — so in practice there is no collision today. However, the gap between the two strategies (multiplier vs. identity) means a developer adding `simple_era_partition(0)` alongside `lone_con_partition(0, ...)` would produce agent ids 0 (from CON) and 0 (from ERA, since `worker_id * 1000 = 0`), which IS a collision. This edge case is not guarded.

**Suggested fix:** Add a module-level comment in the TASK-0412 test block header: "Fixture disjointness contract: `simple_era_partition(w)` uses `next_id = w * 1000` (avoids 0); `lone_con_partition(w, ...)` uses `next_id = w` (starts from 0 for w=0). Do NOT mix `simple_era_partition(0)` with `lone_con_partition(0, ...)` in the same test — they produce overlapping AgentId 0." The simplest long-term fix is to make `simple_era_partition(0)` use `next_id = 10000 * (worker_id + 1)` (i.e., start from 10000, not 0), but that is a fixture change and not strictly necessary now.

### NTH-004: UT-0412-05's border-completeness assertion only checks agent count, not wire resolution

**Category:** Test Quality
**File:** `relativist-core/src/merge/grid.rs`, lines 4830–4835

UT-0412-05 claims to verify D3 (Border Completeness): "After reconstruction, all border references in `n` resolve into actual agents (no dangling FreePorts)." The actual assertion is `assert_eq!(net.count_live_agents(), 2)`. Agent count = 2 does not prove that border wire `bid=1` was resolved into a direct `AgentPort(a0, 0) ↔ AgentPort(b0, 0)` connection — it only proves both agents survived `merge()`. A dangling FreePort would also yield count = 2. The D3 claim in the comment is therefore unverified by the test body. Suggest adding a check that neither agent has a FreePort as its port-0 target after merge (e.g., check `net.get_target(PortRef::AgentPort(a0_id, 0))` is not a `PortRef::FreePort`). This requires knowing the output agent ids after merge, which in the current fixture would be deterministic (both agents live, ids preserved by merge).

## 6. Architecture Assessment

### Placement: `reconstruct` in `merge/grid.rs`

The function is correctly placed. `grid.rs` is the home of the BSP cycle orchestration logic (SPEC-05 §4.5), and `reconstruct` is the final-merge assembly step (SPEC-19 R38 / SPEC-20 §4.2.2). Moving it to `merge/core.rs` would mix the low-level merge primitive with the high-level assembly logic; moving it to `merge/border_graph.rs` would give `border_graph.rs` a dependency on `Partition` and `PartitionPlan`, which are `partition/` types, violating the dependency direction (SPEC-13: `merge/` depends on `partition/`; `border_graph.rs` is a `merge/` sub-module that should remain focused on border state, not partition reconstruction). `grid.rs` is the natural composition point.

### `pub(crate)` with `#[allow(dead_code)]` until Phase B+

This is defensible and matches the pattern established by TASK-0410's `Net::union` (which was also annotated pending its callers). One concern: `reconstruct` is `pub(crate)` but its internal helper `reconstruct_partition_plan_from_collected` is `fn` (private to the module). The `pub(crate)` surface is exactly what TASK-0443 needs, and nothing else in the current codebase needs it. The annotation is correct.

### `super::core::merge` call in `reconstruct` (line 1302)

The call resolves to the `merge/core.rs` function correctly. The notation `super::core::merge` is module-path verbose but unambiguous from inside `merge/grid.rs`. This matches the convention in `run_grid_delta_final_collect` (line 1200), so it is locally consistent.

### Dependency direction

`reconstruct` depends on `BorderGraph` (from `merge/border_graph.rs`), `Partition` and `PartitionPlan` (from `partition/`), and `Net` (from `net/`). These are all legal downward dependencies per SPEC-13. No new upward dependency was introduced.

### `Net::union` NOT used by `reconstruct`

TASK-0412 acceptance criteria say "Semantics: treats `surviving_partitions ∪ reclaimed_partitions` as the input partition set (union via `Net::union` under disjointness precondition)." The implementation does NOT call `Net::union`. Instead it extends the `Vec<Partition>` input to `reconstruct_partition_plan_from_collected`, which passes the combined vector to `merge()` (SPEC-05's full merge, which internally unions the partition subnets). This is semantically equivalent but architecturally different from the spec's stated mechanism.

This is a **minor spec-prose vs. implementation divergence** worth flagging. SPEC-20 §3.8 A8 prose says the reclaimed partitions "are appended to the surviving partition list before the subsequent `split()`", which aligns with the implementation. The earlier cross-spec note in the A8 prose (§3.8 line ~583) references `Net::union` in the context of A7 (TASK-0410), not A8. The implementation's approach of extending the `Vec<Partition>` rather than calling `Net::union` is arguably cleaner because `merge()` already handles the per-partition net assembly — calling `Net::union` inside `reconstruct` would be redundant with what `merge()` does in the next step. This should be documented explicitly in the `reconstruct` Rustdoc: "Note: reclaimed partitions are appended to the `PartitionPlan` input rather than unioned via `Net::union` (TASK-0410/A7) — `merge()` performs the equivalent structural concatenation internally. `Net::union` would be appropriate only if the call site needs a pre-assembled `Net` before `split()`."

This is LOW severity (NTH-level), but note it for the SPEC-19 cross-reference patch that ESPECIALISTA EM SPECS will author.
