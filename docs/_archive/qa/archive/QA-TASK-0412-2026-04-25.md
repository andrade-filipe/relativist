## Findings

### QA-001 (CRITICAL) — Release-build silent topology corruption when two partitions in the merged set share an AgentId range

- **Severity:** CRITICAL — release-build silent D4 (ID Uniqueness) breach reachable through any caller that violates the (debug-only-and-incompletely-asserted) AgentId-disjointness precondition.
- **Category:** Correctness (D-invariant) / Untested-path / Defensive-contract
- **Location:** `relativist-core/src/merge/grid.rs:1281-1296` (the `#[cfg(debug_assertions)]` block) and `relativist-core/src/merge/core.rs:85-127` (`merge()` step 2).

**The bug.** `reconstruct`'s Rustdoc (lines 1253-1258) declares:

> The live AgentId sets of every partition in `surviving_partitions ∪ reclaimed_partitions` are pairwise disjoint (SPEC-20 §3.8 A8 / D4). Violation fires a `debug_assert!` citing SPEC-20 A8 and the conflicting `worker_id`.

The actual debug assert (lines 1281-1296) checks **only `worker_id` disjointness, only between `surviving` and `reclaimed`**. There are three failure modes the assert does NOT catch:

1. **Intra-reclaimed `worker_id` duplication** (already flagged by SF-002 in Stage 4) — two reclaimed partitions with the same `worker_id`.
2. **AgentId overlap with disjoint `worker_id`s** — the proxy's blind spot: caller hands two partitions whose `worker_id` differs (so the assert is happy) but whose `id_range` overlaps. This is what SPEC-20 R30 / A4's `remap_partition_ids` is supposed to prevent, but the precondition is *transitive*: this function trusts that. A future Phase-B-or-later caller (TASK-0440 / TASK-0443) that wires `reconstruct` without first running `remap_partition_ids` (e.g., a bench harness, a fuzz driver, an integration test fixture) walks straight into it.
3. **All asserts disabled in release** — even the partial check above is `#[cfg(debug_assertions)]`. Release benchmarks and production runs (the `--release` builds the whole `bench/` infra exercises) get *zero* protection.

**What happens in release when AgentId ranges overlap?** Trace through `merge::core::merge` (core.rs:85-127):

```rust
for partition in &partitions {                                  // step 2
    for (i, slot) in partition.subnet.agents.iter().enumerate() {
        if let Some(agent) = slot {
            let id = i as u32;
            if result.agents.len() <= i { result.agents.resize(i + 1, None); }
            result.agents[i] = Some(*agent);                    // (*) overwrite!
            ...
            for p in 0..num_ports {
                let target = partition.subnet.get_target(PortRef::AgentPort(id, p));
                ...
                result.ports[idx] = target;                     // (**) overwrite!
```

Line `(*)` is **last-writer-wins by AgentId index** across partitions. If two partitions both carry an agent at slot `i`, the second one's `Symbol` and `Agent` payload silently clobbers the first's. Line `(**)` is the same for that agent's port wiring. The result is a `Net` whose AgentId `i` carries partition-2's `Symbol` but possibly partition-1's port targets (if partition-2 doesn't write that specific port slot, e.g., because partition-2's agent at `i` is an ERA with arity 0 while partition-1's was a CON with arity 2: ports 1 and 2 retain partition-1's targets, which now reference an agent that has been silently mutated to ERA).

**Reproduction (static trace, no cargo run, release-mode behaviour):**

```rust
// Hand-built reclaimed partitions that violate AgentId-disjointness while
// having distinct worker_ids (so the proxy assert is satisfied).
let mut net_a = Net::new();
net_a.next_id = 0;
let a = net_a.create_agent(Symbol::Con);  // id = 0
net_a.connect(PortRef::AgentPort(a, 0), PortRef::FreePort(100));
net_a.connect(PortRef::AgentPort(a, 1), PortRef::FreePort(101));
net_a.connect(PortRef::AgentPort(a, 2), PortRef::FreePort(102));
let p_a = Partition {
    subnet: net_a, worker_id: 0,
    free_port_index: HashMap::new(),
    id_range: IdRange { start: 0, end: 1 },
    border_id_start: 0, border_id_end: 0,
};

let mut net_b = Net::new();
net_b.next_id = 0;                        // OVERLAPPING — caller forgot remap
let b = net_b.create_agent(Symbol::Era);  // id = 0 (collides with p_a)
net_b.connect(PortRef::AgentPort(b, 0), PortRef::FreePort(200));
let p_b = Partition {
    subnet: net_b, worker_id: 1,          // distinct worker_id — proxy assert green
    free_port_index: HashMap::new(),
    id_range: IdRange { start: 0, end: 1 },
    border_id_start: 0, border_id_end: 0,
};

// reconstruct's debug_assert checks worker_id 0 vs {1} — disjoint, no panic.
// In release: no check at all.
let net = reconstruct(&empty_graph, vec![p_a], vec![p_b]);

// Result: net.agents[0] == Some(Era) (last writer);
//         net.ports[0] == FreePort(200), net.ports[1..3] == FreePort(101..102)
//         (partition-A's port-1 / port-2 wiring survives because Era's
//          step-2 loop only writes port 0; CON's earlier writes at
//          port_idx 1 and 2 are NOT cleaned up).
// count_live_agents() returns 1 (one CON-and-then-Era at id 0).
// I6 violation: ERA agent has non-DISCONNECTED at port 1 and port 2.
// In debug: result.assert_all_invariants() at core.rs:170 fires
//   `assert_era_unused_ports_clean` panic — but the panic message is
//   "I6 violated", citing nothing about the actual root cause
//   (overlapping AgentId ranges).
// In release: the panic does NOT happen and the corrupt net flows
//   downstream into split() / serialise() / reduce_all().
```

**Why this is reachable in practice.**

- `reconstruct` is `pub(crate)` and `#[allow(dead_code)]` because Phase A only ships it; Phase B (TASK-0440 / TASK-0443) wires actual departure-recovery callers. The Phase B authors will read the Rustdoc, see "caller MUST renumber via `remap_partition_ids`", and *call it* — but the contract enforcement lives in three places: SPEC-20 R30, SPEC-20 A4, and TASK-0411 (`remap_partition_ids` itself, owned by a separate task). If the Phase B integrator forgets one (e.g., calls `remap_partition_ids` only on partitions returned by transport, but not on partitions reconstructed from `retained_initial`), the corruption is silent.
- The **fixture-rescue PF-001** noted by the reviewer is precisely the symptom: the orchestrator's two fixture-fix passes were needed because hand-built fixtures *did* violate disjointness in subtle ways. Hand-built fixtures are exactly the population that future bench / fuzz / integration tests will keep generating. PF-001 is a developer-process concern; QA-001 is the **product-defect mirror** of the same root cause: the function's "MUST" precondition is enforced as a partial debug assert, not as a total runtime check.

**Why the proxy is unsound under non-pipeline callers.** SPEC-20 R11 says `WorkerId` is never reused across active slots. R30 / A4 says reclaimed partitions are renumbered via `remap_partition_ids` so their `id_range` ends up disjoint. The Stage 4 review (SF-002) correctly notes that R11 makes `worker_id`-disjointness a sound proxy for AgentId-disjointness *along the BSP pipeline*. But the proxy depends on **two separate facts**:

> (i) every partition's `worker_id` is unique in the active set (R11), AND
> (ii) every partition's `id_range` is the canonical one assigned by `compute_id_ranges(partition_index)` for that worker_id (R8 / R13 / R30).

The implementation only depends on (i). A caller that constructs a `Partition` by hand with `worker_id: 99` and `id_range: IdRange { start: 0, end: 1 }` (because they hand-crafted the subnet too) breaks (ii) without breaking (i), and the proxy assert is happy. This is the **core architectural defect**: the function's signature accepts `Partition` (a value type with public-via-`pub(crate)` fields), so any caller can violate (ii) by construction. The proxy is sound *only* under the implicit assumption that all `Partition` values flow through the BSP pipeline — which is true today (Phase A has no Phase B callers wired) but **silently false** the moment Phase B integrators or test authors instantiate `Partition` directly.

**Reachable severity.** This is CRITICAL because:
- the corruption is silent in release,
- the corruption survives a partial debug assert (the worker_id proxy is happy),
- the corruption can survive `assert_all_invariants` if the overwriting agent has the same arity as the overwritten one (e.g., CON ↔ DUP collision at the same id: I1/I2 might still hold because both have 3 ports and the port wiring is mostly preserved; only the `Symbol` field of the Agent struct differs — and `assert_all_invariants` does NOT check Symbol identity post-merge against any reference),
- the input space (a hand-built `Partition` with overlapping `id_range`) is exactly what fixture authors and Phase B integrators will produce.

**Suggested fix (Stage 6).** Three options, in increasing rigour:

1. **Cheap (1-line):** make the `debug_assert!` block also unconditional in release for the structural check at the cost of a single `HashSet<WorkerId>` allocation per call. `reconstruct` is invoked at round boundaries, not on a hot path — the cost is negligible. Drop `#[cfg(debug_assertions)]` and turn `debug_assert!` into `assert!`.
2. **Correct:** check `id_range` overlap directly, not via `worker_id`. Iterate `surviving ∪ reclaimed`, sort by `id_range.start`, verify `prev.end <= curr.start` for adjacent pairs. O(K log K) where K = number of partitions; trivial. This addresses (ii) above.
3. **Defensive (highest cost, recommended for the 3-arg path specifically):** in addition to (2), iterate every partition's `subnet.agents` and verify the actual minted-id set is a subset of the declared `id_range`. Catches the "I lied about my id_range" failure mode that (2) does not. This is the only check that survives a hostile caller; given that `reconstruct` is `pub(crate)` and not yet wired, this is overkill *today* — but TASK-0443's Phase B integration will eventually expose it to fixture-driven callers, at which point this check is the only structural guarantee.

The Stage 4 SF-002 fix (intra-reclaimed `worker_id` check) is necessary but not sufficient — it closes mode (1) of the three failure modes above, leaving modes (2) and (3) open.

### QA-002 (HIGH) — UT-0412-04's panic-fixture confounds two failure modes; does not prove the proxy assert fires for the assertion's stated precondition

- **Severity:** HIGH — the only test that exercises the precondition assert does so under fixture conditions where the proxy and the underlying invariant are *both* violated, so a regression that disables the proxy assert (or mis-cites the wrong invariant) would still pass the test as long as `merge()` itself panics first on the AgentId collision via `assert_all_invariants`.
- **Category:** Test Quality / Confounded fixture
- **Location:** `relativist-core/src/merge/grid.rs:4716-4745` (`reconstruct_panics_on_overlapping_reclaimed_worker_id_debug_build`).

**The bug.** UT-0412-04 builds:

```rust
let p0 = lone_con_partition(0, 100);   // worker_id 0, AgentId 0
let r_collide = lone_con_partition(0, 500);  // worker_id 0, AgentId 0
```

`lone_con_partition` (grid.rs:3735) sets `subnet.next_id = worker_id` and mints exactly one CON, so `worker_id = 0` produces AgentId 0 for both fixtures. The test then calls `reconstruct(&graph, vec![p0], vec![r_collide])` and asserts a panic occurs whose message contains "A8" or "A7".

**Confounded failure modes.** A panic in this fixture can come from any of three sources:
1. The `debug_assert!` in `reconstruct` itself (the test's intended target) — message contains "A8".
2. `merge::core::merge`'s tail invocation `result.assert_all_invariants()` (core.rs:170) — fires if the AgentId collision corrupts the net (e.g., I1 violation if port targets disagree). Message would mention "I1" or similar, NOT "A8".
3. Either of the above can fire under different ordering — the assertion in `reconstruct` is *first*, so under normal flow it wins. But if a future refactor moves the assert (e.g., into the helper, post-merge, etc.) the test becomes silently dependent on `merge()`'s downstream check.

**Why the panic-message check `"A8" || "A7"` is too weak.** The test accepts *either* token. "A7" is the citation for `Net::union` (TASK-0410), unrelated to the 3-arg `reconstruct` precondition. The reviewer's note in REVIEW §6 ("Net::union NOT used by `reconstruct`") confirms A7 is not actually in the call path. Including "A7" as an acceptable token is defensive against panic messages that *might* mention the up-stream A7 contract, but it weakens the regression guarantee: if a future change replaces the A8-citing message with a generic "AgentId-disjoint precondition violated" message, the test would fail (no "A8") even though the structural intent — "the assert fires" — is preserved. Conversely, if a future bug *removes* the proxy assert entirely, `merge()`'s downstream invariant check could still panic (e.g., I1 / I2 / next_id) and would NOT contain "A8" or "A7" — so the test correctly fails. That part is fine. The actual gap:

**Missing test cases.** The current test does not cover:
- (a) `worker_id` colliding but AgentIds disjoint (the proxy's *only* unique purpose — to fire on `worker_id` collision *as a stand-in* for AgentId collision before `merge()` sees it). Construct: `lone_con_partition(0, 100)` survivor + a hand-built reclaimed with `worker_id=0` but `subnet.next_id=1000` and AgentId 1000. The proxy MUST fire; `merge()` would not have caught it.
- (b) Intra-reclaimed `worker_id` collision (SF-002 expansion). Construct two reclaimed partitions with `worker_id=5` each; the surviving set is irrelevant. The current proxy does NOT fire on this case; SF-002's fix requires a new test.
- (c) AgentId collision *without* `worker_id` collision (the unsound-proxy mode from QA-001). Construct two partitions with distinct `worker_id`s but overlapping `id_range`. The proxy is silent; in debug, `merge()`'s downstream invariant check fires (with message NOT containing "A8" / "A7"); in release, NOTHING fires and the corrupt net flows downstream. The current test suite has zero coverage for this mode.

**Suggested fix (Stage 6).** Replace UT-0412-04 with three independent test functions (or add two new tests beside the current one):

### QA-003 (HIGH) — `partitions.sort_by_key(|p| p.worker_id)` after `extend` violates R29's "`partitions[i].worker_id == i`" invariant when reclaimed shares `worker_id` with surviving (release-build silent), and the empty-fast-path's order-preservation is unverified

- **Severity:** HIGH — silent invariant breach reachable when the proxy is bypassed (release) or extended-and-not-yet-fixed (debug, Phase B integration).
- **Category:** Correctness (R29 invariant) / Helper contract
- **Location:** `relativist-core/src/merge/grid.rs:1227-1230` (`reconstruct_partition_plan_from_collected`).

**The bug.** The helper unconditionally extends and re-sorts:

```rust
if !reclaimed_partitions.is_empty() {
    partitions.extend(reclaimed_partitions);
}
partitions.sort_by_key(|p| p.worker_id);
```

Two distinct issues:

1. **R29 (SPEC-19) requires `partitions[i].worker_id == i` after sort** — that is, dense `[0, K_eff)`. In Phase A with the BSP pipeline this holds because `WorkerId` matches `partition_index`. In SPEC-20's elastic mode (R11a, the partition_index decoupling), `WorkerId` is sparse (`{0, 1, 5, 7}` is the spec-cited example) — the sorted order is `[0, 1, 5, 7]` and `partitions[2].worker_id == 5 != 2`. **R29 as literally stated is already false in the elastic regime**; SPEC-20 §3.7 R11a documents this with the dense-vs-sparse distinction. The helper's *current* Rustdoc bullet (line 1213-1214) still says "partitions[i].worker_id == i", which is **stale** under SPEC-20.
   - Concrete consequence: `merge::core::merge` does NOT depend on dense `worker_id`s — it iterates `partitions` and copies AgentIds by their *AgentId index*, not by worker_id. So the merge itself is robust. But:
   - Downstream, `reconstruct` returns a `Net` that is fed into `split()` (in TASK-0443's flow per SPEC-20 §4.2.2 step 4 "reconstruct + split"). `split()` calls `compute_id_ranges(K_eff)` and assumes its output is indexed by *partition_index*, not *WorkerId*. The bridge between "I just produced a Net" and "I now re-split with K_eff_new" is owned by TASK-0443 — but the contract that `reconstruct` is supposed to produce a *clean* `Net` (one whose state is independent of which partitions it came from) is currently held only by `merge()`. If a future refactor exposes `reconstruct`'s output `partitions` vector (e.g., as a return tuple `(Net, PartitionPlan)` for diagnostics), the order is `worker_id`-sorted, not `partition_index`-sorted, and any downstream code that maps `i → partition_index` is wrong.

2. **The `sort_by_key` is stable** (std `slice::sort_by_key` is stable, contrary to `sort_unstable_by_key`), so reclaimed partitions with `worker_id` matching a surviving's tie-break to "after the original" — but this is **never asserted** in code or doc. A future maintainer may "optimise" to `sort_unstable_by_key` (frequently recommended by clippy on small / move-only payloads), and then the order between a surviving partition with `worker_id=5` and a reclaimed partition with `worker_id=5` becomes non-deterministic. Combined with the `merge()` step-2 last-writer-wins behaviour (QA-001), this means the resulting `Net` becomes **build-dependent** for the AgentId-overlap pathological case. Determinism (G1 in SPEC-01) is silently lost.

**Reproduction (static trace).**

```rust
// Two partitions sharing worker_id (wid=3) reach the helper.
// The proxy assert in `reconstruct` would fire in debug if these
// were passed to reconstruct directly — but the helper is private,
// callable from inside the module via the `super::core::merge`
// pathway and from `run_grid_delta_final_collect` (line 1198).
// run_grid_delta_final_collect always passes Vec::new() as the
// reclaimed argument today, so the production callers do not hit
// this. But the helper has no documented precondition that
// reclaimed_partitions has unique worker_ids relative to surviving.
let surviving = vec![p_with_wid(3)];
let reclaimed = vec![q_with_wid(3)];
// The helper accepts both, extends, sorts. Result vector:
//   stable: [p_with_wid(3), q_with_wid(3)]   (intent)
//   unstable refactor: arbitrary order
// merge() then iterates partitions and overwrites slots last-writer-wins.
```

**Suggested fix (Stage 6).**

a. Update the helper's Rustdoc invariant bullet (NTH-003 already requested this) — change "partitions is sorted ascending by worker_id" to "the combined `surviving ∪ reclaimed` list is sorted ascending by `worker_id`. **Caller must ensure `worker_id`s are unique across the union** — duplicates produce build-dependent merge ordering."

b. Add a `debug_assert!` immediately after `sort_by_key` verifying strict monotonicity of `worker_id`s:

```rust
partitions.sort_by_key(|p| p.worker_id);
#[cfg(debug_assertions)]
for w in partitions.windows(2) {
    debug_assert!(
        w[0].worker_id < w[1].worker_id,
        "SPEC-19 R29 / SPEC-20 R11a: partition list contains duplicate \
         worker_id {} after surviving∪reclaimed sort",
        w[0].worker_id
    );
}
```

c. Pin the stability requirement in code: `partitions.sort_by_key(|p| p.worker_id);  // stable sort REQUIRED — see SPEC-19 R29; do NOT switch to sort_unstable_by_key`.

d. (NTH-003 already requested this part.) Update the stale "`partitions[i].worker_id == i`" doc-line to reflect SPEC-20 R11a's sparse-WorkerId reality.

### QA-004 (MEDIUM) — `redex_queue` ordering across the union is unspecified; downstream reduction determinism is build-dependent under non-empty `reclaimed`

- **Severity:** MEDIUM — affects G1 (determinism) only when reclaimed partitions carry non-empty redex queues, which the current Phase A pipeline does not produce, but Phase B integrators may.
- **Category:** Determinism / Helper contract / SPEC-01 G1
- **Location:** `relativist-core/src/merge/core.rs:73-83` (the `redex_queue` debug check inside step 2) and `relativist-core/src/merge/grid.rs:1297-1303` (`reconstruct` call to `merge`).

**The bug.** `merge()` step 2 contains:

```rust
#[cfg(debug_assertions)]
{
    for &(a, b) in &partition.subnet.redex_queue {
        debug_assert!(
            !partition.subnet.is_valid_redex(a, b),
            "Non-stale redex ({}, {}) found in partition {} queue after reduce_all",
            a, b, partition.worker_id
        );
    }
}
```

This asserts every partition has been *fully* reduced before merge — i.e., its redex queue is stale-only. But `merge()` then **discards** the per-partition queues entirely (R9: "Redex queue contains only border redexes"). The result `Net`'s queue is built fresh from the border-restoration pass (step 3, lines 152-165 of core.rs).

**The contract gap for the 3-arg form.** SPEC-20 §4.2.2 delta-mode departure step 4 invokes `reconstruct(border_graph, surviving_partitions, reclaimed_partitions)` where:
- `surviving_partitions` came from the BSP collect — fully reduced (R9 holds);
- `reclaimed_partitions` came from `retained_initial[w]` or `retained_last_acked[w]` — these are **partition snapshots from earlier rounds**, whose redex queues hold *the redexes that were valid at snapshot time*. SPEC-20 R23a / R31 do NOT mandate that `retained_initial[w].subnet.redex_queue` be empty when stored.

If a reclaimed partition has a non-stale redex in its queue (because the snapshot was taken at the start of a round, before reduction), the `debug_assert!` above fires under debug. **In release, the queue is silently discarded** (good for determinism — the result Net's queue is rebuilt from borders only), but the implicit assumption is that reduce_all on the reconstructed `Net` will rediscover any non-border redex via R41's full-scan defence in `verify_no_redexes_full_scan` (grid.rs:1364). That defence is *also* `#[cfg(debug_assertions)]` — release relies on `Net::connect`'s redex-discovery path during `merge()` step 3, which only fires for *border* redexes (step 3 calls `result.connect(port_a, port_b)` for borders, and `connect` auto-detects principal-principal pairs).

**Concrete failure mode for Phase B.** A reclaimed partition snapshot contains a CON-CON redex that was internal to its subnet (not a border, not yet reduced). After `merge()`:
- Step 2 copies both agents into `result.agents` and copies their port wiring (CON aux 0 → CON aux 0). The principal-principal connection is preserved structurally but **not re-inserted into `result.redex_queue`** — `set_port` (step 2's actual write path, core.rs:111) does NOT call `connect()` and does NOT invoke redex discovery.
- Step 3 only iterates `borders`, so it doesn't see this internal redex either.
- Result: a *valid* internal redex exists in the merged `Net` but is missing from `redex_queue`. `reduce_all` on this net advances the queue and converges *without* reducing the missing redex — a silent G1 violation.

In debug, `verify_no_redexes_full_scan` (grid.rs:1364) catches this on the next round if `delta_mode` is on and `R41` is exercised. In release: silent.

**Why the 2-arg call site does NOT exhibit this.** `run_grid_delta_final_collect` (grid.rs:~1198) calls `reconstruct` with `Vec::new()` as the reclaimed argument. Its survivors come from a transport collect AFTER full reduction (R29), so the assumption "survivor redex queues are stale-only" holds in Phase A.

**Suggested fix (Stage 6).** Add to `reconstruct`'s Rustdoc precondition list:

> **Precondition (queue cleanliness):** Every partition in `surviving_partitions ∪ reclaimed_partitions` MUST have a stale-only `redex_queue` (R9). Reclaimed partitions retrieved from `retained_initial` or `retained_last_acked` snapshots that were taken **before** local reduction MUST be locally reduced via `reduce_all(&mut subnet)` BEFORE calling `reconstruct`, OR the caller MUST invoke a full-scan redex rediscovery on the merged `Net` immediately after `reconstruct` returns. Failure to do so produces a silent R41 violation.

This is documentation, not code — but the Rustdoc currently makes no mention of queue state across the reclaimed boundary. Phase B (TASK-0443) authors will read this and miss it.

A code-side fix would add an *unconditional* (not `#[cfg(debug_assertions)]`) redex rediscovery pass at the end of `reconstruct`, but that changes performance characteristics and is out of scope for an isolated 3-arg extension.

### QA-005 (MEDIUM) — `BorderGraph` references that point into reclaimed partitions but the reclaimed partition's `free_port_index` does NOT carry the matching `border_id` produces silent border drops (D3 violation in release)

- **Severity:** MEDIUM — affects D3 (border completeness) only when the BorderGraph and reclaimed partition's `free_port_index` are out of sync, which the spec assumes is impossible by construction but is not enforced by `reconstruct`.
- **Category:** Correctness (D-invariant) / Untested-path / Defensive-contract
- **Location:** `relativist-core/src/merge/core.rs:138-165` (the border restoration loop in `merge()` step 3), and the test gap in EC-3.

**The bug.** `merge()` step 3 iterates `borders` (the `PartitionPlan.borders` map) and looks up each border_id in every partition's `free_port_index`. If a `border_id` is found in *zero* `free_port_index` maps, the `(None, None)` arm fires — interpreted by the SPEC-05 rule R7 as "both sides erased". If found in *one* map, R6 fires — "one side erased". If found in *two*, the wire is restored.

For the 3-arg `reconstruct` flow, the `BorderGraph` was built from a snapshot (the coordinator's surviving border state at departure time). The reclaimed partition is from `retained_initial[w]` — which carries its OWN `free_port_index` from the time it was *created* (R8 of SPEC-04, original split). If the BorderGraph was rebuilt mid-round (e.g., R24d border_id rebase per SPEC-20), the reclaimed partition's `free_port_index` carries the **old** border_ids while the BorderGraph carries the **new** ones. Step 3 looks up the new ids in the old `free_port_index` and finds nothing → `(None, None)` → silently treated as "erased".

**Result:** a border wire that the BorderGraph says exists is silently dropped. The merged `Net` has fewer connections than the abstract net it should represent. This is a **D3 (Border Completeness) violation**: SPEC-01 D3 says "every border in the BorderGraph is reflected in the merged Net". Step 3's `(None, None)` arm conflates "both sides erased by reduction" (legitimate R7) with "both sides invisible due to border_id mismatch" (defect).

**EC-3's gap.** UT-0412-EC-3 (grid.rs:4869) asserts:

```rust
let net = reconstruct(&graph, vec![p0], vec![r0]);
assert_eq!(net.count_live_agents(), survivors_count + r0_count, ...);
```

The graph is `empty_graph()` — zero borders. The test passes because there is nothing to drop. The interesting case — *non-empty* BorderGraph references that the reclaimed partition does not match — is not covered. The reviewer flagged this in their adversarial recommendations (REVIEW §10 sign-off, second bullet) but no test was added.

**Reproduction (static trace).**

```rust
// Surviving partition: no border references at all.
let p0 = lone_con_partition(0, 100);

// Reclaimed partition: also no border references.
let r0 = simple_era_partition(2);

// BorderGraph claims border_id=99 exists between two ports
// (both pointing at agents that DO exist, but the agents'
//  free_port_index does NOT carry border_id=99).
let mut borders_bg = HashMap::new();
borders_bg.insert(99u32, (
    PortRef::AgentPort(0, 0),  // p0's CON, port 0
    PortRef::AgentPort(2000, 0), // r0's ERA, port 0
));
let plan_bg = PartitionPlan {
    partitions: vec![p0.clone(), r0.clone()],
    borders: borders_bg,
    next_border_id: 100,
};
let graph = BorderGraph::from_partition_plan(&plan_bg);

let net = reconstruct(&graph, vec![p0], vec![r0]);
// step 3: for border_id=99, look up in p0.free_port_index → not present;
//         look up in r0.free_port_index → not present;
//         arm (None, None) fires → SILENTLY treated as "both sides erased".
// Result: net.count_live_agents() == 2; the wire that border 99 should
//         have restored is missing. D3 violated; no panic; no debug assert.
```

**Why this is reachable.** SPEC-20 R24d rebases `border_id`s on every re-split. A reclaimed partition's `free_port_index` was built from the *original* split's `border_id`s. If the BorderGraph being passed to `reconstruct` is the post-rebase one, the indices don't match. The spec assumes the rebase is applied to reclaimed partitions before reconstruction (SPEC-20 §4.2.2 step 3 mentions "rebase" as part of the recovery flow), but the function signature does not enforce it.

**Suggested fix (Stage 6).** Two complementary actions:

1. Add EC-3a (or strengthen EC-3): non-empty BorderGraph with a `border_id` that the reclaimed partition's `free_port_index` does NOT carry. Assert that `reconstruct` either (a) panics with a clear message ("BorderGraph border_id X not present in any partition's free_port_index") or (b) the test pins the *current* drop behaviour with a `#[should_panic(expected = "...")]` reverse — i.e., the test acknowledges that the current implementation silently drops the border. Either way, the gap becomes visible to future maintainers.

2. Add a `debug_assert!` inside `merge::core::merge` step 3's `(None, None)` arm: count how many `border_id`s end up in this arm and assert that the count is `<=` the count of borders explicitly marked as erased by reduction. The test data structure for "explicitly marked as erased" does not exist today, so the simplest defensive check is:

```rust
(None, None) => {
    #[cfg(debug_assertions)]
    {
        // Sanity: a (None, None) result for a border listed in the
        // BorderGraph means BOTH sides were eliminated by reduction.
        // This is legitimate only if every partition that ever owned
        // this border_id has reduced to the empty net. We cannot
        // distinguish "erased by reduction" from "lost due to free_port_index
        // staleness" at this point — log a warning under debug builds.
        eprintln!(
            "merge: border {} produced (None, None) — verify both sides \
             were erased by reduction, not by stale free_port_index",
            border_id
        );
    }
}
```

This is opt-in noise; the cleaner architectural fix is to make the precondition explicit in `reconstruct`'s Rustdoc and add a precondition assert.

### QA-006 (MEDIUM) — `merge()`'s `max_next_id` policy can produce a result `Net` with `next_id == u32::MAX`, blocking subsequent `create_agent` calls without a clear error path; reachable via reclaimed snapshots from long-running coordinators

- **Severity:** MEDIUM — release-build silent state in which the next `create_agent` will hit the I3 ceiling unannounced.
- **Category:** Boundary / I3 invariant / Untested-path
- **Location:** `relativist-core/src/merge/core.rs:45-58`.

**The bug.** `merge()` sets `result.next_id = max_next_id` where `max_next_id = max(p.subnet.next_id for p in partitions)`. Reclaimed partitions in the 3-arg flow can carry `subnet.next_id` values from the *original* split (early in the run) or from a long-lived `retained_last_acked` snapshot (late in the run, after many rounds of churn). If any reclaimed partition's `subnet.next_id == u32::MAX`, the resulting merged `Net` inherits it.

**Downstream consequence.** `Net::create_agent` (TASK-0410's QA-001 also flagged this) does `self.next_id += 1` without checking for overflow. In release, `u32::MAX + 1 == 0` (wrapping), so the next `create_agent` returns AgentId 0 and overwrites whatever lives there. The `reconstruct → split → InitialPartition` flow per SPEC-20 §4.2.2 step 4 does NOT call `create_agent` immediately, but `split()`'s `compute_id_ranges` and any defensive identity-bridging *might*. TASK-0411's `remap_partition_ids` is the most likely first downstream caller of `create_agent` post-reconstruct, and it would silently re-mint id 0.

**Why this is the 3-arg path's specific concern.** The 2-arg call site in `run_grid_delta_final_collect` always feeds *current-round* partitions, whose `next_id` is bounded by the round's redex output (which would have to be near `u32::MAX` to trip this — astronomically unlikely in Phase A). The 3-arg path's reclaimed partitions are snapshots that may be *much older* and have spent multiple rounds growing `next_id` (e.g., a CON/DUP-heavy partition's `next_id` doubles per round in profile B).

**This is the exact same root cause as QA-001 in QA-TASK-0410-2026-04-25 (Net::union)**, just exposed through a different code path: the assumption "merging two nets yields a usable next_id" is unchecked at the merge boundary.

**Suggested fix (Stage 6).** Either:

a. Add a precondition check in `reconstruct` (or in `reconstruct_partition_plan_from_collected`):

```rust
let max_next_id = partitions
    .iter()
    .chain(reclaimed_partitions.iter())
    .map(|p| p.subnet.next_id)
    .max()
    .unwrap_or(0);
debug_assert!(
    max_next_id < u32::MAX,
    "SPEC-01 I3: reconstruct would produce a Net with next_id == u32::MAX; \
     no subsequent create_agent is possible without overflow"
);
```

b. Better — make this an unconditional `assert!` (not `debug_assert!`) given that the cost is one comparison and the failure mode is silent corruption.

c. Best — defer to TASK-0410's QA-001 fix (return `Result<Net, NetError>` from `union` with a `NextIdSpaceExhausted` variant) and have `reconstruct` propagate it.

### QA-007 (MEDIUM) — Empty `surviving_partitions` with non-empty `reclaimed_partitions` (EC-1) does not exercise R29's "self-partition" assumption; SPEC-20 R7a's `WorkerId = 0 reservation` for the coordinator self-partition is silently violated

- **Severity:** MEDIUM — only relevant under hybrid mode (SPEC-20 R7a), which is the typical Phase B configuration.
- **Category:** Spec-compliance / Untested-path
- **Location:** `relativist-core/src/merge/grid.rs:4840-4852` (EC-1 test) and `reconstruct`'s Rustdoc.

**The bug.** SPEC-20 R7a: "`WorkerId = 0` is reserved permanently for the coordinator self-partition role." Under hybrid mode, every `reconstruct` call on the coordinator MUST include a partition with `worker_id = 0` representing the coordinator's own self-partition (the survivor side). EC-1 calls `reconstruct(&graph, vec![], vec![r0])` with empty survivors and a single reclaimed partition `r0 = simple_era_partition(0)` — the reclaimed partition claims `worker_id = 0`, which under R7a is the coordinator self-partition slot.

In a real hybrid-mode departure recovery, the reclaimed partition's `worker_id` would be the *departing worker's* id (which under R11 is monotonic from 1 upward). A reclaimed partition with `worker_id = 0` is either (a) the coordinator's own self-partition being reclaimed (which makes no sense — the coordinator does not depart), or (b) a non-hybrid-mode test using R7a's permissive clause.

EC-1 is testing case (b), which is fine *for non-hybrid mode*. But the test does not document this, and `reconstruct`'s Rustdoc does not mention the R7a interaction at all. A future hybrid-mode integrator (TASK-0443) reading EC-1 might infer that `worker_id = 0` in the reclaimed slot is universally legal — and write code that confuses the self-partition slot with a reclaimed-departing-worker slot.

**Suggested fix (Stage 6).** Two options:

1. Add a comment in EC-1's body: `// EC-1 represents non-hybrid-mode (R7a's permissive clause). In hybrid mode, worker_id = 0 in the reclaimed slot is invalid (the coordinator self-partition is never reclaimed).`

2. Add a sibling EC-1a test that exercises hybrid-mode semantics: `reconstruct(&graph, vec![self_partition_with_worker_id_0], vec![r0_with_worker_id_5])` where `r0`'s worker_id is monotonic-from-1 per R11. Assert that `count_live_agents` matches the union and that R7a is not silently violated.

This finding is borderline NTH-severity but I keep it at MEDIUM because the Rustdoc gap (no R7a mention) is exactly the kind of contract-erosion that compounds into Phase B integration bugs.

### QA-009 (LOW) — Order-isomorphism of `reconstruct` is implicit: `reconstruct(graph, vec![p0, p1], vec![r0, r1])` and `reconstruct(graph, vec![p1, p0], vec![r1, r0])` should produce isomorphic merged nets, but no test pins it

- **Severity:** LOW — currently holds because `reconstruct_partition_plan_from_collected` re-sorts; future refactor risks.
- **Category:** Determinism (G1) / Test Quality
- **Location:** Test-coverage gap; no specific code line.

**The bug.** The helper sorts on `worker_id` after extending, so input order does not matter. But this is *implementation behaviour*, not a *contract*. SPEC-19 R29 specifies that the input vector is sorted, but does not specify *that the function tolerates any input order*. fn reconstruct_is_order_independent() {

```rust
    let r0 = simple_era_partition(2);

    let symbols_b: std::collections::BTreeMap<_, _> = net_b.live_agents().map(|a| (a.id, a.symbol)).collect();
    assert_eq!(symbols_a, symbols_b, "reconstruct must be order-independent");
}
```

This subsumes the Stage 4 SF-001 strengthening and adds order-isomorphism in one test.
