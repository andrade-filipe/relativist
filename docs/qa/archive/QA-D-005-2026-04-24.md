## Findings

### F-C1 (CRITICAL) — `pending_new_borders` is drained BEFORE the fix can compute what was promoted

**Severity:** CRITICAL — blocks any straight-line "read `pending_new_borders` after `register_minted_agents`" implementation.

**Repro (static trace):**

`border_graph.rs:724` — `register_minted_agents` begins Phase 3 with
```rust
let candidates = std::mem::take(&mut self.pending_new_borders);
```
and at `border_graph.rs:752` assigns `self.pending_new_borders = still_pending`. The promoted entries **never land back in `self.pending_new_borders`**; they are moved via `resolved_entries` into `self.borders` (Phase 4, line 759).

The reviewer's fix sketch says: "after `register_minted_agents` resolves `PendingNewBorder`s into `border_graph.borders`, synthesise for each newly-added border_id a `new_borders` entry ... where `side_x_port` is the concrete `PortRef` from `BorderState.side_a/b`." The **only** way to enumerate the newly-added border_ids post-call is to either (a) diff `border_graph.borders.keys()` before/after, or (b) change `register_minted_agents`'s signature to return `&[AddBorderEntry]` (or `Vec<u32>`, or `Vec<BorderState>`). The former is O(|borders|) per worker; the latter is a signature change that the developer must design.

**Expected:** The fix has an authoritative enumeration of "borders promoted by THIS `register_minted_agents` call for THIS worker_id" that does not require reconstructing set differences.

**Actual (if naively implemented):** the developer writes `for bid in border_graph.pending_new_borders.iter() { ... }` after the call → loop body runs on the RESIDUAL (still-pending) entries, promoting borders that do not yet exist in `borders` → panic on `partition.free_port_index.insert` is silent overwrite but `apply_border_deltas_to_partition` calls `Net::connect(target, FreePort(bid))` with `target` taken from a `PendingPortRef::Pending` (undefined) → wrong wires or panic.

**Suggested fix:** change `register_minted_agents` to return `Result<Vec<BorderState>, GridError>` (or `Vec<AddBorderEntry>`) giving the caller the exact set of entries promoted in this call. This is the cleanest API and closes the "which borders are new this round" question definitively. Alternative: `register_minted_agents` writes the promoted ids into a caller-provided `&mut Vec<u32>` out-param.

**P1–P5 coverage?** Not covered. Reviewer's fix surface text says "the newly-promoted `BorderState`s" but does not prescribe the enumeration primitive. **Novel adversarial finding.**

### F-C2 (CRITICAL) — Coordinator-side `BorderState.side_a/side_b` carries WORKER-LOCAL AgentIds — blindly applying them to the peer worker injects garbage

**Severity:** CRITICAL — directly causes test failures on cross-worker CON-DUP even after F-C1 is fixed.

**Repro (static trace):**

Follow the CON-DUP data-flow for border_id 100 (principal-to-`FreePort(100)` wire for minted Con on worker_con, slot 1):

1. `border_resolver.rs:823-832` — `emit_external_principal` pushes
   ```rust
   PendingNewBorder {
     border_id: 100,
     side_a: PendingPortRef::Pending { cid=cid_con, slot=SLOT_R, port_slot=0 },  // minted Con on worker_con
     side_b: PendingPortRef::Concrete(t_c1),                                       // pre-existing FreePort on worker_con
     worker_a: worker_con,
     worker_b: worker_con,  // same worker — see F-H6 below
   }
   ```
2. Round N+2: `worker_con` echoes `MintedAgent { request_id, minted_agent_id = 3 }` (worker-local id inside `worker_con.id_range`).
3. `register_minted_agents` substitutes `side_a = PortRef::AgentPort(3, 0)` via `try_resolve_port_ref`.
4. `add_border_states` inserts `BorderState { border_id: 100, side_a: AgentPort(3, 0), side_b: Concrete(FreePort(100)) or a pre-existing AgentPort on worker_con, worker_a = worker_con, worker_b = worker_con }`.

**CASE A — `worker_a == worker_b`:** the fix applies `new_borders = [(100, side_a)]` to worker_con's partition and (because the code tries to mirror) ALSO to worker_con a second time with `side_b`. `apply_border_deltas_to_partition` will silently overwrite `free_port_index[100]` on the second insertion. The `Net::connect` calls will each wire a `FreePort(100)` to a different agent port on worker_con, creating a non-well-formed net (every `FreePort(K)` has at most one endpoint inside a given subnet).

**CASE B — true cross-worker (e.g. CON-DUP 4-worker fixture where `worker_con != worker_dup`):** `side_a` is `AgentPort(3, 0)` in `worker_con`'s arena. `side_b` is an `AgentPort(Y, p)` in `worker_dup`'s arena (or a `FreePort`). If the fix sends `(100, side_a)` to worker_con (correct) AND `(100, side_b)` to worker_dup (correct), each worker gets a local AgentPort. **But if the fix naively swaps or sends `side_a` to BOTH workers**, worker_dup's `Net::connect(AgentPort(3, 0), FreePort(100))` with id 3 in worker_dup's arena points at whatever agent is at worker_dup-local slot 3 — which is a DIFFERENT agent from the one minted on worker_con. Silent corruption.

**Expected:** fix MUST send per-worker-correct port:
- worker_a gets `(border_id, side_a)` ONLY;
- worker_b gets `(border_id, side_b)` ONLY.
And when `side_a`/`side_b` is a `PortRef::FreePort(_)` concrete (CON-ERA dangling case), applying it via `apply_border_deltas_to_partition` is still a FreePort-to-FreePort wire — legal, but the semantic is "keep the dangling marker in place" and the worker's `subnet.connect(FreePort(x), FreePort(bid))` may or may not be what we want here (see F-C3 + F-H7).

**Actual (in reviewer's sketch):** reviewer wrote "for `worker_a` the entry `(border_id, side_a_port)`; for `worker_b` the entry `(border_id, side_b_port)`" — this part is RIGHT. But the sketch does NOT explicitly call out the case where `side_a` or `side_b` is a `PortRef::FreePort(_)` concrete (emit_erasure_principal's `None` branch, border_resolver.rs:1116-1127), where applying the entry via `apply_border_deltas_to_partition` hits the `Net::connect(FreePort(x), FreePort(bid))` path which is a no-op in the port array (net/core.rs:187-194) — but populates `freeport_redirects`, which then surfaces in `rebuild_free_port_index` phase 2 and changes `free_port_index` contents. Verdict: needs an explicit branch OR a debug_assert that `side_a`/`side_b` is `AgentPort` at the point of application.

**Suggested fix / test coverage:**
- Probe test: build a 2-worker CON-DUP fixture where minted agent ids are INTENTIONALLY disjoint (e.g., worker_con.id_range = 0..50, worker_dup.id_range = 100..150). Mint agents 3 and 103. Assert that after the fix, worker_con's `free_port_index[100] == AgentPort(3, 0)` AND worker_dup's `free_port_index[100]` does NOT exist (because border 100 is intra-worker_con per F-H6).
- New UT `reg_minted_agents_returns_promoted_borders_for_coordinator_mirror`: assert that a fix-API-returned `Vec<BorderState>` carries the correct `side_a`/`worker_a` / `side_b`/`worker_b` pairing (NO swapping).
- Rust-level debug_assert inside the new loop: `debug_assert!(matches!(side, PortRef::AgentPort(id, _)) ==> worker_partition.id_range.contains(id), "side must reference the target worker's local arena")`.

**P1–P5 coverage?** Partially — P2 (cross-partition minted-edge reconstruction via merge) exercises the merge-side rebuild, but NOT the round-N+2 worker-partition sync. This is the **single biggest novel risk** the fix can trip over.

### F-C3 (CRITICAL) — `apply_border_deltas_to_partition`'s `new_borders` loop issues `Net::connect(target, FreePort(bid))` which OVERWRITES the principal-port slot of an agent that was (until this moment) wired to `DISCONNECTED`

**Severity:** CRITICAL — interferes with all three asymmetric rules.

**Repro (static trace):**

`helpers.rs:182-185`:
```rust
for (bid, target) in new_borders {
    partition.subnet.connect(*target, PortRef::FreePort(*bid));
    partition.free_port_index.insert(*bid, *target);
}
```

For CON-DUP: `target == AgentPort(3, 0)` and `bid == 100`. Currently (pre-fix), the minted agent 3's principal port is `DISCONNECTED`. So `Net::connect(AgentPort(3, 0), FreePort(100))` is:
- `set_port(AgentPort(3,0), FreePort(100))` → OK, overwrites DISCONNECTED.
- `set_port(FreePort(100), AgentPort(3,0))` → no-op (FreePort is a sentinel with no storage).

This is correct in isolation. **But the fix must run AFTER `register_minted_agents` and BEFORE the NEXT `dispatch_round_start`** — otherwise one of two things breaks:

- If run BEFORE Round N+2's `apply_pending_commutation` (the call that mints the agent): agent 3 does not yet exist. `Net::connect(AgentPort(3, 0), ...)` hits `set_port` which indexes `self.ports[3 * PORTS_PER_SLOT + 0]` — if `self.ports` is `Vec<PortRef>` with `len == next_id * PORTS_PER_SLOT`, this is an **out-of-bounds write** (panic in debug, undefined write in release). Need to confirm `Net::connect` bounds-check (Grep'd `net/core.rs:177` — only `debug_assert_ne!(a, b, ...)`, no bounds check on the slot → likely `ports[idx]` panics if idx exceeds len).

- If run AFTER `apply_pending_commutation` (which happens inside the worker's `handle_round_start` — the fix is coordinator-side): the coordinator has NO direct handle to each worker's post-commutation partition. The worker's partition has already been mutated during `dispatch_round_start`. The fix would have to either (a) push new_borders into the NEXT round's `dispatch_round_start` payload (clean, but that's Round N+3 and the loop may have already converged per F-H8), or (b) push a side-channel "reconciliation" dispatch call (the reviewer's "new `WorkerDispatch::dispatch_apply_promoted_borders` entry point").

**Expected:** the fix defines a clear ordering:
1. Coordinator `register_minted_agents` runs (Round N+2, after worker echo).
2. Coordinator computes per-worker new_borders for the NEWLY promoted set.
3. Coordinator applies to its own cache via `apply_border_deltas_to_partition(partitions_vec[w], ..., new_borders)`.
4. Coordinator plumbs the same per-worker new_borders into Round N+3's `RoundStartDispatch.new_borders` — worker applies in `handle_round_start` at worker.rs:558-563 (which already calls `apply_border_deltas_to_partition`, so the primitive is right; only the plumbing is missing).

The cleanest ordering is fold-into-existing-dispatch: the newly-promoted borders go into `pending_dispatch` for Round N+3. **BUT** — F-H8 shows the loop may break at end of Round N+2 before N+3 fires. So the fix MUST ALSO guarantee one more round's dispatch (force-non-convergence) OR piggyback a separate "reconciliation" phase inside Round N+2's dispatch_round_start output.

**Actual:** reviewer's sketch says "Apply each via `apply_border_deltas_to_partition(partitions_vec[w], &[], &[], &[(bid, side)])`. Also apply the mirror operation to the worker-side partition ... OR fold it into the existing `dispatch_round_start` as a final 'reconciliation round' before `FinalStateRequest`." The "OR" is a choice the developer still has to make — and the failure surface of each choice is different.

**Suggested fix / test coverage:**
- Decision record required: pick ONE of (a) extra round with only new_borders payload, (b) extended last round carrying new_borders AND marking `has_border_activity = true` so the convergence check returns false, (c) out-of-band `dispatch_reconcile_borders` on the WorkerDispatch trait.
- New UT `new_borders_plumbed_into_round_n_plus_3_round_start`: dispatch Round N+2 → observe the NEXT round's pending_dispatch contains per-worker `(border_id, side_x)` entries.
- New UT `worker_applies_reconciliation_new_borders`: call `handle_round_start` with new_borders populated and zero pending_commutations, assert `free_port_index` updated AND `subnet.get_target(AgentPort(mint_id, 0)) == FreePort(bid)`.

**P1–P5 coverage?** Not covered. Reviewer's P1 tests cleanup over-drop on an already-correctly-wired partition; P2 tests merge — neither exercises the worker round-trip. **Novel.**

### F-H6 (HIGH) — CON-DUP intra-worker principal-target path emits `PendingNewBorder { worker_a == worker_b }`; the fix must special-case "same-worker" to avoid double-insert

**Severity:** HIGH — intermittent CON-DUP failure on the 2-of-4 external principals that land on the minting worker.

**Repro (static trace):**

`border_resolver.rs:940-974` — `emit_external_principal` match arms:
- `PortRef::AgentPort(_, _) if new_worker == target_home` → emits into `local_wiring` (LOCAL branch). NOT a PendingNewBorder.
- `_ =>` (everything else, including `PortRef::AgentPort` on a different worker AND any `PortRef::FreePort`) → emits PendingNewBorder with `worker_a = new_worker`, `worker_b = target_home`.

In the CON-DUP fixture, `t_c1 = FreePort(100)`. The fixture's `target_home` for FreePorts is set by the caller (`worker_con` for `t_c1`). So the emitted PendingNewBorder has `worker_a = worker_con, worker_b = worker_con`. **Intra-worker PendingNewBorder.** When promoted, `BorderState.worker_a == BorderState.worker_b == worker_con`.

If the fix naively does:
```rust
partitions_vec[state.worker_a].apply(..., &[(bid, state.side_a)]);
if state.worker_a != state.worker_b {
    partitions_vec[state.worker_b].apply(..., &[(bid, state.side_b)]);
}
```
→ correct, one application.

But if the fix does:
```rust
for w in [state.worker_a, state.worker_b] {
    partitions_vec[w].apply(..., &[(bid, side_for(w, state))]);
}
```
→ worker_con gets `(bid, side_a)` AND `(bid, side_b)`. Second apply overwrites `free_port_index[bid]` and issues `Net::connect(side_b, FreePort(bid))` — but `side_b` for the CON-DUP intra-worker case is `PortRef::Concrete(t_c1) == FreePort(100)`. So the second call is `Net::connect(FreePort(100), FreePort(100))` → trips `debug_assert_ne!(a, b, ...)` at net/core.rs:178 → **panic in debug build.**

**Expected:** fix explicitly guards with `if state.worker_a != state.worker_b` OR the fix never loops over both sides for intra-worker borders.

**Actual:** reviewer's sketch shows the loop structure but is not explicit on the `worker_a == worker_b` case.

**Suggested fix / test coverage:**
- Probe: add `assert_ne!` on the pair to catch the debug-panic path in a controlled UT.
- New UT `reg_minted_agents_intra_worker_promotion_single_apply`: fixture where `worker_a == worker_b`; assert the new loop applies `(bid, side_a)` ONCE.
- For the `side_a == side_b == FreePort(bid)` degenerate, confirm `Net::connect` guard.

**P1–P5 coverage?** NOT covered. E3 in the QA charter flagged it; this is the confirmation that **yes**, emit_external_principal DOES produce intra-worker borders for every FreePort external principal in CON-DUP.

### F-H7 (HIGH) — `apply_border_deltas_to_partition` does NOT validate the `target` AgentPort's owning arena — cross-worker mis-send corrupts `ports[]` silently

**Severity:** HIGH — silent wrong answer (not a panic).

**Repro (static trace):**

`helpers.rs:182-185` — no validation that `target`'s AgentId fits in `partition.id_range` or points at a live agent. `Net::connect` trusts `target` verbatim. If the fix accidentally ships `side_b` (whose AgentId is in worker_b's arena) to worker_a's partition, `Net::connect(AgentPort(worker_b_local_id, 0), FreePort(bid))` either:
- Indexes into worker_a's port array at a slot that's valid (if `worker_b_local_id < worker_a.next_id`) → overwrites the wrong agent's principal port. T-invariant violation visible only at final merge.
- Indexes out of bounds → panic via ports Vec bounds (per net/core.rs, `set_port` is not shown but likely does `self.ports[idx] = value` where idx can exceed len).

**Expected:** the fix plumbs per-worker per-side data such that the `target` is guaranteed local to the worker's arena, or adds a debug_assert.

**Actual:** reviewer's sketch does not prescribe such a validation.

**Suggested fix / test coverage:**
- Add `debug_assert!(partition.id_range.contains(target_agent_id) || matches!(target, PortRef::FreePort(_)))` to the new loop body inside `apply_border_deltas_to_partition`'s new_borders case.
- New UT `apply_border_deltas_cross_arena_target_detected_in_debug`: construct a partition with id_range = 0..50, call `apply_border_deltas_to_partition(..., new_borders=[(100, AgentPort(200, 0))])`, assert `#[cfg(debug_assertions)]` this panics with the new message.

**P1–P5 coverage?** Partial — reviewer's P2 asserts merge reconstructs the wire, but doesn't detect the mid-round corruption that never reaches merge.

### F-H8 (HIGH) — Post-fix `has_border_activity` goes TRUE in Round N+2 → the loop does NOT converge → developer needs a deliberate convergence-ordering decision

**Severity:** HIGH — affects strict AND lenient BSP paths identically.

**Repro (static trace):**

After the fix, at the end of Round N+2:
- Coordinator mirrors the newly-promoted borders into `partitions_vec[w]`; `free_port_index` gains `{100: AgentPort(3, 0), 101: AgentPort(4, 0), 110: AgentPort(2, 0), 111: AgentPort(1, 0)}`.
- Worker-side partitions (via whichever plumbing mechanism) gain the same.

Now in Round N+3, `handle_round_start` calls `rebuild_free_port_index` (worker.rs:650) which scans ports — finds `FreePort(100)` wired to `AgentPort(3, 0)` (principal port, port_slot == 0). Then `compute_border_activity` (worker.rs:664) returns `true` (principal port found in index). `RoundResult.has_border_activity = true`.

`check_delta_convergence` at `grid.rs:747` requires `all_no_border_activity == true`. **With the fix, this is now false at the exact round we expected to converge.** Unless the resolver can turn these principal-port borders into redexes (they CAN'T — the other side is a CONCRETE `FreePort(bid)`, not another `AgentPort(_, 0)`, so `is_principal_pair` returns false; `detect_border_redexes` returns empty), the loop runs forever until `max_rounds` cap.

Wait — the other side. For CON-DUP, `side_b` is `PortRef::Concrete(t_c1) == FreePort(100)`. So `is_principal_pair(AgentPort(3, 0), FreePort(100))` returns `false` (requires BOTH AgentPort with port 0). So `active_redexes` does NOT gain border 100. `graph_has_no_redexes == true`. **BUT** `has_border_activity == true` because worker_con sees a principal port in its free_port_index. Round does NOT converge.

Next round: workers re-dispatch, `reduce_all` has nothing to reduce (no redexes), workers report `has_border_activity = true` again (the state is stable), loop runs until max_rounds hit, `converged = false`, test asserts `v2_metrics.converged` fails. **UT-0385-08 CON-DUP flips from "empty net" failure to "did not converge" failure.**

**Expected:** `compute_border_activity` must NOT flag a border principal-endpoint whose REMOTE side is a Lafont/concrete `FreePort(bid)`. OR: the fix MUST ensure the promoted borders have their REMOTE side resolve to DISCONNECTED (R17 double-disconnected kill) before the next convergence check. OR: the test comparator accepts `metrics.converged = false` when the cap is generous and the residual net is correct.

**Actual:** reviewer's fix sketch is silent on the convergence predicate. The P1–P5 probes don't touch convergence.

**Suggested fix / test coverage:**
- **Option A (smallest blast radius):** tighten `compute_border_activity` — only flag principal-port borders whose `BorderGraph.borders[bid]` has the other side also principal (i.e., only flag genuine cross-worker redexes). This makes `has_border_activity` a strict-redex predicate. Audit: SPEC-19 §3.1 R1 wording says "potentially active — it could form a border redex once paired with the remote side". The CON-DUP post-fix state CAN'T pair — side_b is concrete FreePort — so the predicate is lying. Semantic fix.
- **Option B:** post-promotion, the coordinator also emits a RESOLUTION for each promoted border with `side_a AgentPort(_, 0)` paired against a CONCRETE FreePort — a dangling wire that merges correctly in `merge::core::merge` but does not keep the BSP loop alive.
- New UT `post_fix_con_dup_converges_after_promotion`: run the 4-worker CON-DUP fixture through run_grid_delta, assert `v2_metrics.converged == true` AND `v2_metrics.rounds <= 5` (bound).
- New UT `compute_border_activity_treats_lafont_freeport_peer_as_inactive`: synthesise a partition with `free_port_index = {100: AgentPort(0, 0)}` but a `BorderGraph` marking border 100 as `side_b = FreePort(Lafont)`; assert `compute_border_activity == false`.

**P1–P5 coverage?** NOT covered. **Novel adversarial finding — highest implementation risk.**

### F-M1 (MEDIUM) — D-004 NF-001 multi-slot mint echo + DC-B6 "replace existing" interaction

**Severity:** MEDIUM — likely works but needs explicit test.

**Repro:**

`worker.rs:478-488` — one `PendingCommutation` with 2 target_symbols emits TWO `MintedAgent` echoes, each encoded with a different `(cid, slot)` pair. `register_minted_agents` at Phase 2 (line 715-719) does last-write-wins for duplicate `(cid, slot)`. **Duplicate (cid, slot) pairs cannot occur within a single PC echo** because the encoding is `cid:28 | slot:4` and the worker iterates slots uniquely — so the lenient last-wins branch is unreachable under well-formed input.

But: for CON-DUP the resolver emits TWO CommutationBatches (one per worker; border_resolver.rs:905-917), EACH with 2 target_symbols. So 4 MintedAgents per round arrive at `register_minted_agents`. All 4 are distinct `(cid, slot)` pairs (2 cids × 2 slots). Phase 1 R48 validation passes (each pair matches an outstanding Pending). Phase 3 walks `pending_new_borders` (6 entries: 4 external principals + 2 internal aux-aux crosses), resolves side_a and side_b to concrete AgentPorts in 5 of them (one is wholly-Concrete for FreePorts?), promotes all 6 to `borders`. **No `replace_existing` hits** — the border_ids 100/101/110/111 are freshly allocated by `border_alloc.next()` and haven't appeared before.

**Expected:** the fix's post-promotion sync sees 6 new borders (4 external + 2 aux-aux cross).

**Actual:** the 2 aux-aux-cross PendingNewBorders have BOTH sides as `PendingPortRef::Pending` (border_resolver.rs:874-886); both resolve. Their `side_a`/`side_b` are both AgentPorts on the two different workers. Worker_a = worker_con, worker_b = worker_dup — GENUINE cross-worker. This is the case where F-C2 swap-error is most dangerous. The fix must ship `side_a` to worker_con and `side_b` to worker_dup and NEVER swap.

**Suggested test coverage:**
- UT `reg_minted_agents_promotes_6_borders_on_con_dup`: assert `border_graph.borders.len()` increases by exactly 6, with 4 intra-worker and 2 cross-worker.
- UT `promoted_aux_aux_cross_lands_on_correct_workers`: verify per-partition `free_port_index` keys after the fix.

### F-M2 (MEDIUM) — "Fold into dispatch_round_start" alternative mis-metrizes border_interactions_per_round

**Severity:** MEDIUM — affects G1 `metrics.total_interactions` parity.

If the developer picks the reviewer's alt "fold into the existing `dispatch_round_start` as a final 'reconciliation round'": the new borders flow through Round N+3's `RoundStartDispatch.new_borders`. But:
- Round N+3's `package_resolutions_with_pending` emits no resolutions (no redexes) → `border_interactions_this_round = 0`.
- Round N+3's `local_redexes` — workers just applied new_borders, no pending_commutations, `reduce_all` finds zero → OK.
- `metrics.rounds += 1` — increments rounds count. UT-0385-08 may expect a specific `metrics.rounds` value for parity with v1.

v1's `run_grid` converges in one fewer round (it merges every round, not at-the-end), so `metrics.rounds` may already differ. The current passing 11/12 show this is tolerated. But adding ONE more round for the asymmetric fixtures might push `metrics.total_interactions` off-by-zero (because reconciliation rounds generate 0 interactions) — UT-0385-08 asserts `v1_metrics.total_interactions == v2_metrics.total_interactions`. This **should** still pass since both totals exclude zero-delta rounds. Confirm via UT.
