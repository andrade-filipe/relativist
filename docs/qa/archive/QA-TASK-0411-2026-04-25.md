## 1. Summary

Static-only adversarial sweep over the new primitives plus the `reconstruct_partition_plan_from_collected` cursor-zero hazard flagged in Stage 4 RV-001. The arithmetic in `allocate_border_ids` is sound (no overflow path found); the structural rewrite in `remap_partition_ids` is complete; the test suite covers happy paths and the headline error variants. Adversarial probing surfaces:

1. **One concrete reproducer for RV-001** (QA-001, HIGH) — calling `allocate_border_ids` on a plan returned by `reconstruct_partition_plan_from_collected` collides with border IDs that were already used in the originating split.
2. **Two latent inconsistencies in `remap_partition_ids`** (QA-002 MED on the `agent_count == 0` early-return preserving a stale `free_port_index`; QA-003 MED on the silent-preserve-on-stray-AgentPort branch in the port-rewrite loop, which can mask data corruption).
3. **One arithmetic correctness issue at the empty-input boundary** (QA-004 MED): `agent_count == 0` plus a non-empty `new_range` causes `next_id` and `id_range` to diverge in non-empty/empty cases — the early-return path uses `Net::new()` (next_id = 0) instead of `new_range.start`, breaking the documented invariant "`next_id` is set to `new_range.start + partition.agent_count()`".
4. **Two missing-test gaps that QA cannot construct without the runtime gate** (QA-005, QA-006), covering inverted `IdRange`, and `remap_partition_ids` interaction with `redex_queue` containing pairs where one element is dead.
5. **One protocol/wire concern for the new error variants** (QA-007, LOW): `PartitionError` is not on the wire today, but the new variants are also not given `serde::Serialize/Deserialize` — if a future telemetry path serializes errors (e.g., `RoundResult::partition_error`), the silent absence of derives will surface as a build break.
6. **Two minor docs/contract drifts** (QA-008, QA-009, LOW).

**Top-3 dangerous bugs (one line each):**
- **QA-001 (HIGH)** — `reconstruct_partition_plan_from_collected` returns `next_border_id: 0`; calling `allocate_border_ids(N)` on it collides with border IDs already issued in the original split's `wire_class.border_id_end`.
- **QA-004 (MEDIUM)** — `remap_partition_ids` empty-partition early-return sets `subnet = Net::new()` (`next_id = 0`), violating the documented postcondition `next_id = new_range.start + agent_count` when `new_range.start > 0`.
- **QA-002 (MEDIUM)** — `remap_partition_ids` empty-partition early-return preserves `partition.free_port_index` byte-identical (via struct-update `..partition`), so any pre-existing AgentPort entries — though impossible in a well-formed empty partition — would survive a "remap" pointing at non-existent ids.

**No CRITICAL blocker. Stage 6 may proceed.** RV-001 / QA-001 must be the first item; QA-002, QA-003, QA-004 should land in the same Stage 6 pass.

## 2. Findings

### HIGH

#### QA-001 — `reconstruct_partition_plan_from_collected` produces a plan whose `allocate_border_ids` cursor will collide with the originating split's border IDs (concrete reproducer)

- **Severity:** HIGH (escalation of RV-001 with executable reproducer)
- **Category:** Logic Error / Spec Compliance / Latent Production Hazard
- **File:** `relativist-core/src/merge/grid.rs:1235-1239`
- **Specs:** SPEC-20 §3.8 A3 (R18a), SPEC-04 D3 (Border Completeness), SPEC-04 D4 (ID Uniqueness)
- **Description:** `reconstruct_partition_plan_from_collected` hard-codes `next_border_id: 0` on the returned `PartitionPlan`. RV-001 in the reviewer's pass already flagged this as semantically wrong; this QA report supplies a concrete adversarial sequence that exhibits the breakage *now* (no need to wait for TASK-0440/0443 to land).
- **Adversarial reproducer (sketch):**
  ```rust
  // Build a 2-worker split that uses border IDs 0..3 (3 cut wires).
  // After split() returns, plan.next_border_id == 3 (== wire_class.border_id_end).
  let net = make_net_with_three_border_wires();
  let plan = split(net, 2).expect("split ok");
  assert_eq!(plan.next_border_id, 3);  // OK: cursor sits past last assigned border id

  // Simulate post-round collected partitions and a BorderGraph that retains
  // the same 3 border IDs (no annihilation).
  let border_graph = BorderGraph::from_plan(&plan);
  let collected: Vec<Partition> = plan.partitions.clone();

  // Reconstruct via the helper. Today this is the path used by
  // run_grid_delta_final_collect; tomorrow TASK-0440 will use it for
  // departure recovery.
  let reconstructed: PartitionPlan =
      reconstruct_partition_plan_from_collected(collected, &border_graph, Vec::new());

  // BUG: cursor reset to 0 even though border IDs 0, 1, 2 are live.
  assert_eq!(reconstructed.next_border_id, 0);

  // Now allocate one fresh border ID. Per SPEC-20 R24d this should yield 3
  // (disjoint from {0, 1, 2}). It actually yields 0 — direct collision with
  // the still-live border id 0 in BorderGraph.
  let mut plan = reconstructed;
  let r = plan.allocate_border_ids(1).expect("allocation ok");
  assert_eq!(r, 3..4, "MUST be disjoint from existing border IDs");
  // Above assertion FAILS: r == 0..1, colliding with border id 0.
  ```
- **Expected behaviour:** `reconstructed.next_border_id` should be at least `border_graph.borders.keys().max() + 1` (or, more conservatively, the originating split's `wire_class.border_id_end`). Calling `allocate_border_ids(1)` should yield `3..4` (or higher), never a value already in `borders`.
- **Actual behaviour:** Cursor resets to `0`. `allocate_border_ids` happily issues IDs that are already in the border map. Downstream `merge()` then has two distinct logical wires labelled with the same `border_id`, breaking D3 (border completeness) and producing silent merge corruption.
- **Why this fires today (and not just after TASK-0440):** the final-collect path's `merge()` destructures `PartitionPlan` with `..` and ignores `next_border_id`, so the returned plan is consumed-and-dropped without any `allocate_border_ids` call. The bug fires the moment any successor (TASK-0440 / TASK-0443 / a hypothetical custom orchestrator that retries reduction on the reconstructed plan) calls `allocate_border_ids` on the returned plan. The reviewer's RV-001 correctly identifies this; this QA finding supplies the precise sequence.
- **Fix suggestion:** Per RV-001 Option A — add a `next_border_id: u32` parameter to `reconstruct_partition_plan_from_collected`, supply it from the caller. For the `run_grid_delta_final_collect` path, pass either `0` (with a comment "merge() ignores cursor; harmless") or `border_graph.max_border_id().map(|m| m + 1).unwrap_or(0)` for defensive future-proofing.

  Alternative defensive fix that does not change the signature: derive the cursor inside the function:
  ```rust
  let next_border_id = border_graph
      .borders
      .keys()
      .copied()
      .max()
      .map(|m| m.saturating_add(1))
      .unwrap_or(0);
  ```
  Reviewer's RV-001 correctly flags that this *underestimates* the cursor when borders have been removed via `remove_border` mid-round — the explicit-parameter Option A is preferred.

- **Test gap:** No test in `partition_amendments.rs` constructs a plan via `reconstruct_partition_plan_from_collected` and then calls `allocate_border_ids` on it. The reviewer's recommendation in §6 of REVIEW-TASK-0411 explicitly asks QA to add this scenario; we recommend it land alongside the RV-001 fix as `UT-0411-EC-RV001-collision`.

#### QA-002 — `remap_partition_ids` empty-partition early-return preserves `free_port_index` byte-identical via `..partition`, leaving stale AgentPort entries pointing at non-existent ids

- **Severity:** MEDIUM
- **Category:** Logic Error / Silent Inconsistency
- **File:** `relativist-core/src/partition/remap.rs:69-76`
- **Description:** The early-return path for `agent_count == 0`:
  ```rust
  if agent_count == 0 {
      return Ok(Partition {
          subnet: Net::new(),
          id_range: new_range,
          ..partition          // <-- preserves free_port_index, border_id_start/end, worker_id
      });
  }
  ```
  Struct-update `..partition` keeps `free_port_index` exactly as supplied. For a *well-formed* empty partition this map is necessarily empty (no live agents → no `AgentPort` targets). However, a *malformed* input (post-bug, post-corruption, post-fuzz) with `count_live_agents() == 0` but a non-empty `free_port_index` would survive the remap with that stale map intact. The non-empty path (lines 164-177) defends against this by `filter_map`-dropping entries whose target id is not in `id_map`; the early-return path skips that defence.
- **Reproducer (constructed):**
  ```rust
  let mut malformed = Partition {
      subnet: Net::new(),                         // 0 live agents
      worker_id: 0,
      free_port_index: HashMap::from([
          (42u32, PortRef::AgentPort(99, 1)),     // dangling: agent 99 does not exist
      ]),
      id_range: IdRange { start: 0, end: 0 },
      border_id_start: 0,
      border_id_end: 1,
  };
  let new_range = IdRange { start: 100, end: 100 };
  let result = remap_partition_ids(malformed, new_range).unwrap();
  // BUG: result.free_port_index still contains (42 -> AgentPort(99, 1)),
  //      but the subnet is brand-new and has no agent 99.
  assert!(result.free_port_index.is_empty(),
          "expected stale entries to be dropped");  // FAILS
  ```
- **Impact:** Subsequent merge/union operations consult `free_port_index` to wire FreePort markers; a stale AgentPort entry is a silent dangling reference — type-checks pass, but `Net::union` precondition (D4) is violated invisibly.
- **Fix:** Either (a) clear `free_port_index` in the early-return (`free_port_index: HashMap::new()`) on the rationale that an empty subnet has no boundary, or (b) `debug_assert!(partition.free_port_index.is_empty(), "empty subnet must have empty free_port_index")` before the early return. Option (a) is safer (always-correct postcondition); option (b) is louder (catches malformation in dev).

#### QA-003 — `remap_partition_ids` silently preserves an `AgentPort(stale_id, port)` reference in the port-rewrite loop when `id_map` does not contain the stale id

- **Severity:** MEDIUM
- **Category:** Silent Data Loss / Inconsistent Failure Policy
- **File:** `relativist-core/src/partition/remap.rs:130-143`
- **Description:** The port rewrite loop:
  ```rust
  PortRef::AgentPort(ref_id, port) => {
      if let Some(&mapped) = id_map.get(&ref_id) {
          PortRef::AgentPort(mapped, port)
      } else {
          // Reference to an agent outside the partition — should not occur
          // in a well-formed partition (...), but we preserve it unchanged
          // to avoid data loss.
          old_target
      }
  }
  ```
  The fallback "preserve unchanged" branch silently emits an `AgentPort(stale_old_id, port)` into the new ports array — referring to an id from the *old* arena that has no corresponding agent in the *new* arena. This is the textbook "dangling pointer through ID rebase" pattern. The comment claims "should not occur in a well-formed partition", which is a correct claim about `split()`'s output, but `remap_partition_ids` is also reachable from departure-recovery (TASK-0440) where the input partition has been munged by `cleanup_t1_violations` and similar passes. RV-005 already flags the *inconsistency* between this branch and the `free_port_index` drop-policy; QA escalates to MEDIUM because the dangling reference does not just disappear — it actively corrupts the new arena.
- **Reproducer (constructed):**
  ```rust
  // Build a partition whose ports array references a "ghost" id 999 not in agents[].
  let mut subnet = Net::new();
  let a = subnet.create_agent(Symbol::Con);   // id == 0
  // Inject a corrupted port: agent 0 aux1 -> AgentPort(999, 0).
  // (In real life this can arise from `cleanup_t1_violations` racing with
  // a delete; specs do not forbid the intermediate state.)
  subnet.ports[0 * PORTS_PER_SLOT + 1] = PortRef::AgentPort(999, 0);

  let p = Partition {
      subnet,
      worker_id: 0,
      free_port_index: HashMap::new(),
      id_range: IdRange { start: 0, end: 1 },
      border_id_start: 0,
      border_id_end: 0,
  };
  let new_range = IdRange { start: 100, end: 101 };
  let result = remap_partition_ids(p, new_range).unwrap();

  // Agent 0 became agent 100. BUG: agent 100 aux1 still points to old id 999.
  assert_eq!(
      result.subnet.ports[100 * PORTS_PER_SLOT + 1],
      PortRef::AgentPort(999, 0),  // <-- stale; "should not occur" but does
  );
  ```
- **Impact:** Same as QA-002 — a downstream reduction that follows the link from agent 100 will read `AgentPort(999, _)` and either panic on out-of-bounds index (if `new_arena_len` is smaller than 999), or silently skip via the `if old_port_idx < partition.subnet.ports.len()` guard at line 122 (which uses *old* lengths) and cascade the corruption.
- **Fix:** Two-part:
  1. In production builds, replace the silent-preserve branch with a hard `Err(PartitionError::InvariantViolation(format!(...)))` — or, less invasively, set the slot to `DISCONNECTED` and return an error if the resulting partition is ill-formed.
  2. Add a `debug_assert!(id_map.contains_key(&ref_id), "stray AgentPort target {} in partition; not in id_map", ref_id)` — mirrors RV-005's recommendation for `free_port_index` and unifies the two policies.

#### QA-004 — `remap_partition_ids` empty-partition early-return violates the documented `next_id = new_range.start + agent_count` postcondition

- **Severity:** MEDIUM
- **Category:** Logic Error / Postcondition Violation
- **File:** `relativist-core/src/partition/remap.rs:69-76`
- **Specs:** Rustdoc lines 38-39: "`next_id` is set to `new_range.start + partition.agent_count()`."
- **Description:** When `agent_count == 0`, the early-return path returns `Partition { subnet: Net::new(), id_range: new_range, ..partition }`. `Net::new()` initialises `next_id` to `0` (per `core.rs:84-92`). When `new_range.start > 0` (e.g., `new_range = IdRange { start: 500, end: 500 }`), the resulting partition has `id_range.start == 500` but `subnet.next_id == 0`. Any subsequent `subnet.create_agent(...)` will mint id `0`, in violation of the documented invariant that the partition's `next_id` should equal `new_range.start + 0 = new_range.start = 500`.
- **Reproducer (constructed):**
  ```rust
  let empty = Partition {
      subnet: Net::new(),
      worker_id: 0,
      free_port_index: HashMap::new(),
      id_range: IdRange { start: 0, end: 0 },
      border_id_start: 0,
      border_id_end: 0,
  };
  let new_range = IdRange { start: 500, end: 600 };
  let result = remap_partition_ids(empty, new_range).unwrap();

  assert_eq!(result.id_range.start, 500);          // OK
  assert_eq!(result.subnet.next_id, 500);          // BUG: actual is 0
  ```
- **Impact:** Once the reclaimed-and-renumbered partition starts minting agents (e.g., during `apply_pending_commutation` for CON-DUP), the very first minted id will be `0`, colliding with surviving partitions whose ranges overlap `0..500`. SPEC-20 R30 (ID uniqueness preservation) is silently violated by exactly the case `remap_partition_ids` was added to handle.
- **Fix:** Replace the early-return with:
  ```rust
  if agent_count == 0 {
      let mut subnet = Net::new();
      subnet.next_id = new_range.start;
      return Ok(Partition {
          subnet,
          id_range: new_range,
          free_port_index: HashMap::new(),  // also resolves QA-002
          ..partition
      });
  }
  ```
- **Note:** The non-empty path (line 183) sets `next_id: new_range.start + agent_count` correctly. The bug exists *only* on the empty-partition fast-path.

#### QA-005 — Inverted `IdRange { start > end }` is silently treated as empty by `len()` / `is_empty()` (RV-002 expansion + adversarial path)

- **Severity:** MEDIUM
- **Category:** Silent Failure / Hidden Behaviour
- **File:** `relativist-core/src/partition/types.rs:31-38`
- **Description:** RV-002 already flags this; QA confirms a reachable adversarial path that turns the silent-zero into a *correctness* hazard rather than just a documentation gap:
  ```rust
  let p = make_p0();                     // 4 agents
  let inverted = IdRange { start: 1000, end: 100 };  // malformed input
  // inverted.len() == 0  (saturating_sub)
  // 4 > 0 → NewRangeTooSmall { partition_size: 4, range_size: 0 }
  let result = remap_partition_ids(p, inverted);
  // Caller sees "range too small (0)"; actual problem is "range inverted".
  ```
  The error variant name (`NewRangeTooSmall`) misleads the caller into thinking the range is too small. The caller will allocate a *bigger* range — which, if also inverted, produces the same misleading error.
- **Worse case:** `IdRange { start: 1000, end: 100 }` paired with an empty partition (`agent_count = 0`) returns `Ok` (since `0 > 0` is false). The empty-partition early-return then hands back a partition with `id_range = inverted_range`, propagating the malformed range silently into the rest of the system.
- **Fix:** RV-002's recommended `debug_assert!(self.start <= self.end, ...)` in `IdRange::len`/`is_empty` is necessary but not sufficient. Add a second `debug_assert!` at the top of `remap_partition_ids`:
  ```rust
  debug_assert!(
      new_range.start <= new_range.end,
      "remap_partition_ids: new_range is inverted (start={} > end={})",
      new_range.start, new_range.end
  );
  ```
  Or upgrade to a real check that returns a new `PartitionError::InvalidIdRange` variant (additive, no wire impact).

#### QA-006 — `redex_queue` filter-map drops pairs where one element is not in `id_map`, silently losing in-flight redexes during a remap

- **Severity:** LOW
- **Category:** Silent Data Loss
- **File:** `relativist-core/src/partition/remap.rs:151-160`
- **Description:**
  ```rust
  let new_redex_queue: VecDeque<(u32, u32)> = partition
      .subnet
      .redex_queue
      .iter()
      .filter_map(|&(a, b)| {
          let new_a = id_map.get(&a)?;
          let new_b = id_map.get(&b)?;
          Some((*new_a, *new_b))
      })
      .collect();
  ```
  The `?` early-return means a pair `(a, b)` where `a` is live in this partition but `b` is not (e.g., `b` references an agent in a sibling partition via a stale enqueue) is *silently dropped*. The Rustdoc does not document this, and it is inconsistent with the port-rewrite loop, which *preserves* (incorrectly — see QA-003) such cross-partition references.
- **Impact:** In a well-formed partition, both elements of a redex pair are intra-partition (proved by SPEC-04 split's "intra-partition redexes only" invariant). In a *real-world* call site (departure recovery), the redex queue may contain pairs scheduled before reduction stalled; if `cleanup_t1_violations` has invalidated one side, the entire pair is discarded with no diagnostic. SPEC-19 R26 (id_range exhaustion) already covers a related scenario, so this is more of a "consistency wart" than an active bug.
- **Fix:** Add a `debug_assert!` on the asymmetric case:
  ```rust
  .filter_map(|&(a, b)| {
      let mapped_a = id_map.get(&a);
      let mapped_b = id_map.get(&b);
      debug_assert!(
          mapped_a.is_some() == mapped_b.is_some(),
          "redex_queue entry ({}, {}) crosses partition boundary; SPEC-04 forbids this",
          a, b
      );
      Some((*mapped_a?, *mapped_b?))
  })
  ```

#### QA-007 — New `PartitionError` variants are not `serde::Serialize/Deserialize`-derived; if a future telemetry path includes errors in `RoundResult`, the build will silently break the wire format

- **Severity:** LOW
- **Category:** Forward-Compatibility / Wire-Serde Hygiene
- **File:** `relativist-core/src/error.rs:48-71`
- **Description:** `PartitionError` does not derive `Serialize`/`Deserialize`. Today this is correct — errors are not wire payloads. However, the spec brief explicitly raised the question "Do they survive PROTOCOL_VERSION semantics? Are they Serialize-derived?". Confirmed: they are *not* Serialize-derived, by design. This is fine *now*; it becomes a footgun the moment SPEC-22 or a future task adds a `RoundResult::partition_error: Option<PartitionError>` field (or any other wire payload that wants to carry an error variant). The footgun is *silent at the spec layer* (no test catches it) and *loud at the build layer* (compile error when the derive is missing).
- **Action:** No change needed today. Document in TASK-0411's closing notes that any future wire-borne `PartitionError` must add `serde::Serialize/Deserialize` derives to all variants; the `available: u32`/`requested: u32`/`partition_size: u32`/`range_size: u32` fields are already serializable, so the change is mechanical when needed.

#### QA-009 — `Partition::agent_count` cast to `u32` is silent on `usize > u32::MAX` (RV-004 confirmation; QA agrees with severity MEDIUM in review)

- **Severity:** LOW (downgraded from RV-004's MEDIUM in review because the architectural impossibility on 32-bit platforms is structurally enforced; RV-004's severity is preserved as MEDIUM in the unified Stage 6 list)
- **Category:** Defensive Contract
- **File:** `relativist-core/src/partition/types.rs:93-95`
- **Description:** Already covered by RV-004; QA confirms with one additional probe: the cast is the only failure mode that escapes both the early-return guard (`agent_count == 0`) and the size-check guard (`agent_count > range_size`) — so a (theoretical) `usize` overflow truncates to `u32::MAX as u32 = u32::MAX`, comparing against `range_size: u32`, which on a typical successor (`new_range = IdRange { start: 0, end: u32::MAX }` → `len = u32::MAX`) would *just barely* compare as not-too-small. The truncation is silent; the precondition is silently bypassed; the subsequent `vec![None; arena_len]` allocation would either OOM-abort the process or panic on `arena_len` arithmetic overflow at line 102. Fix per RV-004: add the debug_assert.
