### QA-001 (CRITICAL) — Wildcard arm in `transition()` silently swallows all 9 new SPEC-20 events; no test asserts the lost-event contract

- **Severity:** CRITICAL — every new SPEC-20 event reaches a dead-end without producing actions. `WorkerJoined` fired during `WaitingForResults` (R10b) is lost; TASK-0436's R10b test will go green for the wrong reason if the QA-001 mitigating contract isn't established now.
- **Category:** Logic Error / Lost Event / Test Coverage Gap
- **Location:** `coordinator.rs:501–507` (the wildcard `(state, event) => { tracing::warn!(...) }`).
- **Reproduction (static trace, no cargo run):**
  ```rust
  let mut ctx = CoordinatorContext::new("127.0.0.1:9000".parse().unwrap(), 2);
  ctx.state = CoordinatorState::WaitingForResults;
  let actions = transition(&mut ctx, CoordinatorEvent::WorkerJoined(7));
  // Expected (per SPEC-20 §4.1.4 R10b):
  //   ctx.state == WaitingForResults
  //   actions contains QueueWorkerForNextWindow(7), LogJoin(7)
  // Actual:
  //   ctx.state == WaitingForResults  ✓
  //   actions == []                    ✗ (warn-only)
  // The join is LOST. The pending_connections_queue receives nothing.
  ```
- **Repeat for all 9 new events:** every one of `WorkerJoined`, `WorkerLeft`, `WorkerConnectionLost`, `MembershipWindowClosed`, `SelfPartitionReduced`, `SelfPartitionPanic`, `InitialWaitTimeout`, `SoloReductionComplete`, `SoloReduceBatchComplete` from any state hits the wildcard. Verified by static read of `transition()` at `coordinator.rs:354–508` — none of the existing arms match a new SPEC-20 event variant by name.
- **Why this is a CRITICAL bug, not a "deferred to TASK-0436" non-issue:**
  1. **The function `transition()` is `pub` and the enum `CoordinatorEvent` is `pub`.** Any caller — test, runtime, integration code — can invoke `transition(&mut ctx, CoordinatorEvent::WorkerJoined(id))` today. There is no compile-time gate (e.g., `#[non_exhaustive]` would help; SF-004 raised this and got LOW; QA elevates because the impact under MF-001's fix is non-deferred).
  2. **The reviewer's MF-001 fix changes the timer-id literal but does not change the wildcard contract.** After MF-001 lands, `WorkerJoined` is still silently dropped.
  3. **The `tracing::warn!` does NOT raise an error**, does NOT push an action, does NOT mutate `ctx.state` — it is a pure no-op observable only via log scraping. Production builds may silently absorb this in low-verbosity log configurations.
  4. **The 11 UT-0414 tests do not exercise any (state × new-event) cell** — UT-0414-04..08 are *constructor* tests that build events without firing them through `transition()`. There is **no test that asserts what happens** when a new event meets the existing FSM in any state. Verified by reading lines 824–1057.
  5. **The current behaviour creates a test-passes-for-the-wrong-reason trap for TASK-0436.** TASK-0436's authors will write `let actions = transition(&mut ctx, CoordinatorEvent::WorkerJoined(7));` and `assert!(actions.iter().any(|a| matches!(a, CoordinatorAction::QueueWorkerForNextWindow(_))));` — but if the new arm is *missing or has a typo*, the wildcard absorbs the event and the assertion fails with "actions list is empty" — debuggable but slow. Worse, if the developer adds the arm but accidentally uses `_` in a guard that always falses, the wildcard catches the fall-through and the test still fails — also debuggable. **The genuinely dangerous case is silent fall-through from a partially-implemented arm that returns an empty vec by mistake.** A pinning test today would catch it.
- **Suggested fix:**
  1. **(Mandatory)** Add a regression test that asserts the contract `transition(ctx_in_any_state, any_new_event)` — for now in the type-surface task — *must* either (a) produce at least one action, OR (b) be explicitly enumerated as a `(state, event) → no-action` row in a documented matrix. The minimum acceptable implementation for TASK-0414:

     ```rust
     #[test]
     fn ec_3_new_events_in_existing_states_produce_warn_no_action_per_doc() {
         // Pin TODAY's behavior: every new SPEC-20 event in every existing state
         // hits the wildcard arm and emits zero actions. TASK-0436 MUST replace
         // this with explicit transitions; if any (state, new_event) cell still
         // emits zero actions after TASK-0436, that is a contract violation.
         let new_events = || vec![
             CoordinatorEvent::WorkerJoined(7),
             CoordinatorEvent::WorkerLeft(7, LeaveKind::AfterResult),
             CoordinatorEvent::WorkerConnectionLost(7),
             CoordinatorEvent::MembershipWindowClosed,
             CoordinatorEvent::InitialWaitTimeout,
             CoordinatorEvent::SoloReductionComplete,
             CoordinatorEvent::SoloReduceBatchComplete,
         ];
         let states = [
             CoordinatorState::Init, CoordinatorState::WaitingForWorkers,
             CoordinatorState::Partitioning, CoordinatorState::Dispatching,
             CoordinatorState::WaitingForResults, CoordinatorState::Merging,
             CoordinatorState::CheckTermination,
             // Done / Error excluded — terminal states accept FatalError only.
         ];
         for state in &states {
             for event in new_events() {
                 let mut ctx = make_ctx();
                 ctx.state = state.clone();
                 let actions = transition(&mut ctx, event);
                 // CURRENT contract (TASK-0414): wildcard absorbs, no actions emitted.
                 // TASK-0436 MUST flip this to non-empty for every cell it owns.
                 assert!(actions.is_empty(),
                     "QA-001: TASK-0414 baseline violated — state {:?} now emits actions for a new event; \
                      did TASK-0436 land partially? Update this test to assert the new contract.",
                     state);
             }
         }
     }
     ```
     This test *asserts the bug*. When TASK-0436 lands, the test will fail and the developer must consciously update it — that is the point. It locks in awareness.
  2. **(Recommended)** Add `#[non_exhaustive]` to `pub enum CoordinatorEvent` so external crate callers get a compile error when matching on it without a wildcard, *and* add a TODO comment above the wildcard arm: `// NOTE(TASK-0436): Replace this catch-all with exhaustive arms; any new SPEC-20 event reaching here is a missing transition row that test ec_3 will detect.`
  3. **(Defence-in-depth)** Change the wildcard arm body to push an explicit "unexpected" action sentinel (e.g., `CoordinatorAction::LogTransition { from: ctx.state.clone(), to: ctx.state.clone() }` — i.e., a no-state-change log) so that even a silent fall-through emits at least one action and the FSM contract "every event produces ≥1 action" holds. This is a *minor behaviour change* but it pins the invariant.
- **Connection to SF-001/SF-004:** SF-001 (LOW) flagged the diagnostic clarity gap; SF-004 (LOW) flagged the `#[non_exhaustive]` gap. QA combines them into a single CRITICAL because the consequence in TASK-0436's test design is non-trivial: *silent test-passes-for-wrong-reason*.

### QA-002 (HIGH) — `CoordinatorEvent::SelfPartitionPanic(String)` accepts unbounded payload; coordinator already in distress can OOM via panic-message clone

- **Severity:** HIGH — reachable on any panicked self-partition; release-build behavior is to allocate verbatim.
- **Category:** Resource Exhaustion / Adversarial Input / Defensive Contract
- **Location:** `coordinator.rs:201–203` (event variant); also `coordinator.rs:351` (`let from = ctx.state.clone();` — but `ctx.state` is small; the actual amplifier is the `String` payload itself entering via `Debug` formatting in `tracing::error!`).
- **Static trace:**
  ```rust
  // SPEC-20 R3a: "A panic or unexpected termination of the self-partition
  // task MUST surface to the FSM via the new SelfPartitionPanic(String) event."
  // SPEC-20 makes no statement about the bound on this String.
  //
  // Realistic adversarial scenario:
  //   - spawn_blocking task panics inside reduce_n() during a recursive
  //     net traversal; the panic message captures the full call stack +
  //     net dump in the panic hook (e.g., RUST_BACKTRACE=full + a panic
  //     hook that includes Net::Debug). Result: panic_message ~ 4 GiB.
  //   - The runtime catches the panic, captures the message via
  //     tokio::task::JoinError::into_panic() -> Box<dyn Any + Send>::downcast::<String>(),
  //     wraps as CoordinatorEvent::SelfPartitionPanic(msg).
  //   - transition() matches (WaitingForResults, FatalError(_)) — wait,
  //     SelfPartitionPanic is not FatalError, so it hits the wildcard.
  //     wildcard arm: tracing::warn!(state = ?state, event = ?event, ...);
  //     Debug-formatting `event` clones+escapes the entire 4 GiB String.
  //     -> coordinator OOM during shutdown, no actions emitted.
  ```
- **Impact:**
  1. **Coordinator OOM during fatal-handling code path.** The exact moment when graceful shutdown matters most.
  2. **Even after MF-001 + QA-001 are fixed**, when TASK-0436 routes `WaitingForResults × SelfPartitionPanic → Error` (per SPEC-20 §4.1.4 R3a row at line 692), the event still carries the unbounded payload. The transition arm will likely include `tracing::error!(error = %msg, "self-partition panic")` per the `FatalError` precedent at line 491 — same OOM amplifier.
  3. **The `from = ctx.state.clone()` pattern at line 351 is innocent** (state is small enum); the amplifier is the `event`'s `Debug` materialisation in tracing macros.
- **Why current tests miss it:** UT-0414-04 and EC-1 both construct `SelfPartitionPanic` with `"boom"` and `String::new()` — short, bounded payloads. There is no test with a multi-MB payload, no test asserting payload-length bounds, no test asserting `Debug` formatting truncates.
- **Suggested fix:**
  1. **(Preferred)** Change the variant to `SelfPartitionPanic(Box<str>)` AND document a hard cap in the Rustdoc: "`String`/`Box<str>` payload MUST be ≤ 64 KiB; runtime SHOULD truncate longer panic messages to the first 4 KiB before constructing the event." Update SPEC-20 R3a in a separate spec amendment to bound the payload (cross-task tracker for `especialista-specs`).
  2. **(Compromise)** Keep `String` but add a constructor helper: `pub fn self_partition_panic(msg: impl Into<String>) -> Self { ... }` that truncates `msg` to 4 KiB with a "[truncated]" suffix. Production code uses the helper; the raw variant stays for tests. This puts the bound at the call boundary without an enum-shape change.
  3. **(Minimal — TASK-0414 scope)** Add a test:
     ```rust
     #[test]
     fn self_partition_panic_payload_size_documented() {
         // Pin the current "no enforcement" state — replaces with truncation
         // assertion when QA-002 lands.
         let huge = "X".repeat(64 * 1024 * 1024); // 64 MiB
         let event = CoordinatorEvent::SelfPartitionPanic(huge.clone());
         match event {
             CoordinatorEvent::SelfPartitionPanic(s) => {
                 assert_eq!(s.len(), 64 * 1024 * 1024,
                     "QA-002: payload is unbounded; if this fails, SPEC-20 R3a \
                      enforced a cap — update test to match.");
             }
             _ => unreachable!(),
         }
     }
     ```
     Then file a follow-up against SPEC-20 to add the cap. The test pins the *current* state and forces a conscious flip later.
- **Note:** This complements MF-002 (LeaveKind placement) — both are "the type surface accepts something the wire/runtime layer cannot safely consume." Worth folding into a single "ENUM-SURFACE BOUNDS" review pass.

### QA-005 (MEDIUM) — `CoordinatorContext::clone()` is not derived; `from = ctx.state.clone()` at line 351 only clones `state` — but no test asserts that `ctx` mutations during a wildcard fall-through preserve invariants

- **Severity:** MEDIUM — partial-mutation hazard during invalid transitions.
- **Category:** State Machine / Invariant Preservation
- **Location:** `coordinator.rs:307–343` (`CoordinatorContext` definition; only `Debug` derived); `coordinator.rs:350–510` (`transition()`).
- **Static trace:**
  - `CoordinatorContext` derives only `Debug`, not `Clone` (verified at line 307). This is fine *if* the FSM transitions are atomic.
  - But: in the `WaitingForWorkers × WorkerConnected` arm at lines 366–387, `ctx.connected_workers += 1` happens *unconditionally before* the count check. If the same worker_id is delivered twice (e.g., spurious TCP duplicate via QA-007 below), `connected_workers` increments twice without dedup. The current FSM has no `WorkerSet` to dedupe.
  - Symmetric concern: `ctx.partitions_received += 1` at line 427 in the `WaitingForResults × PartitionReturned` arm; no dedup if a worker sends `PartitionReturned` twice (network retry under SPEC-06 R30 timeout).
  - **The new SPEC-20 events that hit the wildcard arm do NOT mutate `ctx`** (the wildcard body is `tracing::warn!` only) — that is correct. But QA-001 already raised that issue from a different angle. QA-005 is the complementary question: are the *existing* arms idempotent under spurious event repetition?
- **Why this matters under SPEC-20:**
  - SPEC-20 §3.5 R10b says `WorkerJoined` events that arrive mid-round must be queued for the next window. After QA-001 is fixed and TASK-0436 lands the R10b arm, that arm will likely call `actions.push(CoordinatorAction::QueueWorkerForNextWindow(id))` — which buffers in the runtime, not in `ctx`. So the FSM itself stays clean.
  - But what about `WorkerConnected`? If a worker reconnects under SPEC-20 R0d (full-rejoin under mode-mismatch), the `WorkerConnected` event may fire again for an id that was already connected. The current arm increments `connected_workers` unconditionally — silent over-count.

### QA-008 (MEDIUM) — `CoordinatorEvent` and `CoordinatorAction` are `pub` but lack `#[non_exhaustive]`; external matchers will silently miss new SPEC-20 variants if `relativist-core` is consumed as a library

- **Location:** `coordinator.rs:155` (`CoordinatorEvent`), `coordinator.rs:229` (`CoordinatorAction`).

    ```rust

    ```
    compiles today (exhaustive). After TASK-0418/TASK-0432 lands, the new variants are silently absorbed by the wildcard `_ =>` if any (or fail to compile if not). The compile-failure path is acceptable; the silent-absorption-by-wildcard path is the hazard.

### QA-009 (MEDIUM) — `Net` payload in `CoordinatorEvent::MergeComplete { net, ... }` and `CoordinatorAction::WriteOutput(Net)` is moved by value through the FSM; an adversarial 1 GiB net allocates twice during transition (`from = ctx.state.clone()` is small, but the `Net` is large)

- **Category:** Resource Exhaustion / Memory Profile

- **Static trace:**
  - The transition arm for `Merging × MergeComplete` at lines 454–487 destructures `event` to get `net`, then *moves* `net` into `WriteOutput(net)` or `InvokeSplit { net, … }`. So it's move-only — no clone. Good.

     impl fmt::Debug for CoordinatorEvent {

         }
     }

The current `#[derive(Debug)]` materialises the entire structure. Prefer `tracing::warn!(event = ?event.kind())` (a `kind()` accessor returning the variant name) over `tracing::warn!(event = ?event)` for these variants."
- **Connection to QA-002:** Both findings expose the same root: the FSM's `tracing::*!(event = ?event, ...)` pattern is unbounded under adversarial event payload sizes. A unified mitigation (custom Debug for large-payload variants) addresses both.
