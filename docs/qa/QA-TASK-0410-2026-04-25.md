### QA-001 (CRITICAL) ‚Äî `merged_next_id == u32::MAX` lets the union output corrupt itself on the next `create_agent` (D4 violation)

- **Severity:** CRITICAL ‚Äî release-build silent D4/I3 invariant breach reachable without violating `union`'s documented precondition.
- **Category:** Correctness (D-invariant) / Untested-path / Boundary
- **Location:** `relativist-core/src/net/core.rs:392` (`merged_next_id` computation) and `relativist-core/src/net/core.rs:115-118` (`create_agent`'s `self.next_id += 1`).
- **Reproduction (static trace, no cargo run):**
  ```
  let mut a = Net::new();
  a.next_id = u32::MAX - 1;          // legitimate state after a long run
  let mut b = Net::new();
  b.next_id = u32::MAX;              // legitimate state after a long run

  let c = a.union(b);                // OK: merged_next_id = u32::MAX
  assert_eq!(c.next_id, u32::MAX);   // confirmed by reading core.rs:445

  // Now feed `c` to a downstream caller (e.g. SPEC-20 ¬ß4.2.2 step 5: re-split).
  // Or, more directly:
  let mut c_mut = c;
  let id = c_mut.create_agent(Symbol::Con);
  // create_agent does:
  //   let id = self.next_id;    // id = u32::MAX
  //   self.next_id += 1;        // DEBUG: panic. RELEASE: wrap to 0.
  ```
- **What happens in each build profile:**
  - **`debug_assertions` ON (cargo test default):** `next_id += 1` panics with `attempt to add with overflow`. The arena `agents.resize(u32::MAX as usize + 1, None)` immediately above also blows up: `(u32::MAX as usize) + 1 == 4_294_967_296`, an attempt to allocate 4 GiB of `Option<Agent>` (~64 GiB). Allocation fails ‚Üí process aborts.
  - **Release (cargo build --release / cargo test --release):** Integer overflow wraps. `id = u32::MAX`, `self.next_id = 0`. Then `agents.resize((u32::MAX as usize) + 1, None)` will OOM-panic, but if the resize succeeds on a hypothetical large-RAM box, agent `u32::MAX` lives in the arena and `next_id == 0`. The very next `create_agent` returns `id = 0` ‚Üí **D4 (ID Uniqueness) violated** because either (a) agent 0 was already live in `c` (slot collision panic ‚Äî depending on existing state), or (b) agent 0 was free (slot silently overwritten with the new agent). Either way, the post-`union` net is one step from arena corruption.
- **Impact:**
  - **D4 (ID Uniqueness)** ‚Äî directly breached.
  - **I3 (`next_id > any live AgentId`)** ‚Äî silently breached after wrap.
  - **G1 (Grid Determinism / convergence)** ‚Äî downstream `reduce_n` over a net with duplicated AgentIds is undefined; `is_valid_redex` would happily re-fire stale rules.
  - **The bug is reachable from documented-precondition-honouring callers.** The disjoint-AgentId precondition does not bound `next_id`. SPEC-20 ¬ß4.2.2 explicitly composes `union` with `remap_partition_ids` (TASK-0411), which renumbers partitions into disjoint ranges ‚Äî those ranges can stack toward `u32::MAX` over a long elastic-grid run.
- **Why the existing tests miss it:**
  - TEST-SPEC-0410 lists no `u32::MAX` boundary case in EC-1..3 (deliberate scope decision per Stage 4 reviewer note line 145).
  - `Net::union`'s `debug_assert!` (core.rs:415) only fires on **agent slot overlap**, not on `next_id` overflow risk.
  - `create_agent`'s overflow on `next_id += 1` is a pre-existing issue, but `union` makes it materially easier to reach by composing two near-`u32::MAX` nets.
- **Suggested fix (in priority order):**
  1. Add an explicit `debug_assert!` (and ideally a release-build runtime check that returns `Result<Net, NetError>` or saturates) at `core.rs:392`:
     ```rust
     debug_assert!(
         next_id_a < u32::MAX && next_id_b < u32::MAX,
         "SPEC-20 A7 / SPEC-01 I3: Net::union would produce next_id == u32::MAX, \
          which makes the result one create_agent() call from D4 violation. \
          self.next_id={}, other.next_id={}", next_id_a, next_id_b
     );
     ```
  2. Document the boundary explicitly in the Rustdoc `# Precondition` block: "`max(self.next_id, other.next_id) < u32::MAX` ‚Äî the result must have at least one fresh AgentId available; otherwise downstream `create_agent` violates I3."
  3. Add a regression test (release-build resilient): `ec_4_union_next_id_near_u32_max_panics_in_debug_or_returns_safe_in_release` ‚Äî drive `force_agent_id_range` with a tail of `u32::MAX - K` for some small `K`, attempt the union, assert the debug panic OR the saturation behaviour.
  4. Long term (out of TASK-0410 scope): `create_agent` itself should switch from `self.next_id += 1` to a checked-add returning `Result<AgentId, NetError::ArenaExhausted>`. Track as new task on the SPEC-22 backlog.

### QA-002 (HIGH) ‚Äî `force_agent_id_range` fixture is too permissive: green tests DO NOT prove `union` is safe under realistic `remap_partition_ids` output

- **Severity:** HIGH ‚Äî passes do not generalise to TASK-0411's production output.
- **Category:** Untested-path / Test Quality
- **Location:** `relativist-core/tests/net_union.rs:79-99` (the fixture); cross-check `relativist-core/src/net/core.rs:399-407` (`agents_target_len` / `ports_target_len`).
- **Reproduction (static trace):**
  - `force_agent_id_range(net, 100, &[Con, Dup, Era])` resizes `agents` to `len=103` with `[None; 100] ++ [Some(_); 3]` and `ports` to `len=309` with `[DISCONNECTED; 309]`. Slots 0..100 are `None`, ports 0..300 are `DISCONNECTED`.
  - `Net::union` then sees `agents_a.len() == 103` and `next_id_a == 103`. **All slots in the range `[0..100)` are `None`, so the agent-overlap `debug_assert!` (line 415) silently passes** ‚Äî there's nothing to overlap with.
  - **But `remap_partition_ids` (TASK-0411) is unlikely to produce an arena with this exact shape.** TASK-0411 will renumber partition agents into a disjoint range, and the *production* `Partition` will likely keep its `Net` compact: `agents.len() == agent_count`, with NO leading `None` padding. Or alternatively it may produce a sparse arena with `None` padding *between* renumbered ids if it remaps in-place. Either way the assumption baked into the fixture is "the leading prefix of `agents` is `None`", and **that assumption is not promised by SPEC-20 ¬ß4.2.2 / ¬ß3.8 A4 / R13**.
- **Concrete escape case the fixture misses:**
  ```
  // net_a (compact): agents=[Some(0), Some(1)], next_id=2, ports=[‚Ä¶6 entries‚Ä¶]
  // net_b: produced by remap_partition_ids that renumbers in-place ‚Äî agents=[None, None, Some(2), Some(3)], next_id=4, ports=[12 entries‚Ä¶]
  // Disjointness: a.live_ids = {0,1}, b.live_ids = {2,3} ‚Äî disjoint. ‚úì
  // Union: agents_target_len = max(2, 4) = 4. agents.resize(4, None).
  //   Loop: idx=0 slot=None ‚Üí skip; idx=1 slot=None ‚Üí skip; idx=2 slot=Some(_) ‚Üí agents[2] = Some.
  //   But wait ‚Äî agents[2] is currently None (resize put it None), so the assert passes.
  //   What if a.live_ids included 2? Then a.next_id ‚â• 3 ‚â• 2 disjoint precondition is already violated.
  // OK: the disjoint-AgentId check protects against this. ‚úì
  ```
  So the fixture is *technically* sound for the union, but it doesn't tell us whether REAL `remap_partition_ids` output (which the developer of TASK-0411 has not yet defined) would be congruent.
- **The actual risk:** Stage 6 lands `union` with this fixture; Stage 7+ lands TASK-0411 with a slightly different post-condition; `union(remap_partition_ids(p_a), remap_partition_ids(p_b))` exposes a corner the fixture didn't model ‚Äî most likely "ports vector has gaps" or "agent vector has trailing `None`s past `next_id` from a removed agent in the source partition that survived the renumbering."
- **Suggested fix:**
  1. **Add a fixture variant** `compact_net_with_id_range(base, symbols)` that produces a Net with `agents.len() == symbols.len()` (no leading `None` padding) and `next_id == base + symbols.len()` ‚Äî and add a UT that asserts `union(compact, compact_disjoint)` works. This proves union doesn't accidentally rely on the leading-`None` padding.
  2. **Add a fixture variant** `sparse_net_with_residue` that produces a Net with one `Some` agent followed by a removed slot (`None`) followed by another `Some` ‚Äî i.e., the shape `remap_partition_ids` will produce after a partition shrinks via `remove_agent`. Add a UT.
  3. **Document the test-fixture contract** at the top of `net_union.rs`: "These fixtures simulate ONE specific shape of `remap_partition_ids` output. TASK-0411's actual output may differ; the union impl MUST also be correct under (a) compact arenas, (b) sparse arenas with internal `None` slots, (c) trailing-`None` arenas. Add a regression test BEFORE TASK-0411 lands."
- **Why this is HIGH not CRITICAL:** the bug is hypothetical until TASK-0411 lands; the union is correct *for the fixture shape*. But the bundle's risk profile is "this is the FIRST DEV task of 50; if its tests bake in a specific arena shape, every subsequent task inherits the same blind spot."

### QA-003 (HIGH) ‚Äî `freeport_redirects` collision is documented as "other wins" but completely unexercised; the Rustdoc soundness argument is also wrong (RV-003 confirmed and expanded)

- **Severity:** HIGH ‚Äî undocumented behaviour reachable from any caller that has called `connect(FreePort, FreePort)` on either input.
- **Category:** Correctness (Determinism) / Untested-path / Documentation
- **Location:** `relativist-core/src/net/core.rs:439` (`redirects_a.extend(redirects_b)`) + Rustdoc lines 354-356.
- **Reproduction (static trace):**
  ```
  // Setup: both inputs have a FreePort-to-FreePort wire.
  let mut a = Net::new();
  let id_a = a.create_agent(Symbol::Con);     // agent at id 0
  // (we deliberately don't connect id_a; we just need redirects_a populated)
  a.connect(PortRef::FreePort(7), PortRef::FreePort(8));
  // After this: a.freeport_redirects = {7 ‚Üí FreePort(8), 8 ‚Üí FreePort(7)}

  let mut b = Net::new();
  // Force agent id 100 so disjoint precondition holds
  force_agent_id_range(&mut b, 100, &[Symbol::Con]);
  b.connect(PortRef::FreePort(7), PortRef::FreePort(99));
  // After this: b.freeport_redirects = {7 ‚Üí FreePort(99), 99 ‚Üí FreePort(7)}

  let c = a.union(b);
  // What is c.freeport_redirects[7]?
  // Code at core.rs:439 is `redirects_a.extend(redirects_b)`.
  // HashMap::extend on key collision OVERWRITES, so c.freeport_redirects[7] = FreePort(99).
  // c.freeport_redirects[8] = FreePort(7) (only in self).
  // c.freeport_redirects[99] = FreePort(7) (only in other).
  // So 7 ‚Üí 99, 99 ‚Üí 7, but 8 ‚Üí 7 (orphan: nothing points back to 8).
  ```
- **Impact:**
  - The 7‚Üî8 wire from `self` is now ASYMMETRIC: `freeport_redirects[7] = FreePort(99)` and `freeport_redirects[8] = FreePort(7)`. The redirect graph has a one-way edge 8‚Üí7 with no return path. `rebuild_free_port_index` (SPEC-05) would resolve this incorrectly ‚Äî the post-merge wiring depends on which side is queried first.
  - Worse: the Rustdoc claims (line 355-356) "Under the disjoint-`AgentId` precondition the targets cannot reference colliding agents." This is **false** in two ways:
    1. The precondition is on `AgentId`, not on border-`u32` keys (already noted by RV-003 in the Stage 4 review).
    2. Even if both inputs used disjoint AgentIds, the *targets* of `freeport_redirects` can be arbitrary `PortRef`s (FreePort or AgentPort) that need not be agent-disjoint ‚Äî the targets are border-id-resolved later, not AgentId-resolved.
- **Why downstream may not catch it:**
  - SPEC-20 ¬ß4.2.2 calls `union` then `split`. `split()` rebuilds border IDs from scratch and **discards `freeport_redirects`** (per its standard SPEC-04 mechanism). So the asymmetric redirect map is observationally invisible *for that one call site*.
  - **But `Net::union` is `pub`**: any other caller (test, future task, external integration) gets the asymmetric map. There is no `#[doc(hidden)]` or `pub(crate)` gate.
- **Why the tests miss it:** `build_net_a` / `build_net_b` / `build_collision_net_with_id_5` never call `connect(FreePort, FreePort)` ‚Äî only `connect(AgentPort, FreePort)`. So `freeport_redirects` is empty on every input across all 10 tests. Verified by static read of `tests/net_union.rs:30-68`. The collision policy is **completely uncovered**.
- **Suggested fix:**
  1. Add `EC-4` to `tests/net_union.rs` exercising the FreePort-FreePort collision ‚Äî assert "other wins" or specify a different policy. Recommended skeleton:
     ```rust
     #[test]
     fn ec_4_union_freeport_redirect_collision_other_wins() {
         let mut a = Net::new();
         a.connect(PortRef::FreePort(7), PortRef::FreePort(8));
         let mut b = Net::new();
         b.connect(PortRef::FreePort(7), PortRef::FreePort(99));
         let c = a.union(b);
         assert_eq!(c.freeport_redirects.get(&7), Some(&PortRef::FreePort(99)),
                    "RV-003 / QA-003: other wins on key collision");
         assert_eq!(c.freeport_redirects.get(&8), Some(&PortRef::FreePort(7)));
         assert_eq!(c.freeport_redirects.get(&99), Some(&PortRef::FreePort(7)));
     }
     ```
  2. Replace the wrong soundness argument in the Rustdoc (already RV-003's recommendation; my version is more pointed):
     > "`freeport_redirects`: union of the two maps via `HashMap::extend`; on key collision, `other`'s entry overwrites `self`'s. **Border-`u32` keys are NOT covered by the disjoint-`AgentId` precondition** (see EC-3 for a duplicate-border-id case; see EC-4 for a duplicate-redirect-key case). The asymmetric-redirect outcome is observationally invisible **only when** the union result is immediately round-tripped through `split()` (SPEC-20 ¬ß4.2.2 step 4 ‚Üí step 5), which discards `freeport_redirects`. **Other callers MUST NOT rely on the redirect map being symmetric after union.**"
  3. (Optional, defence-in-depth) Add a `debug_assert!` checking `redirects_a.keys().collect::<HashSet<_>>().is_disjoint(&redirects_b.keys().collect::<HashSet<_>>())` BEFORE `extend`, with a message saying "if this fires, see EC-4 ‚Äî call site must round-trip through split()". This makes the assumption explicit at the call boundary.

### QA-004 (MEDIUM) ‚Äî `agents_target_len` does not include `agents_b.len()`; `Some(_)` residue past `agent_capacity` would panic with index-out-of-bounds (precondition asymmetry)

- **Severity:** MEDIUM ‚Äî only reachable if I3 is violated on the right input.
- **Category:** Safety / Defensive contract
- **Location:** `relativist-core/src/net/core.rs:399`.
- **Static trace:**
  ```rust
  let agents_target_len = std::cmp::max(agents_a.len(), agent_capacity);
  // NOT: max(agents_a.len(), agents_b.len(), agent_capacity)
  ```
  This is sound IFF I3 holds on `other` (because then `agents_b[i].is_some()` ‚üπ `i < other.next_id ‚â§ merged_next_id == agent_capacity ‚â§ agents_target_len`). But if I3 is violated on `other` ‚Äî e.g., a fixture or future caller leaves a `Some(agent)` at index `idx > other.next_id` ‚Äî the loop at line 413 will `agents[idx] = Some(_)` with `idx ‚â• agents_target_len`, **panicking with `index out of bounds`** rather than a useful "you violated I3 on `other`" message.
- **Why the loop body uses unchecked indexing:** line 420 is `agents[idx] = Some(agent)`, which uses `Vec::index_mut`. On out-of-bounds, this panics.
- **Why the existing test suite misses it:** every fixture respects I3.
- **Suggested fix:**
  1. Either expand the target-length computation to `max(agents_a.len(), agents_b.len(), agent_capacity)` (cheap; fixes the panic by silently growing the result).
  2. Or add a `debug_assert!` before the loop: `debug_assert!(agents_b.len() <= agents_target_len, "SPEC-01 I3 violated on `other`: agents.len()={} > next_id={}", agents_b.len(), next_id_b);`.
  3. Recommend (1) ‚Äî it's load-bearing for "Net::union accepts any structurally-valid Net" and removes a false contract dependency.

### QA-005 (MEDIUM) ‚Äî Tail-residue corruption: non-DISCONNECTED targets in `ports_a` past `self.next_id * 3` survive into the result without sanity check

- **Severity:** MEDIUM ‚Äî only reachable via private `set_port` misuse or fixture irregularity, but the union is a `pub` boundary so it must defend.
- **Category:** Safety / Defensive contract
- **Location:** `relativist-core/src/net/core.rs:432-436` (the port-copy loop).
- **Static trace:**
  ```rust
  for (idx, target) in ports_b.into_iter().enumerate() {
      if target != DISCONNECTED && idx < ports.len() {
          ports[idx] = target;
      }
  }
  ```
  This loop **only writes** when `ports_b[idx]` is non-DISCONNECTED. Consequence: any non-DISCONNECTED target in `ports_a[idx]` for `idx ‚â• self.next_id * 3` (i.e. past self's "claimed" range) survives in the result, even if no agent owns that port slot.
- **Realistic trigger paths:**
  - Future task uses `Net::union` after manually growing `ports` via `ports.resize` and writing to slots beyond `next_id * 3`. The Net's encapsulation does not formally forbid this (`ports` is `pub` per core.rs:32).
  - Existing `force_agent_id_range` fixture writes to `net.ports` directly ‚Äî if a future test author uses `force_agent_id_range` followed by direct `net.ports[k] = AgentPort(...)` writes for indices beyond `max_id * 3`, the residue survives union.
  - Production `remove_agent` always DISCONNECTs both sides via `disconnect`, so production-only paths are safe today.
- **Suggested fix:**
  1. Add a `debug_assert!` after the destructure verifying that for every index `i ‚â• next_id_a * 3`, `ports_a[i] == DISCONNECTED`. This pins the "no residue past next_id" invariant as part of the union contract.
  2. Same for `ports_b`.
  3. Update Rustdoc `# Field handling` `ports` paragraph to enumerate the invariant: "Caller MUST guarantee that for both inputs, every port slot at index `‚â• next_id * 3` is `DISCONNECTED`. Production paths (`create_agent`, `connect`, `remove_agent`, `disconnect`) maintain this; direct field manipulation MUST follow the same convention."
- **Connection to RV-004:** This is the second half of RV-004 ‚Äî RV-004 noted that ports_a slots in OTHER's range must be DISCONNECTED; QA-005 notes that ports_a slots **past anyone's range** must also be DISCONNECTED. Both should be folded into the same `debug_assert!` cluster.

### QA-006 (MEDIUM) ‚Äî `redex_queue` ordering is observable at the public API and undocumented; non-SPEC-20 callers may be surprised

- **Severity:** MEDIUM ‚Äî observable via the public `redex_queue: VecDeque<‚Ä¶>` field; surface area for future tasks composing differently.
- **Category:** Determinism / Documentation
- **Location:** `relativist-core/src/net/core.rs:438` (`redex_a.extend(redex_b)`); Rustdoc line 348.
- **Static trace:** `redex_a.extend(redex_b)` puts `self`'s entries first, then `other`'s (FIFO append). The Rustdoc says "the two queues are concatenated (`self`'s first)" ‚Äî correct, but it doesn't say what *behaviour* hinges on this, nor does any test pin the ordering.
- **Adversarial scenario (reachable):**
  - SPEC-22 (or any future caller) might want to use `Net::union` outside the ¬ß4.2.2 round-trip-through-split pattern. If that caller relies on a specific redex-ordering for determinism (e.g., to make `reduce_n(budget)` reduce a specific redex first), the current "self's first" policy becomes a load-bearing API choice. With no test pinning it, a future refactor could swap to "other's first" without anyone noticing.
- **Why the tests miss it:** UT-0410-03 / UT-0410-04 / UT-0410-05 use multiset comparisons that are insensitive to ordering. Verified by static read of test bodies.
- **Suggested fix:** Add `EC-5` to `tests/net_union.rs`:
  ```rust
  #[test]
  fn ec_5_union_redex_queue_self_first_concatenation() {
      let mut a = Net::new();
      let a0 = a.create_agent(Symbol::Con);
      let a1 = a.create_agent(Symbol::Dup);
      a.connect(PortRef::AgentPort(a0, 0), PortRef::AgentPort(a1, 0));
      // a.redex_queue = [(0, 1)]
      let mut b = Net::new();
      force_agent_id_range(&mut b, 100, &[Symbol::Con, Symbol::Dup]);
      b.connect(PortRef::AgentPort(100, 0), PortRef::AgentPort(101, 0));
      // b.redex_queue = [(100, 101)]
      let c = a.union(b);
      let queue: Vec<_> = c.redex_queue.iter().copied().collect();
      assert_eq!(queue, vec![(0, 1), (100, 101)],
                 "redex_queue concatenation order: self first, then other");
  }
  ```
- **Note:** This corresponds to RV-006 (the Stage 4 LOW); QA elevates it to MEDIUM because the absence of a test (not just the absence of a Rustdoc paragraph) is the real risk.

### QA-007 (MEDIUM) ‚Äî UT-0410-07 is `#[cfg(debug_assertions)]`-only; release-build callers have ZERO protection from precondition violation

- **Severity:** MEDIUM ‚Äî documented behaviour but the practical safety net is missing in production.
- **Category:** Safety / Untested-path
- **Location:** `relativist-core/tests/net_union.rs:298-329` (the `#[cfg(debug_assertions)]` gate).
- **Static trace:** The only mechanism enforcing the disjoint-AgentId precondition is `debug_assert!` at `core.rs:415-419`. In release builds the assert is compiled out ‚Üí silent agent overwrite at line 420 (`agents[idx] = Some(agent)` blindly overwrites whatever was there). Symbol could swap (Con ‚Üí Era), wires would orphan, internal-wires would reference a different symbol's port count.
- **Realistic trigger:** A bug in `remap_partition_ids` (TASK-0411) that off-by-ones the disjoint range allocation. In production builds this manifests as silent net corruption with no panic; the operator only sees downstream confluence failures or G1 violations many rounds later, with no traceback to the union call.
- **Why the tests miss it:** UT-0410-07 exists, but it is gated. There is no release-build smoke test that simulates "precondition violation in production".
- **Suggested fix (pick one):**
  1. **(Preferred)** Promote the disjoint-AgentId check to a release-build `#[cold]` runtime check that returns `Result<Net, NetError::PreconditionViolation>` on violation, OR wrap `Net::union` in a debug helper wrapper that does the runtime check and remove the `debug_assert!`. ### QA-008 (MEDIUM) ‚Äî `Net::empty()` vs `Net::new()` divergence risk: no test pins their byte-equality, so a future divergence goes undetected (RV-001 expansion)

### QA-009 (LOW) ‚Äî `freeport_redirects` carries a `PortRef` target that may point at the dropped `other.root` after union; harmless under SPEC-20 but observability footgun

- **Severity:** LOW ‚Äî no soundness impact under documented call pattern.
- **Category:** Future-Proofing / Documentation
- **Location:** `relativist-core/src/net/core.rs:439`.
- **Static trace:** `freeport_redirects` map values are arbitrary `PortRef`s ‚Äî including `AgentPort(other_root_id, 0)` if the caller had set that as a redirect target before union. - **Why this is LOW:** SPEC-20 ¬ß4.2.2 calls `split()` immediately after `union`, and `split()` ignores `freeport_redirects` for root-reconstruction. No production caller is affected.
- **Suggested fix:** None code-wise. Add a one-line note to the Rustdoc `# Field handling` `freeport_redirects` bullet: "Note: redirect targets pointing at `other.root` are preserved verbatim despite `other.root` being dropped; downstream `split()` discards the redirect map and re-establishes roots from scratch."

### QA-010 (LOW) ‚Äî `Net::union` consumes both inputs by value; cloning an input that fails the precondition still triggers the panic ‚Äî caller can't do dry-run validation

- **Severity:** LOW ‚Äî ergonomic.
- **Category:** Future-Proofing / API Design
- **Location:** `relativist-core/src/net/core.rs:373` (`pub fn union(self, other: Net) -> Net`).

- **Why this is LOW:** No production caller will validate-before-commit; the upstream `remap_partition_ids` is already the validation step.
- **Suggested fix:** None now. Track as future API expansion if a test or external caller needs dry-run.
