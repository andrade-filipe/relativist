### QA-001 (HIGH) ã `handle_phase_timeout(worker_index, ...)` assumes per-worker timeout granularity that `collect_results` does NOT provide

- **Severity:** HIGH ã the helper signature is structurally incompatible with the only caller it can ever have, and TASK-0438's per-worker timer rewrite is documented as part of TASK-0438's own scope, not TASK-0413's.
- **Category:** API design / temporal-coupling / spec-vs-impl mismatch
- **Location:** `relativist-core/src/protocol/coordinator.rs:109-131` (`handle_phase_timeout` signature) vs `:318-377` (`collect_results` shape) vs `docs/backlog/TASK-0438-departure-detection-timeout-connloss.md:44` ("Per-worker vs aggregate timeout: R18 mandates per-worker detection (not a single aggregate timer)").
- **Reproduction (static trace, no cargo run):**
  Read `collect_results` (lines 318-377):
  ```rust
  pub async fn collect_results(
      worker_streams: &mut [TransportStream],
      round: u32,
      max_payload_size: u32,
      collect_timeout: Duration,
  ) -> Result<...> {
      let collect_future = async { /* sequential recv_frame loop, no per-worker timer */ };
      tokio::time::timeout(collect_timeout, collect_future)
          .await
          .map_err(|_| ProtocolError::Timeout {
              phase: "collect_results",
              elapsed: collect_timeout,    // <-- single value, no worker_index
          })?
  }
  ```
  The `Timeout` error variant carries an `elapsed` and a `phase`, but **no `worker_id` and no `worker_index`** ã because the timer is aggregate. The for-loop iterates `for stream in worker_streams.iter_mut()`, with no `enumerate()`, so even at the call site no usize index is produced.
  
  Now read `handle_phase_timeout`:
  ```rust
  pub(crate) fn handle_phase_timeout(
      worker_index: usize,                 // <-- where would this come from?
      elapsed: Duration,
      elastic_departure: bool,
  ) -> ConnectionLossOutcome { ... }
  ```
  TASK-0438's *own* acceptance criterion at line 19 says: "`PhaseTimeout` fires per-worker when their result is overdue: emit `CoordinatorEvent::PhaseTimeout(worker_id)`." TASK-0438's note at line 44 says: "Per-worker vs aggregate timeout: R18 mandates per-worker detection (not a single aggregate timer)."
  
  **This means TASK-0438 will need to first rewrite `collect_results` to use a per-worker timer (e.g., `tokio::select!` or `tokio::join!` per stream) BEFORE it can supply a `worker_index` to `handle_phase_timeout`.** The TASK-0413 helper signature is therefore not just a "wired later" placeholder ã it is a *future-tense fiction* whose only valid call site does not exist and whose enabling refactor is part of TASK-0438's scope.

- **Why this matters now (and is HIGH, not LOW):**
  Phase-A spec-driven development justifies Phase-A code only when the surface is *stable* ã i.e., downstream tasks can consume it without further changes. If TASK-0438 lands and immediately needs to rewrite `handle_phase_timeout`'s signature (e.g., to accept a `WorkerId` directly, or to fold into a per-worker recv-with-timeout loop that doesn't have a `worker_index`), then TASK-0413's surface contract was a fiction. The cost is small *now* (10 LoC + tests), but compounds: REVIEW-TASK-0413 already has 6 findings dependent on the current shape (RV-001 wants to add `kind` to the *current* enum, which would also need to be redone).
  
  The same critique applies more weakly to `handle_connection_loss` ã TCP I/O errors are detected per-stream (during `recv_frame` or `send_frame`), so a `worker_index` *could* be supplied via `enumerate()`, but no current call site does this either.

- **Suggested fix (one of):**
  1. **Preferred:** widen the signature *now* to `WorkerId` (a `u32` newtype-or-alias from `partition::types`) instead of `usize`; this matches what TASK-0438 will actually pass, and it implicitly closes QA-002 (no usize ã u32 cast). Update UT-0413-02, -05, -07, -10 fixtures to construct `WorkerId(N)` instead of `0`/`3`/`5`. The change is mechanical:
     ```rust
     use crate::partition::WorkerId;
     pub(crate) fn handle_connection_loss(
         worker_id: WorkerId,           // was: worker_index: usize
         error_description: &str,
         elastic_departure: bool,
     ) -> ConnectionLossOutcome { ... }
     pub(crate) enum ConnectionLossOutcome {
         Abort(String),
         RecoveryTriggered { worker_id: WorkerId },   // was: worker_index: usize
     }
     ```
  2. **Minimal:** at minimum, document in the function-level Rustdoc that `worker_index` is a "logical placeholder" and call out **explicitly** that TASK-0438 will need to (a) refactor `collect_results` to a per-worker recv-with-timeout shape AND (b) plumb a `WorkerId` (not a `usize`) into the helper. Without this note, TASK-0438's author will read the existing UT-0413 tests (which pass `0`, `3`, `5` etc.) and assume the helper signature is final.
  3. **Document at minimum:** add a `# Phase-A note` block to both Rustdocs:
     ```
     # Phase-A note (TASK-0413)
     The `worker_index: usize` parameter is a placeholder. TASK-0438 will refactor
     `collect_results` to per-worker recv-with-timeout (per its own acceptance
     criterion line 19 and design note line 44) and switch this parameter to
     `WorkerId` (= u32). Tests that hard-code `worker_index = 0/3/5` will need to
     be migrated to `WorkerId` constructors at that time.
     ```

- **Severity rationale:** HIGH because Phase-A surface stability is the entire justification for shipping a gate-only task; if the surface is already known not to survive downstream wiring, the test-floor protection it offers is illusory.

### QA-002 (HIGH) ã `worker_index: usize` ã `WorkerId(u32)` cast in downstream TASK-0438 will silent-truncate on 64-bit hosts when `worker_index > u32::MAX`

- **Severity:** HIGH ã silent data loss on a documented hot path (`as` cast in Rust truncates without panic in release builds), no boundary test, no `debug_assert!`, and the existing precedent (`as WorkerId` at lines 1025 / 1035) demonstrates the truncating cast is already the project's established pattern.
- **Category:** Integer overflow / boundary / silent truncation
- **Location:** `relativist-core/src/protocol/coordinator.rs:71` (`worker_index: usize`), `:49` (`worker_index: usize` in the enum); the *future* call site in TASK-0438 will be where the truncation lands; `relativist-core/src/partition/types.rs:11` confirms `pub type WorkerId = u32`.
- **Reproduction (static trace, no cargo run):**
  ```
  // 64-bit host (Linux x86_64, macOS, Windows-amd64).
  let huge: usize = 4_294_967_296;        // u32::MAX + 1 = 0x1_0000_0000
  let outcome = handle_connection_loss(huge, "err", true);
  // outcome = RecoveryTriggered { worker_index: 4_294_967_296 }
  
  // TASK-0438 will need to map worker_index -> WorkerId for FSM dispatch.
  // Existing precedent in this same file (line 1025):
  //     worker_id: i as WorkerId,    // i: usize, WorkerId: u32
  // The `as` cast TRUNCATES in release builds without panic. So:
  let wid = huge as WorkerId;             // wid = 0u32 (low 32 bits of huge)
  // CoordinatorEvent::WorkerConnectionLost(WorkerId(0)) is dispatched.
  // -> wrong worker is reclaimed.
  ```
- **What happens in each build profile:**
  - **debug_assertions ON (cargo test default):** `as` casts in Rust never panic, even in debug. The truncation is silent in both profiles.
  - **release (cargo build --release):** identical truncation, no warning, no panic.
- **Impact:**
  - SPEC-20 ôÏ3.3 R18 reclaim path operates on the wrong `WorkerId`.
  - SPEC-20 ôÏ3.3 R19 reclaim path operates on the wrong `WorkerId`.
  - `RemoveWorker(id)` (SPEC-20 ôÏ4.1.3 transition row line 695) removes a different worker than the one that disconnected.
  - **D6 (Worker Set integrity) silently violated** ã the reclaimed partition is the wrong one.
- **Why the existing tests miss it:**
  - UT-0413-01..10 only use `worker_index ã {0, 1, 2, 3, 5}` ã well below `u32::MAX`.
  - No test probes `worker_index = u32::MAX` or `worker_index = usize::MAX`.
  - The helper's own enum carries `usize`, so even a defensive `WorkerId::try_from(worker_index)` is impossible until the call-site cast happens ã which is in TASK-0438.
- **Suggested fix (in priority order):**
  1. **Best:** apply QA-001's preferred fix (switch the parameter and the enum field to `WorkerId` directly). This eliminates the cast entirely and the entire problem class.
  2. **Alternative:** if `usize` must be preserved (it shouldn't), add a `debug_assert!(worker_index <= u32::MAX as usize)` to both helpers, AND a regression test:
     ```rust
     #[test]
     #[should_panic(expected = "worker_index <= u32::MAX")]
     fn test_handle_connection_loss_panics_on_oversized_worker_index_in_debug() {
         let huge = u32::MAX as usize + 1;
         let _ = handle_connection_loss(huge, "err", true);
     }
     ```
     and a release-build saturation note in the Rustdoc.
  3. **Document at minimum** the precondition `worker_index <= u32::MAX` in the function Rustdocs so TASK-0438's author has a tripwire.
- **Severity rationale:** HIGH because the silent truncation lands on the *exact* downstream task (TASK-0438) that TASK-0413 was supposed to enable, and the established `as WorkerId` pattern (lines 1025, 1035) means a defensive author won't reach for `try_from` by default.

### QA-004 (MEDIUM) ã Double-logging risk: `tracing::warn!` inside helpers + likely call-site `tracing::error!`/`warn!` on the same event = two log lines per departure

- **Severity:** MEDIUM ã observability hygiene; not a correctness bug but an SLO/log-volume risk that lands on every elastic departure.
- **Category:** Observability / cross-cutting
- **Location:** `relativist-core/src/protocol/coordinator.rs:78-84` and `:117-123` (the `tracing::warn!` calls inside the helpers). Also, the existing `recv_frame` failure paths in `collect_results` and `accept_workers` typically log before propagating; once TASK-0438 wires the helper, the call site will receive the I/O error, log it, then call `handle_connection_loss` which logs again.
- **Reproduction (forward-looking trace):**
  ```rust
  // TASK-0438's likely call shape (informed guess from collect_results @ line 318):
  match recv_frame(stream, ...).await {
      Err(io_err) => {
          tracing::error!("recv_frame failed: {}", io_err);        // call site logs
          let outcome = handle_connection_loss(idx, &io_err.to_string(), grid_config.elastic_departure);
          // helper logs AGAIN with `tracing::warn!` if elastic_departure=true
      }
      Ok(_) => { ... }
  }
  ```
  Result: per departure, two log lines with overlapping content (the error string is duplicated; the worker_index appears twice). For a long-running benchmark with many elastic departures, log volume doubles.
- **Impact:**
  - Log aggregators (e.g., a future `tracing-subscriber` JSON sink) ingest duplicated events.
  - Operators reading logs see the same event twice and may infer two events occurred.
  - SPEC-20 ôÏ11 mentions metric emission; if a metric counter is incremented in both places, the count is double.
- **Why the existing tests miss it:** UT-0413 tests do not capture log output (no `tracing-test` subscriber). The double-log is invisible to the test suite.
- **Suggested fix (one of):**
  1. Remove the `tracing::warn!` from inside both helpers; defer all logging to the (future) call site, which has the richer context (round number, worker_id, etc.). The helpers become *pure* (modulo `tracing` is already side-effecting via the global subscriber).
  2. Add a `silent: bool` parameter or a `tracing::warn!`-eliding feature flag, allowing the call site to suppress the helper's emission. Heavier and uglier.
  3. Document in the helper Rustdoc: "**Observability:** this helper emits a `tracing::warn!` on the elastic path. Call sites SHOULD NOT also log the underlying I/O error ã the helper's emission is intended to be the canonical record." Lightest fix; relies on TASK-0438 author reading and respecting the note.
- **Severity rationale:** MEDIUM ã not a correctness bug; observability discipline. Recommendation #3 is the lightest acceptable fix.

### QA-005 (MEDIUM) ã `format!("phase timeout after {:.2}s", ...)` is now an implicitly contractual error message (UT-0413-08), but no Rustdoc declares it stable

- **Severity:** MEDIUM ã the error string format is pinned by UT-0413-08's `assert_ne!` semantics (different elapseds ã different strings), but no Rustdoc documents the format as stable, and no upstream contract tells log aggregators or future test code to key on a stable substring.
- **Category:** API contract / format-string stability / observability
- **Location:** `relativist-core/src/protocol/coordinator.rs:129`:
  ```rust
  ConnectionLossOutcome::Abort(format!("phase timeout after {:.2}s", elapsed.as_secs_f64()))
  ```
  vs UT-0413-08 (`:1501-1513`) which asserts that two different elapseds produce *distinct* abort strings (i.e., the elapsed value must be embedded). UT-0413-04 (`:1422-1438`) asserts the substring `"600"` appears for `Duration::from_secs(600)` ã a stricter contract than UT-0413-08, because it pins the *digit representation*.
- **Reproduction (forward-looking):**
  Suppose a future maintainer changes the format to `"phase timeout: {} ms"` (different precision, milliseconds). 
  - UT-0413-04 fails (substring `"600"` not present in `"phase timeout: 600000 ms"` ã actually it IS present in this case, but a switch to seconds-rounded `"phase timeout: 10 minutes"` would not contain "600").
  - UT-0413-08 still passes (two different durations still produce different strings).
  
  So UT-0413-04 partially pins the format; UT-0413-08 alone would not catch a regression to a non-numeric representation. **The format-string contract is split across two tests with no Rustdoc pinning it.**
  
  Worse: a downstream user (not yet existing, but plausible) might parse the abort string to extract the elapsed time for retry logic. There is no API to query the elapsed value separately from the abort string.
- **Impact:**
  - If a future change to the format string lands without re-reading UT-0413-04, the test still passes (e.g., `"timeout @ 600.00s"` contains `"600"`), but log aggregators keying on `"phase timeout after"` break silently.
  - The `Abort(String)` variant flattens structured data (the `Duration`) into a string, losing the structured value entirely.
- **Why the existing tests miss it:**
  - UT-0413-04 pins `"600"`, not `"phase timeout after"`. The leading phrase is unprotected.
  - No test asserts the exact prefix or the `{:.2}s` precision.
- **Suggested fix (one of):**
  1. **Best:** widen the `Abort` variant to `Abort { description: String, elapsed: Option<Duration> }` so the elapsed is preserved structurally. Heavier; requires updating UT-0413-04 and the connection-loss path (which has no `elapsed`). Probably defer to a future task.
  2. **Recommended:** add a Rustdoc note to `handle_phase_timeout` declaring the format string contractual:
     ```
     # Format string stability
     The `Abort(String)` returned on the v1 path has the format
     `"phase timeout after {N}s"` where `{N}` is `Duration::as_secs_f64()`
     formatted with `{:.2}` precision. This format is contractual for upstream
     log aggregators and is pinned by UT-0413-04 / UT-0413-08. Do not change
     without a coordinated update.
     ```
  3. **Strengthen UT-0413-04** to also assert the `"phase timeout after"` prefix:
     ```rust
     assert!(desc.starts_with("phase timeout after"),
         "format-string prefix is contractual; got: {desc}");
     ```
- **Severity rationale:** MEDIUM ã silent format drift is a low-frequency but high-ambiguity bug class.

### QA-008 (LOW) ã `error_description.to_owned()` allocates on every connection-loss event; cold path today, hot path under elastic-stress benchmarks

  In an elastic stress benchmark (e.g., a Phase-2.3 chaos test with frequent worker disconnects), each disconnect produces a `String` allocation. - **Impact:** ~1 allocation per departure event. At 100 departures/sec (elastic stress), 100 allocs/sec. Not a tight loop; not a regression.
- **Why the existing tests miss it:** no benchmark stresses the elastic path yet.
- **Suggested fix:** **None now.** Track as future API expansion. Leave a `// TODO: consider Cow<'_, str> for elastic-stress paths` comment near the allocation.
- **Severity rationale:** LOW ã performance hint, not correctness.

### QA-009 (LOW) ã `ConnectionLossOutcome::Abort` is not symmetric across panic-poisoning of underlying I/O errors

- **Severity:** LOW ã adversarial corner; no production trigger today.
- **Category:** Panic-resilience / unwind-safety
- **Location:** `relativist-core/src/protocol/coordinator.rs:90` and `:129`. Both branches construct an owned `String` via `to_owned()` / `format!`.
- **Reproduction (adversarial trace):**
  - `error_description: &str` is the `&str` from `io_err.to_string()` or similar. - **Why the existing tests miss it:** no test feeds a `&str` from a panicking source. UT-0413-NN feed string literals.
- **Suggested fix:** **None.** Leave a one-line note in the function-level Rustdoc: "`error_description` is consumed by `to_owned()`; this helper is panic-free given a legal `&str` argument."
- **Severity rationale:** LOW ã adversarial-only.

### QA-010 (LOW) ã Concurrent invocation of the helpers is safe (pure functions), but the `tracing::warn!` macro touches the global subscriber and is not contention-free

- **Severity:** LOW ã observability under contention.
- **Category:** Concurrency / shared global state
- **Location:** `relativist-core/src/protocol/coordinator.rs:78-84` and `:117-123` (the `tracing::warn!` calls).
- **Reproduction (adversarial trace):**
  Suppose two workers disconnect simultaneously. Two `tokio` tasks each call `handle_connection_loss`. Both helpers are pure (no shared state in TASK-0413's code), but both call `tracing::warn!`, which acquires the global subscriber's lock (depends on subscriber implementation; `tracing-subscriber`'s default fmt layer uses a per-call `parking_lot::Mutex`). Under contention, this serializes the two emissions.
  
  Result:
  - The warn emissions are serialized (not interleaved garbled output, good).
  - The `RecoveryTriggered { worker_index }` returns are independent (good).
  - But the *order* of the warn lines in the log may not match the order of the FSM events ã TASK-0438's FSM dispatcher might receive `RecoveryTriggered` for worker 5 before it does for worker 3, even though worker 3's warn appeared first in the log.
- **Impact:** observability anomaly only; the FSM correctness is unaffected because each `RecoveryTriggered` is dispatched independently.
- **Why the existing tests miss it:** no concurrent test is part of TASK-0413's scope (the helpers are sync). 
- **Suggested fix:** **None now.** Document in the helper Rustdoc: "**Concurrency:** the helpers are pure and may be called from multiple tasks concurrently. The `tracing::warn!` emission is serialized by the global subscriber; log line order may not match return-value dispatch order under contention."
- **Severity rationale:** LOW ã observability edge case under high concurrency.
