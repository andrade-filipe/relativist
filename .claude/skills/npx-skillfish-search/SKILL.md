# npx-skillfish-search

Inject the `npx skillfish` CLI as a sampling and discovery channel for the `skill.fish` community registry. Read-mostly: this skill enumerates community Skills and surfaces their SKILL.md shapes; it never edits files itself, and any actual install or write goes through the architect. Sibling to `npx-skills-search` (which targets the parallel `skills.sh` marketplace).

## Commands

| Command | Purpose |
|---|---|
| `npx skillfish search <query> [-l/--limit <n>]` | Non-interactive search of the skill.fish registry; returns `name / slug / owner / github / url / description / stars`. Add the global `--json` flag for machine-readable output. The primary sampling call. |
| `npx skillfish list [--project\|--global\|--agent <name>]` | Enumerate installed skills across detected agents (project, global, or filtered to one agent). Combine with global `--json` for automation. |
| `npx skillfish add <owner/repo> [skill-name]` | Install a skill from a GitHub repo. Flags: `--global` (→ `~/.claude`), `--project` (→ `./.claude`), `--path <p>` (install one skill from a multi-skill repo), `--all`, `--force`, `-y/--yes`. Also supports `owner/repo/plugin/skill` syntax. State-touching — confirm with user first. |
| `npx skillfish remove [skill]` | Uninstall a skill. State-touching — confirm with user first. |
| `npx skillfish update` | Update installed skills. State-touching — confirm with user first. |
| `npx skillfish bundle [--global\|--project]` | Write the currently installed skills out to a `skillfish.json` manifest (lockfile-style snapshot). **Unique to skillfish — no analogue in `npx skills`.** Read-only on the skills themselves; writes only the manifest. |
| `npx skillfish install [--global\|--project] [-y] [--dry-run]` | Read `skillfish.json` and install the listed skills. Declarative restore from a manifest. State-touching unless `--dry-run` — confirm with user first. |
| `npx skillfish submit <repo> [-y]` | Submit a repo to the skill.fish registry so others can discover it. **Unique to skillfish.** Touches the PUBLIC registry — confirm with user first. |
| `npx skillfish init [--name --description --author --version --license -y --project\|--global]` | Scaffold a new local skill. |

Footnote on flags: `--json` is a **global flag** on skillfish (works on every subcommand for automation), not per-subcommand — different from `npx skills` where `--json` is attached to specific subcommands like `list`. The `--path <p>` flag and the `owner/repo/plugin/skill` syntax let you install a single skill out of a multi-skill repo, which the `npx skills add` surface does not expose directly.

## When to fire

Three concrete scenarios that map to the trigger categories in the description:

1. **Sampling before authoring (Authoring + Sampling triggers).** Architect is about to write `.claude/skills/<new-name>/SKILL.md`. Run `npx skillfish search <closest-keyword> --limit 5 --json` first; pull 3-5 community SKILL.md examples from the returned `github` URLs to calibrate description density and trigger-list shape; then hand off to `superpowers:writing-skills` (or author directly under the architect write-boundary) with the samples in context. Pair with `npx skills find <term>` from the sibling skill to cover both registries.
2. **Discovery for a vague ask (Search/find + Discovery triggers).** User asks "is there a skill.fish skill for postgres migrations?" or "what's on skill.fish for code review?". Run `npx skillfish search <term> --limit 10` non-interactively; surface the top hits with star counts and `github` URLs; let the user decide whether to install or just inspect. If the user did not name skill.fish specifically, also run the sibling `npx skills find` so coverage is symmetric.
3. **Declarative bundle / restore workflow (Declarative-restore triggers).** User wants to snapshot the currently installed skills (e.g., to reproduce on another machine or to pin a known-good set). Run `npx skillfish bundle --project` (or `--global`) to write `skillfish.json`, then on the target machine `npx skillfish install --project --dry-run` to preview and `npx skillfish install --project -y` to apply. This `bundle` → `skillfish.json` → `install` loop is **unique to skillfish** and the cleanest declarative-restore path; `npx skills` has no direct equivalent (its closest neighbors are `experimental_sync` / `experimental_install`).

## Discipline / guardrails

- **Community quality varies — sampling, not authority.** A high-star skill on skill.fish is evidence of community fit, not framework fit. The framework's own glossaries (`framework-glossary`, `techniques-glossary`, `workflows-glossary`) and tier system override any community pattern that conflicts. Same posture as the sibling `npx-skills-search` skill toward skills.sh.
- **Do NOT auto-install.** Always confirm with the user before running `npx skillfish add`, `remove`, `update`, `install`, or `submit`. These touch global or project state (and `submit` touches the public registry); silent state mutation is out of scope for this skill.
- **Sampling, not copying.** Extract the shape or idea from a community SKILL.md (description density, trigger categories, body-section layout), then author the artifact per this framework's conventions (tier-aware frontmatter, REF citations, anti-triggers, write-boundary discipline). Do not lift a community SKILL.md verbatim.
- **Write-boundary preserved.** This skill does NOT bypass the architect-only `.claude/skills/*` write boundary declared in `CLAUDE.md`. After sampling, the actual write to `.claude/skills/<name>/SKILL.md` goes through the `architect` meta-agent under a `goal-execution` envelope — not through this skill, and not through `general-purpose`.
