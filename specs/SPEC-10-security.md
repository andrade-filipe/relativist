## 1. Purpose

This spec defines the security model for Relativist v1: the three-tier security architecture (development, private network, production), token-based authentication for worker registration, optional TLS 1.3 encryption via feature flag, message size enforcement, connection limits, and defensive measures against common distributed system vulnerabilities. Security in Relativist is layered on top of the wire protocol (SPEC-06) and transport abstraction (SPEC-13, R28-R30) without modifying their interfaces.

## 2. Definitions

Terms defined in SPEC-00 (Glossary) are used without redefinition. Terms introduced or refined in this spec:

| Term | Definition |
|------|-----------|
| **Security Tier** | One of three predefined security configurations (Development, Private Network, Production) that define which authentication, encryption, and integrity mechanisms are active. Selected implicitly by which CLI flags and features are enabled. (Relativist) |
| **Authentication Token** | A 256-bit (32-byte) cryptographically random value generated by the coordinator at startup, base64-encoded (44 characters). Workers present this token during registration to prove authorization. Per-session lifetime (no rotation). (Relativist) |
| **TLS (Transport Layer Security)** | Cryptographic protocol providing encryption and authentication over TCP. Relativist uses TLS 1.3 exclusively, via rustls + tokio-rustls, feature-gated under `tls`. |
| **mTLS (Mutual TLS)** | TLS mode where both server and client present certificates for mutual authentication. Excluded from v1 (PESQ-017 L3). |
| **CSPRNG (Cryptographically Secure Pseudo-Random Number Generator)** | A random number generator suitable for security-sensitive operations. Relativist uses `rand::rngs::OsRng` for token generation. |
| **Message Size Limit** | Maximum allowed payload size (in bytes) for a single wire protocol frame. Messages exceeding this limit are rejected before memory allocation. Default: 256 MiB (SPEC-06, R9; PESQ-019 L2). |
| **Server TLS** | TLS mode where only the server (coordinator) presents a certificate. Workers verify the coordinator's certificate but do not present their own. Sufficient for v1 (PESQ-017 L3). |

## 3. Requirements

### 3.1 Three-Tier Security Model

**R1.** Relativist MUST support three security tiers, selected by the combination of CLI flags and enabled features. **(MUST)**

| Tier | Auth | Encryption | Integrity | Bind Default | Use Case |
|------|------|-----------|-----------|-------------|----------|
| **1 (Development)** | None | None | CRC32 (SPEC-06, R10) | `127.0.0.1` | Local testing, Docker Compose single host |
| **2 (Private Network)** | Token | None | CRC32 (SPEC-06, R10) | Explicit `--bind` | LAN, VPN, private cloud |
| **3 (Production)** | Token | TLS 1.3 | TLS (subsumes CRC32) | Explicit `--bind` | Untrusted network, cloud |

Source: PESQ-018 L5, PESQ-023 D4.

**R2.** The system MUST NOT require any security configuration for Tier 1 (development). A plain `relativist coordinator --workers 2 --input net.bin` MUST work without `--token` or TLS flags, binding to localhost. **(MUST)**

**R3.** The coordinator MUST automatically detect the active tier based on the provided flags and log it at startup at `INFO` level:
- No `--token` and no TLS flags: Tier 1
- `--token` provided and no TLS flags: Tier 2
- `--token` provided and TLS flags provided: Tier 3
**(MUST)**

**R4.** TLS without token authentication (TLS flags but no `--token`) SHOULD be rejected with a configuration error. TLS provides encryption but without token auth any host that trusts the CA can register as a worker. **(SHOULD)**

### 3.2 Network Binding

**R5.** The coordinator MUST bind to `127.0.0.1` by default. This prevents accidental exposure of the service to the network. **(MUST)**

Source: PESQ-019 L1, PESQ-023 D4.

**R6.** To bind to a non-localhost address, the operator MUST explicitly provide the `--bind` CLI flag (e.g., `--bind 0.0.0.0:9000`). **(MUST)**

**R7.** When the coordinator binds to `0.0.0.0` (all interfaces), it MUST emit a `WARN`-level log message: `"Binding to all interfaces (0.0.0.0). Ensure authentication is enabled for non-trusted networks."` **(MUST)**

**R8.** The coordinator SHOULD refuse to bind to `0.0.0.0` without token authentication (Tier 1 on a public address) and emit an error. This MAY be overridden with a `--insecure` flag for development convenience. **(SHOULD)**

### 3.3 Token Authentication

**R9.** The coordinator MUST generate a 256-bit (32-byte) authentication token using `rand::rngs::OsRng` (a CSPRNG) when the `--token auto` flag is provided or when `--token <value>` is not specified but token auth is required (Tier 2/3). **(MUST)**

Source: PESQ-018 Section 2.2.

**R10.** The generated token MUST be base64-encoded (standard alphabet, 44 characters) for human readability and transport. **(MUST)**

**R11.** The coordinator MUST display the generated token in the startup log at `INFO` level exactly once. After the initial display, the token value MUST NOT appear in any subsequent log entry, metric, error message, or health endpoint response. **(MUST)**

Source: PESQ-019 Section 2.6.

**R12.** The coordinator SHOULD write the generated token to a file (default: `./relativist-token`) with mode `0600` (owner read/write only). The path MUST be configurable via `--token-file`. **(SHOULD)**

**R13.** Workers MUST provide the token via one of two mechanisms, in order of precedence:
1. `--token <value>` CLI flag
2. `RELATIVIST_TOKEN` environment variable

If both are provided, the CLI flag takes precedence. **(MUST)**

Source: PESQ-018 L2.

**R14.** Workers MUST include the token in the `Register` message sent to the coordinator upon connection (SPEC-13, R25, Worker FSM Init->Idle transition). The token MUST be transmitted as raw bytes (32 bytes), not base64-encoded, in the serialized message. **(MUST)**

**R15.** The coordinator MUST validate the token in every `Register` message using constant-time comparison to prevent timing side-channel attacks. **(MUST)**

**R16.** If token validation fails, the coordinator MUST:
1. Send a `RegisterNack` message with reason `"authentication failed"` (no further detail).
2. Close the TCP connection.
3. Log the rejection at `WARN` level with the remote address (but NOT the rejected token value).
**(MUST)**

**R17.** If token validation succeeds, the coordinator MUST send a `RegisterAck` message and proceed with normal worker registration (SPEC-13, R21). **(MUST)**

**R18.** Token authentication MUST be per-session. The token is generated once at coordinator startup and remains valid until the coordinator shuts down. There MUST be no token rotation, refresh, or expiry mechanism in v1. **(MUST)**

Source: PESQ-018 L4.

**R19.** The `Register` message (SPEC-06 Message enum) MUST be extended to include an `auth_token: Option<[u8; 32]>` field. When running in Tier 1 (no auth), this field MUST be `None`. When running in Tier 2 or 3, this field MUST be `Some(token_bytes)`. **(MUST)**

```rust
/// Token type for worker authentication.
/// 256-bit (32-byte) random value, generated via OsRng.
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct AuthToken([u8; 32]);

impl AuthToken {
    /// Generate a new random token using a CSPRNG.
    pub fn generate() -> Self;

    /// Decode a token from a base64-encoded string.
    pub fn from_base64(s: &str) -> Result<Self, TokenError>;

    /// Encode the token as a base64 string.
    pub fn to_base64(&self) -> String;

    /// Constant-time comparison to prevent timing attacks.
    pub fn verify(&self, other: &AuthToken) -> bool;

    /// Return the raw bytes.
    pub fn as_bytes(&self) -> &[u8; 32];
}
```

### 3.4 TLS Support

**R20.** TLS support MUST be feature-gated under the `tls` Cargo feature. When the `tls` feature is not enabled, all TLS code MUST be compiled out entirely. The binary without `tls` MUST have no dependency on `rustls`, `tokio-rustls`, or `rustls-pemfile`. **(MUST)**

Source: PESQ-017 L2, SPEC-13 R37.

**R21.** When TLS is enabled, Relativist MUST use `rustls` (pure Rust, no C dependencies) with `tokio-rustls` for async TLS integration. **(MUST)**

Source: PESQ-017 L1.

**R22.** TLS MUST use TLS 1.3 exclusively. TLS 1.2 fallback MUST NOT be permitted. The `rustls::ServerConfig` and `rustls::ClientConfig` MUST be configured to accept only TLS 1.3 cipher suites. **(MUST)**

**R23.** Relativist v1 MUST implement Server TLS only: the coordinator presents a certificate and the workers verify it. Workers do NOT present client certificates (no mTLS). **(MUST)**

Source: PESQ-017 L3.

**R24.** TLS MUST wrap the existing wire protocol (SPEC-06) transparently. The framing format (length + CRC32 + payload), message serialization, and `Transport` trait interface (SPEC-13, R28) MUST NOT change when TLS is enabled. TLS operates at the TCP stream level, below the framing layer. **(MUST)**

Source: PESQ-017 L5.

**R25.** The coordinator MUST accept TLS configuration via CLI flags:
- `--tls-cert <path>`: Path to the PEM-encoded certificate file.
- `--tls-key <path>`: Path to the PEM-encoded private key file.
Both flags MUST be provided together; providing only one MUST result in a configuration error. **(MUST)**

**R26.** Workers MUST accept TLS configuration via CLI flag:
- `--tls-ca <path>`: Path to the PEM-encoded CA certificate for verifying the coordinator's certificate.
When `--tls-ca` is provided, the worker MUST initiate a TLS handshake when connecting to the coordinator. **(MUST)**

**R27.** Self-signed certificates MUST be supported. Workers verify the coordinator's certificate against the provided CA cert (`--tls-ca`), which may be self-signed. No external PKI infrastructure is required. **(MUST)**

Source: PESQ-017 L4.

**R28.** When TLS is enabled, `TcpTransport` (SPEC-13, R29) MUST perform the TLS handshake after the TCP connection is established but before any wire protocol messages are exchanged. **(MUST)**

```rust
/// TLS configuration for the coordinator (server side).
/// Only available when the `tls` feature is enabled.
#[cfg(feature = "tls")]
pub struct TlsServerConfig {
    /// rustls ServerConfig, pre-built from cert + key.
    pub config: Arc<rustls::ServerConfig>,
}

#[cfg(feature = "tls")]
impl TlsServerConfig {
    /// Load certificate and private key from PEM files.
    pub fn from_pem_files(
        cert_path: &Path,
        key_path: &Path,
    ) -> Result<Self, SecurityError>;
}

/// TLS configuration for workers (client side).
/// Only available when the `tls` feature is enabled.
#[cfg(feature = "tls")]
pub struct TlsClientConfig {
    /// rustls ClientConfig, pre-built from CA cert.
    pub config: Arc<rustls::ClientConfig>,
}

#[cfg(feature = "tls")]
impl TlsClientConfig {
    /// Load CA certificate from a PEM file.
    pub fn from_ca_pem(ca_path: &Path) -> Result<Self, SecurityError>;
}
```

### 3.5 Message Size Limits

**R29.** The receiver MUST enforce a maximum message size before allocating memory for deserialization. The length field in the frame header (SPEC-06, R6) MUST be checked against the configured maximum before reading the payload. **(MUST)**

Source: PESQ-019 L2, SPEC-06 R9.

**R30.** The default maximum message size MUST be 256 MiB (`268_435_456` bytes). This MUST be configurable via `--max-message-size` CLI flag (accepting values in bytes or human-readable format like `256M`). **(MUST)**

**R31.** When a message exceeds the maximum size, the receiver MUST reject it with a `ProtocolError::MessageTooLarge` error (SPEC-13, R16) and close the connection. The oversized payload MUST NOT be read from the socket. **(MUST)**

### 3.6 Connection Limits

**R32.** The coordinator SHOULD enforce a maximum number of concurrent TCP connections (default: 1024). Connections beyond this limit MUST be rejected immediately with a TCP reset. **(SHOULD)**

**R33.** The coordinator SHOULD enforce an idle connection timeout (default: 30 seconds). A connection that has not sent or received any wire protocol message within the timeout period MUST be closed. **(SHOULD)**

**R34.** The coordinator MAY implement rate limiting for registration attempts: at most N registration attempts per source IP within a time window (e.g., 10 attempts per 60 seconds). Rejected attempts MUST be logged at `WARN` level. **(MAY)**

### 3.7 Defensive Measures

**R35.** The use of bincode for deserialization MUST be documented as a security advantage: bincode deserializes into fixed Rust structs defined at compile time, eliminating the class of arbitrary-code-execution deserialization attacks that affect Java, Python, and other languages with polymorphic deserialization. **(MUST)**

Source: PESQ-019 L3.

**R36.** After the initial token display at startup (R11), the token value MUST NOT appear in:
- Subsequent log entries at any level (DEBUG, TRACE included).
- Prometheus metrics or labels.
- Error messages returned to clients.
- Health or readiness endpoint responses.
- Core dumps or panic messages (use `Zeroize` or avoid including token in panic paths).
**(MUST)**

Source: PESQ-019 Section 2.6.

**R37.** Error messages sent to unauthenticated clients MUST NOT reveal internal state. Specifically:
- Failed authentication: `"authentication failed"` (no indication of whether the token was wrong, expired, or malformed).
- Protocol errors: `"protocol error"` (no stack traces or internal details).
**(MUST)**

**R38.** The `Message` enum SHOULD include version or magic bytes as the first field to allow fast rejection of non-Relativist connections (e.g., port scanners, accidental connections). This MAY be implemented as a protocol version prefix in the first message of each connection. **(SHOULD)**

### 3.8 What is NOT in v1

**R39.** The following security features MUST NOT be implemented in v1. Each is explicitly excluded with justification. **(MUST NOT)**

| Feature | Justification | Source |
|---------|--------------|--------|
| Mutual TLS (mTLS) | Requires PKI infrastructure and client certificate distribution; server TLS + token auth is sufficient | PESQ-017 L3, SPEC-13 R49 |
| Token rotation | Per-session token is sufficient; sessions last minutes to hours | PESQ-018 L4, SPEC-13 R49 |
| Byzantine fault tolerance | Redundant computation not justified for research prototype | PESQ-019 L5, DISC-007 v2 |
| Certificate rotation | Manual cert management is acceptable for TCC scope | PESQ-017 Section 2.3 |
| Access control lists (ACLs) | All workers are homogeneous; no capability differentiation needed | PESQ-018 Section 3 |
| Encryption at rest | Net data is ephemeral computation state, not sensitive data at rest | Out of scope |
| HMAC message integrity | CRC32 is sufficient for trusted networks; TLS provides integrity for untrusted networks; HMAC as a middle ground adds complexity for minimal benefit in v1 | PESQ-018 L3 |

## 4. Design

### 4.1 Security Module Structure

The `security` module (SPEC-13, R5) contains all security-related types and functions:

```
src/security/
├── mod.rs       # SecurityConfig, tier detection, re-exports
├── token.rs     # AuthToken, generation, validation
└── tls.rs       # TlsServerConfig, TlsClientConfig (feature-gated)
```

### 4.2 Security Configuration

```rust
/// Security configuration, assembled from CLI flags and environment.
#[derive(Debug, Clone)]
pub struct SecurityConfig {
    /// The active security tier (detected, not directly configured).
    pub tier: SecurityTier,
    /// Authentication token. None for Tier 1.
    pub token: Option<AuthToken>,
    /// TLS server config. None when `tls` feature is disabled or no TLS flags.
    #[cfg(feature = "tls")]
    pub tls_server: Option<TlsServerConfig>,
    /// TLS client config. None when `tls` feature is disabled or no TLS flags.
    #[cfg(feature = "tls")]
    pub tls_client: Option<TlsClientConfig>,
    /// Maximum message size in bytes.
    pub max_message_size: u32,
    /// Maximum concurrent connections.
    pub max_connections: usize,
    /// Connection idle timeout.
    pub idle_timeout: Duration,
}

/// The three security tiers.
#[derive(Debug, Clone, Copy, PartialEq, Eq, serde::Serialize)]
pub enum SecurityTier {
    /// Tier 1: No auth, no TLS, localhost only.
    Development,
    /// Tier 2: Token auth, no TLS.
    PrivateNetwork,
    /// Tier 3: Token auth + TLS 1.3.
    Production,
}
```

### 4.3 Token Lifecycle

```
Coordinator startup:
  1. Parse --token flag:
     a. --token auto       -> generate new token via AuthToken::generate()
     b. --token <base64>   -> decode via AuthToken::from_base64()
     c. (no --token flag)  -> token = None (Tier 1)
  2. If token is Some:
     a. Log token at INFO: "Worker authentication token: <base64>"
     b. Write to --token-file (default: ./relativist-token) with 0600 perms
  3. Detect tier based on token + TLS flags (R3)

Worker startup:
  1. Read token from --token flag or RELATIVIST_TOKEN env var
  2. If token is Some, include in Register message
  3. Connect to coordinator (with or without TLS)
  4. Send Register { auth_token: Some(token_bytes), ... }

Coordinator on Register:
  1. If self.token is Some:
     a. If msg.auth_token is None -> reject (R16)
     b. If msg.auth_token != self.token (constant-time) -> reject (R16)
     c. If msg.auth_token == self.token -> accept (R17)
  2. If self.token is None:
     a. Accept unconditionally (Tier 1)
```

### 4.4 TLS Integration with Transport

TLS integrates at the `TcpTransport` level (SPEC-13, R30):

```
Without TLS:
  TCP socket -> FrameReader/FrameWriter -> Message

With TLS:
  TCP socket -> TLS handshake -> TlsStream -> FrameReader/FrameWriter -> Message
```

The `FrameReader`/`FrameWriter` (SPEC-06 Section 4.3) operate on `AsyncRead + AsyncWrite`. Both `TcpStream` and `TlsStream<TcpStream>` implement these traits. The framing layer is unaware of whether TLS is active.

### 4.5 Connection Acceptance Flow

```
Coordinator accept loop:
  1. Accept TCP connection
  2. Check connection count against max_connections (R32)
     - If exceeded: close immediately, log WARN
  3. If TLS enabled: perform TLS handshake
     - On failure: close, log WARN with remote addr
  4. Set idle timeout timer (R33)
  5. Read first message (expect Register)
     - If not Register: close, log WARN
  6. Validate token (R15-R17)
     - If failed: send RegisterNack, close
  7. Send RegisterAck, add to worker pool
```

## 5. Rationale

### 5.1 Why Three Tiers Instead of Always-On Security

Requiring TLS and authentication for every execution would create friction during development and local testing. The three-tier model (PESQ-018 L5) follows the principle of progressive security:

- **Tier 1** enables `cargo run` during development with zero configuration.
- **Tier 2** adds authentication for multi-machine deployments on trusted networks, with a single shared secret that is trivial to distribute.
- **Tier 3** adds encryption for untrusted networks, requiring certificate management but providing full transport security.

This mirrors HTCondor's approach (PESQ-005): HTCondor historically defaulted to no authentication but added IDTOKENS for modern deployments. Relativist adopts the layered model from the start.

### 5.2 Why Token Auth Over More Sophisticated Mechanisms

HTCondor's IDTOKENS (PESQ-005) support signed claims, capability-based access, expiry, and revocation. Relativist's single shared token is simpler because:

1. There is one coordinator, one computation, one token per session.
2. Workers are homogeneous (no capability differentiation).
3. Sessions are short-lived (minutes to hours for TCC benchmarks).
4. The operator controls all machines.

JWT, OAuth2, or signed tokens add complexity with no benefit at this scale (PESQ-018 Section 1.1).

### 5.3 Why Server TLS Only (No mTLS)

mTLS requires distributing client certificates to every worker, either via a PKI or manual file distribution. Server TLS + token auth achieves the same effective security for Relativist's threat model:

- **Server TLS** ensures workers connect to the real coordinator (not an impostor).
- **Token auth** ensures only authorized workers can register.
- **mTLS** would additionally prove worker identity cryptographically, but token auth already serves this purpose for a trusted operator scenario.

mTLS is a candidate for v2 if Relativist expands to multi-tenant or cross-organization deployments (PESQ-017 L3).

### 5.4 Why HMAC is Excluded from v1

PESQ-018 L3 proposes HMAC-SHA256 as a middle ground between CRC32 and TLS. However:

- On trusted networks (Tier 2), CRC32 detects accidental corruption; deliberate tampering is not a concern.
- On untrusted networks (Tier 3), TLS provides integrity that subsumes HMAC.
- HMAC would be useful only for the narrow case of "untrusted network without TLS," which is a configuration Relativist should discourage rather than support.

This avoids adding a cryptographic dependency (`hmac`, `sha2`) to the always-on build.

## 6. Test Requirements

**T1.** Token generation MUST be tested: two consecutive calls to `AuthToken::generate()` MUST produce different tokens (statistical test: probability of collision is 2^-256, so any collision indicates a broken RNG). **(MUST)**

**T2.** Token serialization roundtrip MUST be tested: `AuthToken::from_base64(token.to_base64()) == token` for all valid tokens. **(MUST)**

**T3.** Token validation MUST be tested:
- Correct token: `RegisterAck` returned.
- Wrong token: `RegisterNack` returned, connection closed.
- Missing token when required (Tier 2/3): `RegisterNack` returned, connection closed.
- Token provided when not required (Tier 1): accepted (token ignored).
**(MUST)**

**T4.** TLS handshake MUST be tested (feature-gated under `tls`):
- Valid cert + valid CA: handshake succeeds, messages exchanged.
- Valid cert + wrong CA: handshake fails, connection refused.
- Expired cert: handshake fails.
**(MUST)**

**T5.** Message size limit enforcement MUST be tested: a frame with a declared length exceeding `max_message_size` MUST be rejected without reading the payload. **(MUST)**

**T6.** Connection idle timeout MUST be tested: a connection that sends no messages for longer than `idle_timeout` MUST be closed by the coordinator. **(SHOULD)**

**T7.** Unauthorized registration rejection MUST be tested: a worker connecting without the correct token in Tier 2/3 MUST be disconnected after `RegisterNack`. The coordinator MUST continue accepting connections from other workers. **(MUST)**

**T8.** Localhost default binding MUST be tested: a coordinator started without `--bind` MUST only be reachable from `127.0.0.1`. **(SHOULD)**

**T9.** Tier detection MUST be tested: verify that the correct `SecurityTier` is detected for each combination of flags. **(MUST)**
